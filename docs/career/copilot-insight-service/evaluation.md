# Query 评测与发布

## 评测范围

`/insights/query` 的评测对象是从用户 Query 到 Top 12 邮件候选的检索链路：

```text
Query 与过滤条件
→ Outlook Search 召回
→ 权限过滤
→ 去重与重排
→ Top 12
→ Snippet 和 Citation
```

评测不要求 Query 接口生成最终答案。它回答五个问题：

1. 目标邮件是否进入候选；
2. 目标邮件是否排在前面；
3. 未授权邮件是否发生任何形式的泄露；
4. Citation 和 Snippet 是否对应正确邮件；
5. 质量达标时，延迟和下游成本是否可接受。

最终回答的事实完整性和忠实度属于 Copilot 端到端评测，Query 评测只检查模型收到的检索证据。

## 评测数据

离线评测集包含 **4000 条 Query**，来自脱敏后的真实问题模式和人工补充的边界场景。每条样本绑定固定邮箱快照，并保存：

- Query、绝对时间、Person ID 和当前邮件 Anchor；
- 应召回的 Message ID 与 Conversation ID；
- 相关性等级：核心证据、相关背景或不相关；
- 当前用户可访问和不可访问的邮件集合；
- 预期 Citation；
- Query 类型和难度标签。

数据覆盖按主题查找、按人员查找、时间范围、当前邮件追问、同名人员、共享邮箱、长时间范围和无结果请求。评测时固定邮箱快照和 Outlook Search 版本，保证旧版本与新版本看到相同数据。

训练权重和调整阈值使用开发集，发布判断使用独立测试集。生产事故和用户反馈中的新失败样本会补入回归集，但已经看过的样本不会替代独立测试集。

## 召回指标

### Raw Recall@200

先检查 Outlook Search 原始召回阶段：

```text
Raw Recall@200 =
前 200 个原始候选中的相关邮件数
÷ 样本中全部相关邮件数
```

该指标为 **97.6%**。如果目标邮件没有进入前 200，后续打分无法补救，问题应归因于 Query、过滤条件或 Outlook Search，而不是重排公式。

### Recall@12

权限过滤、Conversation 去重和重排后，检查最终 Top 12：

```text
Recall@12 =
Top 12 中的相关邮件数
÷ 样本中全部相关邮件数
```

线上版本的 Recall@12 为 **93.2%**。发布门槛是不得低于 **92%**，并且相对当前基线的下降不能超过 **0.5 个百分点**。

对于只有一封目标邮件的 Query，额外统计 Hit@12，即目标邮件是否出现在 Top 12。对于存在多封证据的 Query，使用 Recall@12，避免“找到其中一封”掩盖证据遗漏。

## 排序质量

召回只关心有没有找到，排序还要检查相关邮件是否足够靠前。我们使用三个指标：

| 指标 | 含义 | 线上结果 | 发布门槛 |
| --- | --- | ---: | ---: |
| Top 3 Hit Rate | 至少一封核心证据进入前三名 | **86.4%** | **85%** |
| MRR | 第一封核心证据排名倒数的平均值 | **0.81** | **0.79** |
| NDCG@12 | 根据核心证据、相关背景和不相关三级标签评价整体顺序 | **0.87** | **0.85** |

MRR 适合用户通常只需要第一封关键邮件的场景；NDCG@12 能检查多封相关邮件之间的整体顺序。权重或时间衰减修改后，三个指标要一起看，不能用 Recall 提升掩盖首个结果变差。

报告还按人员查询、时间查询、当前邮件 Anchor、共享邮箱和 Query 长度分桶，防止总体指标掩盖某一类请求退化。

## 权限泄露

权限检查是发布阻断项，目标值固定为 **0**。评测集包含：

- 当前用户无权访问的邮箱和文件夹；
- 已撤销权限的共享邮箱；
- 跨租户 Message ID、Conversation ID 和 Person ID；
- 用户曾经有权限但评测时已经失去权限的对象；
- 混合了可访问和不可访问邮件的搜索结果。

泄露检查不只看最终 Message ID。以下任何字段来自未授权对象都算失败：

- Subject；
- Snippet；
- Sender；
- Citation；
- 未授权对象造成的命中数量变化。

离线权限测试和预发布测试都必须 **100% 通过**。发现一条泄露就停止发布，不使用平均分或错误率容忍。

## Citation 指标

Query 返回的 Citation 用于让 Copilot 或 Outlook 客户端定位原始邮件。它包含两个确定性指标：

- **Citation Validity**：Citation 中的 Message ID 可以在同一用户身份下打开，且 Tenant 和 Mailbox 匹配；
- **Snippet Consistency**：返回的 Snippet 确实来自 Citation 指向的邮件版本。

Citation Validity 必须为 **100%**，Snippet Consistency 为 **99.9%**。邮件在检索后被删除或移动时，Citation 会在读取阶段返回对象已变化，而不能静默指向另一封邮件。

## 性能指标

性能评测在固定并发和邮箱数据规模下运行，既看端到端延迟，也看造成延迟的下游调用：

| 指标 | 线上结果 | 发布门槛 |
| --- | ---: | ---: |
| P50 | **180 ms** | **250 ms** |
| P95 | **620 ms** | **700 ms** |
| P99 | **1.4 s** | **1.5 s** |
| 超时率 | **0.2%** | **0.3%** |
| `partial` 比例 | **0.7%** | **1%** |
| 平均搜索页数 | **1.7** | **2.0** |
| 平均下游请求数 | **1.8** | **2.1** |

P95/P99 从服务端收到请求开始，到 Top 12 响应序列化完成为止，包含 Token 验证、OBO、Outlook Search、权限校验、重排和序列化。

性能回归必须与 Recall@12 一起判断。减少分页、缩短默认时间范围或提前停止都能降低延迟，但只要 Recall@12 超过允许退化范围，就不能发布。

## 发布流程

每次修改 Query 构造、分页、权限过滤、打分权重、K 值或连接池配置，都执行同一套流程：

```text
固定邮箱快照回放
→ 计算召回、排序、权限和 Citation
→ 固定并发性能测试
→ 与当前生产基线比较
→ 5% 流量灰度
→ 观察线上质量代理指标和 P95/P99
→ 逐步放量
```

离线发布门槛要求：

- Recall@12、Top 3 Hit Rate、MRR 和 NDCG@12 全部达标；
- 权限泄露为 0；
- Citation Validity 为 100%；
- P95、P99、超时率和 `partial` 比例不超过门槛。

灰度期间按版本比较无结果率、Conversation 后续读取率、Query 改写率、用户重试率、下游页数和延迟。任一权限或 Citation 错误立即回滚；质量或性能持续越过门槛时停止放量。

## 失败归因

一次端到端回答错误不能直接归因于模型。通过分层指标定位第一次偏离：

- Raw Recall@200 失败：Query、过滤条件或 Outlook Search 召回问题；
- Raw Recall 成功但 Recall@12 失败：权限过滤、Conversation 去重或打分问题；
- Top 12 正确但答案遗漏：Conversation 读取、Context 组装或模型生成问题；
- Citation 指向错误对象：结果规范化或对象版本问题；
- 离线质量不变但线上超时增加：分页、下游容量、连接池或服务资源问题。

这种分层方式让 Query 检索改进与后续 Copilot 生成评测能够衔接，同时避免用最终答案分数掩盖具体检索故障。
