# Microsoft Purview 工作流

## 项目是什么

Microsoft Purview 是面向企业数据治理的产品。所在团队维护其中的工作流能力，让管理员围绕数据资产配置审批、通知和自动处理步骤，例如访问申请、策略确认和人工复核。

2022 年 5 月加入微软后，主要负责这套已经在线运行的后端系统。最初的工作与大模型无关，是典型的业务后端和工作流平台开发。

## 系统处在什么位置

工作流的上游包括 Purview 管理界面和其他数据治理服务。用户或系统事件创建工作流实例后，服务根据定义推进节点，并调用审批、通知和资产相关服务。运行状态需要提供给前端和运维系统查询。

```text
Purview 管理界面或治理事件
→ 工作流定义与实例服务
→ 审批、通知和自动处理节点
→ 数据资产与权限相关服务
→ 状态、结果和审计记录
```

## 主要工作

日常工作包括工作流定义、实例状态、异步节点、失败重试、权限校验和服务运维。系统已经有真实客户和历史数据，因此除了增加节点，也需要处理卡住的实例、重复通知、版本兼容和发布后的告警。

这段经历建立了后续 Agent 工程需要的后端基础：长任务不能只属于某个进程；每一步需要稳定状态；远程调用失败和结果未知要区别处理；重试不能产生重复业务结果。

## 开始接触大模型

2023 年初，团队开始尝试把 OpenAI 接入工作流。第一批场景是对审批材料做摘要、分类，或者根据已有字段生成说明。工作流仍由程序确定步骤，模型只是其中一个处理节点。

这个阶段最重要的认识是：调用模型不等于 Agent。输入输出明确、执行路径固定时，把模型作为 Workflow 中的智能节点更简单，也更容易控制。

## 这个项目可以深入到哪里

- Workflow、单次模型调用和 Agent 的区别；
- 异步任务、状态机、幂等、重试和结果未知；
- 怎样在已经上线的业务流程中接入模型；
- 哪些步骤适合交给模型，哪些规则和状态继续由程序控制；
- 模型节点怎样沿用原有工作流的权限、审计和发布体系。

这个项目为后续 Agent 工程提供了后端基础，也能够具体说明：**系统使用了大模型，不代表它就是 Agent。**

## 后续可以展开的文档

项目内容增加后，可以继续拆分：

- 业务与团队：Purview 数据治理场景、工作流用户和团队分工；
- 系统设计：工作流定义、实例状态、节点调度和上下游接口；
- 可靠执行：幂等、重试、结果未知、卡住实例和人工处理；
- 模型接入：摘要与分类节点、Prompt、数据边界和发布方式；
- 部署与生产运行：服务拓扑、计算资源、存储、容量、监控和值班事故。
