# 邮箱整理端到端设计

## 核心难点

邮箱整理不是把一句自然语言直接转换成移动操作。用户经常只说“把最近需要我跟进的邮件整理一下”，但没有说清时间范围、判断标准、动作和目标文件夹；哪些邮件符合条件又只有搜索之后才能确定。

这部分需要模型根据每轮结果决定澄清、继续搜索还是读取详情。但移动、归档和加旗标会真实修改邮箱，不能只依赖模型判断。项目因此把一次任务分成开放探索和确定性执行两个阶段：

```text
开放探索
用户目标 → 必要澄清 → 搜索 → 读取 → 确定候选

确定性执行
动作计划 → 用户确认 → 权限和版本检查 → 写操作 → 逐项结果
```

Microsoft 365 Copilot 平台负责 Conversation、Planning 和确认，已有邮件 Extension 负责权限、资源版本与幂等执行。我的工作是配置 Agent Instructions、Context、可用工具和错误后的后续行为，把两段连接起来。

## 一次完整任务

以“把最近两周需要我跟进的邮件移到 Follow Up，但排除已经回复过的”为例：

1. 从用户输入取得时间、筛选条件、排除项、动作和目标 Folder；
2. 缺少动作或目标 Folder 时先澄清，不明确写操作就不继续；
3. 调用 Insight Query，返回最多 Top 12 候选；
4. 根据 Snippet 判断是否需要按 Message ID 读取详情；
5. 把候选 Message ID、资源版本、动作和目标 Folder 交给平台 Planning；
6. 用户确认后调用已有邮件写 Extension；
7. 根据逐项 Tool Result 告知成功、失败和状态未知的邮件。

第一版只支持移动、归档和加旗标三类动作。删除、发送邮件和修改收件人没有接入 Agent，避免在能力尚未稳定时引入不可逆或向外发送的操作。

## 必要澄清

Agent 不会为了补齐所有字段连续提问。当前 Folder 和明确时间可以直接用于只读搜索；只有以下信息会改变实际写入时才追问：

- “整理一下”没有说明移动、归档还是加旗标；
- 目标 Folder 不存在或有同名候选；
- 用户前后给出的时间或排除条件冲突；
- 搜索结果明显超出用户原本描述的范围。

如果仍无法确定写操作，Agent 返回候选列表并停止，不自行修改邮箱。

## 搜索与读取

Insight Query 先返回 Top 12 轻量候选，包括 Message ID、Conversation ID、Snippet、时间和 Citation。大多数候选可以通过 Snippet 排除；需要判断“我是否回复过”、行动项归属或邮件最新状态时，才读取完整邮件。

搜索工具只发现候选，读取工具只取得已知邮件详情，写工具只接收确认后的对象。这个边界同时体现在工具名称、Description 和场景 Prompt 中，避免模型拿搜索结果直接执行写入。

## 计划、确认与修改

计划至少展示筛选条件、动作、目标 Folder、影响邮件数量和代表性邮件。用户确认由平台完成，项目不实现 Plan 存储或确认服务。

如果用户看到计划后说“不要移动 Alice 的邮件，时间改成最近一个月”，旧计划不再执行。Agent 更新条件、重新搜索并生成新范围，再由平台重新确认。确认只表示用户同意当时的计划，不代表邮件状态不会变化。

## 执行失败

三个状态最容易出错：

- `version_conflict`：邮件在确认后发生变化。Agent 重新读取冲突邮件，范围变化时重新确认；
- `partial_success`：批量中部分邮件已成功。Agent 保留成功项，只处理剩余邮件，不重放整个批次；
- `result_unknown`：暂时无法判断写入是否完成。Agent 保留原 Operation ID 查询结果，不创建一个新的写请求。

这样可以避免两类实际风险：用户确认后继续执行已经过期的计划，以及网络结果不明时重复移动或标记同一封邮件。

## 任务结束

任务在用户只要候选列表、取消计划、写操作完成，或者工具和权限问题使任务无法安全继续时结束。部分成功时也会结束当前执行并明确列出剩余项，不在后台无限重试。

## 验证结果

邮箱整理使用固定 Query 和多轮 Scenario 回归。多轮 Scenario 会覆盖用户修改时间、排除项或目标 Folder，以及版本冲突和部分成功。当前统一评测基线为：

| Metric | 基线 |
| --- | ---: |
| `lm_checklist` | 87.6% |
| `tool_call` | 93.8% |
| `safety` | 0.0% |
| `latency_p95` | 7.4 秒 |

失败时从同一个 Run 依次检查用户约束、搜索结果、读取邮件、平台确认范围、写 Tool Call 和逐项 Tool Result，找到第一次偏离后修复。修复先跑邮箱整理切片，再跑全量 Golden Set。

## 当前限制

邮箱整理依赖 Insight 和邮件写 Extension，两者不可用时只能返回已经取得的只读结果或未执行的计划。第一版不支持长期后台整理，也不保存跨任务记忆；用户开始新任务后，邮件事实仍需从 Outlook 重新读取。
