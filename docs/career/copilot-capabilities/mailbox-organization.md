# 邮箱整理端到端设计

## 为什么选择 Agent

邮箱整理不是把一条自然语言指令直接翻译成一次移动操作。用户经常只说“把最近需要我跟进的邮件整理一下”，其中缺少时间范围、目标文件夹和“需要跟进”的判断标准。Agent 需要根据当前文件夹和会话约束探索邮件，再根据搜索结果决定是否改写查询、读取详情、向用户澄清或形成动作计划。

这部分适合由模型做开放判断，但实际写入仍由确定性系统控制：Microsoft 365 Copilot 平台负责 Planning、确认和 Tool Call 调度，邮件 Extension 负责权限、资源版本与幂等执行。项目负责让 Instructions、Context、工具和 Tool Result 契约形成一条可检查的任务链。

如果筛选条件、对象和动作都能预先枚举，固定 Workflow 会更简单；这个场景使用 Agent，是因为自然语言目标不完整、候选邮件取决于检索结果，模型需要在每轮 Observation 后决定澄清、改写查询、读取详情还是形成计划。但确认和写入仍保留固定业务规则，不能把整条链路都交给模型。

```text
开放判断
用户目标 → 澄清 → 搜索 → 读取 → 候选选择

确定性执行
结构化计划 → 用户确认 → 权限与版本检查 → 写操作 → 逐项结果
```

划词解释和附件总结的路径相对固定；邮箱整理是这个项目中最完整的 Agent 场景，因为后续步骤取决于前一轮 Tool Result，并且涉及多轮状态、Planning、确认、写入和失败恢复。

## 一次完整任务

以“把最近两周需要我跟进的邮件移到 Follow Up，但排除已经回复过的”为例，初始 Context 只有当前 Folder、用户目标和同一任务中已经确认的约束，不会把整个文件夹预加载给模型。

```text
用户目标
→ 提取时间、筛选条件、排除项、动作和目标 Folder
→ 检查会改变执行路径的缺失信息
→ 使用 Insight Query 搜索候选邮件
→ 根据候选摘要改写查询或读取必要详情
→ 形成包含目标邮件与动作范围的计划
→ 平台展示计划并取得用户确认
→ 平台调用已有邮件写 Extension
→ Agent 根据逐项 Tool Result 汇总结果
```

### 最小澄清

Agent 不会把所有缺省字段都变成追问。只读探索可以使用当前 Folder 和自然语言中能够确定的时间范围；以下信息缺失或冲突时才追问：

- 动作不明确，例如“整理一下”既可能表示移动，也可能表示加旗标；
- 目标 Folder 不存在或存在同名候选；
- 用户给出的时间和当前会话约束冲突；
- 写入范围可能明显超出用户预期。

已经确认的条件保留在当前任务状态中，后续不重复询问。连续澄清仍无法形成明确动作时停止在只读结果，不自行写入。

### 候选搜索与按需读取

第一次搜索返回轻量候选，包含 Message ID、Conversation ID、Snippet、时间和 Citation。Agent 可以根据结果：

- 候选过多时收紧时间、人物或主题条件；
- 候选不足时在用户目标允许范围内放宽查询；
- 需要判断“我是否已经回复”或行动项归属时，按 Message ID 读取详情；
- 已有证据足够时停止检索，不为增加置信感重复调用工具。

搜索工具只负责候选发现，读取工具负责已知邮件详情，写工具只消费已经确认的目标对象。工具边界同时写在名称、Description、Schema 和场景 Instructions 中，避免一个工具承担多个模糊职责。

### 计划与确认

平台 Planning 能力承载计划状态。项目要求模型提交给计划的业务信息至少能让用户检查：

- 使用了什么 Folder、时间范围和筛选条件；
- 将执行移动、归档还是加旗标；
- 目标 Folder；
- 影响邮件的范围和数量；
- 代表性邮件与排除条件。

项目没有实现 Plan 存储或确认服务。我的工作是在平台支持的计划结构内提供必要的 Message ID、动作、目标和理由，并配置写工具的风险类别。工具白名单只决定模型能否看到某个工具；具体调用是否需要确认，由平台根据工具风险配置和本次动作范围执行。

## 多轮状态与 Replanning

平台保存 Conversation 和 Plan，项目定义哪些业务信息仍然有效：

- 用户目标、时间、Folder 和排除项可以在同一任务中继承；
- 搜索候选和读取结果只作为当前计划的证据；
- Message 资源版本、权限结果和旧计划不能被长期复用；
- 用户改变范围、动作或目标 Folder 后，旧计划与确认失效。

例如用户看到计划后说“不要移动 Alice 的邮件，时间改成最近一个月”，Agent 不在旧计划上直接删一项，而是：

```text
更新用户约束
→ 使旧 Plan Version 失效
→ 按新时间和排除项重新搜索
→ 形成新的动作范围
→ 重新取得确认
```

如果确认后执行前部分邮件发生变化，写 Extension 返回 `version_conflict`。Agent 保留未受影响对象，重新读取冲突项；新结果改变动作范围时，平台形成新计划并重新确认。确认解决“用户是否同意”，资源版本解决“对象是否仍是确认时的状态”，两者不能互相替代。

## 部分成功与结果未知

批量执行后，Tool Result 可能同时包含成功、失败和未知项。Agent 不把批次简化为一次成功或失败：

```text
成功项 → 明确告知已经完成，不再提交
失败项 → 说明稳定错误原因
未知项 → 不宣称成功，也不创建新的写请求
剩余动作变化 → 新计划与新确认
```

`result_unknown` 表示请求结果尚不能确定，不等于“可以换参数再试一次”。Agent 保留原 Operation ID，等待平台或 Extension 查询执行结果。底层幂等记录由 Extension 实现，项目负责防止模型绕过原操作创建语义重复的 Tool Call。

## 任务何时结束

邮箱整理在以下状态之一结束：

- 用户只需要候选列表，不再执行写操作；
- 计划被用户取消；
- 已确认动作全部成功；
- 部分成功，且 Agent 已准确说明成功、失败和未知项；
- 权限、工具不可用或连续澄清后仍无法形成安全动作；
- 达到平台执行预算，由 Agent 返回当前进度和未完成部分。

完成条件写进场景 Instructions，避免模型在已经获得足够结果后继续搜索，也避免写入失败后无限 Replanning。

## 怎样验证这条链路

2025 年 6 月开始参与 Evaluation 后，邮箱整理的代表性失败会进入 Golden Set。固定 Query 检查候选、计划和最终结果，多轮 Scenario 模拟用户修改时间、排除项或目标 Folder，`lm_checklist` 检查最终任务结果，`tool_call` 检查工具与关键参数，`safety` 检查未确认写入和超范围操作。

发生失败时，从同一个 Run 的 Trace 找第一次偏离：

```text
Utterance 与 Folder Context
→ 模型是否提取正确约束
→ 搜索 Query 与候选
→ 读取的 Message
→ 平台 Plan 与确认范围
→ 写 Tool Call
→ 逐项 Tool Result
→ 最终回答
```

修复先在失败 Query 上验证，再运行邮箱整理功能切片和全量 Golden Set。这样 Planning 和错误恢复规则不是只写在 Instructions 中，而是能通过稳定 Scenario 持续回归。
