# LLM 应用、固定工作流与 Agent

大模型进入软件系统后，可以承担不同程度的责任。有时它只处理一次输入；有时它参与一条由程序安排好的流程；有时它还需要根据当前结果判断下一步做什么。

LLM 应用、固定工作流和 Agent 的核心区别在于**控制权**：任务的执行路径由程序决定，还是由模型在运行时参与决定。

## 控制权决定系统形态

三种系统形态可以放在同一条控制权轴线上理解：

| 系统形态 | 执行路径由谁决定 | 模型的主要职责 |
| --- | --- | --- |
| 一次模型调用 | 程序决定任务和输入输出 | 完成一次理解或生成 |
| 固定工作流 | 程序决定步骤和分支 | 处理流程中的一个或多个节点 |
| Agent | 模型根据中间结果参与决定路径 | 选择行动并持续推进目标 |

模型拥有的控制权从左到右逐渐增加，灵活性、不确定性和工程成本也随之增加。**工具调用和 RAG 可以出现在任何一种形态中，并不能单独决定一个系统是不是 Agent。**

## 从一张支持工单说起

假设一家企业软件公司每天收到数万张客户支持工单。工单中包含用户描述、错误截图、产品版本和历史对话。支持工程师需要判断问题类型、检查服务状态，并给出回复或处理建议。

同一个业务，可以采用三种不同的系统形态。

## 一次模型调用

最简单的方案是把工单内容发送给大模型，请它返回摘要、分类和建议回复：

```text
工单内容
  ↓
大模型
  ↓
摘要、分类、回复草稿
```

程序在调用模型之前已经知道要做什么，也知道模型应该返回哪些字段。**模型负责处理内容，但不决定后续流程。**

这种方式适合边界清楚、输入输出明确的任务，例如：

- 摘要一段对话
- 从文本中提取字段
- 判断工单类别
- 改写一封邮件
- 根据给定资料生成草稿

它的系统结构简单，容易测试，延迟和成本也比较稳定。如果一次调用能够可靠地解决问题，就没有必要引入更复杂的执行机制。

## 固定工作流

支持团队随后发现，只看工单文本并不足以生成可靠回复。系统还要查询客户账户、产品版本和当前服务状态。工程师可以把处理过程写成固定步骤：

```text
读取工单
  ↓
提取客户、产品和错误信息
  ↓
查询账户与服务状态
  ↓
检索相关知识库文章
  ↓
生成回复草稿
  ↓
人工确认
```

**固定工作流的步骤和分支由程序提前规定。**大模型可以参与信息提取、检索查询改写和回复生成，普通代码则负责调用服务、传递数据和处理异常。

固定工作流适合步骤可以预先列举的任务。它的优势是可预测：工程师知道每个请求会调用哪些服务，能够分别设置超时、重试和权限，也容易判断失败发生在哪一步。

工作流不一定是一条直线。它可以包含条件和并行分支，例如只有企业客户才查询合同服务，发现全局故障时跳过知识库检索，多个只读查询则同时执行。只要分支规则仍然由程序明确控制，它就是固定工作流。

## Agent

有些工单无法用统一步骤完成。用户可能只说“昨晚升级以后，欧洲办公室都登录不了”，没有提供租户、版本或错误码。系统需要根据已知信息决定调查顺序：先确认租户和部署区域，查看身份服务状态，再比较最近变更；如果没有发现异常，还可能继续查询审计日志或向用户追问信息。

这时，执行路径很难在开发阶段完整写出。工程师可以给模型一个目标和一组受控工具，让它根据每次返回的结果决定下一步：

```text
目标：调查工单并提出处理方案
  ↓
模型判断当前缺少什么信息
  ↓
选择并调用一个或多个工具
  ↓
读取工具结果，更新判断
  ↓
继续调查、请求补充信息，或给出结论
```

**Agent 的关键变化，是模型开始参与控制流程。**程序仍然负责执行工具、保存状态和实施权限，但不再预先写死每一步的顺序。

这种方式适合开放程度较高的任务，例如：

- 跨多个系统调查未知原因的问题
- 根据现场结果不断调整搜索方向
- 在大量候选工具中选择合适组合
- 处理步骤数量和顺序事先无法确定的任务

灵活性也带来了新的风险。Agent 可能查询无关信息、重复调用工具、错误理解结果，或者在没有足够证据时结束调查。系统因此需要限制最大步骤数和时间，为写操作设置审批，并保存完整执行轨迹以便评测和排查。

## 工具调用不等于 Agent

工具调用只是模型能够请求外部能力的一种接口。固定工作流同样可以让模型生成查询参数，再由程序调用搜索服务；程序也可以强制模型在某一步从几个工具中选择一个。

判断系统是否具有 Agent 特征，要看模型的选择是否会改变后续执行路径。如果模型只在程序规定的位置填充参数，它仍是工作流中的一个智能组件；如果模型根据环境反馈持续决定下一步，系统才具有更强的 Agent 性。

RAG 也不是独立于这三种形态之外的第四类系统。一次模型调用可以带上预先检索的文档，固定工作流可以包含检索步骤，Agent 也可以在需要时主动调用知识搜索。RAG 解决的是信息从哪里来，而不是流程由谁控制。

## 如何选择

架构选择可以从两个问题开始。

### 执行路径能否预先写出

能够列出稳定步骤时，固定工作流通常更合适。只有任务需要根据中间结果不断改变方向时，Agent 的动态决策才产生明显价值。

### Agent 的复杂度是否值得

动态执行会增加模型调用、状态管理、Trace、评测和故障恢复的成本。任务最好还有可以观察的完成信号，例如测试通过、告警解除或所需证据齐全；否则，系统很难发现 Agent 已经走偏。

错误代价也会影响采用范围。内部调查可以允许 Agent 探索更多路径，直接退款或修改生产环境则不适合由 Agent 独立控制。这里要决定的是**哪些任务或子任务值得使用 Agent**；进入 Agent 内部后，具体判断、业务规则和高风险操作如何分给模型、程序与人工，属于下一层设计。

| 任务特征 | 更合适的起点 |
| --- | --- |
| 单步、输入输出明确 | 一次模型调用 |
| 多步但路径稳定 | 固定工作流 |
| 路径依赖中间结果，难以穷举 | Agent |
| 开放调查后包含高风险动作 | Agent 只负责调查，动作留给确定性流程 |

## 常见的组合方式

生产系统往往不是三选一，而是分层组合：

```text
确定性的业务工作流
  ├─ 单次模型调用：摘要和字段提取
  ├─ RAG：检索产品文档
  └─ Agent：调查少数复杂工单
         ↓
      人工审批后执行操作
```

大部分普通工单可以走成本稳定的固定流程，只有信息不足、跨系统或原因未知的工单才进入 Agent 调查。这样既保留了动态处理能力，也避免让所有请求承担 Agent 的延迟、成本和不确定性。

合适的设计通常从最简单的结构开始。一次调用不够时增加工作流，固定路径确实无法覆盖任务时再引入 Agent。工程价值不在于系统拥有多少自主性，而在于是否把 Agent 用在真正需要动态路径的任务上。

确定某个子任务需要 Agent 后，还要继续拆解它的内部责任：哪些开放判断由模型完成，哪些确定规则由程序执行，哪些高风险决定交给人工。系统形态选择回答“在哪里使用 Agent”，决策边界设计回答“Agent 内部的模型能走多远”。

## 一分钟复述

LLM 应用、固定工作流和 Agent 的核心区别是控制流归谁。一次模型调用中，模型只完成一次理解或生成；固定工作流中，程序确定步骤和分支；Agent 中，模型根据中间结果参与决定下一步。工具调用和 RAG 都可以用于这三种形态，不能单独作为 Agent 的判断标准。路径能否预先写出，以及动态决策带来的收益是否值得额外复杂度，决定一个任务或子任务应该采用哪种形态。
