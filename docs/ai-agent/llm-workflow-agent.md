# LLM 应用、固定工作流与 Agent

一家企业软件公司每天收到大量支持工单。有的用户只是询问产品用法，有的需要查询账户状态，还有人只说：“昨晚升级后，欧洲办公室都登录不了。”

这几类工单都可以使用大模型，却不该采用同一种系统结构。选择的关键在于：**模型只处理内容，还是还要根据中间结果决定下一步？**

## 输入输出明确时，一次调用就够了

工单摘要和分类通常有清楚的输入输出。程序把工单内容发给模型，模型返回摘要、类别和回复草稿。后面做什么，程序早已知道。

```text
工单内容 → 模型 → 摘要、分类、回复草稿
```

模型负责理解文字，不负责安排流程。这种方式简单、容易测试，延迟和成本也比较稳定。若一次调用已经可靠完成任务，就没有必要增加循环和状态管理。

## 步骤稳定时，用固定工作流

生成可靠回复可能还要查询客户账户、产品版本和服务状态。若这些步骤长期不变，程序可以按固定顺序执行：提取客户信息，查询账户和服务，检索知识库，最后生成回复并交给人工确认。

模型可以参与其中几个节点，例如提取字段、改写检索词和生成草稿。步骤与分支仍由程序决定。

工作流也可以有条件和并行处理：企业客户才查询合同服务；发现全局故障时跳过知识库；几个只读查询可以同时进行。只要这些规则预先写在程序中，它仍然是固定工作流。

**任务有多步，不代表它一定需要 Agent。路径能预先写清时，工作流通常更可靠。**

## 路径依赖现场结果时，才需要 Agent

“升级后欧洲办公室无法登录”没有提供租户、版本和错误码。调查顺序会随着结果变化：先确认区域服务状态；若服务正常，再检查发布记录；发现可疑配置后，继续比较客户配置和错误日志。

模型在每一步都要回答同一个问题：“根据目前结果，下一步最值得查什么？”它的选择会改变后续路径，这时系统才表现出 Agent 特征。

程序并没有退出。它仍然执行工具、保存状态、检查权限，并限制最大步骤数和时间。变化只是路径不再完全由开发者提前写死。

**Agent 的关键不是多次调用模型，而是模型根据反馈参与控制后续路径。**

## 工具和 RAG 不能单独证明这是 Agent

固定工作流也可以让模型生成搜索词，再调用知识库；一次模型调用也可以带上预先检索的文档。工具调用解决“模型能做什么”，RAG 解决“信息从哪里来”，它们都不直接决定流程由谁控制。

判断一个系统是否具有 Agent 特征，可以看模型的选择会不会改变后续步骤：

- 模型只在程序指定的位置填写参数，它是工作流中的智能节点；
- 模型读取工具结果后持续选择下一步，系统才具有更强的 Agent 性。

## 真实系统经常混合三种方式

支持系统不必三选一。大部分普通工单走固定流程，其中的摘要和分类使用一次模型调用；只有信息不足、跨多个系统或原因未知的工单进入 Agent 调查。若调查建议生产变更，再交给确定性流程和人工审批。

这样既保留 Agent 处理未知问题的能力，也不会让所有工单承担它的延迟、成本和不确定性。

## 选择依据

先问任务路径能否预先写出：单步且输入输出明确，从一次调用开始；多步但路径稳定，使用固定工作流；只有下一步必须依赖现场反馈时，才引入 Agent。

再看动态决策是否值得。Agent 需要更多模型调用、状态、Trace 和故障恢复。任务最好有可观察的完成信号，例如测试通过、告警解除或证据齐全。高风险动作即使出现在 Agent 任务中，也应留给程序规则和人工审批。

可以把选择记成三句话：

- **一次调用处理一个明确输入输出。**
- **固定工作流处理可预先安排的多步过程。**
- **Agent 处理需要根据现场结果改变方向的任务。**

确定某个子任务需要 Agent 后，下一步才是划分它内部的责任：哪些判断交给模型，哪些规则留给程序，哪些决定由人承担（[Agent 和 Workflow 有什么区别？]({{ site.baseurl }}/docs/interview/ai-agent/runtime/#agent-vs-workflow)）。
