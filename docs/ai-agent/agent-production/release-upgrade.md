# 如何安全发布和升级 Agent

数据生成 Agent 准备升级。新的 Skill 会生成 `enterprise-saml-v2` 模板，测试环境中的 MCP 已经支持它。生产环境却还有一些旧 MCP 实例，只认识 `enterprise-saml`。离线评测全部通过，灰度开始后，部分上传请求仍被拒绝。

团队回滚了应用代码，问题却没有完全消失，因为有些任务已经用新 Skill 生成了 v2 数据，有些 MCP 实例也已经升级。

这次事故说明：**发布 Agent 时，要追踪真正共同决定行为的版本，并提前准备整组恢复方案。**

## 到底发布了什么

一次任务的行为可能同时受到应用代码、模型与推理配置、项目规则、Skills、MCP 工具、权限策略和目标平台 API 影响。

若只记录应用提交号，失败后仍然不知道任务使用了哪版 Skill、哪个模型，以及模型看到了哪份工具 Schema。

团队可以为每次发布保存一份清单：

```yaml
release: data-agent-2026.07.2
model: claude-version-A
rules: project-rules-v18
data_generation_skill: v3.2
mcp_schema: v2
upload_service: v4.6
execution_policy: v7
```

每条任务 Trace 关联这份清单。这样，同一个评测样本失败时，工程师才能重现当时的环境。

**一次 Agent 发布，通常是一组版本的组合。**

## 先检查组件能不能配合

新 Skill 会生成 `enterprise-saml-v2`，旧 MCP Schema 不接受这个值。这类问题有明确的接口关系，可以在完整 Agent 评测之前发现。契约测试依次确认 Skill 可能生成的值、工具 Schema 接受的范围，以及 MCP 能否把数据正确传给平台。

接口检查通过后，仍要运行 Agent 回归评测。即使 Schema 没变，模型升级也可能选择不同工具；Skill 的一句说明改变，也可能让 Agent 选择另一套模板。

因此，发布前需要两种验证：

- 契约测试检查字段、枚举和接口是否兼容；
- 回归评测检查组合起来以后，Agent 是否仍然完成真实任务。

前者发现“接不上”，后者发现“接得上，但做错了”。

## 新旧版本如何过渡

引入 v2 模板时，可以分五步完成：

1. MCP 与平台先同时接受 v1 和 v2；
2. 确认旧 Agent 仍能正常运行；
3. 让少量任务开始使用新 Skill；
4. 等待仍在使用 v1 的旧任务结束；
5. 最后移除 v1 支持。

顺序很重要。上游先生成新格式，而下游尚未准备好，生产中必然出现不兼容。

工具返回结构变化也可以这样处理：先增加新字段并保留旧字段，等所有 Agent 迁移完成，再删除旧字段。

## 正在运行的任务怎么办

数据生成任务可能等待用户确认或平台恢复，跨越几个小时。若任务中途切换 Skill 和工具版本，前半段与后半段会遵循不同规则。

任务创建时应绑定发布版本。例如，`T-2048` 创建时绑定 `data-agent-2026.07.2`；即使中途等待平台恢复，继续执行时仍使用同一组版本。

旧版本需要保留到长任务结束。若旧版本存在严重安全问题，不能继续使用，系统应暂停相关任务，检查已经发生的上传和运行，再明确迁移到新版本或停止任务。

任务之间切换版本相对简单；任务执行到一半再切换，就需要专门处理已有状态。

## 先让谁使用新版本

离线评测覆盖不了所有真实项目和平台状态。新版本可以先用于测试项目，再交给内部团队试用，然后开放少量真实项目，最后扩大任务类型和流量。

观察时不能只看接口是否返回成功。数据生成 Agent 还要检查模板是否正确、上传目标是否匹配、是否出现重复操作，以及用户最终有没有完成平台运行。

发布前就要约定停止条件：跨项目写入、权限绕过和重复上传应立即停止；大量 Schema 拒绝说明兼容问题；普通成功率的小幅波动则需要结合样本量判断。

## 回滚时还要处理已经发生的动作

假设灰度失败，团队把新任务切回旧版本。此时仍有几个问题：

- 哪些任务已经生成 v2 数据？
- 哪些数据已经上传？
- 旧 MCP 能否继续处理这些数据？
- 正在等待的任务应该继续、迁移还是停止？

所以，回滚方案要包含两部分：让新任务回到上一组可用版本；检查新版本已经留下的文件、上传和平台任务，必要时迁移或补偿。

旧版本也不能在发布完成后立即删除。要等长任务结束，并确认不再需要回滚，才能移除旧 Skill、工具 Schema 和服务实例。

**切回旧版本只能阻止问题继续扩大，已经发生的业务动作仍需单独处理。**

## 一份简短的发布记录

每次发布记录六个问题就够了：

| 问题 | 需要记录的内容 |
| --- | --- |
| 为什么改 | 对应的需求或失败 |
| 发布什么 | 模型、规则、Skill、工具和服务版本 |
| 怎样验证 | 契约测试和回归评测结果 |
| 谁先使用 | 灰度范围 |
| 何时停止 | 触发暂停或回滚的异常 |
| 怎样恢复 | 旧版本和长任务的处理方式 |

生产异常发生后，这份记录与 Trace 一起回答：问题是否由本次变更引入，哪些任务受影响，应该恢复哪一组版本。

## 记住发布的是一组协作关系

安全发布可以概括为：记录完整发布组合，分别检查接口和 Agent 行为；让下游兼容后再升级上游；从低风险任务开始放量；出错时切回旧组合，并处理已经产生的状态。

真正重要的是，每次变化都能被识别，影响范围有限，出错后有一条清楚的恢复路径。
