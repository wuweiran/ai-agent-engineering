# 如何保证生产 Agent 的可靠性

周一上午，许多团队同时使用数据生成 Agent。每个任务都要生成文件、运行校验、上传数据，再启动平台任务。很快，平台开始限流，一台 Agent 服务实例也在这时重启。

工程师需要先回答一个朴素的问题：刚才那些任务怎么办？

- 哪些请求已经收到？
- 哪些文件已经生成？
- 哪些上传可能成功了？
- 重启后的实例应该从哪里继续？

**生产可靠性的重点，是让系统始终知道任务已经做过什么，下一步怎样做才不会重复或丢失。**

## 流量突然增加时

数据生成不是一次很快的接口调用。一个任务可能占用多轮模型请求、文件工作区和平台配额。若一百个请求同时开始，最先耗尽的可能是模型限额，也可能是上传连接。

因此，应用收到请求后先检查身份、项目和任务大小，再创建 `task_id` 放入队列。等有空闲执行能力时，工作进程才开始运行。

队列只能吸收短时间的高峰。等待任务持续增加时，系统要停止接收、降低任务规模，或明确告诉用户当前无法处理。把所有请求都留在队列里，只会把立即失败变成很久以后失败。

这里需要记住的是：**系统只能接收自己有能力完成或合理排队的任务。**

## 实例重启时

任务 `T-2048` 已生成文件并通过校验，正在等待上传。若这些进度只存在服务进程里，实例重启后，新的工作进程只能从头再来。

应用需要在每个重要阶段保存状态。任务 `T-2048` 可能依次经历 `queued`、`generating`、`validating` 和 `uploading`；平台暂时不可用时进入 `waiting_platform`，最终结束为 `succeeded` 或 `failed`。

状态还要记录生成的文件版本、校验报告、上传幂等键和平台任务 ID。工作进程退出后，另一个实例读取这些信息，才知道哪些步骤可以跳过，哪些外部结果需要确认。

两个实例也不能同时接管同一任务。常见做法是让工作进程在处理任务时持有一段有期限的执行权；它失联并过期后，其他实例才能接手。

**已经接收的任务要属于业务系统，不能属于某台服务实例。**

## 上传超时时

Agent 发起上传，平台保存了数据 `D-1042`，但响应在网络中丢失。Agent 只看到“超时”。

```text
发送上传请求，携带幂等键 K
  ↓
平台保存 D-1042
  ↓
响应丢失
  ↓
Agent 收到超时
```

此时不能马上再上传一次。应用应把状态记为“结果未知”，然后用幂等键 `K` 查询平台：

- 找到 `D-1042`，说明前一次已经成功，可以继续；
- 确认没有创建，才重新上传；
- 暂时无法查询，就让任务等待，而不是猜测。

相同问题还会出现在启动平台任务、发送消息和创建退款申请中。

**超时只说明没有收到结果，不说明动作没有发生。结果未知时先查证。**

## 平台限流时

平台要求三十秒后重试。如果一千个任务都在第三十秒同时醒来，平台马上又会被压垮。

任务可以进入 `waiting`，在平台给出的时间基础上稍微错开恢复时间。连续失败时，等待时间逐步增加。参数错误和权限不足则不应重试，因为时间不会改变它们。

还要从平台整体观察重试。Agent 即使换了几个工具，只要最后都调用同一个故障平台，仍然在增加压力。

平台持续不可用时，可以暂停新的上传，同时继续允许文件生成和本地校验。这样，上传故障不会占满所有工作进程，也不会让其他平台的任务一起停摆。

**重试要给依赖恢复的机会，不能把一个故障放大成重试高峰。**

## 用户看到什么

异步处理和等待对系统有利，却可能让用户以为任务卡住。界面至少要把状态说清楚：

| 状态 | 用户看到的含义 |
| --- | --- |
| `queued` | 请求已收到，正在等待执行 |
| `running` | 正在生成、校验或上传 |
| `waiting` | 正在等待平台恢复、用户信息或审批 |
| `succeeded` | 数据和平台任务已完成 |
| `failed` | 无法继续，可以查看已有结果和原因 |
| `cancelled` | 已停止后续步骤 |

取消任务通常只能停止后续动作。已经上传的数据不会自动消失，除非平台本身提供删除或补偿能力。

清楚的状态也能减少重复提交。用户知道任务正在等待平台恢复，就不需要再创建一份相同任务。

## 怎样验证系统真的能恢复

正常路径成功，只能证明功能可用。可靠性要在故障中验证。

可以在测试环境做几个有针对性的演练：上传过程中重启工作进程；让平台成功后丢掉响应；让模型 API 连续限流；让等待中的任务被取消。

每次只检查三个结果：

1. 任务有没有丢失？
2. 外部动作有没有重复？
3. 恢复后是否从正确位置继续？

如果这三点成立，系统才真正具备了恢复能力。

## 记住任务不能失去来路

生产可靠性可以用一句话复述：请求到来时不过量接收，开始执行后保存关键进度，外部结果不确定时先查询，依赖故障时避免集中重试，恢复后从正确位置继续。

它不要求每个任务都成功。它要求任务失败或系统重启时，工程师仍然能说清已经发生什么，以及下一步怎样做才安全。
