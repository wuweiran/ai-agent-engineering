# 如何保证生产 Agent 的可靠性

上午十点，许多团队同时使用数据生成 Agent。每个任务要调用模型生成数据，通过校验工具检查，再由 MCP 上传到平台并启动运行。目标平台很快开始限流，一台 Agent 服务实例随后重启，还有几个上传请求其实已经成功，只是响应没有返回。

如果所有任务都立即重试，平台会承受更大压力；如果服务把超时直接记成失败，用户会再次提交并产生重复数据；如果任务状态只存在实例内存中，重启后所有进度都会丢失。

**生产可靠性的核心，是控制系统接收多少工作，让已接收的任务具有可靠状态，并在依赖恢复后安全继续。**

```text
流量进入
  ↓
容量保护：接收、排队或拒绝
  ↓
持久执行：每一步都有状态
  ↓
依赖故障：限制重试和影响范围
  ↓
恢复后查证结果并继续
```

## 先保护系统接收任务的能力

用户请求到来时，Agent 不一定能够立即完成。一次任务可能占用多轮模型调用、代码执行环境和平台配额。如果请求都直接进入执行，短时流量就可能耗尽连接、内存或模型限额。

长任务通常更适合异步处理：服务先校验请求并创建任务，返回 `task_id`，工作进程再从队列取得任务。队列吸收短时波动，但不能无限增长；系统仍要知道自己能同时执行多少任务，以及排队多久后请求已经失去价值。

```text
用户提交
  ↓
校验身份、配额和任务大小
  ↓
有容量 → 创建任务并排队
容量紧张 → 延迟接收、降级或明确拒绝
  ↓
返回 task_id 和当前状态
```

拒绝并不一定比排队差。一个预计等待两小时、但用户十分钟后就不再需要的数据任务，只会浪费后续资源。容量保护要让系统在过载时保持可预测，而不是接受所有请求后一起超时。

**队列用来平滑流量，容量上限用来阻止排队本身成为故障。**

## 已接收的任务不能依赖单个实例

服务实例可能因为部署、故障或扩缩容随时退出。已经接收的任务需要把目标、进度、外部资源和恢复位置保存在持久存储中，而不是只存在进程内存或模型会话里。

数据生成任务可以经历：

```text
queued
  ↓
running: generating_data
  ↓
running: validating
  ↓
running: uploading
  ↓
waiting: platform_rate_limited
  ↓
running: verifying_upload
  ↓
succeeded / failed
```

工作进程领取任务时还要有所有权或租约。实例失联后，租约到期，另一个实例才能接管；两个实例不能同时继续同一项写操作。

恢复时不应重放全部过程，而是读取最后一个可靠状态，确认外部动作的真实结果，再从明确位置继续。这与单任务状态设计一脉相承，只是生产系统还要解决多个工作进程如何安全接管任务。

## 超时不等于动作失败

Agent 上传数据时，平台可能已经保存文件，但响应在网络中丢失。此时任务状态是“结果未知”，不是“上传失败”。

```text
发送 upload(dataset, idempotency_key=K)
  ↓
平台保存数据 D-1042
  ↓
响应丢失
  ↓
Agent 收到超时
```

可靠处理方式是使用稳定幂等键，并提供按幂等键或资源 ID 查询结果的接口。恢复后先查找 `K` 对应的数据：存在则记录 `D-1042` 并继续，不存在且确认前次未执行，才重新上传。

同样的原则适用于启动平台任务、发送消息和创建退款。**结果未知时先查证，只有确认未执行后才能安全重试。**

## 重试要给依赖恢复空间

平台限流时，如果一千个任务都在一秒后重试，下一秒仍会形成相同高峰。重试需要拉开时间，并限制总次数：

```text
收到 rate_limited(retry_after=30s)
  ↓
任务进入 waiting
  ↓
在建议时间基础上加入少量随机偏移
  ↓
到期后少量任务逐步恢复
```

等待时间通常随连续失败增加，让依赖有恢复空间；随机偏移避免大量任务同时醒来。错误也要分类：限流和短暂不可用可能重试，参数错误和权限不足则不会因等待而改变。

重试预算还应覆盖整条任务，而不只是单个工具。若 Agent 换了三个查询工具，却都在调用同一个故障平台，系统仍然是在持续放大同一依赖的压力。

**重试的目标不是尽快再次调用，而是在不扩大故障的前提下等待成功机会。**

## 局部依赖故障不能拖垮整个服务

数据生成 Agent 可能依赖模型 API、对象存储、校验服务和运行平台。某一个平台变慢时，占满所有工作进程和连接，会让不依赖该平台的任务也无法执行。

工程师可以按依赖和任务类型隔离容量。例如，上传阶段使用独立并发上限；平台连续失败时暂停新的上传，但仍允许数据生成和本地校验；低优先级批量任务让位于交互任务。

```text
生成与本地校验 ── 继续
上传到故障平台 ── 限制并发、进入等待
查询已有任务   ── 保留少量容量
其他平台任务   ── 不受影响
```

这种隔离不是让故障消失，而是控制影响范围。恢复后也要逐步放量，避免积压任务瞬间压垮刚恢复的依赖。

## 用户需要看到真实任务状态

异步和等待提高了系统可靠性，却会让用户失去即时结果。产品需要展示真实状态，而不是一直显示“处理中”：

| 状态 | 用户可以理解的含义 |
| --- | --- |
| `queued` | 请求已接收，尚未开始 |
| `running` | 正在生成、校验或上传 |
| `waiting` | 等待平台恢复、用户信息或审批 |
| `succeeded` | 数据和平台任务达到完成条件 |
| `failed` | 无法继续，已有结果和原因可查看 |
| `cancelled` | 用户或系统已停止后续执行 |

用户取消任务时，系统还要区分“停止后续步骤”和“撤销已经发生的外部动作”。已经上传的数据不会因为队列任务取消而自动消失，除非平台支持明确的补偿操作。

清楚的状态也能减少用户重复提交。若用户知道任务正在等待平台限流恢复，就不必再次创建相同任务。

## 恢复演练比正常成功更能验证可靠性

正常路径跑通不能证明生产可靠。工程师需要主动制造关键故障：

```text
上传过程中重启工作进程
平台成功后丢弃响应
让模型 API 连续限流
让目标平台短暂不可用
在任务等待时取消或重新部署
```

然后检查三个结果：任务是否丢失，外部动作是否重复，恢复后是否从正确位置继续。还要观察积压是否持续增长、恢复时是否产生重试高峰，以及其他任务是否被连带拖慢。

这些演练可以在测试环境和故障注入中完成。生产事故中的真实 Trace 则补充评测没有覆盖的组合，并转化为新的可靠性回归场景。

## 三步理解生产可靠性

1. **接收任务时保护容量：能执行才接收，不能执行就排队、降级或拒绝。**
2. **任务执行时保存可靠状态：实例重启和结果未知都不会导致进度丢失或副作用重复。**
3. **依赖故障时限制影响：分类错误、错开重试、隔离容量，并在恢复后逐步继续。**

可靠性不是保证所有任务立即成功，而是让系统在压力和故障下仍然知道自己接受了什么、已经做了什么、接下来能安全做什么。
