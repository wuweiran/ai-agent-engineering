# Agent 监控与故障处理

数据生成 Agent 发布了新版登录数据 Skill。模型 API、MCP 和上传平台的成功率都在正常范围，已经被系统接收的任务中，最终完成平台运行的比例却持续下降。用户提交没有报错，文件也能上传，问题直到平台运行阶段才暴露。如果系统只监控接口状态，这类退化通常要等到用户集中投诉才会被发现。

Agent 监控负责持续采集指标、日志、Trace 和任务状态，发现用户任务或系统运行的异常；故障处理负责确认影响范围、控制影响、定位根因、恢复任务并防止问题再次发生。在软件工程中，这套能力常被统称为“可观测性与事故处理”。本文使用更直接的“监控与故障处理”，重点讨论生产环境中怎样发现异常和采取行动。

```text
用户任务与系统指标发现异常
→ 按任务、阶段和版本确认影响范围
→ 停止可疑版本或高风险动作
→ 使用指标与 Trace 定位第一次异常
→ 回滚或修复
→ 处理事故期间遗留的业务状态
→ 验证用户任务恢复
→ 复盘并补充评测、程序规则和发布检查
```

## 监控对象

Agent 监控需要同时覆盖**用户结果、Agent 执行和系统运行**。三层指标回答不同问题，也必须通过任务和版本标识关联起来。

| 层面 | 主要指标 | 主要问题 |
| --- | --- | --- |
| 用户结果 | 任务完成率、成功交付率、人工接管率、重试率、放弃率 | 用户是否真正完成目标 |
| Agent 执行 | 阶段成功率、等待与失败分布、工具和参数正确率、重复副作用、虚假完成 | Agent 在哪个阶段、以什么方式偏离 |
| 系统运行 | 模型错误与限流、队列长度与等待时间、MCP 和平台错误率、P95/P99 延迟、资源使用率 | 哪个依赖或基础设施正在异常 |

数据生成 Agent 的业务任务从接收、生成、校验、确认、上传到平台运行，每个阶段都要有可识别的状态。模型请求成功或 HTTP 返回 200，只能说明某个组件仍在响应，不能证明任务已经完成。生产监控应从用户结果开始，再用 Agent 执行和系统指标解释变化。

越权、错误写入、重复副作用和虚假完成等严重失败需要单独计数和告警，不能被平均完成率掩盖。延迟与成本也应按成功任务统计，否则大量快速失败会让平均耗时看起来更好。

三层信息通过业务任务或业务对象 ID、Agent 运行 ID、发布版本、Tool Call ID 和平台资源 ID 关联。聚合指标用于判断影响规模和共同特征，Trace 用于还原单个任务为什么失败。缺少任何一侧，工程师都很难同时判断“影响有多大”和“问题发生在哪里”。

## 告警设计

告警应该表示**需要工程师采取行动的异常**，不需要为每次模型调用和工具调用分别告警。常见告警信号包括：

- 用户任务完成率明显偏离正常范围；
- 某个任务阶段持续积压或失败；
- 新版本的失败率显著高于基线或旧版本；
- 高风险写操作出现越权、重复或结果未知；
- 队列、模型、MCP 或业务平台达到错误率、延迟和容量阈值。

固定阈值适合权限失败、重复写入等低容忍事件；完成率和延迟通常需要结合历史基线、任务类型、流量和持续时间判断。一个失败任务通常进入 Trace 调查，不一定触发事故；同一失败机制在特定版本或任务类型中持续聚集，才说明系统性退化。

告警内容至少要包含异常指标、开始时间、影响任务类型或版本，以及进入相关任务和 Trace 的入口。只有“成功率下降”而没有分组维度和调查入口，会让值班工程师重新做一遍数据关联。

## 影响范围

收到告警后，第一步是确认**从什么时候开始、影响哪些任务、集中在哪个阶段和版本**。常用拆分维度包括：

- 任务类型、租户、项目和区域；
- Agent、模型、Prompt、Skill 和工具版本；
- 任务阶段、错误类型和外部资源状态；
- 客户端、后端 Worker 与业务平台环境。

当前事故按版本和任务类型拆分后显示：基础登录任务正常，问题集中在新版 Skill 处理的企业 SAML 任务；这些任务大多已经上传，却在平台运行阶段失败。队列等待、模型错误和 MCP 延迟没有同步上升，平台也能正常运行旧版任务，因此没有证据支持重启所有服务或扩大平台容量。

影响范围应先由聚合指标确定，再选择失败任务和正常任务作为对照进入 Trace。先挑一条显眼 Trace 猜根因，容易把单个项目的数据问题误判为系统事故。

## 影响控制

根因调查可能需要时间，生产故障不能等待完整结论后再处理。影响控制的目标是**阻止新的损失，同时保留调查和恢复所需的现场**。

在当前案例中，团队停止让新的企业 SAML 任务使用新版 Skill，将新任务切回上一组已验证版本；基础登录和其他数据类型继续运行。已经使用新版生成但尚未上传的任务暂停在当前状态，保留文件、任务阶段和发布版本。

若故障涉及跨项目写入、权限绕过或数据破坏，应立即关闭相关写工具、冻结可疑任务并保护审计记录。若问题只是特定任务类型的平台运行失败，则可以只隔离可疑版本和任务，不必停止所有生成与只读能力。

清空队列、覆盖文件或批量重跑可能暂时降低错误数量，却会破坏 Trace、丢失业务状态并产生重复副作用。停止 Agent 也只会阻止后续模型和工具调用，已经创建的数据集、退款申请或平台运行仍要根据业务语义处理。

## 根因定位

影响得到控制后，工程师结合聚合指标与对照 Trace 定位第一次异常。指标负责缩小范围，Trace 负责比较失败任务和正常任务的执行链。

企业 SAML 的新版与旧版任务都正确识别了任务类型，也取得相同平台模板。两组轨迹最早在校验阶段分叉：旧版执行完整业务校验，新版为了缩短大文件处理时间，只执行了结构校验。结构校验确认字段和数量正确，却没有验证 SAML 证书与身份字段的组合关系，所以模型和 MCP 调用都成功，数据也成功上传，直到平台运行时才被拒绝。

团队检查新版 Skill，发现其中规定“大文件通过结构校验后即可上传”，却没有保留企业 SAML 必须完整校验的例外。使用同一任务、模型、工具响应和平台环境重放，只恢复完整校验后，平台运行重新成功，根因假设得到确认。

第一次异常发生在哪一层，决定修复位置。Context 缺少事实就修复装配或检索，Skill 方法错误就修复 Skill，Tool Call 正确而执行错误就检查 MCP 或业务服务，状态没有更新则检查持久化和恢复逻辑。最终出错的组件不一定是根因所在的组件。

## 业务状态恢复

回滚只阻止新任务继续使用问题版本，事故期间已经产生的业务状态还需要逐类处理。当前案例留下三类任务：

| 任务状态 | 恢复方式 |
| --- | --- |
| 已生成但尚未上传 | 使用完整规则重新校验；内容未变化时保留原确认，内容变化时生成新版本并再次确认 |
| 已上传但尚未运行 | 根据 `dataset_id` 查询真实状态，确认是否可以继续，不能重复创建数据集 |
| 已运行失败 | 保留原 `run_id` 作为事故证据；修复后的内容使用新的数据集和运行记录 |

任务记录决定哪些步骤已经完成、哪些副作用已经发生、哪些操作仍能安全继续。恢复时优先读取业务对象和外部资源状态，不依赖原会话中的自然语言结论。

用户也需要看到明确状态，例如“受版本问题影响，正在重新校验”，而不是只收到一次模糊失败。批量恢复应记录处理范围、执行版本、结果和异常任务，便于审计和再次接管。

## 恢复验证

回滚命令成功、模型错误率下降或 MCP 恢复 200，都不足以宣布故障结束。服务恢复必须由**真实任务重新完成**来证明。

恢复验证至少包括：

1. 新进入的受影响任务重新执行正确步骤；
2. 任务能够到达完整业务终态，用户完成率恢复到正常范围；
3. 没有新的受影响任务，失败和等待队列持续下降；
4. 事故期间遗留任务都有明确处理结果；
5. 重复写入、越权和虚假完成等严重失败没有出现；
6. 关键延迟、容量和成本没有形成新的退化。

用户结果通常晚于底层接口恢复。异步任务需要走完校验、确认、上传和平台运行阶段，因此恢复判断要覆盖完整业务链，而不是在回滚完成后立即关闭故障。

## 复盘与预防

故障复盘需要把一次事故转化为可验证的系统改进，而不是停在“加强监控”或“让模型更谨慎”。复盘至少回答：

- 哪个指标首先发现问题，是否足够及时；
- 影响范围和用户损失是什么；
- 哪项措施阻止了问题继续扩大；
- 第一次异常和根因位于哪个系统层；
- 已有任务怎样恢复，是否发生额外副作用；
- 哪项评测、程序规则或发布检查能够阻止同类问题再次发生。

企业 SAML 必须完整校验已经成为稳定业务规则，团队将它加入上传服务的确定性检查：任务没有对应的完整校验记录，服务端就拒绝上传。模型仍然负责理解和修复数据，但不能跳过这个条件。

失败任务经过脱敏后加入回归评测集；契约测试明确不同校验模式的含义；发布灰度增加按任务类型和版本统计的平台运行完成率；停止条件也包括“新版任务在下游集中失败”。事故期间使用的查询和仪表盘应保留，缩短下一次类似问题的确认时间。

Agent 监控与故障处理可以概括为：**用用户结果、Agent 执行和系统运行三层指标发现异常，按任务阶段和版本确认影响，先控制影响，再通过指标与 Trace 定位根因，恢复已有业务状态，并用真实任务验证恢复。**故障只有在用户任务重新完成、受影响状态得到处置，并且失败转化为评测与程序规则后，才算真正处理完毕。
