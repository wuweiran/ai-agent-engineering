# Agent 可观测性与事故处理

数据生成 Agent 发布了新版登录数据 Skill。模型 API、MCP 和上传平台的成功率都在正常范围，值班工程师却收到另一类告警：已经被系统接收的任务中，最终完成平台运行的比例持续下降。

用户提交没有报错，文件也能上传，问题直到平台运行阶段才暴露。如果团队只监控接口状态，这次退化可能要等到用户集中投诉才会被发现。

**生产观测从用户任务是否完成开始，再沿 Agent 行为和底层系统寻找原因。**

## 告警从用户结果开始

数据生成 Agent 的价值落在用户能否取得符合要求的数据并完成平台运行。模型调用成功或 HTTP 返回 200，只能说明链路中的一个环节正常。因此，团队以业务任务为观测单位，记录任务从接收、生成、校验、确认、上传到运行的状态变化。

用户完成率下降后，工程师再查看另外两个层面：

| 层面 | 这次事故要回答的问题 |
| --- | --- |
| 用户结果 | 哪些用户没有完成任务，失败前已经等待多久 |
| Agent 行为 | 任务停在哪个阶段，使用了哪版 Skill，怎样校验、重试和结束 |
| 系统运行 | 模型、队列、MCP 和平台是否报错、限流或变慢 |

三层信息需要通过 `task_id`、Agent 运行 ID、发布版本和平台资源 ID 关联。只有聚合指标而没有任务入口，团队知道“情况变差了”，却找不到可以还原的执行过程；只有 Trace 而没有整体指标，又很难判断一条失败代表偶发现象还是大面积事故。

这里不需要为每次工具调用都设置告警。告警应当指向需要工程师采取行动的变化，例如用户完成率明显偏离正常范围、任务在同一阶段持续堆积，或者高风险写操作出现重复。

## 先确定影响范围

值班工程师先找出异常从什么时候开始，再按任务类型、发布版本、项目和失败阶段拆分。结果显示：基础登录任务正常，问题集中在新版 Skill 处理的企业 SAML 任务；这些任务大多已经上传，却在平台运行阶段失败。

这一步把范围从“整个 Agent 不稳定”缩小到“特定版本下的一类任务”。同时，队列等待、模型错误和 MCP 延迟都没有同步上升，平台也能正常运行旧版任务，因此暂时没有理由重启所有服务或扩大平台容量。

定位影响范围时，不宜先挑一条最显眼的 Trace 猜根因。工程师需要先确认共同特征，再选择受影响任务和正常任务作为对照。否则，一个项目自身的数据问题可能被误认为系统事故。

**指标用来判断哪里正在发生异常，Trace 用来解释一项任务为什么异常。**

## 根因未明时先控制影响

调查完整根因可能需要时间，生产事故不能等结论完全确定后才处理。

团队先停止让新的企业 SAML 任务使用新版 Skill，把新任务切回上一组已验证版本。基础登录和其他数据类型继续运行，避免一次局部退化中断整个产品。已经使用新版生成、但尚未上传的任务暂时停在当前状态，保留文件和发布版本，等待后续处理。

如果事故涉及跨项目写入、权限绕过或可能破坏数据，止损范围还要更大：立即关闭相关写工具，冻结可疑任务，并保护审计记录。当前案例只造成平台运行失败，因此不需要删除数据或停止所有只读与生成功能。

止损的目标是限制新影响，同时保存调查所需的现场。随意清空队列、重跑任务或覆盖文件，可能暂时降低错误数量，也会破坏 Trace 并产生重复写入。

## 从一组任务找到第一次异常

工程师选择几条新版失败任务，与相同类型的旧版成功任务比较。两组任务都正确识别了企业 SAML，也取得了相同平台模板。最早的差异出现在校验阶段：旧版执行完整业务校验，新版为了缩短大文件处理时间，只执行了结构校验。

结构校验确认字段和数量正确，却没有检查 SAML 证书与身份字段的组合关系。因此，MCP 调用都成功，上传也成功，平台真正运行时才拒绝这些数据。

进一步检查新版 Skill，团队发现其中写着“大文件通过结构校验后即可上传”，却没有保留企业 SAML 必须完整校验的例外。使用同一任务和工具环境重放，只恢复完整校验后，平台运行重新成功，根因假设得到确认。

这次事故不是模型 API 或上传平台故障。第一次偏离发生在新版 Skill 所定义的方法中，平台失败只是最后的表现。

## 回滚之后还要处理已有任务

把新任务切回旧版只阻止问题继续扩大。事故期间已经留下三类任务：尚未上传的候选文件、已经上传但尚未运行的数据集，以及已经运行失败的任务。

团队没有把它们全部从头重跑。尚未上传的文件使用完整规则重新校验，通过后仍保留原有用户确认；若修复改变了文件内容，则生成新版本并再次请求确认。已经上传的数据先根据 `dataset_id` 查询状态，避免重复创建。运行失败的任务保留原 `run_id` 作为事故证据，修复后的数据使用新的数据集和运行记录。

工作进程可以根据持久任务状态继续，用户也能看到“受版本问题影响，正在重新校验”，而不是只收到一次模糊失败。

**回滚恢复执行版本，任务记录决定已有业务状态怎样处理。**

## 服务恢复的判断标准

旧版恢复后，模型和 MCP 指标很快保持正常，但这并不能直接关闭事故。团队先确认新进入的企业 SAML 任务重新执行完整校验，再等待这些异步任务走到平台运行阶段，观察用户完成率是否回到正常范围。

恢复判断同时包括：没有新的受影响任务，等待队列开始下降，旧版任务能够完成，已有失败任务得到明确处理。用户结果通常晚于底层接口恢复，因此不能在回滚命令成功后立即宣布结束。

事故期间的临时查询和仪表盘也要保留下来。下一次出现类似现象时，值班工程师可以直接从任务类型、版本和失败阶段缩小范围。

## 从事故到系统改进

复盘没有停在“Skill 少写了一条例外”。企业 SAML 必须完整校验已经是稳定业务规则，团队把它加入上传服务的确定性检查：任务没有对应的完整校验记录，服务端就拒绝上传。模型仍可判断如何修复数据，却不能跳过这个条件。

这次失败任务经过脱敏，加入回归评测；契约测试确认不同校验模式的含义；发布灰度增加按任务类型观察的平台运行完成率；停止条件也明确包含“新版任务在下游集中失败”。

复盘需要留下可以改变系统的结论：为什么现有观测能够或不能及时发现，哪项止损有效，根因落在哪个系统层，以及什么验证能够阻止同类问题再次发布。笼统地要求“加强监控”或“让模型更谨慎”，不会改变下一次事故的结果。

## 观测最终要支持行动

生产 Agent 的一次事故会跨过用户任务、模型决策、工具调用和业务平台。底层服务健康只能说明组件仍在响应，不能证明用户目标已经完成。

有效的生产观测把用户结果连接到任务阶段、Agent 行为、发布版本和底层依赖。事故发生后，工程师先确定谁受到影响，限制问题继续扩大，再用对照 Trace 找到第一次异常；恢复版本之后，还要处理已经产生的文件、数据集和运行记录。

事故处理完成的标志，也不只是图表恢复正常。真实任务重新完成，受影响状态得到处置，失败被转化为评测、程序规则和发布检查，系统才真正从这次事故中恢复。
