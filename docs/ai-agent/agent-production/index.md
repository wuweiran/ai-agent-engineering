# 如何保障 Agent 的生产运行

数据生成 Agent 在测试时一次只处理几个任务。上线后的第一个周一上午，几十个团队同时提交请求：模型 API 开始限流，上传平台偶尔超时，一台服务实例又恰好重启。

这时，团队要处理的问题已经超出“Agent 能不能生成正确数据”。有些任务还在排队，有些已经生成文件，有些可能上传成功却没有收到响应。另一些退化更隐蔽：模型和工具接口都正常，用户却越来越难完成平台运行。

**Agent 上线后，工程师既要看见真实任务是否正在退化，也要让故障中的任务可以安全恢复，并限制每次系统变化的影响。**

## 先让任务经得起故障

假设上传请求超时。若系统直接重试，可能产生两份数据；若直接报失败，用户又会重新提交；若任务只保存在进程内存中，实例重启后连上一步做到哪里都不知道。

可靠运行需要把几个动作接起来：请求太多时有序接收，任务开始后保存进度，外部调用结果不确定时先查证，依赖恢复后再从正确位置继续。

队列、持久状态、幂等键和错开重试都在解决这条链路中的具体问题。最重要的判断是：**系统必须说得清已经接收什么、已经做过什么、还能安全做什么。**

## 再控制每次系统变化

几周后，团队更新了数据生成 Skill，让它使用新版模板。测试环境中的 MCP 已支持新字段，部分生产实例却仍使用旧 Schema，于是同一版本的 Agent 在不同任务中得到不同结果。

Agent 的行为会同时受到模型、规则、Skill、工具和业务服务影响。发布时需要记录真正生效的组合，先验证这些组件能否配合，再让少量低风险任务使用。运行正常后逐步扩大范围；出现异常则恢复上一组经过验证的配置。

这里最值得记住的是：**一次 Agent 发布往往包含多项共同决定行为的版本。**只记录应用代码的提交号，通常不足以复现一次任务。

## 先发现用户任务正在退化

Agent 可能仍在正常调用模型和 MCP，用户完成率却持续下降。团队要从任务结果发现异常，再按任务类型、阶段和发布版本缩小范围，并进入相关 Trace 调查。

事故发生后，根因调查和止损要同时进行。工程师先暂停可疑版本或高风险动作，保留任务现场，再定位第一次异常。回滚之后还要处理已经生成、上传或运行的业务状态，直到真实任务恢复。

## 质量体系和生产保障如何衔接

离线评测帮助团队判断一个版本是否值得发布，Trace 帮助定位失败原因。真实生产环境还会出现评测中没有的流量、数据和依赖故障。

版本通过评测后逐步发布。生产观测发现新的异常，事故处理限制影响并恢复服务，确认过的失败再进入 Trace、修复和回归评测。质量体系控制改动进入生产前的风险；生产保障发现并限制问题发生后的影响。两者不断循环，Agent 才能从一次成功演示变成长期可用的软件。

## 生产运行要持续回答什么

这一阶段的工作围绕三个问题展开：用户任务正在退化时，系统能否及时发现并找到受影响范围；故障发生后，任务能否知道自己走到哪里并安全继续；系统变化时，能否先限制影响，出错后恢复到已知可用版本。

它们分别对应生产观测与事故处理、任务可靠性，以及发布与升级。延迟、成本、安全和容量同样重要，可以在具体生产问题出现时继续深入，不必一次全部展开。
