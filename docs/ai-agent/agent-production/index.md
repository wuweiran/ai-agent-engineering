# 如何保障 Agent 的生产运行

数据生成 Agent 已经通过评测并开始服务真实用户。上午十点，几十个团队同时提交任务；模型 API 开始限流，目标平台偶尔超时，一部分上传其实已经成功，但 Agent 没有收到响应。与此同时，团队准备升级 Skill 和 MCP 工具，单次任务的平均成本也持续上涨。

这些问题不能靠提高模型能力解决。Agent 上线后，还需要像其他生产软件一样管理容量、故障、变更和资源，只是它的行为还会受到模型决策、上下文增长和多轮工具调用影响。

**保障生产运行，就是让大量真实任务在依赖故障和系统变化中仍然可靠执行，并以可接受的资源代价持续提供质量。**

```text
真实流量与外部故障
        ↓
保护容量，保存任务，安全恢复
        ↓
控制模型、Skill 和工具的变化
        ↓
治理延迟与成本
        ↓
持续满足用户和业务目标
```

## 可靠运行：故障不应变成失控

单个 Agent 任务已经设计了重试、等待和终止，但生产系统还要同时处理许多任务。突发流量可能耗尽模型配额，目标平台故障可能让所有任务一起重试，实例重启还可能中断正在执行的工作。

工程师需要保护系统的接收能力，让任务有可恢复状态，并限制一个依赖故障的影响范围。例如，任务先进入有容量上限的队列；上传超时后通过幂等键查询真实结果；平台限流时延后重试，而不是让所有任务立即再次调用。

可靠性的目标不是让每个任务永不失败，而是：**故障发生时，系统知道哪些任务已经接收、哪些动作是否成功、哪些任务可以恢复，并且不会因局部问题拖垮整体。**

## 安全变更：Agent 发布不只是一份代码

Agent 的行为由多项版本共同决定：

```text
模型 + 系统规则 + Skills + MCP 工具 + 业务服务 + 运行策略
```

任何一项变化都可能改变工具选择、任务路径和成本。模型升级可能让原来的提示失效，Skill 修改可能改变模板选择，MCP 返回结构变化则可能让 Agent 误读结果。

生产变更需要记录这些版本之间的组合，并经过回归评测、测试环境验证和逐步发布。发现严重退化时，团队要能恢复到上一组已知可用的组合，而不只是回滚应用代码。

## 资源治理：质量必须能够长期承担

一次 Agent 任务的延迟和成本来自整条链路：装配上下文、调用模型、等待工具、重试和平台运行。只优化单次模型调用，未必改善用户等待时间；为了减少 Token 删除关键上下文，也可能让失败和重试更多。

工程师要从任务价值出发设置资源边界：低价值任务允许多少轮判断，高成本运行是否需要确认，哪些只读查询可以并行，哪些结果可以缓存，以及什么时候应换用固定工作流。

**成本和延迟不是独立于质量的两个数字，而是提供目标质量所付出的资源代价。**优化的目标是在不破坏任务成功和安全边界的前提下，减少无效步骤和等待。

## 三者会相互影响

生产运行中的三个问题不能孤立处理：

- 为了可靠性增加重试，可能放大流量和成本；
- 为了降低延迟并行调用工具，可能触发依赖限流；
- 为了升级模型提升质量，可能改变步骤数和工具参数；
- 为了快速回滚切换旧版本，还要保证旧 Skill 与当前 MCP 接口兼容。

因此，工程师观察的不只是服务是否在线，还要连接用户结果、Agent 行为和系统资源：用户是否完成任务，Agent 为何等待或失败，底层模型和工具消耗了多少容量。

## 生产保障与质量体系的关系

质量体系回答“这个版本是否值得发布、失败为什么发生”；生产保障回答“已经发布的版本能否在真实流量和故障中持续服务”。两者形成循环：

```text
评测和回归控制发布风险
  ↓
生产系统承载真实任务
  ↓
监控发现容量、故障和质量变化
  ↓
Trace 定位具体原因
  ↓
改进并重新评测
```

一次平台超时既可能暴露可靠性缺陷，也会产生新的评测场景。一次模型升级既是质量改进，也是一项需要灰度和回滚的生产变更。

## 三个问题理解生产运行

1. **面对流量和故障，已接收的任务能否安全继续或恢复？**
2. **模型、Skill、工具和服务变化时，影响是否可验证、可限制、可回滚？**
3. **系统能否以业务可以接受的延迟和成本持续提供目标质量？**

这三个问题分别对应可靠性、变更管理和资源治理。它们把 Agent 从一个可以演示的能力，变成能够长期服务真实用户的软件系统。
