# Agent 执行与业务生命周期

数据生成 Agent 已经能够生成文件、调用校验器并修复错误。用户关闭会话后，已经确认的文件仍要等待上传；平台运行失败后，也可能由另一次 Agent 执行继续分析。同一项业务工作会跨越会话、Agent 执行和外部平台运行。

接入业务系统的关键，是把两种生命周期连接起来：**业务系统记录用户委托的工作和正式业务结果，Agent Runtime 记录 Agent 为推进这项工作进行的执行。**

## 业务工作独立于 Agent 执行

业务系统首先要用自己的对象表达用户关心的工作。数据生成产品可以创建业务任务 `T-2048`，记录目标项目、文件版本、用户确认、`dataset_id`、`platform_run_id` 和最终交付状态。即使没有 Agent，这项任务仍可以由人工或固定 Workflow 完成。

订单和退款场景不一定需要额外创建一个名为“业务任务”的通用对象。订单、退款申请、审批记录和退款交易已经能够表达各自的业务生命周期，就应继续由这些领域对象承载状态。

Agent 任务表示 Agent 围绕一个目标进行的一次可跟踪执行，是 Runtime 管理的逻辑单元。当前目标、执行步骤、工具结果、等待原因和恢复位置等状态，可以单独保存，也可以合并进业务任务；客户端 Runtime 还可以保存在本地。会话则提供交互与消息历史。三者可能一一对应，也可能有不同的生命周期，概念上的区分不要求数据库机械拆成多张表。

```text
业务任务 T-2048：生成并运行一份测试数据
├─ Agent 任务 A-901：生成并校验候选文件
├─ 用户确认文件 v2
├─ Agent 任务 A-902：上传文件并启动平台运行
└─ Agent 任务 A-903：分析运行 R-8821 的失败并生成修复版本
```

同一个 Agent 任务也可以在等待用户确认后恢复，不要求每个阶段都新建任务。需要拆分时，通常是因为执行目标、恢复位置、Agent 版本或审计边界已经不同。**业务任务可以没有 Agent 任务，也可以关联一次或多次 Agent 执行。**

## 业务入口启动 Agent 执行

Agent 可以由用户操作、工单、告警、定时事件或已有 Workflow 触发。入口先创建或找到对应的业务对象，再把这次执行需要的目标和引用交给 Runtime：

```text
业务入口
→ 创建或读取业务任务
→ 绑定用户、租户和资源范围
→ 启动一次 Agent 执行
→ Runtime 组装 Context，开始执行
```

交给 Runtime 的信息通常包括业务任务或业务对象 ID、当前操作者、资源 ID、完成条件和允许的动作。若这次执行需要独立恢复、取消或审计，系统可以为它分配 Agent 任务 ID；简单场景也可以与业务任务共用记录，或者由客户端 Runtime 保存执行状态。业务事实仍保存在原系统中，模型需要时通过工具读取最新状态，不必把业务对象完整复制到 Agent 状态里。

身份与权限只是任务入口的一项约束。模型可以理解“为当前项目生成数据”，项目范围必须由可信入口绑定；后续每次真实动作仍由业务服务重新鉴权。这部分边界与普通后端系统一致，Agent 接入没有改变它。

## 工具调用连接两种生命周期

模型通过 [工具]({{ site.baseurl }}/docs/ai-agent/agent-design/tools/)提出查询或动作，Runtime 把调用与当前业务任务关联，再交给业务服务执行。

```text
模型请求 upload_dataset
→ Runtime 附加 task_id、用户身份和幂等键
→ 平台服务检查权限、配额和文件版本
→ 返回 dataset_id 与执行状态
```

工具返回的 `dataset_id`、`refund_request_id` 和 `platform_run_id` 具有两种用途。它们进入下一轮 Context，帮助模型判断下一步；也写回业务对象，使界面、审批流程和后续执行能够找到已经发生的动作。

并非所有工具结果都要写入业务数据库。搜索候选、临时日志和模型中间判断可以留在 Agent Trace 或任务状态中。已经创建的业务资源、用户确认、审批结果和影响业务流程的状态变化，则必须回到拥有该生命周期的业务系统。

**工具调用是 Agent 执行与业务状态的连接点，稳定资源 ID 是连接两边的依据。**

## 等待期间业务流程仍可推进

Agent 可能创建一项待审退款申请，然后结束当前执行。退款申请随后由主管审批，支付服务再完成退款。审批期间可以没有模型调用，用户也不必守在原会话中。

```text
Agent 任务 A-301：分析资格并创建申请
  ↓ 产生 refund_request_id
退款申请 RR-109：等待审批 → 已批准 → 退款中 → 已退款
  ↓ 如需解释异常，再启动
Agent 任务 A-302：读取退款状态并分析问题
```

等待审批、依赖恢复或异步平台运行时，业务对象保存当前阶段。条件满足后，系统可以恢复原 Agent 任务，也可以创建新的 Agent 任务继续处理。选择取决于原执行是否仍有清楚的恢复位置，以及新的工作是否仍属于同一个 Agent 目标。

这也解释了“转人工”的位置。模型可以通过工具创建人工任务，某个业务工具背后也可以自动进入审批流；Runtime 还可以在预算耗尽或连续失败后创建交接记录。人工步骤属于业务流程，模型是否实时参与由具体阶段决定。

## 三种完成状态不能混为一谈

一次模型生成结束，只表示这一轮不再输出或调用工具。[Agent 任务完成]({{ site.baseurl }}/docs/ai-agent/agent-design/execution-control/)表示 Runtime 确认该执行目标已经满足。业务完成则由对应业务对象和业务规则确认。

例如，Agent 成功创建退款申请后，本次 Agent 任务可以完成，退款业务仍处于“待审批”；数据生成 Agent 成功启动平台运行后，业务任务仍要等待运行结果；模型生成一段“上传成功”的文字，也不能代替平台返回的 `dataset_id`。

```text
模型本轮结束
≠ Agent 任务完成
≠ 业务任务或业务对象完成
```

Runtime 可以检查 Agent 是否交付了约定结果，业务系统负责确认订单、退款、数据集或平台运行是否真正到达终态。Agent 的完成信号不能越过业务状态机。

## 恢复从业务事实继续

Agent 执行中断后，系统先读取关联的业务对象，再决定怎样继续。若 `dataset_id` 已经存在，不能因为会话中没有成功回复就再次上传；若退款申请仍在等待审批，后续 Agent 执行只能读取状态或补充材料；若外部动作结果未知，应使用幂等键和业务请求 ID 查询真实结果。

仅凭这些业务事实，系统可以启动一次新的 Agent 执行，重新组装 Context 和规划后续步骤。若要恢复原 Agent 任务，从原来的步骤继续，还要保存当前目标、已确认事实、等待条件和恢复位置。客户端 Agent 可以把这些状态保存在本地；后台运行、跨设备恢复或由其他 Worker 接管时，则要放进下一执行者能够访问的共享耐久存储。

两边至少需要通过已有的标识建立关联：

```text
业务任务或业务对象 ID
↔ 可选的 Agent 任务、会话或运行 ID
↔ Tool Call 与幂等键
↔ 外部资源 ID
```

并非每种实现都有独立的 Agent 任务 ID。关联记录只需让系统区分模型做过什么、业务系统确认了什么，以及后续还能安全执行什么。停止 Agent 只能阻止后续模型和工具调用，已经创建的业务对象不会自动撤销；需要撤销时，必须由业务系统提供明确的取消或补偿操作。

一个完整的业务接入可以这样概括：业务系统创建并保存用户真正关心的工作，按需要启动 Agent 执行；Runtime 让模型通过受控工具推进工作；重要工具结果同时更新执行状态和业务对象；最后分别判断 Agent 执行是否结束、业务结果是否完成。执行状态可以与业务任务合并保存，也可以由客户端或独立任务存储管理，具体位置取决于谁负责恢复。这样替换 Agent、重建会话或改用人工处理时，业务流程仍然能够继续。
