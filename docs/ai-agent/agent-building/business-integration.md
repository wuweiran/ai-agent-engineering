# 如何把 Agent 接入真实业务系统

团队用 Claude Code、项目规则、Skills 和 MCP 做出了数据生成 Agent。它能根据一句话生成文件并调用校验器，但还不是完整产品：它不知道当前用户能操作哪个项目，生成文件没有正式版本，上传后也没有地方保存平台运行 ID。

接入业务系统后，工程师应该能够回答三个问题：谁在操作什么资源，任务和外部动作怎样保持连续，用户最后怎样取得业务结果。

## 一句话之外还有业务身份

用户输入“生成 500 个企业 SAML 登录测试用户”，只表达了意图。应用还要从可信系统取得登录身份 `U-318`、目标项目 `login-load-test` 和运行环境 `eu-test`；是否自动运行则由用户在产品界面中明确选择。

模型可以识别任务类型和缺失信息，不能自行决定身份和资源范围。即使它正确生成了 `project_id`，服务端仍要检查 `U-318` 是否有权操作该项目。

退款 Agent 的入口不同：它从当前工单取得客户和订单范围。故障调查可能由告警自动创建，初始信息已经包含服务、区域和时间窗口。业务入口决定哪些内容由用户表达，哪些必须由系统绑定。

## 每次真实动作都要带着身份

数据生成 Agent 可能使用 `get_project_capabilities`、`get_data_template`、`validate_dataset`、`upload_dataset`、`create_platform_run` 和 `get_run_status` 等 MCP 工具。模型选择这些动作时，工具处理器要把请求与当前用户和任务结合起来。

例如，模型请求上传到项目 P。工具先取得当前 `task_id` 和用户，确认任务属于该用户、项目 P 在授权范围内，再由平台检查配额、环境和数据版本，最后返回 `dataset_id`。

项目规则和 Skill 可以提醒模型不要访问其他项目，却不能实施授权。故障调查的只读工具也要限制租户和日志范围；退款工具还要检查订单状态、金额和审批。

**模型选择业务动作，工具绑定调用身份，业务服务作出最终授权。**

## 业务任务要在会话之外存在

数据生成会经历生成、校验、用户确认、上传和平台运行，应用后端需要创建自己的任务对象：

```text
task_id: T-2048
├─ 用户、项目和环境
├─ Agent 运行 ID
├─ 输入文件与生成版本
├─ 校验报告和确认记录
├─ dataset_id 与 platform_run_id
└─ 当前状态、等待原因和最终结果
```

Agent 会话记录模型和工具如何交互，`T-2048` 则记录产品向用户承诺的任务。即使会话重建，应用仍能回答现在进行到哪里，已经上传什么，还能否继续。

文件也要有业务版本。草稿 v1 校验失败后，Agent 生成候选 v2；用户确认 v2，平台上传后得到数据集 `D-1042`，启动运行后得到 `R-8821`。运行失败时，Agent 能基于对应数据和日志继续分析。

短时客服建议可以附着在工单上，真实退款由退款申请和审批记录承载；故障调查则需要独立任务保存事实、假设和交接。业务对象应放在最能表达其生命周期的系统里。

## 工具反馈要同时更新模型和任务

底层平台返回的异常不一定能指导下一步。工具层要转换成稳定业务含义：

| 结果 | 后续处理 |
| --- | --- |
| `validation_failed` | 返回字段、位置和约束，让 Agent 修复文件 |
| `permission_denied` | 停止重试，提示用户检查项目权限 |
| `rate_limited` | 任务进入等待，并保存恢复时间 |
| `outcome_unknown` | 保存幂等键并查询结果，不能重复创建 |
| `run_failed` | 保存失败阶段和日志 ID，再判断能否修复 |

反馈既进入下一轮上下文，也更新业务任务。上传结果未知时，界面应显示“正在确认平台状态”，不能直接显示失败并鼓励用户重新提交。

资源 ID 是连接 Agent 与业务状态的关键。只有一段“上传成功”的文字，无法支持恢复、审计和后续运行。

## 界面围绕用户需要作出的决定

数据生成工具的重要交互点包括：信息不足时补充目标，校验失败时查看问题与修复进度，上传前确认文件摘要，平台运行中查看阶段，失败后选择修复或停止。

用户不需要看每次模型调用。他需要知道任务到了哪里，系统已经做了什么，现在为什么等待，以及自己要决定什么。

客服坐席关心退款资格和主管审批；值班工程师关心故障证据、当前假设和接管入口；数据工具用户关心文件预览、校验结果和运行产物。

取消任务也需要业务语义。停止 Agent 只能阻止后续执行，已经上传的数据或创建的退款申请不会自动撤销，除非业务系统提供补偿能力。

## 数据生成任务最终怎样跑通

用户选择项目和环境并描述任务后，应用先验证身份、创建 `task_id`。Agent 加载项目规则和相关 Skill，通过 MCP 查询项目能力与模板，在工作区生成数据。平台校验器返回具体问题，Agent 修复后，应用保存新的文件版本。

用户确认上传与运行时，服务端再次检查身份、配额和数据版本。MCP 返回 `dataset_id` 与 `run_id`，应用保存状态，Agent再根据运行结果结束或继续修复。

Claude Code 提供模型循环和文件环境，MCP 提供业务能力，应用后端管理用户与任务，平台服务守住权限和确定规则。任何单一组件都不能独自完成这条业务链。

## 换一个业务，落点也会变化

| 场景 | 权威状态 | 关键工具 | 主要交互 |
| --- | --- | --- | --- |
| 数据生成 | 任务、文件版本、数据集和平台运行 | 文件、校验、上传、运行 | 预览、确认、下载 |
| 客服退款 | 工单、订单、退款申请和审批 | 订单、资格、待审申请 | 建议、资格、主管审批 |
| 故障调查 | 调查事实、假设、证据和交接 | 监控、发布、日志、配置 | 证据进度、等待、人工接管 |

通用原则没有改变：身份不能由模型决定，权威状态不能只存在于对话，高风险动作不能绕过业务服务。但这些原则最终落在哪个数据库、API、文件目录和页面中，要由业务流程决定。

接入是否完整，可以用三句话检查：每次工具执行都绑定了真实身份和资源范围；任务、产物与外部动作在会话之外可追溯；用户能够补充信息、作出审批并取得业务结果。
