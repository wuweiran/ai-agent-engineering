# Agent 实现方式

团队确定任务需要 Agent 后，很容易开始比较框架功能：谁支持记忆、MCP、多 Agent 或工作流。真正影响长期开发的却是另一件事：团队需要控制哪部分，又愿意长期维护哪部分。

**选择实现方式，就是划分 Agent 平台与业务团队的责任。**

## 内部数据生成工具可以复用成熟产品

用户是公司工程师，任务围绕文件、模板、脚本和运行平台展开。他们已经接受命令行或开发工具式交互，Claude Code 提供的搜索、编辑和终端能力正好匹配任务环境。

一种实际结构是：Claude Code 负责模型循环和文件工作，`CLAUDE.md` 保存项目规则，Skills 描述不同数据类型的生成方法，MCP 连接模板、校验、上传和平台运行。平台服务继续实施权限、配额和真实执行。

团队不用重写 Agent 循环和文件工具，主要开发领域规则、Skills、MCP Server 与平台集成。与此同时，交互方式和运行环境会受到 Claude Code 能力边界影响。

## 客服退款更适合应用内 Agent

客服坐席在现有工作台中处理工单。Agent 读取订单和支付信息、生成建议，偶尔创建待审退款申请。任务短、用户身份明确、风险较高，文件和终端能力没有多少价值。

应用后端可以装配工单、客户和订单上下文，通过 Agent SDK 或受控模型循环调用少量业务工具。支付与审批服务继续掌握资格、金额和执行规则。

团队保留了现有用户界面和业务状态，又不必自己处理所有底层模型消息和工具循环。这是许多业务 Agent 的实际平衡点。

## 故障调查需要更多运行控制

生产故障调查由告警或工单触发，可能运行几十分钟，查询监控、发布、日志和配置系统，还要在值班人员之间交接。

团队可以用 Agent SDK 配合自有任务系统，或者使用托管 Agent 循环，让业务系统保存调查状态。若日志和凭据不能离开企业环境，工具执行还要放在自有基础设施中。

这类场景对异步执行、状态恢复和环境隔离的要求更强，因此团队承担的运行责任也更多。

## 自建 Runtime 的适用条件

直接基于模型 API 自己实现上下文装配、工具循环、状态、权限、恢复和 Trace，会得到最大的控制力，也带来最大的维护成本。

通常只有几个原因值得这样做：执行与审批顺序必须高度定制；需要和已有工作流引擎深度融合；数据和工具执行有特殊隔离要求；或者团队正在建设供多个业务复用的 Agent 平台。

平台少一个功能，并不自动意味着要自建 Runtime。高风险动作可以留在固定工作流中，Agent 只处理开放调查；产品外围也可以增加一层薄应用来补充状态和交互。

## 实现方式的选择依据

首先看用户在哪里工作。内部开发者已经使用终端和仓库，代码 Agent 很自然；客服坐席工作在工单页面，切换到命令行会破坏流程。

再看任务依赖什么环境。文件生成需要工作目录和代码执行，退款只需要几个业务 API，故障调查需要跨系统查询和长任务状态。

第三看哪些状态和控制必须由团队掌握。涉及资金、生产变更、租户数据和长时间恢复时，业务系统通常必须保存权威状态、权限和审批。

最后看控制力是否值得维护成本。自建能力只有持续解决真实约束，才值得长期承担模型接口、停止原因、上下文和工具协议的变化。

## 真实系统经常组合多种方式

数据生成工具可以让 Claude Code 处理文件和模型循环，自有应用负责用户、任务和产物，MCP Server 连接平台。客服系统可以用 SDK 管理 Agent，把退款执行留在已有工作流。故障调查也可以由托管 Agent 执行查询，企业任务系统保存审批和恢复点。

低风险能力可以从最小闭环开始：支持一个测试项目和一种模板，生成后校验，用户确认再上传。高风险边界则不能留到以后补。退款助手即使第一版只生成建议，也要从一开始正确绑定坐席和租户；真正创建申请前，资格检查和审批必须存在。

选择实现方式时，记住三点：成熟产品、SDK 和自建 Runtime 的区别在于复用多少通用能力；业务流程、运行环境和权威状态比功能清单更重要；最终架构往往由多种方式组合而成。
