# 一个数据生成 Agent 是怎样构建的

某企业的测试平台需要不同格式的数据。工程师过去会查模板文档、编写 JSON 或 CSV、运行校验脚本，再把文件上传到平台。平台运行失败后，他们还要读取报告，修改数据并重新提交。

团队希望把这段工作交给 Agent：用户描述测试场景，系统生成数据、完成校验，经用户确认后上传并运行。由于每次任务的模板、错误和修复方法都可能不同，后续步骤要根据实际反馈决定。

下面是一个模拟工程案例。它不代表某家公司的内部实现，重点是说明一个 Agent 应用怎样从可运行的原型逐步补齐业务闭环。

## 先缩小第一版的承诺

“支持所有测试数据”无法直接实现。不同产品的字段、规模和验证方法相差很大，平台写入也有真实成本。第一版只支持测试项目中的登录数据，包括基础登录和企业 SAML 两种任务。

团队把完成条件定义为：生成内容符合用户描述，通过对应类型的完整校验，上传到用户选择的项目，并取得一次成功的平台运行。缺少环境、用户数量或认证方式时，Agent 必须先询问；上传和运行前需要用户确认。

还有几条边界从第一版就由软件执行：不能读取真实客户数据，不能覆盖已有数据集，不能操作当前用户无权访问的项目。Agent 可以判断怎样生成数据，无权改变这些规则。

范围缩小后，团队可以用少量真实任务验证整条链路，而不必先建设一个通用数据平台。

## 为什么先复用 Claude Code

这项任务天然围绕文件展开，又需要反复运行脚本和修改内容。Claude Code 已经提供模型循环、文件工作区和终端工具，团队没有必要重新实现这些通用能力。

他们只补充与业务有关的部分：

| 组成 | 在这个项目中负责什么 |
| --- | --- |
| `CLAUDE.md` | 说明工作区规则、数据边界、必须校验和上传前确认等稳定要求 |
| Skills | 保存不同数据类型的生成方法，例如 SAML 字段关系和常见修复步骤 |
| MCP 工具 | 查询项目能力与模板，调用校验、上传和平台运行服务 |
| 应用后端 | 绑定用户和项目，保存任务、文件版本、确认记录及平台资源 ID |
| 业务平台 | 实施权限、配额、幂等和真实运行规则 |

这些内容不能全部写进一段 Prompt。稳定项目规则和具体生成方法的变化频率不同；模板与项目能力是动态事实，应该在任务中查询；用户身份与权限则必须来自可信的业务系统。

**复用成熟 Agent 产品，省下的是通用执行环境；任务状态、权限和业务结果仍然要由应用补齐。**

## 一次任务怎样跑通

用户 `U-318` 在应用中选择项目 `login-load-test` 和环境 `eu-test`，输入：“生成 500 个企业 SAML 登录测试用户，上传后运行。”

应用先检查用户能否操作这个项目，随后创建任务 `T-2048`。Agent 取得任务目标，加载登录数据 Skill，通过 MCP 查询项目支持的模板和环境约束，然后在独立工作区生成候选文件 v1。

完整校验发现部分记录缺少证书类型。工具返回具体字段、记录位置和约束，Agent 据此生成 v2，而不是重新猜测整份数据。v2 通过后，应用保存校验报告、文件摘要和内容哈希，并把任务置为 `waiting_confirmation`。

用户在界面中看到记录数量、认证类型、目标项目和校验结果，确认上传 v2。服务端再次核对用户、项目、文件哈希和任务状态，再调用上传工具。平台返回数据集 `D-1042`，启动运行后又返回 `R-8821`；这两个 ID 都写入任务记录。

假如平台运行报告指出用户标识不符合该环境的区域规则，任务不会只留下“运行失败”。Agent 可以读取 `R-8821` 对应的结构化错误，生成新的候选版本并再次校验。因为文件内容已经变化，应用会重新请求用户确认，再创建新的数据集和运行。

这段过程里，模型负责理解错误并选择修复方法，程序负责身份、状态、确认与写入规则，平台负责决定数据能否真正运行。端到端结果来自几部分协作，不来自某一轮模型回答。

## 会话关闭后，任务仍然存在

原型阶段，团队曾把进度都留在 Claude Code 会话和工作目录中。用户关闭窗口后，应用不知道文件是否已经确认；服务重启后，也无法判断上传请求是否成功。

因此，产品中的权威记录变成了 `T-2048`，而不是某段对话。任务保存当前阶段、Agent 运行 ID、候选文件版本、已确认的内容哈希、校验报告、上传幂等键、`dataset_id` 和 `run_id`。

会话可以重建，工作进程也可以更换。恢复任务时，系统根据这些记录决定下一步：未确认的文件继续等待用户；已有 `dataset_id` 的任务不能再次上传；结果未知的上传先用幂等键向平台查询。

**Agent 会话解释执行过程，业务任务承载系统对用户的承诺。**

## 原型中的问题怎样改变设计

最早的版本把所有生成规则都放在 `CLAUDE.md` 中。规则增加后，基础登录任务也会加载大量 SAML 细节，修改一种数据方法还可能影响其他任务。团队随后把稳定的项目边界留在 `CLAUDE.md`，把按任务加载的生成方法拆进 Skills。

上传最初由一个通用脚本完成。脚本持有范围过大的凭据，只返回一段文本，超时后也无法确认平台是否已经创建数据集。团队把它替换为 MCP 工具：调用绑定当前任务身份，输入使用结构化 Schema，结果明确区分成功、权限不足、限流和结果未知，并返回可追踪的资源 ID。

另一次失败发生在企业登录任务。Agent 正确识别了 SAML 需求，旧 Skill 却指定了基础登录模板。Trace 找到第一次偏离后，团队修正 Skill，并把模板映射中已经稳定的部分交给程序校验，而不是只在 Prompt 中增加一句提醒。

这几个改动没有追求一开始设计出完整架构。每次真实失败都回答了同一个问题：缺失的能力应该落在方法、工具、任务应用，还是业务平台中？

## 上线前怎样建立证据

团队从支持范围中挑选少量有差异的任务：基础登录、企业 SAML、信息不足、项目无权限、校验失败，以及上传响应丢失。每个样本都检查最终数据、目标项目和平台运行，也检查是否跳过确认、重复上传或在结果未知时直接重试。

评测失败时，Trace 保留当时使用的规则、Skill、工具 Schema 和发布版本，以及关键工具参数、返回结果和状态变化。修复原问题后，还要重跑相邻任务，避免为了 SAML 修改的规则破坏基础登录。

第一版先开放给内部测试项目。新版本也先进入这些项目，确认用户完成率、校验失败、重复写入和平台运行结果没有异常，再逐步扩大范围。若工作进程在上传后重启，系统还要通过任务记录和幂等键恢复，而不能让用户重新提交整项任务。

## 最终交付的是一条业务链

这个案例没有重新实现 Claude Code，也没有把所有控制交给它。Claude Code 处理适合模型参与的文件工作和动态步骤，Skills 提供领域方法，MCP 连接受控业务能力，应用后端管理身份、任务与用户确认，平台服务守住真实资源。

从原型到产品，真正推动系统成熟的是把每个失败放回合适的系统层：方法会版本化，工具有清楚契约，任务可以恢复，写操作能够追踪，质量可以重复验证。

**一个 Agent 应用真正完成时，用户取得了可验证的业务结果，系统也能说清这个结果怎样产生、失败后怎样继续。**
