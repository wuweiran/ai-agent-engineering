# 如何实现一个 Agent 应用

系统设计确定了模型负责什么、需要哪些信息和工具。到了实现阶段，工程师仍然不能照着一张“标准 Agent 架构图”施工。

客服退款、故障调查和数据生成都可以使用 Agent。退款助手嵌在客服工作台里，几秒内给出建议；故障调查可能持续几十分钟，并在值班人员之间交接；数据生成依赖文件工作区、校验器和异步运行平台。它们需要的应用结构自然不同。

**实现 Agent 应用，要从用户原本怎样完成任务出发，再决定复用哪些 Agent 能力、补上哪些业务系统。**

## 数据生成工具需要什么

用户希望描述一种测试场景，让 Agent 生成数据、校验、上传并在平台运行。

Claude Code 已经提供模型循环、文件编辑和终端环境。团队可以用 `CLAUDE.md` 说明项目规则，用 Skills 保存数据生成方法，再用 MCP 接入模板、校验、上传和运行平台。

这些能力仍不足以构成产品。应用还要知道当前用户能操作哪个项目，保存每版生成文件和校验报告，在上传前请求确认，并记录平台返回的 `dataset_id` 和 `run_id`。

在这个场景中，Agent 配置和 MCP 很重要，文件产物、异步任务和用户确认同样重要。

## 换成退款助手，结构就不同了

客服坐席已经在工单系统中工作。Agent 读取当前工单、订单和支付记录，给出处理建议，必要时创建待审退款申请。

这里通常不需要通用文件系统和终端。更重要的是把 Agent 嵌进原有工作台，由应用后端绑定坐席与租户，只提供订单查询、资格检查和待审申请等少量工具。真实退款继续由支付和审批服务控制。

照搬数据生成工具的工作区和 Skills 结构，不会让退款助手更完整，只会增加不需要的组件。

## 故障调查又是另一种任务

故障调查由工单或告警触发，需要查询监控、发布、日志和配置系统。任务持续时间长，可能等待依赖恢复，还要把证据和未解决问题交给下一位值班工程师。

这类应用更需要独立的调查任务、持久状态和跨系统只读工具。若日志与凭据不能离开企业环境，工具执行还要放进受控基础设施中。

三个场景说明，同一个设计概念会落在不同地方：退款状态主要存在订单和审批服务中；故障调查状态存在调查任务中；数据生成还要管理文件版本和平台运行。

## 实现时真正要交付什么

一个可用应用至少要把四部分接起来：

| 部分 | 解决的问题 |
| --- | --- |
| Agent 配置 | 模型实际看到哪些规则、方法和工具 |
| 业务工具 | 模型怎样读取真实数据、请求真实动作 |
| 任务应用 | 状态、产物和外部资源怎样保持连续 |
| 用户交互 | 用户怎样提交、追问、审批、取消和取得结果 |

四部分不要求各自成为独立服务。短时客服建议可以直接附着在工单上；长任务则可能需要任务数据库和异步工作进程。工程师应根据任务时长、风险和恢复要求选择结构。

## 成熟产品、SDK 和自建 Runtime

团队还要决定哪些通用能力由外部平台提供。成熟 Agent 产品适合任务环境与其能力接近的场景；Agent SDK 适合需要自定义产品入口和业务状态的应用；只有执行控制、隔离或平台复用要求足够强时，才值得自建 Runtime。

真实系统往往是组合。数据工具可以让 Claude Code 负责文件工作，让自有后端管理用户和任务，让 MCP 连接平台；退款助手可以用 SDK 管理 Agent 循环，把高风险退款留在现有工作流。

实现是否完整，可以看三个结果：模型是否真的获得完成职责所需的规则和能力，权威状态与副作用是否由合适的软件管理，用户是否能够走完整个业务流程并取得可验证结果。
