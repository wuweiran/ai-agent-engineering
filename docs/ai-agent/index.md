# AI Agent 工程

大模型可以理解自然语言、日志和文档，普通程序则擅长保存状态、执行规则和控制权限。AI Agent 工程要做的，是让两者共同完成一个真实任务。

假设用户提交了一张工单：“昨晚升级后，欧洲办公室都登录不了。”系统无法预先知道应该先查监控、发布记录还是客户配置。模型可以根据已有信息选择调查方向，程序负责查询服务、保存结果并限制权限。新结果回来后，模型再决定下一步。

这就是 Agent 最基本的工作方式：

```text
目标和当前信息
      ↓
模型选择下一步
      ↓
软件执行并返回结果
      └── 结果变化后再次判断
```

模型参与决定路径，是 Agent 与固定工作流最重要的区别。软件仍然守住执行边界：它检查工具参数和权限，记录任务状态，并在达到完成条件、时间上限或审批节点时停止循环。

## 工程师需要完成哪些工作

从业务想法到长期运行的产品，可以归纳为四个阶段。

| 阶段 | 要解决的问题 |
| --- | --- |
| 明确业务任务 | 谁遇到了什么问题，什么结果算成功，是否真的需要 Agent？ |
| 构建 Agent 应用 | 模型负责哪些判断，需要什么信息和工具，怎样接入现有业务？ |
| 验证与改进质量 | 哪些真实任务做对了，失败从哪一步开始，修改后是否有回归？ |
| 保障生产运行 | 流量、故障和版本变化出现时，任务能否恢复，影响能否控制？ |

这四个阶段是本知识库理解 AI Agent 工程的主线。具体文章会继续拆解其中的问题，但主线不会随着文章数量变化。

在掌握系统设计的基本问题后，[从 Claude Code 深入 Agent 系统](claude-code/)会用一个成熟产品解释 Context 组装、工具循环、会话压缩和子 Agent 等 Runtime 机制。它是一组实现案例，不增加新的工程阶段。Token、提示词、RAG、幻觉和训练对齐等模型侧基础放在[大模型基础](../llm/)中，避免把所有与模型有关的内容都误当成 Agent Runtime。

## 始终要问的一件事

Agent 的灵活性来自模型，可靠性来自模型与软件之间清楚的分工。

含糊描述的理解、未知问题的调查方向，可以让模型参与；租户权限、退款金额、幂等写入和资源上限，仍应由程序控制。生产回滚、大额退款等高风险决定还需要人承担责任。

因此，衡量一个 Agent 系统的标准，不是它能自主运行多少步，而是它能否在可接受的成本和风险下，稳定完成用户真正需要的任务。
