# 一个 Agent 任务如何运行

普通程序按照开发者写好的步骤执行，Agent 则需要在运行中反复观察结果、选择下一步。外表上，它可能只是一个聊天窗口；在系统内部，一次请求往往经历目标解析、上下文组装、模型决策、工具执行、状态更新和终止判断。

```text
用户目标
  ↓
加载上下文与任务状态
  ↓
模型决定下一步
  ↓
程序校验并执行工具
  ↓
记录结果，更新状态
  ↓
继续、完成、失败或等待人工处理
```

这条链路中，**模型提出决策，运行时负责把决策安全地变成动作。**

## 目标让任务可以结束

用户的原始请求通常还不是一个可执行的目标。“帮我处理这个线上故障”没有说明调查范围、期望产物和什么情况下可以停止。系统需要把它整理成更明确的任务，例如：

- 找出最近一次发布后登录错误增加的原因；
- 给出证据、影响范围和修复建议；
- 不修改生产环境；
- 无法确认原因时列出下一步人工调查项。

**目标不仅说明要做什么，也定义什么叫完成。**没有完成条件，Agent 很容易继续搜索无关信息，或者拿到一个似是而非的结果就提前结束。

代码 Agent 是一个直观的行业场景。用户交给它一条 GitHub issue 时，真正可执行的目标通常包括修改代码、保持接口兼容、通过现有测试并说明改动。测试结果和代码差异为任务提供了相对明确的完成信号，这也是代码任务较适合 Agent 的原因之一。

## 上下文是模型本轮看到的信息

模型不会自动知道公司系统、代码库和任务历史。每次决策前，运行时需要选择并组织模型当前所需的信息，这就是**上下文**。它可能包含：

- 系统指令和安全规则；
- 用户目标；
- 当前任务状态；
- 工具定义；
- 最近的工具结果；
- 检索到的文档或代码。

上下文不是把所有资料一次塞给模型。无关信息会占用上下文窗口，还可能干扰判断。代码 Agent 通常先查看仓库结构和相关文件，而不是把整个仓库放进一次请求；企业知识助手也常根据问题检索少量相关文档，而不是读取全部知识库。

**上下文回答“模型现在知道什么”，状态回答“任务现在进行到哪里”。**两者经常一起传给模型，但职责不同。

## 状态让多步任务保持连续

一次模型调用结束后，模型本身不会替业务系统可靠保存任务进度。运行时需要记录已经发生的事情，例如：

```text
任务状态
├─ 已确认：问题只影响欧洲区域
├─ 已检查：身份服务状态正常
├─ 已发现：昨晚有配置发布
├─ 待检查：配置差异和回滚记录
├─ 已调用工具：3 次
└─ 剩余时间：6 分钟
```

状态可以存在数据库、工作流引擎或会话存储中。它既包括业务进度，也包括运行控制信息，例如步骤数、重试次数、审批状态和截止时间。

长时间运行的研究、编码和数据分析 Agent 都依赖状态。用户关闭页面后，任务仍可能继续；服务重启后，也应该能够知道哪些步骤已经完成，而不是从头重复昂贵操作。

## 模型决策不等于动作已经发生

模型读取目标、上下文和状态后，可能选择搜索文档、查询服务状态、读取文件，或者向用户追问。它输出的是一个**工具请求**，而不是直接操作外部系统。

运行时收到工具请求后，还需要：

1. 校验工具名称和参数；
2. 检查调用者权限；
3. 判断是否需要人工批准；
4. 执行真正的 API 或程序；
5. 捕获超时、限流和业务错误；
6. 把结果以稳定格式返回给模型。

这种分工形成了重要边界：**模型决定“想做什么”，软件决定“能不能做、怎样做”。**

例如，客服 Agent 可以建议退款，但退款服务仍应验证订单、金额和操作权限。高金额退款还可以要求人工确认。即使模型受到错误提示或误解了用户意图，它也不能绕过业务系统的授权规则。

## 工具把判断连接到现实世界

工具可以是搜索、数据库查询、企业 API、文件读写、代码执行或浏览器操作。好的工具并不只是把底层接口原样暴露给模型，而是提供清楚、边界明确的业务能力。

例如，与其提供一个可以任意执行 SQL 的工具，不如分别提供“查询客户资料”和“查询订单状态”；与其让模型拼接任意 HTTP 请求，不如提供“创建工单评论”并在服务端校验身份和内容。

GitHub Copilot 等代码助手能够读取代码、搜索仓库、运行终端命令和修改文件。真正完成编码任务的能力并不只来自模型，而来自模型与这些工具组成的执行环境。

## 反馈驱动下一轮决策

工具结果返回后，Agent 要判断结果意味着什么。搜索为空可能需要改写查询，测试失败可能要求检查错误日志，权限不足则可能需要请求用户授权。

```text
模型：运行测试
工具：3 个测试失败
模型：读取失败日志
工具：发现返回字段改名
模型：修改兼容逻辑
工具：全部测试通过
模型：总结改动并结束
```

**反馈不是日志的同义词，而是能够改变下一步决策的环境信息。**如果工具只返回“失败”，模型很难恢复；如果它返回错误类型、相关字段和可重试性，模型就能选择更合理的下一步。

这也是工具设计的重要原则：结果要让模型理解发生了什么，但不要把大量无关原始数据全部塞回上下文。

## 终止条件防止循环失控

Agent 不能只依赖模型说“我完成了”。运行时需要检查任务是否应该终止。常见结果包括：

- **完成**：目标和验收条件已经满足；
- **失败**：关键工具不可用，或任务确定无法完成；
- **等待人工处理**：需要审批、补充信息或更高权限；
- **受限终止**：达到最大步骤数、时间或成本预算。

代码 Agent 可以用测试是否通过、目标文件是否修改作为部分完成证据；客服任务可以检查必填字段和审批记录；研究任务则可能需要引用覆盖、来源质量和结论完整性评测。

**模型负责提出完成结论，系统负责验证并执行终止。**即使无法自动判断全部质量，也应该有步骤、时间和费用上限，避免模型在相似工具调用之间反复循环。

## Trace 连接运行、评测和排障

多步任务出错时，只保存最终回答远远不够。Trace 会记录每轮模型输入、决策、工具调用、结果、耗时和状态变化，使工程师能够还原任务经过。

同一个错误答案可能来自完全不同的原因：上下文遗漏了关键文档，模型选择了错误工具，工具返回旧数据，或者终止条件过早触发。只有完整轨迹才能区分这些问题。

Trace 也为评测提供基础。团队可以回放失败任务，建立回归用例，并比较新模型、提示词或工具版本是否真正改善了结果。

## 把整条链路连起来

一次 Agent 任务并不是模型连续“思考”这么简单。它由两个相互配合的部分组成：

- **决策循环**：目标、上下文、模型决策、工具反馈；
- **运行时控制**：状态、权限、执行、终止和 Trace。

模型擅长处理无法完全预先写死的判断，运行时则把每一步限制在可执行、可恢复、可观察的工程边界内。Agent 的可靠性取决于整条链路，而不是某一次模型回答是否看起来聪明。
