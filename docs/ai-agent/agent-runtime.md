# 一个 Agent 任务如何运行

用户对代码 Agent 说：“修复登录错误并运行测试。”界面里只有一句话，系统内部却要经历多轮读取、判断和执行。

Agent 先了解仓库规则和相关代码，模型选择下一步，运行时检查并执行工具。测试失败后，结果重新进入上下文，模型再决定读取日志还是修改代码。循环持续到测试通过、任务失败或需要人工处理。

**模型负责提出下一步，运行时负责让这一步安全执行，并把结果带回下一轮。**

## Agent 系统和一次任务

“Agent”有时指一套可以反复工作的系统，有时指这套系统围绕一个目标进行的一次运行。

Claude Code 是代码 Agent 产品。它包含 Claude 模型、组织执行的运行时，以及搜索文件、编辑代码、运行终端等工具。用户每次交付一个目标，就启动一次具体任务。

```text
Claude Code：可反复接收目标的 Agent 系统
├─ Claude 模型：理解信息并选择下一步
├─ 运行时：管理上下文、状态、权限和结束条件
└─ 工具环境：搜索、编辑、执行命令和测试

“修复登录错误并运行测试”：其中一次任务
```

一次任务可能经过多轮模型判断和工具反馈，最终完成、失败、达到限制，或停下来等待用户。

## 任务先要有结束标准

“帮我处理线上故障”没有说明调查范围、交付物和停止条件。系统要把它整理成可执行目标，例如：找出最近发布后登录错误增加的原因，说明影响范围和证据，不修改生产环境；无法确认时，列出待调查项并转人工。

**目标既说明要做什么，也说明什么时候可以停。**否则 Agent 可能不断搜索，也可能拿到一个似是而非的线索就提前结束。

代码任务比较适合观察这个边界。修改目标文件、保持接口兼容和通过测试，都是能够检查的完成信号。

## 每轮只给模型当前需要的信息

模型不会自动知道代码库、业务系统和之前发生了什么。每次判断前，运行时要组织本轮上下文，其中可能包括任务目标、系统规则、当前状态、工具说明和最近结果。

代码 Agent 通常先查看仓库结构和相关文件，不会把整个仓库一次放进模型。企业支持 Agent 也只查询当前客户、区域和故障时间相关的数据。

上下文回答“模型现在知道什么”。任务状态回答“任务已经进行到哪里”。例如，状态可以记录：问题只影响欧洲区域，身份服务已经检查，昨晚存在配置发布，下一步需要比较配置差异。

状态保存在数据库、工作流引擎或会话存储中。服务重启后，系统能够据此继续，不必依靠模型回忆整段对话。

## 模型的决定还不是现实动作

模型可能输出“查询服务状态”或“运行测试”。这是一项工具请求。运行时还要检查工具名称、参数、当前用户权限和是否需要审批，然后才调用真正的服务或程序。

客服 Agent 即使建议退款，支付服务仍要检查订单、金额和操作者权限。高金额退款还可以要求人工批准。

**模型决定想做什么，软件决定能不能做，并确认执行结果。**这条边界让模型能够处理开放判断，又不能绕过业务规则。

## 工具结果推动下一轮

假设代码 Agent 运行测试，三个用例失败。只返回“失败”，模型很难判断下一步；若返回失败用例、关键错误和日志位置，它就可以继续定位。

```text
模型请求运行测试
  ↓
工具返回三个失败用例
  ↓
模型读取相关日志
  ↓
工具返回字段改名的位置
  ↓
模型修改兼容逻辑并重新测试
```

工具结果能够改变下一步时，它才形成有效反馈。搜索为空可能让模型换查询词，权限不足会让任务请求授权，测试通过则可能满足完成条件。

## 谁来决定结束

模型可以判断“已经找到原因”，运行时还要检查必要证据是否存在、外部动作是否完成，以及步骤、时间和费用是否达到上限。

任务通常会落到几种结果：完成；等待补充信息或审批；确定无法继续；达到运行限制后停止。开放质量无法全部自动判断时，还可以保留人工审查。

**模型建议结束，系统确认任务是否具备结束资格。**

## 出错后怎样还原过程

只保存最终回答，无法解释任务为什么失败。相同的错误结论可能来自不同原因：关键文档没有进入上下文，模型选错工具，工具返回旧数据，或者结束条件过早满足。

Trace 记录每轮输入来源、模型决定、工具调用、结果和状态变化。工程师可以据此回放失败任务，找到第一次偏离，并把它保留成回归评测样本。

一个 Agent 任务最终由两部分配合完成：模型根据上下文和反馈作出开放判断；运行时管理状态、权限、执行和结束。系统的可靠性取决于整条链路，而不取决于某一次回答看起来有多聪明。
