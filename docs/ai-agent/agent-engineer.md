# AI Agent 工程师在做什么

AI Agent 工程师的工作常被简化成“接入大模型”和“调整 Prompt”。在真实项目中，模型调用只是系统的一部分。工程师要把一个模糊的业务愿望，转化成可以执行、评测并长期可靠运行的软件能力。

这项工作可以沿着一条连续链路理解：

```text
定义任务
  ↓
选择系统形态
  ↓
建设执行能力（设计 Agent 系统）
  ↓
建立质量体系
  ↓
保障生产运行
```

这里的每一步都是一组具体工作：定义任务要明确用户、输入、动作和完成条件；选择系统形态要划分程序与模型的控制权；建设执行能力要设计上下文、状态、工具、权限和运行控制；质量体系负责 Trace、评测和安全验证；生产运行保障则处理版本、故障、规模和成本。

## 定义任务：把愿望变成可验证目标

业务方可能提出“做一个能帮助客服的 Agent”或“让 Copilot 自动处理内部请求”。这些描述没有界定用户、输入、允许的动作和成功标准，无法直接用于设计系统。

工程师需要继续拆解：

- 谁在什么场景使用它？
- 当前流程最耗时的环节是什么？
- 系统只给建议，还是可以执行操作？
- 什么结果算成功？
- 错误发生时谁承担后果？

例如，Klarna 等公司公开讨论过 AI 客服助手处理客户咨询的场景。对工程团队而言，真正的问题不会停留在“模型能否聊天”，而会落到咨询解决率、转人工条件、可支持的业务类型，以及退款或账户操作需要什么权限。

**Agent 项目的第一步不是选模型，而是定义任务边界和可验证的结果。**

## 选择系统形态：决定模型拥有多少控制权

任务明确后，工程师要判断使用一次模型调用、固定工作流还是 Agent。

工单摘要可能只需要一次调用；已知步骤的报销审核更适合固定工作流；跨代码、日志和监控系统调查故障，才可能需要模型动态决定查询顺序。

这一步决定了后续工程复杂度。模型控制权越高，系统越需要状态管理、工具治理、执行轨迹和终止保护。工程师的职责不是尽量把产品做成 Agent，而是**把必要的决策权交给模型，把高风险和确定性逻辑留在软件中。**

## 建设执行能力：设计 Agent 系统

“建设执行能力”是对 Agent 系统设计的概括。它不是简单接入模型和几个 API，而是把业务任务落实成一套由模型参与决策、由软件负责执行和约束的运行机制。工程师需要具体设计：

- **决策边界**：哪些步骤允许模型动态选择，哪些规则仍由程序控制；
- **上下文**：每轮判断需要哪些规则、事实、历史和工具说明；
- **状态与记忆**：怎样保存任务进度，以及哪些经验可以跨任务复用；
- **工具与执行环境**：Agent 可以查询或修改什么，参数、结果和错误怎样表达；
- **权限与人工协作**：哪些调用可以自动执行，哪些需要审批或禁止；
- **运行控制**：失败后怎样重试、恢复或回滚，任务何时必须终止。

这些部分共同构成 Agent 的决策与执行闭环。上下文决定模型依据什么作出判断，工具决定它能够观察和改变什么，状态让多步任务保持连续，运行时则负责权限检查、真实执行和终止控制。

代码 Agent 是一个公开可见的例子。GitHub Copilot、Claude Code 等产品不仅调用模型，还要提供代码搜索、文件编辑、终端和测试环境。模型决定查看或修改什么，宿主程序负责真正操作文件、执行命令并展示差异。

工程师在这里做的大量工作与后端和分布式系统相似：定义 API 契约、保证幂等、处理超时和限流、隔离租户权限，并让异步任务可以恢复。

## 建立质量体系：从“看起来不错”走向可测量

Agent 输出不稳定，单看几个演示很难判断系统是否改善。工程师需要和产品经理、领域专家共同建立评测体系。

评测通常从真实任务样本开始，并区分不同失败原因：

- 是否理解了目标；
- 是否选择正确工具；
- 工具参数是否正确；
- 是否使用了关键证据；
- 是否完成任务；
- 是否违反权限和安全规则；
- 延迟与成本是否可接受。

代码 Agent 可以通过测试结果、静态检查和代码评审衡量一部分质量；搜索问答可以检查引用是否支持答案；客服助手则需要观察解决率、转人工率和不当操作。

**评测的作用不仅是给最终答案打分，还要定位执行链路在哪一步失效。**因此，评测通常与 Trace、日志和任务回放一起建设。

## 保障生产运行：让真实任务长期可靠执行

产品上线后，工作才进入长期阶段。工程师需要与应用、平台和 SRE 团队共同处理：

- 模型或提示词升级带来的回归；
- 工具接口和外部服务变化；
- 峰值流量、限流与排队；
- Token 和工具调用成本；
- 长任务超时与卡死；
- 权限事故和异常写操作；
- 用户反馈与失败样本回收。

传统服务通常可以通过状态码和指标快速定位问题，Agent 的失败则可能跨越多轮决策。工程师需要从 Trace 中判断，是上下文选择错误、模型规划偏离、工具故障，还是终止条件不合理。

生产运行中的问题也会反过来改变架构。常见路径被识别后，团队可能把原本由 Agent 自由决定的步骤固化成工作流；只有少数开放任务继续保留动态决策。**Agent 系统不是设计完成后就固定不变，而是在真实失败中重新划分模型与程序的边界。**

## 与其他角色如何协作

AI Agent 工程与传统软件角色有大量重叠。角色名称并不能说明每天在做什么，更实际的区分方式是看他在修改哪一层系统、交付什么结果。

| 角色 | 日常工作 | 常见产出 |
| --- | --- | --- |
| 模型研究人员 | 分析模型能力缺口，准备训练与评测数据，进行训练、微调和推理实验，比较不同模型版本 | 模型版本、训练数据、能力评测报告和推理方法 |
| AI Agent 工程师 | 把业务任务拆成可执行目标，设计上下文、工具和运行控制，查看 Trace，构造评测集并根据失败样本调整 Agent | Agent 设计、工具契约、上下文策略、评测集和任务质量指标 |
| 应用与后端工程师 | 开发用户界面、业务 API、数据库和异步任务，把订单、账户等现有服务接入 Agent，并保证确定性业务规则正确执行 | 产品功能、业务服务、数据模型、任务队列和可靠性指标 |
| 平台工程师 | 建设供多个团队复用的模型网关、Agent Runtime、工具注册、权限、Trace 和评测基础设施，处理容量与成本 | 平台 API、SDK、运行环境、观测平台和服务等级目标 |
| 产品经理和领域专家 | 访谈用户，梳理现有流程和例外情况，定义可处理任务、转人工条件与成功标准，标注真实案例 | 产品需求、任务分类、业务规则、验收样本和质量标准 |
| 安全与合规人员 | 进行威胁建模和数据分类，审查工具权限、敏感数据与审计记录，测试越权和提示注入风险，处理安全事件 | 权限策略、风险评审、审计要求和事件处置方案 |

以客服 Agent 错误创建退款申请为例，领域专家先确认“预授权未完成”不应进入退款流程；AI Agent 工程师回放 Trace，发现模型误解了支付查询结果；后端工程师在支付工具中补充稳定的交易状态，并在退款服务中增加确定性校验；平台工程师确保 Trace 能记录工具版本和权限决策；安全人员检查相同缺陷是否可能导致越权或高金额误操作。只有当大量任务都暴露出模型本身无法区分支付状态时，模型研究人员才可能通过训练或模型评测介入。

这次修复同时改变了 Agent 行为、工具接口和业务服务，却由不同角色承担。AI Agent 工程师主要对特定任务中的决策与执行质量负责；应用和后端工程师保证产品及业务系统可靠；平台工程师提供跨团队复用的基础设施；产品和领域专家定义什么结果在业务上正确。

实际团队不会总按职位名称整齐分工。小团队里，一位工程师可能同时负责前端、后端、上下文和评测；大型公司则可能有专门的模型平台、Agent Runtime、应用和安全团队。AI Agent 工程师更像一种问题领域，而不是边界完全固定的职位。

## 核心能力

把前面的工作合在一起，AI Agent 工程师需要连接三类能力：

- **理解模型**：知道模型擅长什么、如何提供上下文，以及常见失败方式；
- **建设软件系统**：能够设计 API、状态、异步任务、权限、可观测性和故障恢复；
- **理解业务与评测**：能够定义成功标准，用真实任务判断系统是否创造价值。

模型能力决定系统的可能性，软件工程决定它是否可靠，业务理解决定它是否值得构建。AI Agent 工程师的核心工作，就是在三者之间持续做出可验证的工程取舍。
