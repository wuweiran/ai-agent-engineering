# AI Agent 工程师在做什么

业务方提出：“做一个能帮助客服处理复杂工单的 Agent。”AI Agent 工程师不会从选择模型和编写 Prompt 开始。他首先要弄清楚，客服究竟在哪个环节遇到困难，系统可以执行到什么程度，出了错由谁处理。

这项工作可以分成四个稳定阶段：明确业务任务，构建 Agent 应用，验证与改进质量，保障生产运行。它们不是一次走完的瀑布流程；生产中的失败会不断推动团队重新调整任务和系统边界。

## 明确真正要解决的任务

“帮助客服”还不能直接开发。工程师需要和产品经理、客服专家一起确认：谁在什么场景使用，当前最耗时的环节是什么，系统只给建议还是可以执行操作，什么结果算成功，错误会造成什么后果。

例如，同样是客服场景，工单摘要可能只需要一次模型调用；标准退货审核适合固定工作流；跨账户、日志和服务状态调查未知故障，才需要 Agent 根据结果改变查询方向。

**Agent 项目的起点是可验证的业务任务，而不是模型。**系统形态只是任务明确后作出的选择。

## 把任务变成可以运行的应用

确定使用 Agent 后，工程师先划分模型和软件的责任。模型负责哪些开放判断？每轮需要看到什么？任务进度保存在哪里？可以使用哪些工具？什么情况下继续、等待或结束？

这些答案随后要落实到真实产品中。实现方式随业务变化：

- 数据生成工具可以复用 Claude Code 的文件和终端环境，通过 `CLAUDE.md`、Skills 与 MCP 接入平台；
- 客服退款 Agent 更适合嵌入现有工作台，由应用后端装配工单上下文，支付与审批服务执行规则；
- 长时间故障调查需要独立任务、持久状态、跨系统查询和人工接管入口。

工程师的实际产出可能是 Agent 配置、Skill、MCP Server、任务 API、状态表、审批页面和产物管理。大量工作与后端和分布式系统相似：定义接口、处理超时、保证幂等、隔离租户权限，并让任务在服务重启后恢复。

**构建 Agent 应用的重点，是让模型能力、任务生命周期和现有业务系统形成闭环。**

## 用真实任务验证质量

Agent 在几个演示中表现不错，不代表它能稳定服务用户。工程师要和领域专家把真实任务整理成评测样本，定义端到端成功和不可违反的边界。

数据生成 Agent 不仅要生成格式正确的文件，还要选对业务模板、上传到正确项目并成功运行。最终结果错误时，Trace 帮助团队判断第一次偏离发生在任务理解、上下文、Skill、工具还是平台执行。

找到根因后，工程师修改相应层次，重跑失败样本和完整回归集。质量工作因此不是给最终答案打一个分数，而是建立“发现—定位—修改—验证”的循环。

## 让上线后的系统经得起变化

产品上线后，模型、规则、Skill、工具和业务接口都会变化；真实流量还会带来排队、限流、实例重启和外部依赖故障。

工程师需要保证已接收任务不会丢失，结果未知的写操作不会重复，平台故障不会让所有任务一起重试。发布新版本时，还要记录真正生效的版本组合，逐步放量，并准备处理长任务和已有副作用的回滚方案。

生产中的常见失败会反过来改变系统。某条模型判断逐渐稳定后，可以固化为程序规则；固定分支持续膨胀仍覆盖不了新情况时，则可能把局部判断交给模型。

**Agent 系统会在真实失败中持续重新划分模型、程序和人的责任。**

## 不同角色怎样共同完成工作

AI Agent 工程与传统软件角色有大量重叠。角色名称不如实际修改的系统层和交付物有解释力。

| 角色 | 日常工作 | 常见产出 |
| --- | --- | --- |
| 模型研究人员 | 分析模型能力缺口，准备训练与评测数据，进行训练和推理实验 | 模型版本、训练数据和能力评测 |
| AI Agent 工程师 | 定义 Agent 任务，设计和实现执行链路，通过评测与 Trace 改进系统 | Agent 配置、工具契约、评测集和任务质量指标 |
| 应用与后端工程师 | 开发界面、业务 API、数据库和异步任务，实施确定性业务规则 | 产品功能、业务服务和任务系统 |
| 平台工程师 | 建设模型网关、Runtime、工具注册、Trace 和评测基础设施 | 平台 API、SDK、运行环境和观测平台 |
| 产品经理和领域专家 | 梳理用户流程、例外情况、转人工条件和成功标准 | 产品需求、业务规则和验收样本 |
| 安全与合规人员 | 审查工具权限、敏感数据和审计记录，测试越权与提示注入 | 权限策略、风险评审和处置方案 |

例如，客服 Agent 错误创建退款申请时，领域专家先确认正确业务规则；Agent 工程师回放 Trace，发现模型误解了支付结果；后端工程师补充稳定的交易状态，并在退款服务中增加校验；平台工程师确保 Trace 能记录工具版本和权限决定；安全人员检查是否存在越权或高金额误操作。只有大量任务都显示模型本身无法区分支付状态时，模型研究人员才需要介入。

小团队里，一位工程师可能同时负责界面、后端、Prompt 和评测；大型公司会拆成应用、Runtime、平台和安全团队。AI Agent 工程师更像一个问题领域，而不是边界固定的职位。

## 需要连接三类能力

AI Agent 工程师既要理解模型怎样使用上下文和工具，也要能够建设 API、状态、权限和故障恢复，还要理解业务成功标准并建立评测。

模型能力决定哪些事情可能做到，软件工程决定它能否可靠运行，业务理解决定它是否值得构建。AI Agent 工程师的日常工作，就是在三者之间做出能够被真实任务验证的取舍。
