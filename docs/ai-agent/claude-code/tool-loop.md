# Claude Code 工具循环

Claude 读取代码、修改文件和运行测试，看起来像模型直接操作了计算机。真正的边界在模型响应之后：模型只生成结构化工具请求，Claude Code Runtime 决定这项请求能否执行，调用本地工具，再把结果送入下一轮模型调用。

这条链路把语言模型的建议变成了可检查的软件动作。

## 一轮工具调用在消息中长什么样

假设 Claude 想读取 `src/auth.ts`。在 Claude Messages API 的工具协议中，过程可以简化为两次请求。

第一次请求包含用户任务和 Read 工具定义。模型返回一个 `tool_use` 内容块：

```json
{
  "role": "assistant",
  "content": [
    {
      "type": "tool_use",
      "id": "toolu_01",
      "name": "Read",
      "input": {"file_path": "src/auth.ts"}
    }
  ],
  "stop_reason": "tool_use"
}
```

`stop_reason: "tool_use"` 表示模型正在等待宿主执行工具，并不表示任务已经结束。Runtime 读取工具名称和参数，完成检查和执行，再发起下一次模型请求。消息历史中要同时保留刚才的 Assistant 响应和对应结果：

```json
[
  {"role": "user", "content": "修复登录测试失败"},
  {
    "role": "assistant",
    "content": [
      {
        "type": "tool_use",
        "id": "toolu_01",
        "name": "Read",
        "input": {"file_path": "src/auth.ts"}
      }
    ]
  },
  {
    "role": "user",
    "content": [
      {
        "type": "tool_result",
        "tool_use_id": "toolu_01",
        "content": "...文件内容..."
      }
    ]
  }
]
```

`tool_use_id` 把结果绑定到原请求。模型看到结果后，可能继续读取调用方、修改文件，也可能认为信息已经足够并返回文本。

Claude Code 的内部 Transcript 比这个例子包含更多元数据，但核心循环来自同一工具协议：保存完整 Assistant 内容块，执行调用，追加 `tool_result`，再次请求模型。

## Runtime 不会直接相信工具请求

模型给出的工具名和参数都是待处理输入。以 Bash 为例，Claude 可能请求运行测试，也可能生成写文件、访问网络或删除资源的命令。执行前会经过多层检查。

Claude Code 的关键顺序是：

1. `PreToolUse` Hook 先运行，可以拒绝、修改或要求确认这次调用；
2. 权限系统按 `deny`、`ask`、`allow` 的优先级匹配规则；
3. 当前权限模式决定剩余请求是自动执行、询问用户，还是拒绝；
4. 获准后，工具才在本地环境或沙箱中运行；
5. 成功触发 `PostToolUse`，失败触发 `PostToolUseFailure`；
6. 工具结果、错误和 Hook 提供的附加信息返回给模型。

Hook 返回 `allow` 只能跳过普通交互确认，不能越过显式 Deny 规则。`PreToolUse` 返回拒绝则在任何权限模式下都能阻止执行。这个优先级让组织策略可以收紧模型权限，项目中的 Prompt 无法把它重新放宽。

**模型提出动作，Runtime 对动作进行裁决。**这比在 System Prompt 中写“请谨慎操作”多了一条可强制执行的边界。

## “看得见工具”和“用得了工具”也要区分

权限规则不仅影响执行，还可能改变模型看到的工具集合。

若配置裸规则 `deny: ["Bash"]`，Bash 会从模型的 Context 中移除，模型根本不会选择它。若只拒绝 `Bash(git push *)`，Bash 仍然可见，Runtime 在具体命令匹配时阻止执行。

两种方式解决不同问题：

- Agent 永远不需要某项能力时，从工具集合中移除，减少误选和 Context 成本；
- Agent 需要工具的大部分能力，只需阻止特定动作时，保留定义并在执行层做细粒度判断。

业务 Agent 也会遇到同样选择。客服助手可以完全看不到“直接退款”，只保留“创建待审申请”；也可以暴露退款工具，但对高金额调用要求主管确认。前者缩小模型的决策空间，后者保留能力并提高执行门槛。

## 专用文件工具的控制边界

Claude Code 同时提供 Read、Edit、Write、Grep、Glob 和 Bash。很多操作都能通过 Bash 完成，单独设计工具仍有价值。

Read 的参数是路径和读取范围，Runtime 可以应用文件访问规则，并针对图片、PDF 或 Notebook 返回合适内容。Edit 要求旧字符串精确且唯一，文件内容已经变化时会拒绝写入，避免在过期假设上修改。Grep 和 Glob 是只读能力，可以更放心地并行执行。Bash 接收通用命令字符串，能力最广，风险和解析成本也最高。

专用工具给 Runtime 留下清楚的控制点：

- 参数可以按动作校验；
- 只读与写入可以采用不同权限；
- 界面可以把 Diff、命令和问题渲染成不同交互；
- Trace 可以直接统计搜索、修改和执行，而不必先解析任意 Shell。

这说明工具边界并不等于底层函数边界。**需要单独授权、校验、展示或审计的动作，值得拥有专用工具。**

## 工具失败也必须返回给模型

测试命令退出非零、Edit 找不到旧字符串、MCP 返回权限不足，都属于环境反馈。Runtime 不能吞掉错误再让模型假装继续。

工具失败时，`tool_result` 应带有错误标记和足够解释。模型据此可以修正参数、重新读取文件、等待依赖，或者告诉用户无法继续。不同错误需要不同语义：

| 工具结果 | 合理的下一步 |
| --- | --- |
| 文件与预期内容不一致 | 重新读取当前版本，再决定怎样修改 |
| 测试断言失败 | 根据失败用例检查实现或测试 |
| 权限被拒绝 | 选择低风险路径或等待用户授权 |
| 临时限流 | 等待后重试 |
| 外部写入结果未知 | 查询真实状态，不能直接重做 |

Claude Code 的代码任务大多可以通过文件和测试重新确认。业务工具涉及退款、上传或部署时，`tool_result` 还要返回资源 ID、幂等键和权威状态，否则 Agent 无法安全恢复。

## 一次响应可以请求多个工具

模型可以在同一条 Assistant 消息中返回多个互不依赖的 `tool_use`，例如并行读取两个文件。Runtime 执行后，应把所有对应的 `tool_result` 放进同一条 User 消息，再继续调用模型。

这要求工具契约能够说明哪些调用可以并行。多个只读搜索通常互不影响；两个修改相同文件的工具调用则可能冲突。专用工具比任意 Bash 更容易让 Runtime 作出这种判断。

并行工具调用的价值不只在速度。它减少模型等待多个串行往返，也让同一判断所需的证据一起进入下一轮。

## 工具循环的结束条件

每轮模型响应大致有两种结果：

- 出现 `tool_use`，Runtime 继续检查、执行并回传结果；
- 模型返回最终文本并以 `end_turn` 结束当前回合。

Claude Code 还允许用户中断运行，权限请求也可能让任务暂停。长任务中的最大轮次、时间和资源限制属于 Runtime 的执行策略，不能依赖模型自觉遵守。

代码任务常用测试结果作为结束证据，但模型输出最终文本并不自动证明测试通过。成熟业务 Agent 更需要把“模型停止调用工具”和“任务满足业务完成条件”分开：前者是模型回合结束，后者还要由任务状态和确定性规则确认。

## 从这条循环带走什么

Claude Code 的工具循环可以缩成四个技术事实：模型返回 `tool_use`；Runtime 在模型之外实施权限与 Hook；真实工具产生 `tool_result`；新结果进入下一次模型调用。

这四步对应 Agent 设计中的责任边界。模型负责选择动作，Runtime 负责裁决和执行，环境提供可以验证的反馈。业务 Agent 的工具可能变成订单、监控和发布 API，循环本身没有变化。
