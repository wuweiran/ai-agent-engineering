# 如何评测 Agent

一个团队基于 Claude Code、项目规则、Skills 和 MCP 工具构建了数据生成 Agent。用户描述任务，Agent 选择模板、生成数据、调用校验工具，再上传到平台运行。团队修改一条 Skill 后，几个熟悉案例仍然成功，但这不足以说明新版本更好。

Agent 的路径会随输入和工具反馈变化。同一个任务可能这次一次生成成功，下次却选错模板或重复上传。评测需要把这种不确定性变成可以重复比较的工程证据。

**Agent 评测从真实任务出发，定义可观察的成功条件，运行后判断结果，并按首个错误环节解释失败。**

```text
真实任务
  ↓
定义成功和不可违反的边界
  ↓
运行 Agent，检查结果与关键过程
  ↓
定位首个错误环节
  ↓
修复后加入回归评测
```

## 把真实任务变成评测样本

“生成一份有效数据”过于抽象。一个可执行样本需要固定用户目标、已知环境、约束和期望结果：

```yaml
name: 为欧洲测试环境生成登录压测数据
user_request: 生成 500 个企业登录测试用户，并上传后运行
fixtures:
  project: login-load-test
  target_environment: eu-test
  templates: [basic-login, enterprise-saml]
constraints:
  - 不包含真实客户信息
  - 不覆盖已有数据集
expected:
  template: enterprise-saml
  record_count: 500
  platform_run_status: succeeded
```

样本不需要预先规定 Agent 的每一步。先读取模板还是先查询项目规则，都可能合理。需要固定的是用户真正想完成什么、系统当时具有什么条件，以及哪些边界绝不能违反。

评测集也不应只收录顺利的演示任务。少量样本可以覆盖三类重要差异：常见任务是否稳定完成，信息不足时是否正确追问，以及权限不足或平台异常时是否安全退出。每类选取能代表不同失败机制的案例，比收集大量只有措辞变化的输入更有价值。

## 沿着交付链定义成功

数据通过 Schema 校验，不代表用户任务已经完成。Agent 可能选错业务模板，或者把正确数据上传到了错误环境。

成功条件要沿着真实交付链检查：

```text
理解用户目标
  ↓
生成符合业务含义的数据
  ↓
通过结构校验
  ↓
上传到正确项目和环境
  ↓
平台运行得到预期结果
```

在这个案例中，最重要的判断是：数据是否表达正确业务场景，结构和数量是否正确，上传目标是否匹配，以及平台是否成功运行且没有越权或重复副作用。

这些条件不能用平均分互相抵消。上传到错误项目是严重失败，即使数据格式完全正确，也不能算任务成功。

**先判断端到端任务是否完成，再用局部指标解释成功或失败。**

## 选择最可靠的判定方式

不同成功条件适合不同判定方式。

结构、数量、上传目标、运行状态、重复提交和权限记录都有明确答案，应由程序自动检查。数据是否符合用户描述的业务意图，可能需要领域专家判断；样本增多后，也可以使用经过人工校准的模型评分器。

合理的顺序是：

```text
确定性检查
  ↓ 通过
领域语义再由人工或模型判断
```

模型评分器不是天然正确。团队要用一组人工判定样本检查它能否识别关键错误，并定期抽查。能够用 Schema、平台状态和权限记录精确判断的问题，不应先交给另一个模型打分。

## 检查结果，也守住关键过程

Agent 最终可能生成正确数据，却先读取了无权访问的项目，或者重复上传后碰巧采用最后一次结果。只检查最终文件会漏掉这些问题。

过程评测不应强制一条唯一工具路径，只检查真正影响可靠性的边界，例如：

- 是否访问正确的项目范围；
- 是否在上传前通过必要校验；
- 写操作是否避免重复副作用；
- 结果未知时是否先查证；
- 高成本运行是否经过要求的确认；
- 满足完成条件后是否及时结束。

**结果评测保护用户价值，关键过程评测保护软件边界；不要把执行风格偏好写成硬性步骤。**

## 找到首个错误环节

一次平台运行失败可能来自不同位置：

```text
目标理解错误
  ↓
选择错误模板
  ↓
生成内容违反业务约束
  ↓
工具参数或上传目标错误
  ↓
平台执行失败
  ↓
Agent 误读结果或提前结束
```

真正有用的归因是找到**第一个使任务偏离正确路径的环节**。如果 Agent 选错模板，后续数据校验失败只是结果；如果工具参数正确而平台服务不可用，则不应修改 Prompt 或 Skill。

每个失败样本至少应保留最终结果、首个错误环节、违反的边界和对应 Trace。团队可以统计各类失败，但不能只看一个总分。成功率小幅上升，可能同时掩盖新出现的跨项目上传，这种版本仍然不能发布。

## 用回归评测比较版本

评测的价值在于比较变化。修改模型、`CLAUDE.md`、Skill、MCP 工具或运行策略后，在相同任务和尽量相同的环境中运行新旧版本：

```text
版本 A 运行代表性任务
  ↓
修改系统
  ↓
版本 B 运行同一批任务
  ↓
比较成功、严重失败、步骤和成本
```

重要任务要重复运行，区分稳定能力与偶然成功。尤其要关注低频但不可接受的越权和重复写入，而不只是平均成功率。平台状态、模型版本和测试数据也要记录，避免把环境变化误判为系统回归。

生产或评测发现新问题后，团队确认预期结果和首个错误环节，修复对应组件，再把脱敏后的最小复现保留在评测集中：

```text
发现失败
  ↓
确认原因
  ↓
修复上下文、Skill、工具或执行控制
  ↓
原问题通过
  ↓
完整评测集没有不可接受的退化
```

评测集因此随着真实失败逐步增长。它需要负责人和版本；业务规则变化后，过期样本也要更新或删除。

## 四步复述

1. **从真实使用中选择少量有代表性的任务。**
2. **沿交付链定义端到端成功和不可违反的边界。**
3. **用最可靠的方式判断结果，并定位首个错误环节。**
4. **修改系统后重复运行，把确认的问题保留为回归样本。**

Agent 评测不是给一段回答打分，而是持续回答一个工程问题：系统变化后，真实任务是否完成得更可靠，严重风险是否仍被控制，为此增加的步骤、延迟和成本是否值得。
