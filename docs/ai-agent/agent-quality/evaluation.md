# Agent 评测

团队修改了数据生成 Skill。几个熟悉案例仍然成功，但 Agent 的路径会随输入和工具反馈变化：这次一次生成正确，下次可能选错模板或重复上传。

[Agent 评测]({{ site.baseurl }}/docs/interview/ai-agent/#agent-evaluation)要把这种不确定性变成能够重复比较的证据。做法从一条真实任务开始。

## 把真实任务保存成样本

“生成一份有效数据”太抽象。样本要固定用户目标、当时的环境、不可违反的约束和期望结果：

```yaml
name: 为欧洲测试环境生成登录压测数据
user_request: 生成 500 个企业登录测试用户，并上传后运行
fixtures:
  project: login-load-test
  target_environment: eu-test
  templates: [basic-login, enterprise-saml]
constraints:
  - 不包含真实客户信息
  - 不覆盖已有数据集
expected:
  template: enterprise-saml
  record_count: 500
  platform_run_status: succeeded
```

样本不需要规定 Agent 的每一步。先读取模板还是先查询项目规则，都可能合理。真正要固定的是用户想完成什么、系统当时具有什么条件，以及哪些边界绝不能违反。

评测集也要包含信息不足、权限不足和平台异常。每类选择少量能够代表不同失败机制的任务，比收集大量只有措辞变化的输入更有价值。

## 沿着用户交付判断成功

数据通过 Schema 校验，不代表任务已经完成。Agent 可能选错业务模板，也可能把正确数据上传到了错误环境。

一次完整交付包括：正确理解用户目标，生成符合业务含义的数据，通过结构校验，上传到正确项目和环境，并让平台得到预期运行结果。

这些条件有优先级。跨项目上传是严重失败，不能因为字段格式很好而抵消。**先判断端到端任务是否完成，再用局部指标解释原因。**

## 能精确判断的部分交给程序

结构、数量、上传目标、运行状态、重复提交和权限记录都有明确答案，应由程序自动检查。

“数据是否符合用户描述的业务意图”可能需要领域专家判断。样本增多后，可以使用经过人工校准的模型评分器，但仍要定期抽查它能否识别关键错误。

能够通过 Schema 和平台记录精确判断的问题，不需要先交给另一个模型打分。先运行确定性检查，再处理真正需要语义判断的部分。

## 最终结果和关键过程都要检查

Agent 可能生成正确文件，却先读取了无权访问的项目；也可能重复上传后碰巧采用了最后一次结果。只看最终产物会漏掉这些问题。

过程评测不用规定唯一工具路径，只检查真正影响可靠性的边界：是否访问正确项目，上传前是否完成必要校验，写操作是否避免重复，结果未知时是否先查证，高成本运行是否经过确认，满足完成条件后是否及时结束。

模型先读取模板还是先查询规则，通常只是执行风格；越权、跳过校验和重复副作用才是过程错误。

## 失败时找到第一个错误位置

平台运行失败可能经历这样的链路：Agent 先误解目标，随后选错模板，生成内容违反约束，上传后平台报错。平台是最后暴露问题的位置，目标理解才是第一次偏离。

真正有用的归因，是找到第一个让任务离开正确路径的环节。若模板选择已经错误，后续校验失败只是结果；若工具参数正确而平台不可用，则不应该修改 Prompt 或 Skill。

每个失败样本至少保留最终结果、首次偏离、违反的边界和对应 Trace。团队可以统计失败类型，但不能只看总成功率。成功率略有上升，若同时出现跨项目上传，新版本仍然不能发布。

## 用同一批任务比较版本

修改模型、`CLAUDE.md`、Skill、MCP 或运行策略后，在相同样本和尽量相同的环境中运行新旧版本。比较哪些任务改善、哪些退化，是否新增严重失败，步骤和成本是否发生明显变化。

重要任务需要重复运行，以区分稳定能力和偶然成功。低频但不可接受的越权与重复写入，要单独作为发布门槛，不能被平均成功率掩盖。

生产或评测发现新问题后，团队先确认预期结果和第一次偏离，再把脱敏后的最小复现保留在评测集中。修复后，原问题要通过，完整评测集也不能出现不可接受退化。

## Agent 评测闭环

从真实使用中选择代表性任务，沿用户交付定义成功，使用最可靠的方式判断结果，失败时找到第一次偏离，修改后再用同一批任务比较。

Agent 评测不是给一段回答打分。它要持续回答：系统变化以后，真实任务是否完成得更可靠，严重风险是否仍然受控。
