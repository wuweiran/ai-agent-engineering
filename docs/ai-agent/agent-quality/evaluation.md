# Agent 评测

团队修改了数据生成 Skill。几个熟悉案例仍然成功，但 Agent 的路径会随输入和工具反馈变化：这次一次生成正确，下次可能选错模板、跳过确认或重复上传。零散演示无法说明新版本是否真的更可靠。

[Agent 评测]({{ site.baseurl }}/docs/interview/ai-agent/quality-production/#agent-evaluation)是一套可重复的验证机制：让指定版本的 Agent 在固定任务和环境中运行，检查任务结果、关键过程和运行效率，再通过 Trace 解释失败并比较版本。它主要回答三个问题：用户目标是否完成，执行过程是否可靠，完成任务的代价是否合理。

```text
评测集与固定环境
→ 运行指定 Agent 版本
→ 检查任务结果、关键过程和运行效率
→ 通过 Trace 定位第一次偏离
→ 比较版本并决定是否发布
→ 将生产失败补入评测集
```

## 评测对象

Agent 评测要同时检查**任务结果、关键过程与运行效率**。任务结果衡量用户目标是否端到端完成。数据生成 Agent 不只要通过 Schema 校验，还要理解正确的业务意图，把数据上传到正确的项目和环境，并取得预期的平台运行结果。局部步骤成功不能代替最终交付。

关键过程检查影响安全性和可靠性的执行边界。Agent 即使生成了正确文件，只要曾读取无权访问的项目、跳过上传前确认、重复执行写操作或在结果未知时直接重试，这次执行仍然存在严重问题。过程评测不要求唯一工具路径；先读模板还是先查规则通常都可以，越权、跳过校验和重复副作用才是需要约束的错误。

运行效率描述完成任务付出的代价，包括步骤数、工具调用数、延迟、Token 和成本。效率必须建立在结果正确和边界受控的前提上。少调用一次工具却降低任务完成率，不属于有效优化。

这三个层次有明确优先级：**先确认任务完成和严重风险，再比较过程质量与运行效率。**跨项目写入、错误退款和虚假完成等严重失败应设置独立门槛，不能被平均成功率抵消。

## 评测集

评测集由**真实任务、固定环境与边界场景**组成。评测从可重复运行的任务样本开始。“生成一份有效数据”过于抽象，样本需要固定用户目标、执行环境、不可违反的约束和预期结果：

```yaml
name: 为欧洲测试环境生成登录压测数据
user_request: 生成 500 个企业登录测试用户，并上传后运行
fixtures:
  project: login-load-test
  target_environment: eu-test
  templates: [basic-login, enterprise-saml]
constraints:
  - 不包含真实客户信息
  - 不覆盖已有数据集
expected:
  template: enterprise-saml
  record_count: 500
  platform_run_status: succeeded
```

样本不必规定 Agent 的每一步。路径可以随着工具结果变化，评测需要固定的是任务条件、完成标准和安全边界。为了让不同版本可比，依赖的模板、工具返回和业务数据应尽量使用固定 Fixture；必须连接真实平台时，也要记录环境和数据版本。

评测集既要覆盖正常任务，也要覆盖不同失败机制，例如信息不足、权限不足、工具限流、写操作结果未知和平台运行失败。少量机制不同的样本，比大量只有措辞变化的输入更能暴露系统缺陷。重要任务还要重复运行，用来区分稳定能力和偶然成功。

## 评价器

评价器主要包括**确定性检查、人工评审与模型评分**，负责把一次执行转换成可比较的判断。选择评价器时，优先使用最可靠且最容易解释的方法。

结构、数量、上传目标、平台状态、重复提交和权限记录都有明确答案，适合由程序执行确定性检查。程序能够直接查询的数据，不需要先交给另一个模型判断。

业务意图是否理解正确、报告是否完整等开放语义，可能需要领域专家评审。可以把质量拆成相关性、完整性和一致性：是否回答当前问题，是否覆盖全部关键要求，以及回答内部、多轮约束和工具状态是否相互一致。完整性适合将要求拆成 Checklist，一致性需要冲突资料和多轮 Scenario，不能只让评分器给一个模糊总分。样本规模扩大后，可以使用经过人工样本校准的模型评分器，但要保留评分标准、模型版本和定期抽查，防止评分器本身发生偏移。

一次任务通常会组合多种评价器：程序检查业务状态和安全边界，人工或模型检查语义质量，Runtime 指标统计延迟与成本。最终结果应保留各项判断，不能只压缩成一个平均总分。

## Trace 与故障归因

Trace 与故障归因的重点是**定位执行路径的第一次偏离**。最终失败只说明问题在哪里暴露，Trace 用来寻找执行路径第一次偏离正确方向的位置。

平台运行失败可能经历一条连续链路：Agent 误解目标，随后选错模板，生成内容违反约束，最后由平台报错。平台错误是最终现象，目标理解才是第一次偏离。若工具参数正确而平台暂时不可用，问题则属于外部依赖，不应通过修改 Prompt 或 Skill 处理。

每个失败样本至少要保留最终结果、第一次偏离、违反的边界和相关 Trace。归因结果决定改动位置：缺少事实时调整 Context，方法错误时修改 Skill，工具误选时改善名称、描述或 Schema，执行异常时修复 MCP 或业务服务，重复调用和提前结束则检查 Runtime 状态与终止控制。

**评测指出哪里失败，Trace 解释为什么失败。**没有归因的分数只能判断版本变好或变差，无法指导工程修改。

## 版本比较

版本比较需要建立**基线、回归测试与发布门槛**。修改模型、Prompt、Context、Skill、工具或 Runtime 后，应让新旧版本运行同一批样本，并尽量保持环境一致。比较内容包括任务完成率、各类严重失败、过程错误、延迟和成本，也要查看哪些具体任务改善或退化。

新版本先重跑原始失败样本，确认第一次偏离已经消失；再运行相邻任务，检查是否为了一个案例造成过度修复；最后运行完整评测集，验证系统没有出现不可接受的回归。

平均指标适合观察整体趋势，发布门槛负责守住底线。低频但不可接受的越权、错误写入、重复副作用和虚假完成，即使只出现一次，也可以阻止版本发布。

## 线上评测与生产反馈闭环

离线评测能够固定环境，却无法覆盖真实权限、外部服务波动和长尾输入。上线后首先观察用户结果，例如成功交付率、人工接管率、重试率和放弃率；严重失败要单独告警。工具选择率、参数正确率、故障恢复率、P95/P99 完整耗时和每个成功任务的成本，则用于解释用户结果为什么变化。

线上指标需要按任务类型和风险分层，并能够关联到具体任务、Agent 版本和 Trace。否则，简单任务占比提高可能造成成功率上升的假象，团队无法判断 Agent 能力是否真正改善。

生产中出现新问题后，先确认它是否属于产品承诺范围，再提取脱敏的最小复现，固定任务条件和预期结果并加入评测集。修复后，原问题、相邻任务和完整回归集都要重新验证，由此形成持续的质量闭环。

Agent 评测的核心可以概括为：**用真实任务定义成功，用不同评价器检查结果、过程与效率，用 Trace 找到第一次偏离，再以固定评测集比较版本并承接生产反馈。**它最终要证明的不是某段回答得分更高，而是 Agent 在真实业务约束下完成任务更可靠，严重风险仍然受控。
