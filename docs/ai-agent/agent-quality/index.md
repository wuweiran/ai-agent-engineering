# 如何建立 Agent 质量体系

数据生成 Agent 在演示中成功了：用户描述任务，Agent 生成文件，通过本地校验，再上传平台运行。团队修改 Skill 和 MCP 工具说明后，十个手工案例仍然正常，于是准备发布。

真正上线后，有些数据格式正确却不符合用户意图；有些任务运行成功，却重复上传了相同数据；还有些任务结果没有问题，但耗时已经让用户放弃。

几个成功案例只能证明系统有可能工作。团队还需要知道它是否稳定、哪里会失败，以及修改后是否真的更好。

## 先定义什么叫“做得好”

Agent 的质量可以从三个层面观察：最终结果是否完成用户任务，执行过程是否守住软件边界，这种质量能否以可接受的代价持续提供。

| 层面 | 数据生成 Agent 的例子 |
| --- | --- |
| 任务结果 | 数据符合需求，上传到正确项目，平台运行成功 |
| 执行过程 | 选对模板，没有越权读取或重复上传 |
| 真实使用 | 用户等待时间、人工介入和失败率可以接受 |

结果最接近用户价值，应当先判断。过程检查帮助发现尚未造成最终错误、却可能引发事故的行为。真实使用情况则决定原型能否成为产品。

## 评测让不同版本可以比较

模型、规则、Skill、上下文和工具接口都会改变 Agent 行为。团队需要一批可重复运行的真实任务，在改动前后使用同样的成功标准。

评测的目的不是得到漂亮总分，而是回答发布决策：新版本改善了哪些任务，破坏了哪些任务，是否出现越权和重复写入等严重问题。

样本应来自真实使用，既有常见任务，也有信息不足、工具故障和高风险边界。若评测集只有团队最熟悉的演示案例，结果会高估系统能力。

## Trace 解释失败从哪里开始

评测发现“平台运行失败”，原因可能出在用户意图理解、模板选择、数据校验、MCP 参数或平台本身。

Trace 记录系统可观察的过程：模型看到了哪些规则和数据，请求了什么工具，程序执行了什么，任务状态怎样变化。工程师沿着这些记录寻找第一次偏离，而不是把最终错误都归给模型。

**评测告诉团队哪些任务有问题，Trace 帮助找到问题最早出现的位置。**

## 改进后还要重新验证

若 Trace 发现 Skill 使用了错误默认模板，修复就应该落在 Skill，而不是先升级模型或增加更强硬的 Prompt。

原失败样本通过后，还要运行相邻任务和完整回归集，检查修复有没有误伤其他场景。若一条判断已经稳定成业务规则，可以交回程序；仍然依赖开放语义的部分，继续由模型处理。

离线通过后再逐步发布。生产中确认的新失败经过脱敏和最小化，加入回归评测集。这样，团队每解决一个真实问题，质量体系就多一条可以长期验证的证据。

## 质量由多个角色共同定义

领域专家判断数据是否表达正确业务含义，平台工程师确认上传和运行是否可靠，安全人员定义数据与工具权限，产品经理关注用户是否完成任务，Agent 工程师把这些标准落实成评测、Trace 和发布检查。

只由开发者评测，系统往往容易通过技术校验，却没有解决用户任务。质量体系从各方对“什么叫成功”形成可执行共识开始。

## 记住这条循环

质量工作会反复经历：用真实任务评测，沿 Trace 定位第一次偏离，在正确层次修改，运行回归，再从生产中收集新的失败。

它最终服务于一个实际决定：这个 Agent 版本是否比以前更可靠，是否值得让更多真实用户使用。
