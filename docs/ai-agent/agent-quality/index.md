# 如何建立 Agent 质量体系

一个数据生成 Agent 在演示中成功了：用户描述任务，Agent 生成数据文件，通过本地校验，再上传平台运行。团队随后修改了 Skill 和 MCP 工具说明，十个手工案例看起来仍然正常，于是准备发布。

真正上线后，问题可能出现在任何一层：有些数据格式正确却不符合用户意图；有些任务运行成功，却重复上传了相同数据；还有些任务结果没有问题，但耗时和费用已经无法接受。

几个成功案例只能证明系统有可能工作，不能说明它能稳定服务真实用户。**Agent 质量体系要定义什么叫做得好，用可重复的评测发现变化，再通过执行轨迹和生产反馈解释变化。**

```text
定义质量
  ↓
用代表性任务评测
  ↓
通过 Trace 找到第一次偏离
  ↓
在正确的系统层改进
  ↓
回归、发布并观察生产结果
  ↓
确认的新问题进入评测集
```

## 质量不只等于最终答案正确

普通文本生成经常只检查输出内容。Agent 会经过多轮判断和工具执行，即使最终结果相同，过程也可能带来完全不同的风险和成本。

数据生成 Agent 的质量可以从三个层次理解：

| 层次 | 关注的问题 | 例子 |
| --- | --- | --- |
| 结果质量 | 最终是否完成用户任务 | 数据符合需求，平台运行得到预期结果 |
| 过程质量 | Agent 是否用正确方式完成 | 选对模板，没有越权读取或重复上传 |
| 生产质量 | 系统是否适合长期运行 | 延迟、成本、失败率和人工介入可接受 |

结果质量最接近用户价值，应该优先判断。过程质量帮助发现尚未造成错误、但可能在未来引发事故的行为。生产质量则决定一个原型能否成为产品。

**先判断任务是否真正完成，再检查完成过程是否可靠，最后判断这种质量能否以可接受的代价持续提供。**

## 评测让系统变化可以比较

Agent 的模型、Prompt、Skill、上下文策略和工具接口都可能改变行为。团队需要一组可重复运行的任务，在改动前后比较同样的成功条件。

```text
版本 A：运行代表性任务
  ↓
修改 Skill 或 MCP 工具
  ↓
版本 B：运行同一批任务
  ↓
比较任务成功、错误类型、步骤和成本
```

评测不是为了得到一个漂亮的总分，而是回答具体决策：新版本是否值得发布，改善了哪些任务，又破坏了哪些任务。

样本应来自真实使用场景，并包含正常任务、信息不足、工具故障和高风险边界。若评测集只有团队最熟悉的演示案例，结果会高估系统能力。

## Trace 解释为什么成功或失败

评测可以发现“上传后平台运行失败”，却不能单独说明原因。可能是用户意图理解错误、数据模板选择错误、校验器漏检、MCP 参数错误，或平台本身不可用。

Trace 把一次任务还原为执行链：

```text
用户目标
  ↓
装配了哪些上下文和 Skill
  ↓
模型选择什么工具、生成什么参数
  ↓
工具返回什么结果
  ↓
状态怎样变化，任务为何结束
```

它不是为了展示模型的私有思考过程，而是记录系统可观察的输入、决定、动作和结果。团队据此定位应该修改数据规则、Skill、工具契约、模型选择还是终止条件。

**评测发现哪里不好，Trace 解释为什么不好。**

## 根因确认后才进入系统改进

Trace 找到第一次偏离后，工程师要在最接近根因的层次修改：上下文缺少事实就调整装配或检索，Skill 方法错误就修正 Skill，工具参数混淆就修改契约。若一个判断已经能够稳定写成规则，则可以把它从模型收回程序。

修复先通过原失败样本验证，再运行相邻任务和完整回归集。离线通过后，还要限制真实影响范围，逐步发布并观察用户结果、Agent 行为和系统代价。这样才能避免为了一个案例增加 Prompt，却在其他任务中制造新的错误。

## 生产反馈补足离线样本

离线评测无法预见所有用户表达、数据规模和外部依赖故障。上线后还要观察真实任务是否创造价值：用户是否接受生成结果，平台运行是否成功，哪些任务经常转人工，以及延迟和成本是否持续恶化。

生产指标不能替代评测。运行成功率下降说明系统出了问题，但生产流量每天都不同，无法稳定比较两个版本；离线评测可重复，却可能漏掉新场景。两者需要形成循环：

```text
离线评测控制发布风险
  ↓
生产反馈发现未知问题
  ↓
失败任务经过脱敏和确认
  ↓
进入回归评测集
  ↓
修复后重新验证
```

用户投诉也不能直接当成训练数据。团队需要先确认预期结果、失败原因和数据权限，再把案例转化为可重复的评测任务。

## 不同角色共同定义质量

Agent 工程师无法独自决定所有成功标准。数据生成工具中：

- 领域专家判断数据是否表达了正确业务含义；
- 平台工程师确认上传、运行和资源状态是否可靠；
- 安全人员定义数据与工具权限边界；
- 产品经理关注用户是否完成任务以及人工成本；
- Agent 工程师把这些标准落实为评测、Trace 和发布检查。

如果只由开发者评测，系统往往容易通过技术校验，却没有解决用户任务。质量体系的第一步不是选择评分框架，而是让相关角色对“什么叫成功”形成可执行的共识。

## 质量体系最终服务发布决策

一次改动是否发布，可以沿着一条简单链路判断：

1. 代表性任务的结果是否改善或至少没有退化；
2. 高风险过程是否仍满足权限和执行边界；
3. 延迟、成本和人工介入是否仍在可接受范围；
4. 新出现的失败是否已经理解，而不是被平均分掩盖。

不同产品的阈值不同。内部草稿工具可以容忍更多人工修改，自动上传并启动高成本任务的 Agent 则需要更严格的过程检查和审批。

## 四句话复述

**Agent 质量有三个层次：结果是否有用，过程是否可靠，生产代价是否可接受。**

**评测用于稳定发现变化，Trace 用于找到任务第一次偏离的位置。**

**根因确认后，在正确的系统层做最小改动，再用回归和逐步发布验证整体效果。**

**生产失败经过确认后回到评测集，质量体系才形成持续改进闭环。**
