# 如何用 Trace 排查 Agent

评测发现一个数据生成任务失败了：用户要求企业 SAML 登录数据，Agent 最终却使用基础登录模板，平台运行因此报错。

只看结果，很容易说“模型选错了模板”。错误也可能更早发生：上下文没有解释企业登录与 SAML 的关系，Skill 给了错误默认值，MCP 返回了过期模板，或者模型拿到正确信息后没有使用。

**Trace 排查要找到任务第一次偏离正确路径的位置。**

## Trace 要记录哪些事实

一次任务跨越模型和多个软件组件。排查需要知道任务目标和运行版本，本轮装配了哪些规则、Skill 和检索结果，模型请求了什么工具与参数，工具和权限检查返回什么，任务状态怎样变化，又为什么结束。

这不是展示模型的私有思考过程。工程师需要的是系统边界上可以验证的事实：模型看到了什么、请求了什么、程序执行了什么。

模型、Prompt、Skill、工具和数据版本也要记录。否则只能知道当时发生了什么，无法重现相同条件。

## 从最终错误向前看

这次失败的轨迹可能是：平台报错缺少 `saml_certificate`；上传数据使用了 `basic-login` 模板；Agent 调用了 `get_template(name="basic-login")`；模型其实已经判断 `task_type = enterprise_login`；再往前看，Skill 写着“enterprise_login 默认使用 basic-login”。

模型正确识别了任务类型，第一次偏离发生在 Skill。平台只是最后暴露问题的位置。此时升级模型或修改平台校验，都不会解决根因。

阅读关键转折时，可以问：输入到这里是否仍然正确？输出是否符合当时可用的信息和规则？如果这一步正确，后面还会不会失败？

找到第一个“输入正确、输出开始错误”的位置，就得到一个值得验证的根因假设。

## 用小实验确认根因

Trace 只能提出假设。若怀疑 Skill 默认模板错误，可以保持用户请求、模型、工具和平台环境不变，只修正这一条规则，再重跑失败任务。

模型稳定选择 `enterprise-saml`，说明假设得到支持；仍然选错，就继续检查工具定义或模型判断。对外部平台故障，也可以使用保存的工具响应、测试环境或故障注入。

关键是一次只改变一个怀疑因素。若同时修改 Prompt、Skill 和工具，即使任务通过，也不知道哪项真正有效。

## 第一次偏离决定检查哪一层

| 首次偏离 | 优先检查 |
| --- | --- |
| 用户目标一开始就被错误解释 | 输入结构或追问机制 |
| 模型没有看到必要规则或看到过期信息 | 上下文装配与检索 |
| 加载了错误方法或默认步骤 | Skill 内容与触发条件 |
| 选错工具或参数反复出错 | 工具名称、描述和 Schema |
| 工具请求正确，执行结果错误 | MCP 服务、平台 API 或权限实现 |
| 外部动作成功，任务状态没更新 | 状态持久化与恢复逻辑 |
| 证据不足却宣布完成 | 完成条件与执行控制 |
| 输入和接口都清楚，模型仍然不稳定 | 提示方式、模型能力或决策边界 |

这张表不用逐项排查。Trace 已经把范围缩小到一个转折点，工程师只检查附近的系统边界。

## “成功”和空结果也可能误导 Agent

上传工具返回 `ok`，可能只表示文件已接收，并不代表校验通过或平台任务已经创建。Agent 若据此宣布完成，问题可能在返回契约或结束条件。

搜索为空也可能来自错误查询、权限不足或索引延迟。工具结果应区分这些状态，Trace 需要保留查询参数、权限决定和数据版本。

排查时不能只看工具有没有报错，还要检查返回值是否足以支持随后的判断。

## 偶发问题要比较成功和失败轨迹

同一个样本有时成功、有时失败，单条 Trace 只能解释一次运行。可以对比两条轨迹最早从哪里分叉。

若成功运行加载 `enterprise` Skill，失败运行加载 `generic` Skill，问题可能是 Skill 触发条件含糊。若两者直到工具选择才不同，则检查工具描述或模型稳定性。外部服务延迟、模型版本和测试数据也要一起比较。

## Trace 也有数据边界

生产 Trace 不适合无条件保存全部原文。用户数据、凭据和平台返回需要脱敏；大段内容可以保存资源 ID 和摘要；访问按租户和角色限制；记录设置保存期限。

同时，关键版本、工具参数、错误语义和状态变化要保持结构化，否则无法排查。Trace 设计本身也是信息取舍。

一次有效排查只有三步：从失败结果还原关键执行链，找到第一次偏离，再通过对照重放验证根因。它把“Agent 表现不好”转化成可以验证的软件问题。
