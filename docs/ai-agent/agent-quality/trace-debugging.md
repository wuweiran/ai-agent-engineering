# 如何用 Trace 排查 Agent

评测发现一个数据生成任务失败了：用户要求生成企业 SAML 登录数据，Agent 最终却使用基础登录模板，平台运行因此报错。

只看最终结果，很容易得出“模型选错了模板”。但错误可能更早发生：任务上下文没有解释企业登录与 SAML 的关系，Skill 给了错误默认值，MCP 工具返回了过期模板，或者模型明明取得了正确信息却没有使用。

**Trace 排查的目标不是阅读所有日志，而是找到任务第一次偏离正确路径的位置，并用证据解释为什么偏离。**

```text
评测指出失败结果
  ↓
还原关键执行链
  ↓
找到第一次偏离
  ↓
确认它是原因，而不是后续症状
  ↓
确定需要修改的系统层
```

## Trace 记录的是可观察执行过程

一次 Agent 任务会跨越模型和多个软件组件。可用于排查的 Trace 通常包含：

```text
任务目标与运行版本
  ↓
本轮装配的规则、Skill 和检索结果
  ↓
模型输出的工具请求或结构化决定
  ↓
工具参数、权限结果和返回值
  ↓
任务状态变化与结束原因
```

它不要求保存或展示模型的私有思考过程。工程师真正需要的是系统边界上可以验证的事实：模型看到了什么、请求了什么、程序执行了什么，以及结果如何改变后续状态。

每一步还应带上模型、Prompt、Skill、工具和数据版本。否则，同一条 Trace 只能说明当时发生了什么，却无法复现或比较。

## 从失败结果向前寻找第一次偏离

这次平台错误的末端轨迹可能是：

```text
平台运行失败：缺少 saml_certificate 字段
  ↑
上传的数据使用 basic-login 模板
  ↑
Agent 调用了 get_template(name="basic-login")
  ↑
模型判断 task_type = enterprise_login
  ↑
Skill 说明：enterprise_login 默认使用 basic-login
```

最后一次错误发生在平台，但第一次偏离出现在 Skill：模型已经正确识别任务类型，却依据错误规则选择模板。此时修改模型或平台校验器都不会解决根因。

阅读 Trace 时，可以对每个关键转折问三个问题：

1. 这一步的输入是否已经正确？
2. 这一步的输出是否符合当时可用的信息和规则？
3. 如果这一步正确，下一步是否仍会失败？

沿着链路找到第一个“输入正确、输出开始错误”的位置，通常就找到了最值得验证的根因假设。

**最后暴露错误的组件不一定是最先制造错误的组件。**

## 用对照验证根因，而不是凭轨迹猜测

Trace 只能提出根因假设，还需要一个小实验确认。

假设 Skill 的默认模板是问题，可以保持用户请求、模型、工具和平台环境不变，只修正这一条规则，再重跑失败任务。如果模型稳定选择 `enterprise-saml`，说明假设得到支持；如果仍然选错，就要继续检查工具定义或模型判断。

```text
原任务 + 原环境
  ↓ 只改变一个怀疑因素
重新运行
  ↓
错误消失？
  ├─ 是 → 根因假设得到支持
  └─ 否 → 返回 Trace 继续定位
```

对于外部平台故障，不一定能够完整重放，可以使用保存的工具响应、测试环境或故障注入。关键是尽量只改变一个因素，避免同时修改 Prompt、Skill 和工具后无法知道哪项真正有效。

## 把问题定位到系统层，而不是统称模型错误

第一次偏离通常落在某个明确层次：

| 观察到的首次偏离 | 优先检查 |
| --- | --- |
| 用户目标在进入循环前就被错误解释 | 任务定义、输入结构或追问机制 |
| 模型没有看到必要规则或看到了过期信息 | 上下文装配与检索 |
| 加载了错误方法或默认步骤 | Skill 内容与触发条件 |
| 选错工具或参数反复出错 | 工具名称、描述和 Schema |
| 工具请求正确，执行结果错误 | MCP 服务、平台 API 或权限实现 |
| 外部动作成功但状态没有更新 | 状态持久化与恢复逻辑 |
| 证据不足却宣布完成 | 完成条件与执行控制 |
| 所有输入和接口都清楚，模型仍不稳定 | 提示方式、模型能力或决策是否应固化 |

这张表不是要求逐项检查。Trace 已经把范围缩小到一个转折点，工程师只需要检查该位置附近的系统边界。

例如，Agent 请求了正确的 `enterprise-saml` 模板，但 MCP 返回旧版本，这是工具执行问题；如果 MCP 返回正确模板，模型仍生成基础字段，才继续检查 Skill、上下文或模型。

## 空结果和成功结果同样需要排查

明显的异常容易引起注意，语义含糊的“成功”更危险。上传工具返回 `ok`，可能只表示文件已接收，并不代表校验通过或运行已创建。Agent 若把它理解为任务完成，问题在返回契约或终止判断，而不是上传动作本身。

搜索为空也不一定是没有数据。查询条件可能错误、调用者可能无权限，或者索引尚未更新。工具结果应区分这些状态，Trace 则要保留查询参数、权限决策和数据版本。

**排查时不能只看工具是否报错，还要检查返回值是否足以支持 Agent 随后的判断。**

## 多次运行帮助区分稳定缺陷和偶发故障

同一个样本有时成功、有时失败，单条 Trace 只能解释一次运行。工程师需要比较成功和失败轨迹，寻找最早出现差异的地方：

```text
成功轨迹：识别 SAML → 加载 enterprise Skill → 选择正确模板
失败轨迹：识别 SAML → 加载 generic Skill    → 选择基础模板
                                 ↑
                         第一个行为差异
```

如果差异总出现在 Skill 触发，问题可能是触发条件含糊；如果直到模型工具选择时才分叉，则需要检查工具描述或模型稳定性。外部服务延迟、模型版本和测试数据也要一起比较，避免把环境波动误判成逻辑缺陷。

## Trace 本身也需要边界

完整记录有助于排查，也会带来数据和成本风险。生产 Trace 不应无条件保存所有原始内容。工程师需要：

- 对用户数据、凭据和平台返回进行脱敏；
- 用资源 ID 和摘要代替不必要的大段原文；
- 按租户和角色限制 Trace 访问；
- 设置保存期限，并让评测样本与生产记录分离；
- 对关键版本和错误语义保持结构化字段。

缺少关键输入会让问题无法定位，保存所有内容又会扩大敏感数据暴露。Trace 设计本身也是一次信息取舍。

## 三步完成一次 Trace 排查

1. **从评测失败出发，还原影响结果的关键执行链。**
2. **找到第一次偏离，并区分根因与后续症状。**
3. **通过对照重放验证假设，确定需要修改的系统层。**

Trace 的价值不在于让工程师观看 Agent 运行得多复杂，而在于把“Agent 表现不好”转化为一个可以验证的软件问题。根因确认以后，下一步才是选择改动范围，并证明修复没有破坏其他任务。
