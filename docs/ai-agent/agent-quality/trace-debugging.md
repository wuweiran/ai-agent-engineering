# Agent Trace 与故障定位

评测发现一个数据生成任务失败了：用户要求企业 SAML 登录数据，Agent 最终却使用基础登录模板，平台运行因此报错。最终错误只能说明问题在哪里暴露，真正的错误可能更早发生在 Context 装配、Skill、工具选择或业务服务中。

Agent Trace 是一次执行的结构化轨迹。它按照时间和调用关系连接 Context 装配、模型调用、工具执行、状态变化和外部系统结果，使工程师能够重现执行条件并定位第一次偏离。普通日志记录单个组件发生了什么，Trace 更强调一次任务中的前后因果与跨组件关联。

```text
业务任务或业务对象
└─ Agent 执行
   ├─ Context 装配
   ├─ 模型调用
   ├─ 工具调用
   │  └─ 业务服务或外部平台
   ├─ 任务状态变化
   └─ 等待、恢复或终止
```

**Trace 的核心用途是还原执行链、定位第一次偏离并验证根因。**它记录软件边界上可以验证的事实，不要求展示模型未对外返回的内部推理过程。

## Trace 的数据结构

常见可观测系统使用 Trace、Span 和 Event 组织数据。一次 Agent Run 对应一条 Trace；一次模型请求、工具调用或状态写入可以记录为一个 Span；重试、审批和状态切换等时间点则可以记录为 Event。具体系统不必采用完全相同的名称，但要保留层级、顺序和父子调用关系。

每条记录都要关联稳定标识，例如 `trace_id`、运行 ID、业务对象 ID、可选的 Agent 任务或会话 ID。模型请求与 Tool Result 还要通过调用 ID 对应，写操作则关联业务请求 ID、幂等键和外部资源 ID。这样才能把模型请求、Runtime 行为与真实业务结果连接成一条执行链。

时间线只解决“先发生什么”。要重现问题，还需要记录 Agent、模型、Prompt、Skill、工具 Schema、业务数据和运行环境的版本。相同输入在不同配置下可能产生不同路径，缺少版本信息的 Trace 只能查看，无法可靠重放和比较。

## Trace 中的关键事实

Trace 应围绕**执行身份、Context、模型调用、工具执行和状态控制**保存关键事实：

| 类别 | 关键记录 | 主要用途 |
| --- | --- | --- |
| 执行身份 | 业务对象 ID、运行 ID、Agent 与模型版本、环境和时间 | 找到一次执行并重现运行条件 |
| Context | 用户目标、系统约束、任务状态、检索结果与数据版本 | 确认模型实际取得了哪些信息 |
| 模型调用 | 请求 ID、Tool Call、停止原因、Token、延迟和错误 | 判断模型提出了什么动作以及为什么结束本轮 |
| 工具执行 | 工具与 Schema 版本、校验后参数、权限和审批结果、Tool Result、耗时 | 区分模型请求、Runtime 校验和真实执行结果 |
| 状态控制 | 执行前后状态、等待原因、恢复位置、重试次数和终止判断 | 解释任务为什么继续、暂停、恢复或结束 |
| 外部副作用 | 业务请求 ID、幂等键、资源 ID和最终业务状态 | 判断写操作是否成功、重复或结果未知 |

原始消息、检索文档和大段工具输出可能包含敏感数据，也会快速增加存储量。系统可以保存经过脱敏的内容、摘要、哈希或资源引用，但必须确保排查时能够取得当时真正使用的版本。只保存一句“调用成功”无法解释模型看到了什么，也无法确认外部动作是否真的完成。

## 第一次偏离

故障定位的关键不是找到最后一个错误，而是找到**第一次使执行离开正确路径的转折点**。

SAML 任务的 Trace 可能呈现以下执行链：

```text
用户目标：生成企业 SAML 登录数据
→ 模型识别 task_type = enterprise_login
→ Skill 指定默认模板 basic-login
→ 模型调用 get_template(name="basic-login")
→ Agent 生成并上传基础登录数据
→ 平台报告缺少 saml_certificate
```

模型对任务类型的识别仍然正确，Skill 选择默认模板时开始出错，因此第一次偏离发生在 Skill。平台错误只是最终症状。此时升级模型或放宽平台校验都没有处理根因。

沿 Trace 检查每个关键边界时，需要比较该步骤的输入、规则、输出和预期：输入是否仍然正确，输出是否符合当时已有信息，若该步骤保持正确，后续故障是否仍会发生。如果某一步的输入已经错误，就继续向上游追踪；如果输入正确而输出首次错误，就得到一个根因假设。

## 故障归因

第一次偏离出现在哪一层，决定工程师优先检查和修改哪里：

| 第一次偏离 | 优先检查 |
| --- | --- |
| 用户目标被错误解析或缺少必要信息 | 输入结构、任务入口和追问机制 |
| 模型没有取得必要规则，或取得过期事实 | Context 装配、检索与数据版本 |
| 加载了错误方法或默认步骤 | Skill 内容与触发条件 |
| 模型选错工具或持续生成错误参数 | 工具名称、描述、Schema 和模型决策 |
| Tool Call 正确，Runtime 拒绝或改写错误 | 参数校验、权限、审批和工具处理器 |
| Runtime 请求正确，外部结果错误 | MCP 服务、业务 API 或外部平台 |
| 外部动作成功，任务状态没有更新 | 状态持久化、幂等与恢复逻辑 |
| 证据不足却宣布完成 | 完成条件与终止控制 |
| 输入、Context 和接口都清楚，路径仍反复波动 | Prompt、模型能力或决策边界 |

工具返回值的业务语义也属于排查范围。`ok` 可能只表示文件已接收，不代表校验通过或平台运行已经创建；空结果可能表示确实没有数据，也可能来自错误查询、权限不足或索引延迟。Trace 要保留结构化状态和错误语义，不能只记录 HTTP 状态或一段成功文本。

这张表用于归因，不是要求从上到下逐项检查。Trace 已经把故障缩小到一个转折点，工程师只需检查该位置附近的系统边界。

## 对照重放

第一次偏离提供根因假设，还需要通过对照重放验证。若怀疑 Skill 的默认模板错误，可以保持用户请求、模型、工具响应和平台环境不变，只修正这一条规则，再重复运行失败任务。

若 Agent 随后稳定选择 `enterprise-saml`，假设得到支持；若仍然选错，就继续检查 Skill 触发、工具描述或模型决策。由于模型输出存在波动，重要问题通常需要重复运行，不能用单次成功证明修复有效。

外部依赖难以复现时，可以保存并重放 Tool Result、使用固定 Fixture、测试环境或故障注入。实验一次只改变一个怀疑因素；同时修改 Prompt、Skill 和工具即使让任务成功，也无法说明哪项修改真正解决了问题。

## 偶发故障

同一个样本有时成功、有时失败时，需要比较成功与失败 Trace 最早从哪里分叉。若成功运行加载 `enterprise` Skill，失败运行加载 `generic` Skill，应检查 Skill 触发条件；若两条轨迹直到工具选择才不同，应检查工具描述、模型版本和决策稳定性；若调用路径一致而结果不同，则进一步比较外部服务、权限、数据版本和延迟。

这种比较要求两条 Trace 使用相同的任务标识体系和版本字段。只有最终结果而没有中间 Span，或者只保存当前配置而没有运行时版本，都无法定位偶发问题。

## 数据安全与保留

生产 Trace 不应无条件保存全部原文。用户数据、凭据和业务返回需要脱敏；大段内容可以保存资源 ID、内容哈希和受控引用；访问权限按用户、租户和角色隔离；不同记录根据审计和排障需求设置保留期限。

与此同时，调用关系、关键版本、工具参数、错误语义、权限决定和状态变化要保持结构化。过度裁剪会让 Trace 失去重现能力，过度保存则增加隐私、合规和存储风险。Trace 设计需要在可重现性与数据最小化之间取舍。

Agent 故障定位可以概括为三个步骤：**用 Trace 还原跨组件执行链，找到第一次偏离，再通过对照重放验证根因。**这样才能把“Agent 表现不好”转化为可以定位、修复和回归验证的软件问题。
