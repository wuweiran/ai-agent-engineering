# 如何根据评测结果改进 Agent

数据生成 Agent 的 Trace 已经确认：模型正确识别了企业 SAML 登录任务，但 Skill 错误地要求默认使用基础登录模板。根因找到了，工程师仍不能简单增加一句更强硬的 Prompt，然后看失败案例是否通过。

一次修复要回答三个问题：改哪一层最接近根因，怎样证明没有只迎合当前案例，以及如何把新版本安全带到真实用户面前。

**Agent 改进不是不断给模型增加指令，而是在正确的系统层做最小改动，再用回归评测和生产观察验证整体效果。**

```text
已确认的根因
  ↓
选择正确的改动层
  ↓
失败样本验证修复
  ↓
完整回归检查副作用
  ↓
小范围发布并观察
  ↓
新问题回到评测集
```

## 改动应该贴近第一次偏离

SAML 案例的第一次偏离来自 Skill，最直接的修复是更正 Skill 的模板规则和适用条件。若改成升级模型或在系统 Prompt 中重复说明 SAML，不仅扩大影响范围，也掩盖了真正错误。

不同根因对应不同改动层：

| 根因 | 更直接的改动 |
| --- | --- |
| 用户请求缺少必要字段 | 改产品输入或增加追问 |
| 当前决策没有取得关键事实 | 改上下文装配和检索 |
| 任务方法或默认步骤错误 | 改 Skill |
| 工具经常误选或参数混淆 | 重划工具边界、名称和 Schema |
| 工具执行或返回语义错误 | 改 MCP 服务或平台 API |
| 任务重复、无法恢复或提前结束 | 改状态与执行控制 |
| 信息和接口都清楚，模型仍不稳定 | 改提示方式、模型，或收回决策权 |

这并不意味着每次只能修改一个文件，而是要有一个明确的主要假设：哪项改动会消除第一次偏离，为什么。

**修复越靠近根因，影响范围通常越清楚，也越容易验证。**

## 能固化的判断，不必继续依赖模型

有些评测失败会暴露：原本交给模型的判断其实已经稳定成业务规则。

如果模板选择可以准确表示为：

```text
task_type = enterprise_saml
  → template = enterprise-saml
```

那么由程序映射通常比不断增加示例更可靠。模型仍然负责从自然语言识别 `task_type`，程序负责把已确定的类型映射到模板。

```text
用户描述
  ↓
模型识别结构化任务类型
  ↓
程序选择确定模板
  ↓
Agent 继续生成与修复数据
```

这样会在 Agent 内重新划分责任：开放理解仍由模型完成，已经稳定的规则则交还程序。

相反，如果用户表达和模板组合仍然开放，硬编码大量分支会快速膨胀。这时应改善上下文、Skill 或工具契约，而不是过早固化。

## 先修复失败样本，再检查相邻任务

修正 Skill 后，第一步是重跑原始 SAML 样本，确认第一次偏离已经消失。但单个案例通过只能说明局部修复有效。

紧接着要运行相邻任务：基础登录是否仍选择基础模板，其他企业认证方式是否被错误归为 SAML，信息不足时是否仍会追问。相邻样本最容易发现过度修复。

```text
原失败样本
  ↓ 证明问题已修复
同类与边界样本
  ↓ 证明规则没有过度扩张
完整回归集
  ↓ 证明其他能力没有不可接受的退化
```

常见的过拟合修复，是为了一个样本加入关键词规则，结果所有包含“enterprise”的任务都选择 SAML。评测不仅要问“原问题是否通过”，还要问“新规则在哪些输入上不应该生效”。

## 回归不只比较成功率

新版本可能提高任务成功率，却使用更多步骤和 Token；也可能减少普通失败，却引入一次跨项目上传。发布判断需要同时比较：

- 哪些任务改善、保持或退化；
- 是否新增越权、重复写入等严重失败；
- 首个错误环节的分布是否改变；
- 步骤数、延迟、成本和人工介入是否值得。

指标之间有优先级。安全边界和不可逆副作用通常是发布门槛，不能由平均成功率抵消。性能和成本则要结合业务价值判断：任务成功明显改善，增加少量推理步骤可能值得；只改善罕见低价值任务，却让所有请求翻倍变慢，通常不值得。

**回归评测验证的不是“分数有没有上涨”，而是收益是否大于新增的风险和代价。**

## 离线通过后，逐步扩大真实影响

评测环境无法覆盖真实数据规模、用户表达和平台波动。涉及上传和运行的改动，不应从离线通过直接跳到全量发布。

可以逐步扩大影响：

```text
离线回归通过
  ↓
测试项目验证真实 MCP 和平台接口
  ↓
内部用户或指定任务类型
  ↓
小比例真实流量
  ↓
扩大范围，或触发回滚
```

不同产品不一定都使用相同发布技术。关键是限制一次改动的影响范围，并能够比较新旧版本。只生成草稿的 Agent 可以较快扩大；自动上传、覆盖数据或启动高成本运行的 Agent 应更谨慎，必要时先保持人工确认。

发布前还要定义回滚条件。例如，出现跨项目写入立即停止；平台运行成功率持续下降则回滚；成本增长但结果同步改善时，可以继续观察。

## 生产观察要连接用户结果和系统行为

上线后只看 HTTP 成功率不够。数据 Agent 的 MCP 调用全部返回 200，用户仍可能反复修改生成结果，或在平台运行失败后放弃任务。

生产观察应连接三个层次：

```text
用户结果
是否接受数据、是否完成运行
      ↓
Agent 行为
模板选择、重试、转人工和结束原因
      ↓
系统运行
工具失败、延迟、Token 和平台状态
```

当用户完成率下降时，团队可以继续查看是模板误选、平台故障，还是延迟增加导致放弃。没有这种关联，生产指标只能说明“哪里看起来变了”，无法进入下一轮改进。

## 新失败要经过确认再进入评测集

生产中的一次失败可能来自临时平台故障、错误用户输入，也可能暴露评测从未覆盖的新任务。团队需要先确认预期结果、数据权限和第一个错误环节，再决定是否加入回归集。

有价值的生产案例应被转化为最小、可复现和脱敏的样本。无关的用户文本和偶发环境噪声应删除，真正触发缺陷的条件则保留。

```text
生产异常
  ↓
确认是否属于产品承诺范围
  ↓
Trace 定位根因
  ↓
形成最小评测样本
  ↓
修复、回归和再次发布
```

这个循环让评测集逐渐接近真实任务分布，也让 Agent 改进从临时救火变成可积累的工程过程。

## 四步复述

1. **根据 Trace 确认的第一次偏离，在最接近根因的系统层修改。**
2. **能稳定写成规则的判断交给程序，仍然开放的判断留给模型。**
3. **从失败样本扩展到相邻任务和完整回归，比较收益、风险与代价。**
4. **逐步发布，将确认的新失败转化为下一轮评测样本。**

Evaluation 做好以后，AI Agent 工程师的日常工作正是这个循环：诊断、修改、回归、发布和生产反馈。Agent 不是靠一次设计达到稳定，而是在可验证的改动中逐步收紧错误空间。
