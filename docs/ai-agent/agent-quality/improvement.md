# 如何根据评测结果改进 Agent

Trace 已经确认：模型正确识别了企业 SAML 登录任务，Skill 却要求默认使用基础登录模板。根因找到后，工程师还不能简单增加一句更强硬的 Prompt，再看当前案例是否通过。

一次修复要做三件事：在最接近根因的位置修改，证明没有只迎合当前样本，再安全地把新版本交给真实用户。

## 改动要贴近第一次偏离

SAML 案例的错误来自 Skill，直接修复是更正模板规则和适用条件。升级模型或在系统 Prompt 中重复说明 SAML，会扩大影响范围，也会掩盖真正问题。

不同位置的错误，对应不同改动：用户请求缺少必要字段，应修改产品输入或增加追问；当前决策没取得关键事实，应调整上下文；工具经常误选，应重写名称、描述或 Schema；MCP 执行错误，应修复服务或平台 API；任务重复和提前结束，则检查状态与执行控制。

只有信息、Skill 和接口都清楚，模型仍然不稳定时，才主要考虑提示方式、模型能力，或者收回这项决策权。

**修复靠近根因，影响范围才更清楚。**

## 稳定规则可以交回程序

有些失败会暴露：原本交给模型的判断已经稳定成业务规则。

若 `task_type = enterprise_saml` 时，模板总是 `enterprise-saml`，可以让模型只负责从自然语言识别任务类型，程序负责映射模板。这样既保留开放理解，也减少稳定步骤的随机性。

如果用户表达和模板组合仍然开放，硬编码大量分支会快速膨胀。这时更适合改善上下文、Skill 或工具契约。

是否收回模型控制权，要看规则能否稳定表达，而不是看到一次失败就把整个流程写死。

## 一个样本通过还不够

修正 Skill 后，先重跑原始 SAML 样本，确认第一次偏离消失。接着检查相邻任务：基础登录是否仍使用基础模板，其他企业认证有没有被误判，信息不足时是否仍会追问。

这一步很容易发现过度修复。为了一个样本增加关键词规则，可能让所有包含“enterprise”的任务都选择 SAML。

相邻任务通过后，再运行完整回归集。除了任务成功，还要看有没有新增越权、重复写入等严重问题，以及步骤、延迟和成本是否出现不值得的增长。

平均分不能抵消安全边界。总体成功率上升，却新增一次跨项目上传，新版本仍然不应发布。

## 离线通过后逐步放量

评测环境覆盖不了全部真实数据和平台波动。涉及上传与运行的改动，可以先在测试项目验证真实接口，再给内部用户或指定任务类型使用，然后开放少量真实流量。

只生成草稿的 Agent 可以更快扩大范围；自动覆盖数据或启动高成本任务的 Agent，需要更谨慎，必要时继续保留人工确认。

发布前还要定好回滚条件。跨项目写入立即停止，平台运行成功率持续下降则回滚；成本增加但任务质量同步改善，可以结合业务价值继续观察。

## 生产观察要连到用户结果

MCP 调用全部返回 200，用户仍可能反复修改生成结果，或在平台运行失败后放弃任务。只看接口成功率不够。

团队要把三个层面关联起来：用户是否接受数据并完成运行，Agent 选择了什么模板、怎样重试和结束，底层工具、模型与平台发生了什么。

用户完成率下降后，才能继续判断是模板误选、平台故障，还是等待时间太长。生产指标因此不仅用于报警，也要能够回到具体任务和 Trace。

## 新失败变成下一轮回归样本

生产失败可能来自临时平台故障、错误用户输入，也可能暴露从未覆盖的新场景。团队先确认它是否属于产品承诺范围，再用 Trace 找到第一次偏离。

值得保留的案例要脱敏，并去除与缺陷无关的偶然细节，形成能够重复运行的最小样本。修复后，原问题通过，完整评测集没有不可接受退化，才完成一次改进。

AI Agent 工程师的日常质量工作，就是反复完成这条链：确认根因，在正确层修改，检查相邻任务和完整回归，逐步发布，再把生产中的新问题带回评测集（[怎样证明一次 Agent 优化真的有效？](../../interview/ai-agent/#agent-optimization-evidence)）。
