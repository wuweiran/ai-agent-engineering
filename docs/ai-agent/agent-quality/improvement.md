# Agent 质量改进

一次 [Agent 评测]({{ site.baseurl }}/docs/ai-agent/agent-quality/evaluation/)发现多个失败样本：企业 SAML 任务选错模板，信息不足时没有追问，上传超时后又重复提交。工程师不能只修改 Prompt，再看这些样本是否变成成功。最终结果发生变化，不代表根因已经消失，也不能证明其他任务没有退化。

Agent 质量改进是一条基于证据的工程闭环：**固定失败样本和基线，通过 Trace 定位第一次偏离，在负责该问题的系统层修复，再依次完成定向验证、相邻回归、完整回归和逐步发布。**

```text
评测发现失败样本
→ 固定输入、环境、版本和预期结果
→ 通过 Trace 定位第一次偏离
→ 用对照实验确认根因
→ 在对应系统层实施修复
→ 重跑原样本并验证修复机制
→ 运行相邻任务和完整回归集
→ 逐步发布并观察生产指标
→ 将新的生产失败补入评测集
```

## 失败样本与基线

质量改进从可重复的失败证据开始。每个失败样本至少要保存用户目标、环境与 Fixture、不可违反的约束、预期结果、实际结果、Agent 配置版本和执行 Trace。模型、Prompt、Context 装配、Skill、工具 Schema 和业务数据版本都会影响执行路径，缺少这些信息就无法重现当时的条件。

工程师首先确认失败属于哪一类：

- 产品承诺范围内的真实缺陷，需要进入修复流程；
- 外部平台临时不可用，需要检查重试、等待和降级是否正确；
- 评测环境或 Fixture 错误，需要先修复评测基础设施；
- 产品尚未支持的任务，需要明确拒绝或调整产品范围，不能把它混入成功率。

对于偶发失败，还要在同一条件下重复运行，建立修复前的成功率、失败类型和成本基线。只有知道问题出现的条件和频率，后续才能判断变化来自修复，还是模型输出的随机波动。

## 根因定位

根因定位依赖 [Agent Trace 与故障定位]({{ site.baseurl }}/docs/ai-agent/agent-quality/trace-debugging/)。工程师沿执行链寻找**第一次输入仍然正确、输出开始错误的位置**，而不是把最终错误直接归给模型。

企业 SAML 样本可能经历以下链路：

```text
用户要求企业 SAML 登录数据
→ 模型正确识别 task_type = enterprise_login
→ Skill 指定默认模板 basic-login
→ Agent 读取并生成基础登录数据
→ 平台报告缺少 saml_certificate
```

平台报错是最终症状，Skill 给出错误模板才是第一次偏离。此时升级模型、放宽平台校验或在 System Prompt 中重复强调 SAML，都没有直接处理根因。

第一次偏离只能形成根因假设，还需要对照实验确认。可以固定用户请求、模型、工具响应和平台环境，只修正 Skill 中的模板规则，再重复运行该样本。若错误稳定消失，假设得到支持；若仍然失败，就继续检查 Skill 触发条件、工具描述或模型决策。一次同时改动 Prompt、Skill 和工具，即使结果成功，也无法判断哪项改动真正有效。

## 修复位置

修复应落在最接近根因、能够直接负责该行为的系统层。这样影响范围更清楚，也更容易设计回归测试。

| 第一次偏离 | 主要修复位置 |
| --- | --- |
| 用户请求缺少必要信息，Agent 却继续执行 | 产品输入、追问机制或任务入口 |
| 模型没有取得关键事实，或读取了过期内容 | Context 装配、检索过滤或数据版本 |
| 领域方法、步骤或适用条件错误 | Skill |
| 工具经常被误选或参数含义不清 | 工具名称、描述和 Schema |
| Tool Call 正确，执行或返回语义错误 | 工具处理器、MCP 服务或业务 API |
| 写操作重复、结果未知时错误重试 | Runtime 状态、幂等和恢复逻辑 |
| 证据不足却宣布完成 | 完成条件与终止控制 |
| 所有输入和接口都清楚，模型仍然不稳定 | Prompt、模型能力或决策边界 |

有些失败会暴露原本交给模型的判断已经形成稳定规则。若 `task_type = enterprise_saml` 时模板总是 `enterprise-saml`，程序可以负责确定映射，模型只负责从开放表达中识别任务类型。确定性规则交回程序后，更容易测试，也不会受模型波动影响。

如果任务与模板关系仍然开放，硬编码大量分支会让系统难以维护，此时应继续改善 Context、Skill 或工具契约。是否收回模型控制权，取决于规则能否稳定表达，不取决于是否出现过一次失败。

## 定向验证

修复完成后，首先重跑原失败样本。这一步称为定向验证，目标不只是看到最终结果变成成功，还要确认：

- 原来的第一次偏离已经消失；
- 修复后的步骤按照预期机制工作；
- 最终任务满足完整交付条件；
- 权限、确认、幂等和终止等边界没有被绕过；
- 延迟、步骤数和成本没有出现明显异常。

对于模型参与的判断，单次通过通常不够。应在相同条件下重复运行，比较修复前后的成功率和失败分布。若原样本偶尔通过、偶尔仍以相同方式失败，说明修复只降低了发生概率，还没有稳定消除根因。

定向验证通过只能证明原问题得到处理，不能证明这次修改可以发布。针对单个样本增加关键词或强制指令，很容易形成过度修复。

## 分层回归

回归测试按照**原失败样本、相邻任务、完整评测集**逐层扩大范围：

| 层次 | 主要目的 | SAML 修复示例 |
| --- | --- | --- |
| 原失败样本 | 证明已定位的问题得到修复 | 企业 SAML 任务稳定选择正确模板 |
| 相邻任务 | 发现过度修复和边界误判 | 基础登录仍使用基础模板，其他企业认证不会被误判为 SAML |
| 完整评测集 | 检查系统整体质量和严重回归 | 正常、信息不足、无权限、工具故障和结果未知等任务全部重跑 |

相邻任务围绕修复规则的适用边界设计。若为了 SAML 样本增加“包含 enterprise 就选择 SAML”的规则，其他企业认证任务可能被误伤；这类问题通常不会由原失败样本发现。

完整回归要比较三个层次：任务结果是否改善，关键过程是否出现越权、重复写入、跳过确认和提前结束，运行效率是否发生不合理变化。评测环境、Fixture 和版本应尽量一致，否则无法判断差异来自代码修改还是外部条件。

平均分用于观察整体趋势，严重失败用于设置发布门槛。总体成功率提高，若新增一次跨项目写入、错误退款或虚假完成，新版本仍然不应发布。质量改进必须同时满足：目标问题得到修复，其他任务没有不可接受退化，严重风险继续受控。

## 逐步发布

离线评测无法覆盖真实权限、外部服务波动和长尾输入。完整回归通过后，新版本还要按风险逐步发布：先在固定测试环境验证真实接口，再开放给内部用户或指定任务类型，最后逐步扩大真实流量。

发布前应明确观察指标和回滚条件。线上观察仍然分为三个层次：

- **任务结果**：成功交付率、用户接受率、人工接管率、重试率和放弃率；
- **关键过程**：工具与参数正确率、权限拒绝、重复副作用、故障恢复和虚假完成；
- **运行效率**：首个有效反馈时间、P95/P99 完整耗时、步骤数、Token 和每个成功任务的成本。

跨项目写入、错误退款等严重失败应立即停止放量或回滚。成功率持续下降也需要回滚。成本增加是否可以接受，则要结合任务质量和业务价值判断，不能只看单次调用价格。

线上指标必须能够关联到具体任务、Agent 版本和 Trace。只有这样，指标异常才能继续定位到模板误选、平台故障、等待过久或终止条件错误，而不只是得到一个下降的百分比。

## 生产反馈闭环

生产失败可能来自临时故障、错误输入，也可能暴露评测集没有覆盖的新场景。团队先确认它是否属于产品承诺范围，再通过 Trace 定位第一次偏离。值得长期保留的问题需要脱敏并去除无关偶然因素，形成可重复运行的最小样本。

新样本进入评测集后，后续所有相关版本都要接受同一项回归检查。这样，生产问题不只被临时修复，还会变成防止问题再次出现的长期质量证据。

一次完整的 Agent 质量改进应形成以下证据：失败样本能够稳定重现，第一次偏离和根因得到验证，修改落在正确系统层，原样本通过定向验证，相邻任务与完整评测集没有不可接受退化，线上指标也支持继续发布。这就是[证明一次 Agent 优化有效]({{ site.baseurl }}/docs/interview/ai-agent/quality-production/#agent-optimization-evidence)所需要的证据链。
