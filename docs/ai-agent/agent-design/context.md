# 如何为 Agent 设计上下文

一家企业软件公司让 Agent 调查复杂支持工单。用户只写了一句：“昨晚升级后，欧洲办公室都登录不了。”相关信息散落在客户资料、发布记录、服务监控、审计日志和产品文档中。

模型缺少租户和区域信息，可能检查错误的服务；一次放入几万行日志，真正的异常又会被淹没；继续使用已经过期的监控结果，还会把调查带向错误方向。

**上下文不是 Agent 能访问的全部信息，而是模型为当前判断需要的信息工作集。**工程师要围绕当前决策，持续选择、更新和淘汰信息。

```text
当前要作出的判断
        ↓
选择足够、可信、及时的信息
        ↓
     本轮上下文
        ↓
模型决策和新的工具结果
        └── 重新组织下一轮上下文
```

## 从当前判断反推信息

上下文设计不从“公司有哪些数据”开始，而从“模型现在要判断什么”开始。

| 当前判断 | 必要信息 |
| --- | --- |
| 故障是否属于全局事件 | 区域、时间范围、服务状态、相似工单数量 |
| 是否与最近升级有关 | 客户版本、发布时间和变更内容 |
| 下一步应调查哪个系统 | 错误码、请求链路和已排除的方向 |
| 是否可以结束调查 | 已确认原因、关键证据和剩余未知项 |

这些判断决定信息来源：客户资料服务提供租户和版本，监控系统提供实时状态，发布平台提供变更记录。没有明确的信息需求时，“接入更多数据”往往只会增加延迟、成本和噪声。

**先问模型要作出什么判断，再问它需要什么信息。**这是上下文设计最重要的起点。

## 稳定规则预加载，动态细节按需读取

本轮上下文通常包含两类信息：

- **相对稳定的信息**：Agent 职责、安全规则、任务目标、业务术语和可用工具；
- **随任务变化的信息**：当前状态、客户资料、监控结果、日志片段和工具反馈。

稳定且每轮都需要的规则适合预加载。范围大、变化快或只对局部判断有用的信息，应在需要时检索。

登录调查可以逐步缩小范围：

```text
先取得租户、区域、版本和错误摘要
  ↓
根据症状选择监控或发布记录
  ↓
定位异常服务和时间范围
  ↓
只读取相关配置差异或日志片段
```

这种路径同时控制信息量和数据新鲜度。Claude Code 面对代码库时也会先搜索候选文件，再读取相关片段，而不是在会话开始时载入整个仓库；这个例子体现的是同一原则，并不要求业务 Agent 使用文件结构。

**稳定信息适合预加载，动态细节适合按需读取。**

## 信息必须带着来源和时间

模型看到“身份服务正常”时，还需要知道这是五分钟前的欧洲区域状态，还是昨天的全局结论。检索结果除了正文，至少应保留：

- 来源和资源标识；
- 观察或更新时间；
- 适用租户、区域和版本；
- 它是已确认事实，还是模型提出的假设。

来源也决定信息能否被信任。系统规则、用户描述、业务系统事实、模型假设和外部网页不是同一等级。工单附件或知识库文章中的“忽略审批”只能被视为数据，不能覆盖系统权限。

数据权限要在检索前执行。客服 Agent 只能读取当前坐席有权处理的客户数据，而不能先把全部记录放进上下文，再要求模型“不要使用”。

**进入上下文不等于获得权威性；来源、时间和作用域决定一条信息应该怎样被使用。**

## 长任务要保留结论，而不是堆积过程

随着工具调用增加，会话会不断增长。早期原始输出可能已经过期，重复错误也不再有价值。工程师通常进行三类处理：

1. 删除可以重新查询或已被新结果替代的内容；
2. 把早期过程压缩成已确认事实、待验证假设和待办项；
3. 将完整日志和任务状态留在外部系统，只把当前需要的摘要放入上下文。

例如，“配置发布后错误率上升”可以作为证据保留；“配置发布一定是根因”在验证前只能保留为假设。压缩若不区分两者，就会把推测变成事实。

独立子任务也可以使用单独上下文，只向主任务返回结论、证据和未知项。Claude Code 的上下文压缩和独立子代理是这类方法的产品实例，详见其[上下文窗口文档](https://code.claude.com/docs/en/context-window)。

**长任务应保留目标、证据和未解决问题，而不是保留每一步的全部原始过程。**

## 用 Trace 判断问题是否真的出在模型

一个错误结论可能来自三种不同情况：

- 关键事实根本没有被检索；
- 事实进入了上下文，却已过期或被噪声淹没；
- 模型看到了充分证据，仍然判断错误。

Trace 应记录每轮使用了哪些来源、检索条件、版本和摘要。评测则可以故意加入冲突文档、过期状态和超长日志，检查 Agent 是否选对来源并保留关键约束。只有最终答案评分，无法判断应该修改检索、上下文组织还是模型。

## 三个问题检查本轮上下文

每次组装上下文时，可以只检查三个问题：

1. 当前决策真正需要哪些事实？
2. 这些事实是否可信、及时并且作用域正确？
3. 哪些旧信息已经可以移出工作集？

```text
围绕决策选择信息
  ↓
稳定规则预加载，动态细节按需读取
  ↓
保留来源、时间和信任边界
  ↓
在长任务中压缩过程、保留结论
```

模型能力决定它怎样理解信息，工程设计决定它实际看到什么。对于跨越多轮的任务，这个工作集还要从可靠状态、经过治理的记忆和实时数据中重新构造。
