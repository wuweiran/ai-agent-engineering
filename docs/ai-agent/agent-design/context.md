# Agent Context

一家企业软件公司让 Agent 调查复杂工单。用户只写了一句：“昨晚升级后，欧洲办公室都登录不了。”相关信息散落在客户资料、发布记录、服务监控、审计日志和产品文档中。

缺少租户和区域信息，Agent 会检查错误的服务；一次放入几万行日志，真正异常又可能被淹没；继续使用过期监控结果，还会把调查带错方向。

**上下文是模型为当前判断需要的信息工作集，不是系统能够访问的全部资料。**

## 当前判断决定所需信息

上下文设计从当前问题开始。登录故障调查初期，Agent 先用区域、时间范围和服务状态判断影响范围；发现故障接近发布时间后，再读取客户版本和变更记录；定位到具体系统后，Context 转向错误码、请求链路和已排除方向。当前判断发生变化，所需信息和来源也随之变化。若模型还没有明确要判断什么，“接入更多数据”往往只会增加延迟和噪声。

**先确定当前决策，再准备支持它的信息。**

## 稳定规则和动态细节分开处理

Agent 的职责、安全规则、任务目标和业务术语变化较慢，而且经常需要，可以在任务开始时加载。其中，[System Prompt]({{ site.baseurl }}/docs/interview/ai-agent/llm/#system-prompt) 通常承载模型身份、稳定行为边界和输出要求；它是 Context 的一部分，能够影响模型，却不能代替权限和业务校验。客户状态、监控结果和日志片段变化快、范围大，更适合在需要时查询。

登录调查可以先取得租户、区域、版本和错误摘要。若症状指向服务故障，再查询监控；若故障时间靠近发布，再读取发布记录；最后只读取相关时间段的配置差异或日志片段。

Claude Code 处理代码库也采用相似方式：先搜索候选文件，再读取相关片段，不会在会话开始时载入整个仓库。这个例子说明的是按需获取细节，并不要求业务 Agent 使用同样的文件结构。

**稳定信息适合预加载，动态细节适合按需读取。**

## 信息的来源、时效与范围

“身份服务正常”只有正文是不够的。模型还要知道，这是五分钟前的欧洲区域状态，还是昨天的全局结论。

检索结果应保留来源、观察时间和适用范围。系统也要区分已确认事实与模型假设。用户描述、业务数据库、外部网页和系统规则的权威性不同，不能因为它们同时进入上下文就被同等对待。

工单附件或网页中的“忽略审批”只是一段数据，不能覆盖系统权限。客服 Agent 的数据权限也应在检索前检查，不能先把所有客户记录放进上下文，再要求模型不要使用。

**进入上下文不代表获得权威性。模型必须看得出信息从哪里来、什么时候有效、适用于谁。**

## 长任务保留结论，不堆积过程

随着工具调用增加，早期原始输出会占满会话。有些结果已经被新查询替代，有些详细日志可以随时重新读取。

工程师可以删除过时结果，把早期过程压缩成已确认事实、待验证假设和待办项，并把完整日志留在外部系统。例如，“配置发布后错误率上升”可以保留为证据；“配置发布一定是根因”在验证前仍是猜测。

独立子任务也可以使用单独上下文，只向主任务返回结论、证据和未知项。Claude Code 的上下文压缩和独立子代理是这类方法的产品实例，详见其[上下文窗口文档](https://code.claude.com/docs/en/context-window)。

长任务真正需要留下的是目标、证据和未解决的问题，不是每一步的全部原文。

## Context 质量取决于选择与排列

Agent 得出错误结论，可能因为关键事实从未检索，也可能因为结果过期、排序靠后或被噪声淹没；若这些都没有问题，才说明模型在充分证据下仍然判断错误。

检索系统先通过 Rerank 判断候选材料与问题的相关性，再选择真正进入窗口的内容。[Lost in the Middle]({{ site.baseurl }}/docs/interview/ai-agent/context-memory/#lost-in-the-middle) 说明长 Context 中的信息利用可能受位置影响，因此最重要的目标、约束和证据不应埋在大量次要材料中。重排能缓解位置问题，却不能替代准确召回与去噪。

对检索材料进行二次筛选时，可以让模型判断内容是否相关、冲突或过期，再生成回答。这类[自反馈清洗]({{ site.baseurl }}/docs/interview/ai-agent/context-memory/#context-noise-cleaning)与 Self-Refine 思路相近，但同一个模型可能重复自己的误判；高风险信息仍要依赖来源、规则和独立验证，不能让模型自行删除不符合预期的反面证据。

Trace 应记录每轮用了哪些来源、检索条件、排序、版本和摘要。评测可以加入冲突文档、过期状态和超长日志，检查 Agent 是否选对来源并保留关键约束。

## Context Engineering 贯穿信息生命周期

[Agent Context Engineering]({{ site.baseurl }}/docs/interview/ai-agent/context-memory/#agent-context-engineering-strategy)不是单一的 RAG 或摘要技术，而是管理“获取、选择、组织、压缩和隔离”整个信息流：

```text
获取：RAG、记忆检索、工具查询
选择：权限与时效过滤、Rerank、去重
组织：规则、目标、任务状态、证据和近期历史分层组装
压缩：滑动窗口、摘要、旧工具结果裁剪
隔离：子 Agent 使用独立 Context，只返回结论与证据
```

一次有效策略要先定义当前判断和 Token 预算，再按内容性质决定保留方式。稳定规则适合复用固定前缀；目标、约束和待办进入结构化状态；近期交互保留原文；早期过程可以摘要；动态事实重新查询；大型文档和日志留在外部系统。评测则用长任务、冲突资料、过期结果和高噪声输入，检查任务成功、关键信息保留、延迟与成本。

检查一次 Context 时，可以问三个问题：当前判断缺什么事实？已有事实是否可信且及时？哪些旧信息或噪声已经可以移走？

模型能力决定它怎样理解信息，工程设计决定它实际看到什么。跨越多轮的任务，还要从可靠状态、长期记忆和实时数据中重新构造这个工作集。
