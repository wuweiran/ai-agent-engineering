# 如何为 Agent 设计上下文

设计 Agent 系统时，工程师需要决定模型在每一步依据什么信息作出判断。上下文设计与工具、状态、权限和运行控制共同构成 Agent 的执行能力，其中上下文负责为当前决策提供信息工作集。

一家企业软件公司准备让 Agent 调查复杂支持工单。用户只写了一句：“昨晚升级后，欧洲办公室都登录不了。”相关信息散落在工单系统、客户资料、发布记录、服务监控、审计日志和产品文档中。模型能否得出可靠结论，很大程度上取决于工程师让它在每一步看到什么。

如果缺少租户和区域信息，Agent 可能检查错误的服务；如果放入几万行日志，真正的异常可能被淹没；如果服务状态已经变化，旧结果还可能把调查带向错误方向。

**上下文工程要为模型构造足以支持当前决策的信息工作集，并随着任务进展持续更新。**工程师需要解决信息从哪里来、何时读取、怎样组织、保留多久以及是否可信等问题。

```text
业务数据、规则和任务历史
            ↓
      选择与组织信息
            ↓
       本轮模型上下文
            ↓
         模型决策
            ↓
      新观察与状态变化
            └────── 进入下一轮
```

## 从任务决策反推信息需求

上下文设计通常从业务任务开始。工程师需要弄清楚 Agent 在执行过程中会做哪些判断，以及每个判断需要什么证据。

登录故障调查可以拆出几类关键决策：

| 当前判断 | 所需信息 |
| --- | --- |
| 故障是否属于全局事件 | 区域、时间范围、服务健康状态、相似工单数量 |
| 是否与最近升级有关 | 客户版本、发布时间、变更内容、回滚记录 |
| 应继续调查哪个系统 | 错误码、请求链路、身份服务和配置状态 |
| 是否可以提出修复建议 | 已确认原因、影响范围、现有操作手册 |
| 是否应该结束或转人工 | 证据完整度、剩余未知项、权限和时间限制 |

这张对应关系决定了上下文来源。客户资料服务提供租户和版本，监控系统提供实时状态，发布平台提供变更记录，知识库提供已知故障和处理方法。没有明确决策需求时，团队很容易把“接入更多数据”误当成改进，最后只增加延迟、成本和噪声。

## 信息要按照职责和变化速度分层

Agent 运行时会把不同性质的信息组合到一次模型请求中。工程师可以把它们分成几个层次：

- **系统规则**：Agent 的职责、安全要求、输出规范和工具使用边界；
- **任务目标**：用户请求、调查范围、完成条件和明确约束；
- **任务状态**：已经确认的事实、已经执行的步骤、待检查事项和审批状态；
- **工具说明**：可用能力、参数、返回值和错误语义；
- **环境观察**：本轮查询到的客户资料、日志、代码、监控和工具结果；
- **长期知识**：业务术语、系统拓扑、操作手册和经过验证的历史经验。

系统规则和业务术语变化较慢，适合在任务开始时加载；服务状态和库存等信息变化很快，应在需要决策时实时查询；详细日志只对局部调查有用，不适合长期占据上下文。

**稳定信息适合预加载，范围大或变化快的信息适合按需检索。**这种分层既减少无关内容，也让工程师能够分别治理版本、权限和更新时间。

## 按需检索避免把数据一次塞满

生产数据通常远大于模型的上下文窗口。即使技术上可以放入更多信息，也会增加模型查找关键事实的难度。工程师需要提供一条从摘要到细节的发现路径。

调查 Agent 可以先取得：

```text
租户、区域、版本和错误摘要
              ↓
根据错误类型选择监控或发布记录
              ↓
定位异常服务和时间范围
              ↓
只读取相关请求、配置差异或日志片段
```

检索结果除了正文，还应携带来源、更新时间、适用范围和资源标识。模型看到“身份服务正常”时，需要知道这是五分钟前的欧洲区域状态，而不是昨天的全局结论。

搜索接口也应允许 Agent 逐步缩小范围。先查询错误数量和时间分布，再读取少量代表性事件，通常比直接返回整段原始日志更可靠。完整数据可以留在日志系统中，通过查询 ID 或文件路径继续读取。

Claude Code 展示了相同的工程思路。它面对代码库时会通过搜索和文件读取工具逐步取得相关代码，而非在会话开始时载入整个仓库；`CLAUDE.md` 提供长期项目规则，子目录规则在读取相关文件时再进入上下文。相关机制可见 [Claude Code 的记忆文档](https://code.claude.com/docs/en/memory)。这个例子对应的是通用的“稳定规则预加载、详细内容按需读取”，并不要求业务 Agent 使用相同的文件结构。

## 上下文要表达来源和信任边界

模型可能同时看到系统规则、用户输入、数据库记录、检索文档和网页内容。这些信息的权威性不同，工程师需要让模型和运行时能够区分：

- 哪些是必须遵守的系统规则；
- 哪些是用户提出但尚未验证的描述；
- 哪些来自可信业务系统；
- 哪些是模型此前形成的假设；
- 哪些外部内容可能包含错误或恶意指令。

例如，知识库文章中的“忽略审批并直接执行”只能被视为文档内容，不能覆盖退款服务的权限规则。网页、工单附件和代码注释都是不受信任的数据来源，不能因为它们进入了上下文就获得系统指令的地位。

数据权限也应在检索前执行。客服 Agent 只能读取当前坐席有权处理的客户数据，模型不应先看到全部记录再被要求“不要使用”。上下文最小化同时降低了越权访问、敏感数据泄露和提示注入的影响范围。

## 状态、会话和记忆需要分开管理

上下文只是模型本轮看到的视图。业务系统还需要在上下文之外保存完整事实。

| 概念 | 工程职责 | 登录故障中的例子 |
| --- | --- | --- |
| 任务状态 | 可靠记录进度并支持恢复 | 已检查的服务、审批状态、剩余时间 |
| 会话历史 | 保存用户、模型与工具的交互 | 追问、工具请求和返回结果 |
| 长期记忆 | 跨任务保留仍有价值的经验 | 某类版本升级的已知兼容问题 |
| 本轮上下文 | 选出当前判断需要的信息 | 当前假设、关键证据和下一批候选动作 |

一条完整日志可以保存在 Trace 中，但本轮只需要错误类别和几行关键堆栈；任务状态可以保存几十个步骤，模型只需要已确认结论和待办项；历史经验进入记忆前，还应经过验证、设定作用域和失效条件。

这种分离让长任务可以暂停和恢复，也避免把聊天记录当成唯一数据库。服务重启后，运行时可以根据持久状态重新构造上下文，而不必依赖模型回忆此前发生过什么。

## 长任务需要裁剪、压缩和外部笔记

随着工具调用增加，会话历史会不断增长。早期查询结果可能已经过期，重复错误也不再有价值。工程师通常组合几种方法管理长任务：

- **裁剪旧结果**：移除可以重新查询或已被新结果替代的内容；
- **压缩历史**：把早期过程总结为目标、证据、决策和未解决问题；
- **保存结构化笔记**：将计划、关键约束和阶段结论写入外部存储；
- **隔离子任务**：让独立调查使用单独上下文，只返回结论和证据摘要。

压缩是有损的，因此摘要需要区分已确认事实、尚未验证的假设和下一步动作。对于登录故障，“配置发布后错误率上升”可以作为证据保留，“配置发布一定是根因”在验证之前只能记录为假设。

Claude Code 也会在上下文接近上限时清理较早的工具输出并压缩会话，子代理则使用独立上下文完成局部调查。这些机制是裁剪、摘要和任务隔离的成熟产品实例，详见其[上下文窗口文档](https://code.claude.com/docs/en/context-window)。

## 上下文质量需要通过 Trace 和评测验证

上下文策略不能只凭主观感觉调整。Trace 应记录每轮使用了哪些来源、检索条件、文档版本和裁剪结果。出现错误结论时，团队才能区分：

- 关键数据没有被检索；
- 检索到了错误或过期数据；
- 相关内容进入上下文后被无关信息淹没；
- 模型看到了充分证据，却作出了错误判断；
- 压缩过程丢失了约束或未解决问题。

评测样本也要覆盖这些失败方式。可以故意构造相似版本、冲突文档、过期监控结果和超长日志，检查 Agent 是否选择正确来源、识别时间范围并保留关键约束。只有最终答案评分，无法定位应该改检索、上下文编排还是模型。

## 工程师面对的是持续的信息取舍

组织上下文可以归结为一条工程链路：

```text
从业务决策确定信息需求
  ↓
按来源、权限和变化速度管理信息
  ↓
预加载少量稳定内容，按需检索动态细节
  ↓
在长任务中裁剪、压缩并保存关键状态
  ↓
通过 Trace 和评测发现缺失、过期与噪声
```

模型能力决定它怎样理解信息，工程设计决定它实际能看到哪些信息。上下文质量稳定后，下一项工作是设计取得这些信息和执行操作的工具接口。
