# Agent 状态与记忆

登录故障调查已经确认问题只影响欧洲区域，并发现昨晚有一次配置发布。此时发布系统暂时不可用，任务需要等十分钟再继续。

恢复时，系统必须知道哪些结论已有证据、哪个查询需要重试，以及之前有没有执行过外部操作。一个月后再出现相似工单，团队又希望 Agent 能参考已经验证的兼容问题，却不能把上次客户的配置和临时猜测一起带进来。

这两个需求分别对应状态和记忆。**状态服务当前任务，记忆服务未来任务。**（[Agent 的状态和记忆有什么区别？](../../interview/ai-agent/#agent-state-and-memory)）

## 四种信息不要混在一起

| 信息 | 它回答什么 | 登录故障中的例子 |
| --- | --- | --- |
| 会话历史 | 此前发生过什么交互 | 用户消息、工具请求和原始结果 |
| 任务状态 | 当前任务怎样继续 | 已确认事实、待办、审批和重试状态 |
| 本轮上下文 | 模型现在需要看到什么 | 关键事实摘要、当前假设和可用工具 |
| 长期记忆 | 未来任务还值得复用什么 | 已验证的版本兼容问题 |

监控工具返回一份完整响应，它可以进入 Trace；其中“欧洲区域错误率上升”写入任务状态；下一轮只把摘要放进上下文。任务结束后，只有经过验证、未来仍有价值的经验，才可能成为长期记忆。

## 状态要让任务能够继续

任务状态需要记录目标、已确认事实、进度和执行控制。登录调查可以保存为：

```yaml
task_id: INC-2048
status: waiting_dependency
confirmed_facts:
  - fact: 问题只影响 eu-west 区域
    source: monitor-query-817
hypotheses:
  - claim: 配置发布可能影响单点登录
    status: unverified
completed_steps:
  - query_service_health
pending_step: compare_release_configuration
retry:
  tool: get_release_diff
  attempts: 1
  not_before: 2026-07-13T09:35:00Z
```

最重要的不是字段多少，而是已确认事实与待验证假设分开。若恢复时把“配置发布可能相关”当成已确认根因，后面的调查会从错误前提继续。

审批结果、退款申请是否创建、外部任务是否完成，这些权威状态由业务服务、工具结果或人工操作确认。模型可以维护计划和假设，但不能仅凭一句“已经批准”改变权威字段。

## 恢复前验证旧结果

外部工具可能已经执行成功，只是响应丢失。若退款服务创建了申请 `R-3901`，Agent 却只看到超时，系统应该保存“结果未知”和幂等键，恢复后先查询真实结果，不能直接再申请一次。

环境信息还会过期。“租户位于欧洲区域”可以继续使用，“十分钟前身份服务正常”可能需要刷新，“发布系统请求超时”只说明上次没有取得结果。

因此，恢复任务不是重放聊天记录。系统读取目标、事实和待办，确认外部动作的最终状态，刷新已经过期的信息，再从明确位置继续。

**可靠状态使任务可以恢复，同时避免重复副作用。**

## 记忆只留下未来仍值得相信的内容

“版本 4.2.0 至 4.2.2 在特定 SAML 证书格式下可能登录失败”经过事故验证，可能帮助未来调查。它需要附上来源、适用产品和版本，以及何时失效。

```yaml
kind: known_issue
content: 4.2.0 至 4.2.2 在特定 SAML 证书格式下可能登录失败
scope:
  product: enterprise-login
  versions: [4.2.0, 4.2.1, 4.2.2]
source: incident-INC-1880
status: verified
invalid_when: version >= 4.2.3
```

临时服务状态、模型猜测和能够从权威系统实时查询的数据，不适合长期保存。新工单也只有在产品、版本和权限都匹配时，才应取用这条记忆；症状相似仍需检查当前证据。

错误读取影响一次判断，错误写入会污染很多未来任务。用户明确确认的偏好可以保存；从任务中总结的经验可以先进入待审核区；业务规则和安全要求更适合更新正式知识库。软件升级后，旧问题记忆还要及时失效。

**写入记忆时要说明来源、适用范围和失效条件。**（[Agent 的记忆怎样更新和遗忘？](../../interview/ai-agent/#memory-update-forgetting)）

## 上下文只取当前需要的部分

准备比较配置发布时，模型需要看到任务目标、已确认的租户和版本、待验证假设、匹配当前版本的已知问题，以及配置差异工具。它不需要重新阅读全部聊天、无关日志和其他租户的经验。

状态负责保存当前任务的完整进度，记忆负责保存跨任务经验，上下文负责从两者和实时数据中挑选本轮需要的信息。

评测状态设计时，重点检查任务能否暂停恢复、响应丢失后是否避免重复写入、过期环境信息是否刷新。评测记忆时，则检查模型假设会不会被自动保存，过期经验是否仍被使用，不同租户的信息是否隔离。

记住两句话就够了：

- **状态保存当前任务怎样继续。**
- **记忆只保存未来任务仍值得复用的经验。**
