# 如何管理 Agent 的状态与记忆

登录故障调查 Agent 已经确认问题只影响欧洲区域，并发现昨晚有一次配置发布。此时发布系统暂时不可用，任务需要等待十分钟再继续。恢复时，系统必须知道哪些结论已有证据、哪个查询需要重试，以及是否执行过外部操作。

一个月后出现相似工单，团队又希望 Agent 利用此前验证过的兼容问题，但不能把上次客户的配置、临时猜测和过期服务状态带进新任务。

这两个问题分别需要状态和记忆：**状态让同一个任务保持连续，记忆让有价值的经验跨任务复用。**

## 先记住状态和记忆的区别

Agent 系统中有四个容易混淆的概念：

| 概念 | 回答的问题 | 登录故障中的例子 |
| --- | --- | --- |
| 会话历史 | 此前发生过什么交互 | 用户消息、工具请求和原始结果 |
| 任务状态 | 当前任务进行到哪里 | 已确认事实、待办、审批和重试状态 |
| 本轮上下文 | 模型现在需要看到什么 | 关键事实摘要、当前假设和可用工具 |
| 长期记忆 | 未来任务还值得复用什么 | 已验证的版本兼容问题 |

一条监控结果可以完整保存在 Trace 中，关键事实写入任务状态，摘要进入下一轮上下文。只有其中经过验证、未来仍有价值的经验，才可能成为长期记忆。

```text
一次交互产生信息
  ├─ 历史：保留发生过什么
  ├─ 状态：保留任务怎样继续
  ├─ 上下文：选择本轮需要什么
  └─ 记忆：治理后供未来任务复用
```

## 状态要让任务能够继续

任务状态不是聊天摘要，而是可以驱动恢复和执行的结构化记录。最重要的是四类信息：

- **目标**：任务要完成什么，什么情况下结束或转人工；
- **事实**：已经确认什么，证据来自哪里；
- **进度**：完成了哪些步骤，还有哪些待办和阻塞；
- **控制信息**：审批、重试、外部资源、步骤数和截止时间。

登录调查可以保存为：

```yaml
task_id: INC-2048
status: waiting_dependency
confirmed_facts:
  - fact: 问题只影响 eu-west 区域
    source: monitor-query-817
hypotheses:
  - claim: 配置发布可能影响单点登录
    status: unverified
completed_steps:
  - query_service_health
pending_step: compare_release_configuration
retry:
  tool: get_release_diff
  attempts: 1
  not_before: 2026-07-13T09:35:00Z
```

这里最关键的不是字段数量，而是**已确认事实与待验证假设分开**。如果恢复时把“配置发布可能相关”当成已确认根因，后续所有判断都会偏离。

权威状态由业务服务、工具结果或人工操作确认，例如审批是否通过、退款申请是否已经创建。模型可以维护计划和假设，但不能仅凭一句“已经批准”改变权威字段。

## 恢复时既不能重做，也不能盲信旧状态

外部工具可能超时，但动作已经成功。假设退款服务创建了申请 `R-3901`，响应却在网络中丢失，Agent 只看到了超时。系统不能把它简单记成失败并再次退款，而应保存“结果未知”和幂等键，恢复时先查询真实结果。

因此，可靠状态还要记录外部资源 ID、调用结果是否确定，以及恢复点。写操作使用幂等键，状态更新使用版本检查，才能避免服务重启或并发执行导致重复副作用。

暂停后恢复任务，一般需要：

1. 读取目标、事实、待办和控制信息；
2. 确认此前外部动作的最终状态；
3. 重新查询已经过期的环境信息；
4. 从明确的恢复点继续。

“租户位于欧洲区域”可以继续使用，“十分钟前身份服务正常”可能需要刷新，“发布系统请求超时”只说明上次没有取得结果。

**恢复不是重放聊天记录，而是用可靠状态重建当前工作环境。**

## 记忆只保存值得跨任务复用的信息

长期记忆不需要保存 Agent 经历过的一切。它适合保存稳定偏好、已确认问题模式和经过验证的处理经验；临时服务状态、模型猜测以及可以从权威系统实时查询的数据，不应自动成为记忆。

例如：

```yaml
kind: known_issue
content: 4.2.0 至 4.2.2 在特定 SAML 证书格式下可能登录失败
scope:
  product: enterprise-login
  versions: [4.2.0, 4.2.1, 4.2.2]
source: incident-INC-1880
status: verified
invalid_when: version >= 4.2.3
```

一条可用记忆至少需要四类信息：

- **来源**：为什么相信它；
- **作用域**：适用于哪个用户、产品、版本或环境；
- **可信状态**：待验证、已确认还是已废弃；
- **生命周期**：何时创建，何时失效或被替代。

新工单只有在产品、版本和权限都匹配时才使用这条记忆。模型仍需比较当前证据，不能因为症状相似就直接宣布根因。

## 记忆写入比读取更谨慎

错误读取可能影响一次判断，错误写入却会污染许多未来任务。因此，任务中的模型总结不应直接变成长期记忆。

```text
任务产生候选经验
  ↓
检查是否已有权威来源
  ↓
验证证据、作用域和敏感信息
  ↓
自动写入 / 人工确认 / 拒绝保存
  ↓
设置失效条件
```

用户明确确认的偏好可以保存；多次任务总结出的经验可以先进入待审核区；业务规则和安全要求更适合更新正式知识库。软件升级后，旧问题记忆还要失效；用户要求删除数据时，记忆、索引和缓存中的副本都需要清理。

**状态写入关注“任务怎样继续”，记忆写入关注“未来是否仍然值得相信和复用”。**

## 上下文从状态、记忆和实时数据中选择

状态和记忆都不是直接塞给模型的 Prompt。每轮决策前，运行时应根据当前步骤选择相关信息：

```text
任务状态：完整进度和事实
长期记忆：跨任务经验
实时系统：当前客户和服务状态
              ↓ 选择、校验、摘要
本轮上下文：当前判断需要的信息
```

准备比较配置发布时，本轮上下文可以包含：

- 任务目标和禁止生产写入的约束；
- 已确认的租户、区域和版本；
- “配置发布可能相关”这一待验证假设；
- 与当前版本匹配的已知问题；
- 配置差异工具和剩余时间。

它不需要重新包含全部聊天、无关日志和其他租户的经验。状态负责可靠和完整，记忆负责跨任务复用，上下文负责当前相关。

## 用两类测试检查设计

状态主要测试**能否恢复**：任意步骤暂停后能否继续，响应丢失后是否避免重复写入，旧环境信息是否重新查询。

记忆主要测试**是否污染**：模型假设会不会被自动保存，过期经验是否仍被使用，不同租户的记忆是否隔离，删除后是否还有缓存副本。

Trace 要记录状态版本、恢复点、记忆检索条件以及最终进入上下文的内容。这样才能区分任务失败是因为状态丢失、记忆选错、信息过期，还是模型在获得正确信息后仍判断错误。

## 两句话复述

**状态服务当前任务：保存目标、事实、进度和控制信息，使任务可以可靠恢复。**

**记忆服务未来任务：只复用经过验证、具有明确作用域和生命周期的信息。**

运行时再从状态、记忆与实时数据中选择本轮上下文。Agent 因此不必依靠模型“记住”此前发生了什么，也不会让未经验证的历史无限传播。
