# 如何划分模型与程序的决策边界

一家企业软件公司决定用 Agent 调查复杂支持工单。对于“昨晚升级后，欧洲办公室都登录不了”这类问题，调查方向取决于现场证据，很难在开发阶段写成固定步骤。

选择 Agent 只是起点。工程师还要决定：哪些判断交给模型，哪些规则必须由程序执行，哪些决定需要人承担责任。如果把流程全部写死，系统无法适应未知故障；如果把权限、生产变更和完成判断也交给模型，灵活性就会变成风险。

**开放判断交给模型，确定规则留给程序，高风险责任由人承担。**这条分工是决策边界最重要的判断方法。

## 一个任务里有三种责任

登录故障调查可以拆成一组具体环节：理解用户描述、选择调查方向、查询服务、判断证据、提出修复建议，以及执行生产变更。这些环节需要的能力与承担的风险不同。

| 任务环节 | 控制者 | 原因 |
| --- | --- | --- |
| 理解含糊的故障描述 | 模型 | 自然语言表达多样，难以穷举规则 |
| 根据新证据选择调查方向 | 模型 | 路径取决于现场结果 |
| 检查租户权限和调用预算 | 程序 | 规则明确，必须一致执行 |
| 验证结案所需证据是否齐全 | 程序 | 完成条件可以确定性检查 |
| 批准生产回滚 | 人工 | 错误代价高，需要明确责任 |
| 执行回滚并确认结果 | 程序 | 需要权限、幂等和审计 |

同一个动作在不同环境中的归属也可能不同。修改测试环境配置容易恢复，可以允许 Agent 自动执行；修改生产配置则需要审批。边界取决于开放程度和错误代价，而不只是模型“会不会做”。

## 模型处理开放且可验证的判断

模型最适合处理路径难以提前穷举、需要理解非结构化信息的环节。监控显示身份服务整体正常，但错误只集中在使用特定单点登录配置的租户时，模型可以放弃“区域服务中断”的方向，转而比较客户配置和最近发布。

这种选择体现了 Agent 的价值：**模型根据新证据改变下一步。**但开放不等于没有约束。适合交给模型的判断最好还能通过后续证据、测试或人工审查验证，并且一次错误不会立刻造成不可逆损失。

模型形成的原因通常只是待验证假设。它可以提出“配置发布可能改变了证书校验”，但在读取配置差异和错误记录之前，系统不能把这句话保存为已确认根因，更不能直接据此修改生产环境。

## 程序守住必须成立的规则

能够明确写成规则，而且每次都必须执行的逻辑，应留在程序中。最重要的几类是：

- 身份、租户和资源权限；
- 金额、配额、步骤数和时间上限；
- 参数、资源状态和完成条件校验；
- 幂等、并发和数据一致性。

这些规则写进 Prompt 只能影响模型行为，不能形成安全边界。即使系统提示要求“只能查询当前客户”，查询服务仍必须根据调用者身份校验 `tenant_id`。

**模型决定想做什么，程序验证能不能做，并确认动作是否真正成功。**这条边界让模型可以灵活选择路径，又无法绕过业务系统的不变量。

## 人工承担高风险和例外责任

有些决定既无法完全写成规则，也不适合让模型独立承担，例如生产回滚、大额退款、法律承诺和例外授权。它们通常具有三个特征：错误代价高、难以自动验收、需要明确责任人。

登录调查 Agent 可以收集证据并生成回滚建议，值班工程师再审查受影响租户、关键证据、拟执行动作和回滚方案。人工介入不是 Agent 失败后的补丁，而是预先设计好的任务分支：系统要知道何时暂停、交给谁，以及批准后从哪里继续。

## 三者可以共同完成一个环节

模型、程序和人工不是三段互不相干的流程。生产回滚可以由三者共同完成：

```text
模型整理证据并建议回滚
  ↓
程序检查租户、版本和回滚条件
  ↓
值班工程师批准
  ↓
部署服务执行回滚并检查结果
  ↓
模型解释结果，决定是否继续调查
```

客服退款也一样：模型理解诉求并整理证据，规则服务计算退款资格，主管审批高金额申请，支付服务校验订单状态并执行。工程设计的重点不是在“模型还是规则”之间二选一，而是让三者承担不同责任。

## Agent 通常只控制流程的一部分

生产系统常由确定性工作流包围一段开放调查：

```text
程序校验身份并接收工单
  ↓
普通问题走固定流程
  ↓ 复杂或原因未知
Agent 动态调查并整理证据
  ↓
程序检查结果和任务状态
  ↓
低风险建议返回客服
高风险操作进入人工审批
```

模型只在复杂调查中动态选择路径。身份检查、普通请求分流、结果校验和高风险执行仍由程序控制。这样既保留了 Agent 的适应性，也缩小了成本、评测和故障范围。

因此，“产品使用工作流还是 Agent”并不总是二选一。一个工作流可以在局部调用 Agent，一个 Agent 也可以调用固定的审批或退款流程。

## 用两个问题快速判断边界

面对一个环节时，先问两个问题：

1. **它有多开放？**路径越依赖现场信息、越难穷举，越适合模型参与；步骤越稳定、规则越明确，越适合程序。
2. **犯错后能否安全恢复？**越容易验证和撤销，越可以给模型更多空间；越高风险、不可逆或责任重大，越需要程序保护和人工审批。

可以把它记成一张二维图：

| | 错误容易验证和恢复 | 错误高代价或不可逆 |
| --- | --- | --- |
| 路径开放 | 模型判断，程序验证 | 模型提建议，人工决策 |
| 路径稳定 | 固定工作流 | 程序规则与严格审批 |

**开放性决定是否需要模型，错误代价决定模型能走多远。**

## 边界会在生产中移动

决策边界不是上线前决定一次就结束。真实失败会暴露两个方向的问题：

- Agent 总在重复相同路径或产生无意义差异，说明某些步骤应该固化为程序；
- 固定分支持续膨胀，仍覆盖不了新故障，说明某些选择需要模型处理。

如果大多数登录问题都遵循“检查区域状态—检查发布—检查配置”的稳定路径，可以把它固化为工作流，只让模型处理异常分支。反过来，如果错误码无法稳定决定日志来源，可以让模型在受限候选中选择下一步。

成熟系统不会一味增加自主性，而会把已经稳定的决策逐渐收回程序，把仍然开放且可验证的问题留给模型。

## 从边界进入后续设计

决策边界确定了 Agent 的责任范围，也决定了后面的设计：模型负责哪些判断，就为它组织哪些上下文；任务需要跨越哪些步骤，就保存哪些状态；模型允许观察和执行什么，就提供相应的受控工具。

```text
模型负责什么
  ↓
模型需要知道什么
  ↓
系统需要保留什么
  ↓
Agent 可以做什么
```

读者只需要带走两个判断：**开放性决定是否需要模型，错误代价决定模型能走多远。**
