# 如何划分模型与程序的决策边界

前一层设计已经确认：复杂登录故障的调查路径依赖现场证据，适合使用 Agent。接下来要拆开这个 Agent 的内部任务，确定模型、程序和人工各自负责什么。

用户说“昨晚升级后，欧洲办公室都登录不了”。模型可以根据监控和发布记录改变调查方向，但租户权限不能由模型判断是否遵守，生产回滚也不能因为模型认为“很有把握”就自动执行。

**模型负责开放判断，程序守住确定规则，人工承担高风险责任。**这不是在重新选择工作流还是 Agent，而是在已经选定的 Agent 内部分配责任。

## 把 Agent 任务拆成判断、规则和责任

登录故障调查包含不同性质的环节：

| 任务环节 | 控制者 | 原因 |
| --- | --- | --- |
| 理解含糊的故障描述 | 模型 | 需要处理多样的自然语言 |
| 根据新证据选择调查方向 | 模型 | 下一步取决于现场结果 |
| 检查租户权限和调用预算 | 程序 | 必须一致执行且可审计 |
| 检查结案所需证据是否齐全 | 程序 | 完成条件可以确定性验证 |
| 批准生产回滚 | 人工 | 错误代价高，需要明确责任 |
| 执行回滚并确认结果 | 程序 | 需要权限、幂等和状态检查 |

这张表划分的不是几个互相隔离的流程，而是同一个任务里的不同责任。模型可以决定“下一步查看配置差异”，程序仍要检查它能否读取相应租户；模型可以建议回滚，最终操作仍要经过审批和部署服务。

## 模型输出的是判断，不是业务事实

模型适合综合工单、日志和文档，形成假设并选择下一步。例如，监控显示身份服务整体正常，但错误集中在使用特定单点登录配置的租户，模型可以把调查方向转向客户配置和最近发布。

这种动态判断是 Agent 的核心价值。但模型提出“配置发布可能改变了证书校验”时，这仍是待验证假设。只有配置差异、错误记录或实验结果提供了证据，系统才能把它提升为已确认事实。

```text
模型提出假设
  ↓
工具取得证据
  ↓
程序记录来源和验证状态
  ↓
模型根据新事实继续判断
```

**模型可以产生候选结论，业务状态必须区分假设与已确认事实。**这条边界防止模型用一段自然语言间接改变系统事实。

## 程序守住不能协商的不变量

程序负责的不是“模型不会做的事”，而是无论模型多强都必须稳定成立的规则：

- 调用者只能访问有权处理的租户和资源；
- 参数、金额和资源状态必须合法；
- 步骤数、时间和成本不能超过上限；
- 写操作不能因为重试而重复产生副作用；
- 完成或转人工必须满足明确条件。

这些规则写进 Prompt 只能影响模型行为，无法形成执行边界。即使系统提示要求“只能查询当前客户”，查询服务仍要校验调用者身份和 `tenant_id`。

**模型决定想做什么，程序验证能不能做，并确认是否真正成功。**程序因此不仅负责拒绝危险动作，也负责把允许的动作可靠执行并返回明确结果。

## 人工介入要有明确交接

生产回滚、大额退款和例外授权无法总靠固定规则决定，又不适合由模型独立承担。人工负责这些高代价、不可逆或需要组织责任的决定。

Agent 交给人的不应只有一句“是否批准”。登录回滚请求至少要包含：

- 受影响范围；
- 已确认的证据与仍未解决的问题；
- 拟执行的变更；
- 预期效果、风险和恢复方案。

审批也要成为任务状态的一部分。系统需要知道何时暂停、由谁处理、审批是否仍然有效，以及通过或拒绝后从哪里继续。

## 一个生产动作如何由三者共同完成

生产回滚展示了三种责任如何连接：

```text
模型综合证据，提出回滚建议
  ↓
程序检查租户、配置版本和回滚条件
  ↓
值班工程师审查证据并批准
  ↓
部署服务执行回滚，记录资源和状态
  ↓
程序检查错误率是否恢复
  ↓
模型解释结果，决定是否继续调查
```

模型负责开放判断，人工承担是否接受风险的责任，程序在每个边界上实施校验和可靠执行。

客服退款也可以使用同一结构：模型理解诉求并整理证据，规则服务计算退款资格，主管批准高金额申请，支付服务检查订单状态并执行。三者不是互相替代，而是在一次任务中形成责任链。

## 同一能力在不同风险下可以有不同边界

决策边界取决于动作发生的环境。修改测试租户的配置容易验证和撤销，可以允许 Agent 自动执行；完全相同的生产修改则需要审批。

工程师可以用“建议、草稿、执行”三个层级逐步开放能力：

| 能力层级 | 模型权限 | 软件保护 |
| --- | --- | --- |
| 建议 | 生成调查或修复方案 | 不改变外部系统 |
| 草稿 | 创建待审配置或退款申请 | 可查看差异、撤销和拒绝 |
| 执行 | 请求真实操作 | 权限、业务规则、审批和审计 |

任务刚上线、评测数据有限时，可以先停在建议或草稿；随着验证和恢复机制成熟，再允许部分低风险动作自动执行。控制权不是一次性全给或全收。

## 用失败轨迹调整内部边界

上线后，Trace 会暴露责任划分是否合理：

- 模型经常把同一种错误状态误判为不同原因，可以让程序先做结构化分类；
- 模型反复尝试无权执行的动作，说明工具和权限边界暴露得不清楚；
- 人工几乎总是批准某类低风险草稿，可以考虑用确定规则自动放行；
- 相同固定规则不断增加例外，可能需要给模型更多判断空间，但程序不变量仍然保留。

调整的对象是 Agent 内部的具体责任，而不是每次都重新决定整个产品是否属于 Agent。稳定判断可以沉淀为程序规则，仍然开放的判断继续交给模型，高风险责任则随着组织政策保留给人。

## 三句话复述

**模型负责理解现场信息、提出假设和选择下一步，但模型输出不会自动成为业务事实。**

**程序负责权限、业务不变量、可靠执行和完成检查，这些边界不能由 Prompt 代替。**

**人工负责高风险与例外决定，Agent 必须带着证据和可恢复状态完成交接。**

确定这三种责任后，才能继续设计模型需要的上下文、任务必须保存的状态，以及与责任范围匹配的工具。
