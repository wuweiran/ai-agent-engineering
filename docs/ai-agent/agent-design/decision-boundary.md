# 模型、程序与人的决策边界

团队已经确认复杂登录故障适合使用 Agent。现在要拆开内部任务：模型负责调查中的哪些判断，程序守住哪些规则，什么时候必须交给人。

用户说“昨晚升级后，欧洲办公室都登录不了”。模型可以根据监控和发布记录改变方向；租户权限不能由模型决定是否遵守，生产回滚也不能因为模型觉得“很有把握”就自动执行。

**模型负责开放判断，程序守住确定规则，人工承担高风险责任。**

## 从具体环节分配责任

| 任务环节 | 控制者 | 原因 |
| --- | --- | --- |
| 理解含糊的故障描述 | 模型 | 自然语言表达多样 |
| 根据新证据选择调查方向 | 模型 | 下一步取决于现场结果 |
| 检查租户权限和调用预算 | 程序 | 必须一致执行且可审计 |
| 检查结案证据是否齐全 | 程序 | 完成条件可以确定性验证 |
| 批准生产回滚 | 人工 | 错误代价高，需要明确责任 |
| 执行回滚并确认结果 | 程序 | 需要权限、幂等和状态检查 |

这些责任会在同一个环节中交错。模型决定查看配置差异，程序检查它是否有权读取该租户；模型建议回滚，值班工程师批准后，部署服务才执行动作。

## 模型的结论先当作假设

监控显示身份服务整体正常，错误却集中在使用特定单点登录配置的租户。模型据此把调查方向转向客户配置和最近发布，这是 Agent 擅长的开放判断。

它提出“配置发布可能改变了证书校验”时，系统只能把这句话记录为待验证假设。配置差异、错误日志或实验结果提供证据后，才能把它提升为已确认事实。

模型可以提出候选结论，业务状态必须保留来源和验证状态。否则，一段自然语言就可能悄悄改变系统事实。

## 歧义与缺失信息

模型可以理解开放表达，但不能用猜测补齐执行所需的事实。用户只说“把刚才那笔退掉”时，系统仍要确定具体订单和退款范围；存在多个候选时，应提出能够改变后续路径的最小追问。

**是否继续执行取决于信息完整性和动作风险。**低风险回答可以明确说明假设，高风险写入必须取得明确选择，并由工具 Schema 和业务规则检查必要字段。连续澄清仍不能推进时，Runtime 应停止当前动作或转人工（[用户意图模糊或信息不足时，Agent 怎样处理？]({{ site.baseurl }}/docs/interview/ai-agent/runtime/#agent-intent-ambiguity)）。

## 程序守住不会因推理改变的规则

无论模型多强，下面这些规则都必须稳定执行：调用者只能访问有权处理的资源；参数和资源状态必须合法；步骤和时间不能超过上限；写操作不能因重试而重复；完成或转人工必须满足明确条件。

把这些要求写进 Prompt，只能影响模型行为。查询服务仍要校验调用者身份和 `tenant_id`，支付服务仍要检查订单和金额。

**模型决定想做什么，程序验证能不能做，并确认动作是否成功。**

## 高风险决定要有清楚的人工交接

生产回滚、大额退款和例外授权往往无法完全写成规则，又不适合由模型独立承担。

登录调查 Agent 请求回滚时，应同时交付受影响范围、已确认的证据、仍未解决的问题、拟执行变更和恢复方案。系统保存审批人、审批结果和有效期，批准后再从明确位置继续。

人工介入因此是任务中的正常分支，不是系统失灵后的临时补丁。

## 生产回滚中的责任分工

生产回滚可以这样分工：模型综合证据并提出建议；程序检查租户、配置版本和回滚条件；值班工程师决定是否接受风险；部署服务执行回滚并观察错误率；模型再解释结果，判断是否继续调查。

客服退款也遵循同一思路。模型理解诉求并整理证据，规则服务计算资格，主管批准高金额申请，支付服务检查订单状态并执行。

## 自动化程度可以逐步增加

测试环境中的配置修改容易验证和撤销，可以允许 Agent 自动执行；相同的生产修改则需要审批。

团队还可以把能力分成三层：先只生成建议，再允许创建可审查的草稿，最后才开放部分真实执行。新任务刚上线、评测数据有限时，停在建议或草稿通常更稳妥。

随着生产数据增加，边界也会移动。人工几乎总是批准的低风险草稿，可以用确定规则自动放行；模型反复误判的稳定场景，可以先由程序分类；固定规则不断增加例外时，则可能给模型更多判断空间。

决策边界不是一次确定后永远不变，但每次移动都应由真实失败和评测支持。

## 记住三种责任

- **模型理解现场信息、提出假设并选择下一步。**
- **程序执行权限、业务规则、可靠动作和完成检查。**
- **人工处理高风险与例外，并带着证据完成交接。**

确定这些责任后，才能继续设计模型需要的上下文、任务状态和工具。
