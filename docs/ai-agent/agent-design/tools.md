# 如何为 Agent 设计工具

设计 Agent 系统时，上下文决定模型依据什么信息判断，工具则决定它能够取得哪些新信息、执行哪些操作。工程师需要把业务系统的能力转化成受控接口，使模型的动态决策能够进入真实的软件环境。

一个客服 Agent 已经能够理解“客户被重复扣款”的工单，但它还需要查询订单和支付记录，判断是否满足退款条件，并在必要时提交处理申请。工程团队必须决定向模型开放哪些能力、参数怎样表达、错误如何返回，以及哪些操作可以自动执行。

如果只提供一个任意 SQL 和 HTTP 请求工具，模型几乎可以访问所有底层接口，权限和审计很难控制；如果每个字段都做成一个独立工具，模型又会面对大量重叠选择。工具边界直接影响 Agent 的能力、可靠性和风险。

**工具设计要把业务系统的能力包装成模型容易选择、程序能够校验、团队可以治理的契约。**模型提出调用请求，运行时负责执行，业务服务仍然实施权限和数据规则。

```text
业务任务
  ↓
需要读取或执行的能力
  ↓
工具契约：名称、参数、结果、错误、副作用
  ↓
权限检查与真正执行
  ↓
结果进入上下文，驱动下一步决策
```

## 从业务动作划定工具边界

工程师通常先梳理 Agent 为完成任务需要进行哪些观察和操作，而不是直接把现有内部 API 全部暴露给模型。

重复扣款处理可能需要：

- 根据工单身份查询客户和订单；
- 查询指定订单的支付尝试；
- 检查退款资格和历史处理记录；
- 创建退款申请；
- 在工单中写入处理说明；
- 将高金额或异常情况转给人工。

这些动作可以形成 `get_order`、`list_payment_attempts`、`check_refund_eligibility` 和 `create_refund_request` 等工具。模型看到的是与任务一致的业务能力，支付服务仍然保留金额、订单状态、重复退款和操作人权限等确定性校验。

工具粒度可以用三个问题判断：

1. 模型是否能够从名称和描述区分它与其他工具？
2. 运行时是否需要为这个动作设置独立权限、审批或审计？
3. 返回结果是否会直接支持一个明确的后续判断？

若两个工具总是一起调用且权限相同，可以考虑合并；若一个工具内部包含多个风险完全不同的动作，则应拆分。

## 名称和描述要帮助模型选择

模型依靠工具定义判断什么时候调用。描述只写“查询数据”或“处理订单”，会让工具选择依赖猜测。清楚的定义需要说明适用场景、输入范围和不适用情况。

例如：

```text
get_order
根据当前坐席有权访问的 tenant_id 和 order_id 查询订单摘要。
用于确认订单状态、客户归属和支付币种；不返回完整客户资料。
```

这样的描述同时告诉模型：

- 什么时候使用；
- 必须提供哪些标识；
- 返回内容的边界；
- 它不能代替客户资料或退款资格查询。

工具命名还应使用业务中稳定的概念，而不是把内部服务名泄漏给模型。支付平台从 `pay-v2` 迁移到 `ledger-service` 时，“查询支付尝试”这个契约可以保持不变，工具处理器再适配底层变化。

## 参数结构要让程序能够校验

自然语言适合表达意图，不适合直接承担接口约束。工具输入应采用结构化字段，并把枚举、必填项和格式限制尽量写进模式中。

创建退款申请可能需要：

| 参数 | 设计考虑 |
| --- | --- |
| `order_id` | 使用稳定业务标识，不让模型拼接数据库条件 |
| `payment_id` | 明确针对哪次支付，防止退错交易 |
| `amount` | 使用最小货币单位或明确小数规则 |
| `reason_code` | 使用有限枚举，方便策略、统计和审计 |
| `evidence` | 保存支持申请的订单和支付事实，而不是自由发挥的结论 |
| `idempotency_key` | 超时重试时避免重复创建 |

工具处理器必须把模型参数当作不受信任输入。它需要验证类型、业务范围、资源归属和当前状态，不能因为模型“已经分析过”就跳过服务端规则。涉及数据库、文件路径、命令和 URL 时，还要防止注入、越权和路径穿越。

## 返回值要支持下一步判断

工具结果会进入下一轮上下文。返回值若只面向传统程序调用者，模型可能无法理解；若直接返回完整底层响应，又会带来噪声、敏感数据和接口耦合。

一个订单查询结果可以只包含当前任务需要的字段：

```text
order_id: O-1842
status: paid
customer_match: true
currency: EUR
paid_amount_minor: 12900
refunded_amount_minor: 0
payment_attempt_count: 2
```

错误也需要稳定语义：

| 错误 | 模型可以采取的下一步 |
| --- | --- |
| `not_found` | 检查订单号或向用户补充信息 |
| `permission_denied` | 停止查询并请求有权限的人员处理 |
| `rate_limited` | 根据重试时间等待或受限终止 |
| `invalid_state` | 读取当前订单状态，改走其他处理流程 |
| `dependency_unavailable` | 记录失败并转人工，而不是反复调用 |

**好的工具结果不仅说明发生了什么，还要让 Agent 知道下一步有哪些合理选择。**详细原始日志可以保留在日志平台，通过查询 ID 按需读取，不必全部塞回模型上下文。

## 通用能力与专用能力需要组合

专用工具边界清楚，但无法覆盖所有开放任务；通用工具能力强，却更难治理。生产 Agent 往往组合两者。

| 工具形态 | 优势 | 代价 |
| --- | --- | --- |
| 专用业务工具 | 参数稳定、权限清楚、容易审计和评测 | 需要持续维护，难以覆盖未知需求 |
| 通用查询或代码执行 | 灵活，能处理开发时未预见的步骤 | 风险面大，结果格式不稳定，治理更复杂 |

客服 Agent 通常不需要任意 SQL，只需一组受控业务工具。数据分析 Agent 可能需要代码执行，但应运行在隔离环境中，限制网络、文件和凭据。内部运维 Agent 可以保留终端能力，同时把部署、扩容和变更审批提升为独立工具。

Claude Code 是这种组合方式的一个成熟例子。它保留终端工具处理不同项目的构建和测试命令，同时提供 `Glob`、`Grep`、`Read`、`Edit` 等专用文件工具。根据其[工具参考](https://code.claude.com/docs/en/tools-reference)，`Edit` 要求先读取文件并进行唯一精确匹配；专用接口让宿主能够实施比任意 shell 字符串更明确的校验和交互。AI Agent 工程师可以借鉴这种取舍，而不是为业务系统复刻一套代码 Agent 工具。

## 权限要在模型之外执行

提示词可以引导模型少做危险操作，却不能成为安全边界。工具层和业务服务需要根据调用者、资源、环境和动作风险执行权限检查。

工具可以按副作用分层：

- **只读观察**：查询订单、读取监控、检索文档；
- **局部可逆写入**：创建草稿、修改工作区文件、生成待审申请；
- **外部或高风险操作**：真实退款、部署、删除数据、发送消息。

只读工具也需要数据范围控制，防止跨租户读取。可逆操作可以自动执行后展示差异或草稿。高风险操作通常需要明确审批、额度限制、双人复核或由确定性的工作流完成。

[Claude Code 的权限文档](https://code.claude.com/docs/en/permissions)将只读工具、终端命令和文件修改区别处理，并用 allow、ask、deny 规则控制调用。它还明确说明权限由宿主程序执行，`CLAUDE.md` 只能影响模型行为。业务 Agent 同样应把权限放在工具处理器、策略服务和底层 API 中。

## 写操作要考虑超时、重试和并发

Agent 看到超时后可能重新调用工具。如果第一次请求其实已经成功，第二次“创建退款”就会造成重复操作。写工具需要继承分布式系统的可靠性设计：

- 使用幂等键或业务唯一约束；
- 返回资源 ID 和最终状态；
- 区分未执行、已执行但响应丢失、部分完成和确定失败；
- 使用资源版本防止覆盖其他操作人的新修改；
- 对长任务返回任务 ID，让 Agent 查询进度；
- 限制重试次数，并在结果不确定时转人工。

并行调用也需要明确。多个只读查询可以并行降低延迟；依赖前一步结果的写操作必须串行；两个工具若会修改同一资源，则应通过版本检查或锁避免竞态。

## 工具集合要控制规模和重叠

工具越多，模型选择正确工具的难度越高，工具定义本身也会占据上下文。工程师需要持续治理工具目录：

1. 合并用途和参数高度重叠的工具；
2. 将工具按角色、任务和权限分组，只暴露当前需要的部分；
3. 为每个工具写清触发条件和边界；
4. 大型目录使用检索或延迟加载，而不是一次提供全部定义；
5. 通过调用数据发现长期未用、经常误选和返回噪声过多的工具。

企业可能通过 MCP 接入 GitHub、工单、监控和数据平台。Claude Code 的 [MCP 文档](https://code.claude.com/docs/en/mcp)描述了 tool search：模型先发现相关工具，再按需加载完整定义，用于减少大量工具对上下文的占用。业务 Agent 也可以采用注册中心、按域检索或分层路由解决同类问题。

## 工具质量要拆开评测

最终任务失败不能全部归因于模型。工具链路至少包含四个可独立评测的环节：

| 环节 | 评测问题 |
| --- | --- |
| 工具选择 | 该调用时是否调用，是否选对工具？ |
| 参数生成 | 必填项、格式、资源标识和业务范围是否正确？ |
| 工具执行 | 权限、幂等、超时和错误处理是否符合契约？ |
| 结果利用 | Agent 是否正确理解结果并调整下一步？ |

评测样本应覆盖正常路径、空结果、权限不足、限流、服务故障、重复调用和并发状态变化。Trace 则要记录工具版本、输入、权限决策、返回摘要、耗时和重试，使工程师能判断问题在模型、工具契约还是底层服务。

## 工具设计连接模型能力与软件工程

工程师设计工具时，需要连续完成几项取舍：

```text
从业务任务识别必要动作
  ↓
确定专用与通用能力的边界
  ↓
设计可校验的参数和可恢复的返回值
  ↓
按副作用实施权限、审批和隔离
  ↓
处理幂等、超时、并发和版本变化
  ↓
通过 Trace 与评测持续收缩错误空间
```

模型决定想做什么，工具和业务系统决定能不能做、怎样安全地做。可靠的 Agent 工具不是底层 API 的简单转发，而是一层同时服务模型决策和生产治理的软件接口。
