# 如何为 Agent 设计工具

一个客服 Agent 能理解“客户被重复扣款”的工单，但要真正处理问题，还要查询订单和支付记录、判断退款资格，并在必要时提交申请。工程团队必须决定向模型开放哪些能力，以及怎样让这些能力安全执行。

如果只提供任意 SQL 和 HTTP 请求，模型几乎可以触及所有底层接口；如果每个字段都做成工具，模型又会面对大量重叠选择。

**工具是模型与软件系统之间的能力契约：模型选择想做什么，程序校验并执行，结果再支持下一步判断。**

```text
业务任务
  ↓
工具契约：何时调用、输入什么、返回什么
  ↓
程序检查权限和业务规则
  ↓
执行结果进入下一轮上下文
```

## 从业务动作划定工具边界

工具设计不从已有多少内部 API 开始，而从 Agent 完成任务需要哪些观察和动作开始。重复扣款处理可以归纳为四项能力：

- 查询订单摘要；
- 查询订单的支付尝试；
- 检查退款资格；
- 创建退款申请。

它们可以形成 `get_order`、`list_payment_attempts`、`check_refund_eligibility` 和 `create_refund_request`。模型看到的是与任务一致的业务动作，支付服务仍然掌握金额、订单状态和重复退款等规则。

判断两个动作应该合并还是拆分，可以看两件事：模型是否需要分别选择，以及它们是否具有不同权限和副作用。总是一起发生且风险相同的查询可以合并；查询退款资格和执行真实退款则必须分开。

**工具边界应该贴近模型需要选择的业务动作，而不是照搬底层服务结构。**支付平台从 `pay-v2` 迁移到新服务时，“查询支付尝试”这个契约仍然可以保持稳定。

## 契约要让模型选得对、填得对、看得懂

一个工具契约有三个最重要的部分。

### 名称和描述帮助模型选择

“查询数据”无法说明什么时候使用。更清楚的定义是：

```text
get_order
根据当前坐席有权访问的 tenant_id 和 order_id 查询订单摘要。
用于确认订单状态、客户归属和支付币种；不返回完整客户资料。
```

它同时说明适用场景、输入范围、返回边界，以及不能代替什么工具。

### 结构化参数让程序校验

创建退款申请不应只接收一段自由文本。它至少需要稳定的订单和支付标识、金额、原因以及幂等键：

```yaml
order_id: O-1842
payment_id: P-7731
amount_minor: 12900
reason_code: duplicate_charge
idempotency_key: refund-O-1842-P-7731
```

工具处理器要把模型参数当作不受信任输入，检查类型、资源归属和当前状态。模型“已经分析过”不能替代服务端校验。

### 返回值帮助模型继续判断

工具结果不应只是“成功”或一整份底层响应。订单查询可以返回当前任务需要的字段：

```yaml
order_id: O-1842
status: paid
customer_match: true
paid_amount_minor: 12900
refunded_amount_minor: 0
payment_attempt_count: 2
```

错误也要区分语义：`not_found` 提示检查订单号，`permission_denied` 要求停止并转给有权限的人，`invalid_state` 则说明应读取最新订单状态。模型知道发生了什么，才能选择合理的下一步。

**好的工具契约让模型知道何时调用，让程序能够校验输入，让返回值能够驱动下一步。**

## 权限和业务规则必须在工具之外成立

提示词可以引导模型，却不能成为权限边界。工具处理器和业务服务仍要根据调用者、资源和动作风险作出决定。

权限可以围绕三类副作用理解：

| 动作 | 常见控制方式 |
| --- | --- |
| 读取订单和监控 | 自动执行，但限制租户和数据范围 |
| 创建草稿或待审申请 | 自动执行，保留差异并允许撤销 |
| 真实退款、部署或删除 | 额度检查、明确审批或确定性工作流 |

只读并不等于无风险，跨租户读取同样可能造成严重事故。高风险工具即使参数由模型正确生成，也要经过服务端权限和业务规则检查。

Claude Code 区分文件读取、精确编辑和通用终端，并由宿主实施权限规则，是同一设计思想的实例。业务 Agent 不需要复刻代码工具，但可以借鉴“按能力和副作用提供不同接口”的取舍。

## 写工具要按分布式系统设计

Agent 看到超时后可能再次调用工具。假如第一次退款已经成功，只是响应丢失，第二次调用就会造成重复操作。

写工具最重要的不是列出所有异常，而是保证两件事：

1. **重复请求不会重复产生副作用**：使用幂等键或业务唯一约束；
2. **结果不确定时能够查证**：返回资源 ID，并区分确定失败与结果未知。

长任务可以返回任务 ID，让 Agent 查询状态；修改共享资源时使用版本号，避免覆盖其他人的新操作。这些都是普通后端与分布式系统能力，只是在 Agent 会自主重试和改变路径后变得更重要。

## 专用工具与通用工具要有取舍

专用业务工具容易校验、授权和评测，但无法覆盖所有未知需求；通用 SQL、终端或代码执行很灵活，却扩大了风险和结果的不确定性。

客服 Agent 通常只需要受控业务工具，不需要任意 SQL。数据分析 Agent 可能需要代码执行，但应放在限制网络、文件和凭据的隔离环境中。常用、高风险、需要审计的动作，更值得提升为专用工具；真正开放且可恢复的任务，才保留通用能力。

工具也不是越多越好。定义本身会占据上下文，重叠工具会增加误选。大型工具目录应按角色和任务只暴露相关部分，或通过检索按需加载。

## 评测一次工具调用的四个环节

工具相关失败可以沿执行链路定位：

```text
是否选对工具
  ↓
参数是否正确
  ↓
程序是否安全执行
  ↓
模型是否正确使用结果
```

Agent 没有创建退款，可能是没选 `create_refund_request`，也可能是支付 ID 错误、服务拒绝了当前状态，或模型误读成功结果。Trace 要记录工具版本、输入、权限决策、返回摘要和重试，评测才能找到真正需要修改的环节。

## 三个问题检查一个工具

设计工具时，可以集中检查三个问题：

1. **模型能否选对并生成正确参数？**
2. **程序能否实施权限、业务规则和可靠执行？**
3. **结果能否让模型恢复或继续下一步？**

工具不是底层 API 的简单转发。它把 Agent 的业务能力、模型选择和软件治理连接在一起：模型负责提出意图，软件负责守住边界，结构化结果再把现实世界的反馈交还给模型。
