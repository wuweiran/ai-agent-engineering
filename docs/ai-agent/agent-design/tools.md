# Agent 工具

客服 Agent 已经理解“客户被重复扣款”的工单。要继续处理，它还得查询订单和支付记录，判断能否退款，并在必要时提交申请。

团队可以给它任意 SQL 和 HTTP 请求能力，但这样很难限制权限；也可以把每个字段都做成工具，结果是模型面对大量相似选项。

应用把工具名称、描述和参数 Schema 提供给模型，模型返回结构化调用请求，Runtime 执行后再把结果送回模型。这个协议通常称为 Tool Calling 或 Function Calling。模型只提出调用，不会自己执行真实函数（[什么是 Function Calling？]({{ site.baseurl }}/docs/interview/ai-agent/#function-calling)）。

**工具要把业务动作表达成模型容易选择、程序能够校验的接口。**模型提出意图，软件负责执行，业务服务继续掌握权限和规则。

## 先从业务动作出发

重复扣款处理需要的能力并不多：查询订单摘要，查看支付尝试，检查退款资格，创建退款申请。

它们可以分别成为 `get_order`、`list_payment_attempts`、`check_refund_eligibility` 和 `create_refund_request`。模型看到的是客服正在处理的业务动作，不需要知道底层支付服务叫什么、数据存在哪张表里。

工具是否拆开，主要看模型是否需要分别选择，以及动作是否具有不同风险。总是一起发生且权限相同的查询可以合并；检查退款资格和执行退款则必须分开。

支付平台以后从 `pay-v2` 迁移到其他服务，“查询支付尝试”仍可以保持同一工具契约。**工具边界应贴近业务动作，不必照搬底层服务结构。**

## 让模型知道何时使用

名称和描述会直接影响模型的选择。“查询数据”几乎没有帮助，更清楚的定义是：

```text
get_order
根据当前坐席有权访问的 tenant_id 和 order_id 查询订单摘要。
用于确认订单状态、客户归属和支付币种；不返回完整客户资料。
```

这段说明回答了三个问题：什么时候调用，需要什么标识，它不能代替什么工具。

如果模型经常在两个工具之间误选，问题未必是模型能力不足。它们可能使用了相似名称，描述了相同场景，或者本来就应该合并。

## 参数留给程序校验

创建退款申请不能只接收一段自由文本。订单、支付记录、金额和原因都要有明确字段：

```yaml
order_id: O-1842
payment_id: P-7731
amount_minor: 12900
reason_code: duplicate_charge
idempotency_key: refund-O-1842-P-7731
```

结构化参数让程序能够检查类型、枚举、资源归属和当前状态。模型已经分析过订单，不代表服务端可以跳过校验。

`idempotency_key` 还解决了另一个问题：第一次申请可能已经成功，只是响应丢失。Agent 重试时，退款服务可以识别同一请求，而不会创建第二笔退款。

## 返回值要帮助下一步判断

工具只返回“成功”，模型不知道订单是什么状态；直接返回整份底层响应，又会带来噪声和敏感信息。

订单查询可以只返回当前任务需要的字段：

```yaml
order_id: O-1842
status: paid
customer_match: true
paid_amount_minor: 12900
refunded_amount_minor: 0
payment_attempt_count: 2
```

错误也要有稳定含义。`not_found` 提示检查订单号，`permission_denied` 要求停止并交给有权限的人，`invalid_state` 表示订单状态已经不允许当前操作。

**好的工具结果会告诉 Agent 发生了什么，以及下一步还有哪些合理选择。**

## 权限始终由软件执行

提示词可以要求模型谨慎操作，不能代替服务端授权。查询订单时要限制租户范围，创建退款申请时要检查坐席权限，真实退款还可能需要金额限制和主管审批。

不同副作用适合不同控制方式：读取订单可以在授权范围内自动执行；创建待审申请可以自动完成并展示内容；真实退款、部署和删除则需要更严格的规则或人工确认（[工具调用失败后怎样设计降级？]({{ site.baseurl }}/docs/interview/ai-agent/#tool-failure-fallback)）。

Claude Code 将文件读取、精确编辑和通用终端分成不同工具，并由宿主执行权限规则。业务 Agent 可以借鉴这种做法，但不需要复制代码 Agent 的工具集合。

## 通用能力只在需要时开放

客服 Agent 通常不需要任意 SQL。数据分析 Agent 可能确实需要代码执行，因为开发阶段无法预见所有分析步骤；此时要限制网络、文件和凭据，并把高风险动作继续放在专用工具中。

工具也不是越多越好。相似工具会增加误选，完整定义还会占用上下文。工具很多时，可以按角色和任务只提供相关部分，或让 Agent 按需发现。

## 出错后沿调用过程排查

一次退款失败，可以沿四个位置检查：模型是否选对工具，参数是否正确，程序是否安全执行，模型是否正确理解返回值。

例如，Agent 没有创建退款，可能是没选择 `create_refund_request`，也可能是支付 ID 错误、服务拒绝了当前订单状态，或模型把成功结果理解成失败。Trace 要把这些位置区分开，团队才知道应该改工具描述、参数 Schema、业务服务还是结果表达。

设计一个工具时，最后检查三件事：模型能否选对，程序能否守住规则，结果能否支持下一步。工具做到这三点，才真正把模型判断连接到了业务系统。
