# 多 Agent 协作

故障调查需要搜索监控、发布记录和日志。一个 Agent 可以依次完成全部工作，也可以把相互独立的调查交给多个 Agent，最后汇总证据。

多个 Agent 不会自动提高质量。常见的[多 Agent 协作模式]({{ site.baseurl }}/docs/interview/ai-agent/planning-collaboration/#multi-agent-patterns)只有在任务可以清楚拆分，并行、专业化或 Context 隔离的收益高于协调成本时才值得使用。

## 顺序协作适合逐步加工

一个 Agent 收集资料，另一个生成报告，第三个复核结论，这是一条顺序链。每个执行者只需要接收上一步的结构化结果，职责容易解释。

代价是延迟累积，上游遗漏还会传播到后续。复核者如果只看到结论而没有证据，也很难发现早期错误。

## 并行协作适合独立子问题

区域监控、发布历史和客户配置可以同时调查。并行能够缩短等待时间，也让不同 Agent 保持较小 Context。

汇总者需要处理重复、矛盾和部分失败。两个 Agent 同时修改相同文件或业务对象，还会产生写入冲突。并行适合只读探索或能够隔离输出的任务，不能只因为“有多个 Agent”就并行真实副作用。

## 分层协作需要清楚的委派接口

Manager Agent 可以拆分目标并派给 Worker，Worker 返回结论、证据和未知项。层级结构适合子任务数量动态变化或执行者具有不同工具的场景。

Manager 也可能成为瓶颈：它需要理解各项返回，处理失败并决定是否继续。委派消息应明确目标、范围、权限、完成条件和返回格式，不能只说“调查一下”。

## 三层职责把开放判断与系统控制分开

[Multi-Agent 三层架构]({{ site.baseurl }}/docs/interview/ai-agent/planning-collaboration/#multi-agent-three-layer-architecture)不是统一行业规范，而是一种便于解释生产职责的划分：

```text
协调层：目标拆分、依赖调度、结果合并与终止判断
Agent 执行层：各自使用独立 Context、专业工具和能力边界完成子任务
Runtime 与资源层：工具执行、权限、任务状态、通信、Artifact 与 Trace
```

这三层不等于三个模型，也不要求部署成三个服务。Manager 本身可以是 Agent，Runtime 仍要验证它的计划和结束条件；Worker 可以自主调查子任务，却不能仅凭自然语言修改共享任务事实。小系统可以在一个进程中实现三类职责，任务扩大后再按容量和故障边界拆分。

## Agent 间的信息交换

最简单的方式是直接调用并等待结果，适合短而强依赖的子任务。长任务可以通过消息或事件异步返回状态；大型报告、代码和证据保存在 Artifact 中，消息只携带引用。共享状态便于多个 Agent 读取同一任务，但并发写入需要版本和归属规则。事件驱动方式耦合较低，却会带来顺序、重复和调试问题。

无论使用哪种[通信方式]({{ site.baseurl }}/docs/interview/ai-agent/planning-collaboration/#multi-agent-communication)，契约都应包含稳定任务与父任务 ID、发送者和接收者、目标、输入引用、状态、结果或错误、版本和幂等标识。只用一段自然语言说“已经处理好”，无法支持恢复和冲突判断。

跨产品 Agent 通信还可能使用 A2A。它关注 Agent 发现、任务、消息、产物和状态交换。代码编辑器与编码 Agent 之间的 ACP 则解决客户端和 Agent 的交互。它们与 MCP 的边界不同：MCP 主要连接模型应用与工具和数据，不是通用的多 Agent 协作替代品。

## 多 Agent 仍然需要一个任务事实来源

每个 Agent 可以有独立 Context，但总体目标、任务状态、审批和外部动作仍要落到共享的权威系统。否则一个 Worker 认为任务已取消，另一个仍可能继续写入。

协调层还要处理：

```text
谁拥有当前子任务？
结果是否完整且仍然有效？
部分失败是否允许整体继续？
同一动作怎样避免重复？
什么时候应该停止或转人工？
```

这些仍然是任务系统、并发控制和故障恢复问题，不会因为执行者换成 Agent 而消失。

## 不适合拆分的任务

只读取一个已知文件、步骤强依赖前一结果、所有执行者都需要相同完整 Context，或者最终仍由一个模型重复理解全部材料时，拆分只会增加调用和摘要损失。

判断可以压缩成两个问题：子任务能否用清楚的输入与交付物独立描述？它是否会产生大量主任务以后不再需要的中间信息，或能与其他子任务真正并行？两者都成立时，多 Agent 才可能带来价值。

**多 Agent 的核心不是角色数量，而是任务边界、信息接口和副作用归属是否清楚。**
