# 如何控制 Agent 的执行与终止

登录故障调查 Agent 已经查看了欧洲区域的服务状态和昨晚的发布记录。服务整体正常，发布记录却显示一项可疑配置变化。模型可以继续比较配置，也可以搜索更多日志，还可能因为不确定而重复前面的查询。

上下文和工具让 Agent 有能力行动，却不能保证任务会向目标推进。工程师还要控制三件事：这一步是否带来进展，出现问题后能否恢复，以及任务现在为什么结束。

**模型提出下一步和结束建议，运行时依据任务状态、执行结果和完成条件控制是否继续。**

```text
当前目标和状态
      ↓
模型提出下一步
      ↓
运行时检查并执行
      ↓
结果是否带来进展？
      ↓
继续、等待或停止
```

## 完成条件让任务有明确出口

“调查登录问题”只说明了方向，没有说明什么时候可以停止。工程师需要把目标变成可以观察的结果：

```text
目标
找出升级后欧洲客户登录失败的原因，并给出处理建议。

完成条件
- 影响范围已经明确；
- 根因有配置差异或错误记录支持；
- 建议与已确认原因一致；
- 不修改生产环境；
- 证据不足时列出未知项并转人工。
```

其中，影响范围和证据是否存在可以由程序检查，解释是否合理仍可能需要模型或人工判断。完成条件不要求把所有质量都写成规则，而是让“完成”必须有证据。

**目标说明要去哪里，完成条件说明凭什么停下来。**

## 进展意味着任务状态发生变化

登录调查可以这样推进：

```text
未知：是否是区域服务故障
  ↓ 查询服务状态
排除：区域服务整体正常
  ↓ 读取发布记录
发现：故障前有配置发布
  ↓ 比较配置与错误记录
确认：证书校验配置导致登录失败
```

第一次查询没有找到根因，却排除了一个方向，因此仍然有进展。相反，连续使用相同条件查询同一份监控，又没有说明为什么新结果会不同，只是在消耗步骤。

运行时不必理解模型的全部推理。它可以观察任务状态是否增加了事实、排除了假设、减少了待办或改变了外部资源。连续多轮没有这些变化时，系统可以要求模型重新规划；仍无进展则停止并交接。

**推进任务不等于不断调用工具，而是让任务状态发生有意义的变化。**

## 恢复从判断结果是否确定开始

发布记录工具可能限流、权限不足或超时。它们表面上都是失败，恢复方式却不同：限流可以等待，权限不足需要授权，超时则可能只表示没有收到响应。

只读查询超时通常可以安全重试。写操作更谨慎：如果 Agent 创建了回滚申请，但成功响应在网络中丢失，直接重试可能产生重复申请。运行时应保存幂等键或资源 ID，先查询真实结果。

```text
结果是否确定？
  ├─ 确定未执行 → 错误是暂时的，才考虑重试
  ├─ 确定已执行 → 更新状态并继续
  └─ 结果未知   → 先查证，不能直接重做
```

还有一个判断是：再次执行是否可能改变结果。服务限流会随时间恢复，等待有意义；权限不足不会因重复调用消失，应请求授权或转人工。

**结果明确且再次执行可能成功时才重试；结果未知时，先查证而不是重做。**

## 等待和停止都要保留已经取得的进展

调查可能需要用户补充租户信息、值班工程师审批，或等待服务恢复。此时任务没有完成，也没有确定失败。系统应保存等待原因、恢复条件和恢复位置，条件满足后重新检查环境，再继续执行。

运行限制处理另一类问题：任务长时间没有出口。步骤、时间、成本和连续无进展次数本质上都在约束一次任务最多可以消耗多少资源和风险。达到限制时，不应只返回“超时”，而要留下已确认事实、已排除方向、剩余假设和建议的人工步骤。

因此，任务可以归纳为三类出口：

| 出口 | 含义 |
| --- | --- |
| 完成 | 目标和必要证据已经满足 |
| 等待 | 缺少外部条件，保存状态后仍可恢复 |
| 停止 | 无法继续或达到限制，交接已有结果 |

失败和取消都可以是“停止”的具体原因。对于执行控制，更重要的是分清任务已经达成目标、未来仍可恢复，还是不应继续消耗资源。

## 模型建议结束，运行时确认资格

模型能够判断证据是否形成了合理解释，运行时则掌握必填字段、工具状态、审批和资源限制。登录调查结束时，两者可以这样协作：

```text
模型：根因已有配置差异和错误记录支持，建议结束
  ↓
运行时检查：
- 影响范围已经记录
- 证据可以追溯
- 没有结果未知的写操作
- 未违反生产只读约束
  ↓
通过 → 标记完成
不通过 → 返回缺失项，继续或转人工
```

**模型判断结论是否合理，运行时验证任务是否具备结束资格。**无法自动验证的高风险结果仍由人工审查，不能把模型的一句“完成”直接当成最终状态。

## Trace 检查任务为何没有正确结束

执行轨迹主要暴露三种问题：重复工具没有改变状态；不可重试或结果未知的操作被再次执行；证据不足却提前完成，或条件已经满足仍继续搜索。

Trace 应记录每步意图、工具结果、状态变化、恢复理由和终止判断。团队据此调整完成条件、错误语义和运行限制，而不是发现循环后只增加最大步骤数。

## 三个判断控制整条执行链

1. **这一步是否带来进展？**
2. **发生问题后是否可以恢复？**
3. **任务现在为什么结束？**

```text
有进展 → 继续推进
可恢复 → 查证、重试或等待
该结束 → 确认完成，或带着已有结果停止
```

模型提供开放判断，运行时提供可靠控制。两者共同工作，Agent 才不会把持续行动误当成持续前进，也不会把模型想结束误当成任务已经完成。
