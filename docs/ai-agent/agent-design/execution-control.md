# 如何控制 Agent 的执行与终止

登录故障调查已经查看了欧洲区域的服务状态和昨晚的发布记录。服务整体正常，发布记录却显示一项可疑配置变化。模型可以比较配置，也可以继续搜索日志，还可能因为不确定而重复前面的查询。

上下文和工具让 Agent 有能力行动，却不能保证任务会向目标推进。运行时需要持续判断：这一步有没有带来进展，遇到问题后能否恢复，现在是否应该结束。

**模型提出下一步和结束建议，运行时根据任务状态、执行结果和完成条件决定是否继续。**

## 先说清楚凭什么完成

“调查登录问题”只说明了方向。更可执行的目标还要说明：影响范围需要明确，根因要有配置差异或错误记录支持，建议要与已确认原因一致，不允许修改生产环境；证据不足时，列出未知项并转人工。

其中，影响范围和证据字段可以由程序检查，解释是否合理仍可能需要模型或人工判断。完成条件不要求把所有质量写成规则，它只是让“已经完成”必须有依据。

**目标说明要去哪里，完成条件说明凭什么停下来。**

## 工具调用不等于进展

调查开始时，团队还不知道是不是区域服务故障。查询服务状态后，确认服务整体正常，虽然没有找到根因，却排除了一个方向。随后读取发布记录，发现故障前有配置变化；比较配置和错误记录后，才确认原因。

增加事实、排除假设、减少待办或改变外部状态，都算进展。连续使用相同条件查询同一份监控，没有说明为什么新结果会不同，只是在消耗步骤。

运行时不必理解模型的全部推理。它可以观察任务状态是否发生有意义的变化。连续多轮没有新事实和待办变化时，系统可以要求模型调整方向；仍无进展，就停止并交接。

## 失败后先判断结果是否确定

发布记录工具可能限流、权限不足或超时。限流可以等待，权限不足需要授权，超时则可能只表示没有收到响应。

只读查询超时通常可以安全重试。写操作更谨慎：如果 Agent 已经创建回滚申请，只是成功响应丢失，直接重试会产生重复申请。

可以按结果分成三种情况：

| 状态 | 下一步 |
| --- | --- |
| 确定没有执行 | 错误是暂时的，才考虑重试 |
| 确定已经执行 | 更新状态并继续 |
| 结果未知 | 先查询真实结果，不能直接重做 |

还要判断再次执行是否可能改变结果。服务限流会随时间恢复，等待有意义；权限不足不会因重复调用消失，应请求授权或转人工。

**结果明确且再次执行可能成功时才重试；结果未知时先查证。**

## 等待也要成为正式状态

调查可能需要用户补充租户信息、值班工程师批准，或等待服务恢复。任务此时没有完成，也没有确定失败。系统应保存等待原因、恢复条件和恢复位置，条件满足后重新检查环境再继续。

运行限制处理另一种情况：任务长时间找不到出口。步骤、时间、成本和连续无进展次数，都在限制一次任务最多消耗多少资源。达到上限时，系统要留下已确认事实、排除方向、剩余假设和建议的人工步骤。

任务最终有三类出口：完成，表示目标和证据已经满足；等待，表示缺少外部条件但以后可以恢复；停止，表示无法继续或已达到限制，需要交接已有结果。

## 模型建议结束，运行时检查资格

模型可能判断：“根因已经得到配置差异和错误日志支持，可以结束。”运行时随后检查影响范围是否记录、证据是否可追溯、是否存在结果未知的写操作，以及生产只读约束是否被遵守。

检查通过，任务标记完成；检查失败，系统把缺失项交还模型，继续调查或转人工。

**模型判断结论是否合理，运行时检查任务是否具备结束资格。**高风险结果无法自动验证时，仍要保留人工审查。

## Trace 关注三类执行问题

执行轨迹常见的问题很集中：重复工具没有改变状态；不可重试或结果未知的操作被再次执行；证据不足却提前完成，或者条件已经满足仍继续搜索。

Trace 记录每步意图、工具结果、状态变化、恢复理由和结束判断。团队据此修改完成条件、错误语义和运行限制，而不是发现循环后只把最大步骤数调大。

控制执行时，记住三个问题就够了：这一步有没有进展，发生问题后能不能恢复，任务现在为什么结束。
