# 如何设计一个 Agent 系统

团队确认某个任务确实需要 Agent 后，还不能直接把模型、几个 API 和一个循环拼在一起。模型要负责哪些判断，每轮依据什么信息，能够调用哪些能力，发生错误时怎样恢复，都需要被设计成明确的软件边界。

这项工作在 AI Agent 工程师的职责中常被概括为“建设执行能力”。它具体指的是：**把业务目标转化成一套由模型参与决策、由软件负责执行和约束的 Agent 系统。**

```text
定义任务：要解决什么问题，什么算完成
  ↓
选择系统形态：一次调用、固定工作流还是 Agent
  ↓
设计 Agent 系统：模型如何在软件边界内推进任务
  ↓
建立质量体系：怎样判断每一步和最终结果是否正确
  ↓
运营生产系统：怎样处理变化、故障、成本和规模
```

## 先划分模型与程序的职责

Agent 系统允许模型根据中间结果改变执行路径，但不意味着所有决定都交给模型。设计工作首先要区分两类逻辑：

- 路径难以预先穷举、需要理解现场信息的判断，可以由模型参与；
- 权限、金额限制、数据一致性和完成条件等确定性规则，应由程序执行。

例如，调查“升级后欧洲客户无法登录”时，模型可以根据监控结果决定继续查看发布记录还是审计日志。程序仍然负责限制租户数据范围、禁止生产写入、控制最大调查时间，并在证据不足时转给人工。

**Agent 系统设计的核心不是让模型获得尽可能多的自主权，而是为动态决策找到可靠的软件边界。**

## Agent 系统需要设计哪些部分

一次任务的运行链路可以拆成一组相互依赖的设计问题：

| 设计部分 | 要回答的问题 |
| --- | --- |
| 目标与完成条件 | Agent 要交付什么，怎样证明完成，何时应转人工？ |
| 决策边界 | 哪些步骤由模型动态选择，哪些步骤由程序固定控制？ |
| 上下文 | 模型当前需要看到哪些规则、事实、历史和工具说明？ |
| 状态与记忆 | 哪些进度必须可靠保存，哪些经验可以跨任务复用？ |
| 工具与执行环境 | Agent 可以查询什么、修改什么，接口怎样表达？ |
| 权限与人工协作 | 哪些操作可以自动执行，哪些需要审批或禁止？ |
| 运行控制 | 怎样处理循环、超时、重试、并发、恢复和终止？ |
| Trace 与评测 | 怎样还原执行过程，并判断系统在哪一步失效？ |

这些部分不能孤立设计。工具结果会成为新的上下文，任务状态决定恢复后应加载哪些信息，权限规则限制工具能否执行，完成条件则影响运行时何时结束循环。

```text
                 目标与完成条件
                        ↓
任务状态 ──→ 上下文 ──→ 模型决策 ──→ 工具执行
   ↑                                      │
   └──────────── 结果与反馈 ──────────────┘

权限与人工审批：限制每次执行
运行控制：负责重试、恢复和终止
Trace 与评测：记录并验证整条链路
```

## 上下文决定模型依据什么判断

模型不会自动知道业务规则、客户状态或此前步骤。工程师需要从当前决策反推信息需求，选择可信来源，并决定哪些内容预先加载、哪些按需检索、哪些旧结果应该移出上下文。

上下文过少，模型会缺少关键证据；上下文过多，相关事实可能被日志和历史对话淹没；信息过期或来源不可信，也会让后续决策建立在错误前提上。因此，上下文不是简单拼接 Prompt，而是持续维护模型的信息工作集。

## 工具决定 Agent 能观察和改变什么

模型只能提出查询或操作意图，真正的动作由工具处理器、运行时和业务服务完成。工程师需要把底层系统包装成边界清楚的能力，并设计名称、参数、返回值、错误语义和副作用。

工具既扩大 Agent 的能力，也扩大风险面。订单查询可以自动执行，真实退款可能需要额度检查和人工审批；任意 SQL 虽然灵活，却比“查询订单状态”更难授权、审计和评测。工具设计因此同时属于 Agent 能力设计和软件治理。

## 状态、运行时和权限让多步任务可控

上下文只表示模型本轮看到什么，不能代替可靠的任务状态。已经确认的事实、完成的步骤、审批结果和重试次数，需要保存在数据库、工作流引擎或会话存储中。任务暂停或服务重启后，运行时才能据此恢复。

运行时负责把模型提出的下一步变成真实执行：校验参数、检查权限、调用工具、记录结果、更新状态并判断是否继续。对于写操作，它还要处理幂等、超时和结果不确定；对于长任务，则要限制步骤数、时间和成本。

权限和人工审批贯穿整条链路。提示词可以影响模型行为，但不能代替服务端授权。模型即使受到错误信息影响，也不应越过租户隔离、金额限制或生产变更审批。

## Trace 和评测验证设计是否有效

Agent 失败时，最终答案通常无法说明根因。同一个错误结论可能来自不同环节：上下文遗漏证据、模型选错工具、工具返回旧数据、权限策略配置错误，或者终止条件过早触发。

Trace 要记录每轮上下文来源、模型决策、工具调用、权限结果、状态变化和终止原因。评测再使用真实任务与故障样本检查各个环节，而不只是给最终回答打分。生产中的失败会反过来推动工程师调整信息来源、工具边界和模型控制权。

## 从现象定位设计问题

| 运行现象 | 优先检查的设计部分 |
| --- | --- |
| Agent 不知道关键业务事实 | 上下文来源、检索和装配 |
| 长任务忘记进度或重复工作 | 状态、记忆和恢复机制 |
| Agent 无法取得信息或执行操作 | 工具覆盖范围和接口契约 |
| 误操作或读取了不该访问的数据 | 权限、审批和数据隔离 |
| 反复调用相同工具，无法结束 | 反馈语义、运行控制和终止条件 |
| 演示有效，上线后质量无法判断 | Trace、评测样本和生产指标 |

上下文、工具、状态和运行时不是几个并列功能，而是共同构成 Agent 的决策与执行闭环。工程师所谓“设计 Agent”，就是让模型在每一步知道什么、能够做什么、必须遵守什么，并让软件系统能够验证和控制整个过程。
