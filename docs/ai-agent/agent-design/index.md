# Agent 系统设计

团队决定用 Agent 调查“升级后欧洲客户无法登录”的复杂工单。接下来不能直接把模型、几个 API 和循环拼起来：模型可以选择调查方向，租户权限和生产回滚却不能跟着模型自由变化。

系统设计要把一次任务中的责任和信息安排清楚。可以沿着调查过程依次问六个问题：

1. 哪些判断交给模型，哪些规则由程序和人控制？
2. 模型作出当前判断时需要看到什么？
3. 多步任务要把哪些事实和进度保存下来？
4. Agent 可以通过哪些工具观察和行动？
5. 复杂目标怎样拆成步骤，并随证据调整计划？
6. 系统怎样判断任务有进展，以及何时结束？

这些问题互相连接。工具返回的结果会进入上下文，任务状态决定恢复后加载什么，计划表达步骤及依赖，权限规则限制动作能否执行，完成条件决定循环何时停止。单独理解每个概念后，还可以从成熟 Agent 产品的具体机制观察它们怎样组成完整系统。

## 模型、程序与人的分工

调查方向很难预先穷举，可以让模型根据监控和发布记录选择下一步。租户权限、步骤上限和证据字段有明确规则，应由程序检查。生产回滚错误代价高，需要值班工程师批准。

| 控制者 | 在登录调查中的责任 |
| --- | --- |
| 模型 | 理解描述、提出假设、选择下一步调查方向 |
| 程序 | 校验权限、执行工具、保存状态、检查完成条件 |
| 人工 | 批准生产变更和处理证据冲突的例外情况 |

模型获得的不是无限自主权，而是一段有边界的动态判断空间。

## 再决定模型每一步知道什么

模型不会自动知道客户版本、当前服务状态和此前调查结果。运行时要为当前判断选择必要信息。

调查是否属于全局故障，需要区域、时间和服务状态；判断是否与升级有关，需要客户版本和发布记录。稳定规则可以在任务开始时加载，大量日志和实时状态则在需要时查询。

上下文太少会缺证据，太多会淹没关键事实，过期数据还会把调查带错方向。因此，工程师维护的是当前判断需要的信息工作集，而不是可访问数据的全集。

## 任务状态保证调查不会失忆

Agent 查过身份服务、发现配置发布、正在等待发布平台恢复，这些进度要保存在任务状态中。服务重启后，系统才能继续比较配置差异，而不是重新调查一遍。

长期记忆解决另一个问题：以后出现相似工单时，是否要复用已经验证的版本兼容问题。当前服务状态和未经证实的猜测不能因为出现过一次，就永久进入记忆。

## 工具把判断变成现实动作

监控查询、发布记录和配置比较可以包装成边界清楚的工具。模型请求调用，运行时校验参数和权限，业务服务执行真实查询，再返回能够支持下一步的结果。

工具也划定风险。查询订单可以自动执行，真实退款需要资格检查和审批；任意 SQL 虽然灵活，却难以实施细粒度权限和审计。

## 复杂目标需要规划和调整

简单任务可以根据当前结果决定下一步；复杂任务更适合先拆成具有依赖和完成信号的步骤。计划执行后还要根据新证据删除无效方向、增加验证步骤，或者调整执行顺序。

模型可以生成和修改候选计划，Runtime 仍要检查依赖、权限、并发与预算。[Agent Planning](planning/)进一步说明顺序、分层和动态规划怎样取舍。

## 任务推进与结束

每次工具调用都应带来可观察变化：增加事实、排除假设、减少待办或改变外部状态。连续多轮没有新进展时，运行时要让模型调整方向或停止。

模型可以建议“根因已经确认”，程序还要检查影响范围和证据是否齐全。任务也可能等待用户、权限或依赖恢复；达到步骤和时间上限时，则带着已有结果停止并交接。

## 多个 Agent 需要独立任务边界

一个 Agent 足以完成大多数可控任务。子问题能够独立描述，而且并行、专业化或 Context 隔离确有收益时，才值得引入多个 Agent。

[多 Agent 协作](multi-agent/)比较顺序、并行和分层协作，说明委派接口、共享状态与副作用冲突怎样处理。

## 一次失败会指向不同设计位置

| 运行现象 | 优先检查 |
| --- | --- |
| Agent 不知道关键事实 | 上下文来源和装配 |
| 重启后重复调查或重复写入 | 状态与恢复机制 |
| 选错能力或无法理解返回值 | 工具契约 |
| 越权读取或执行高风险动作 | 决策边界、权限和审批 |
| 反复调用工具，无法结束 | 进展判断和终止条件 |

设计 Agent 系统，就是把模型的开放判断安放在软件能够执行、保存、观察和约束的范围内。[Claude Code 的 Agent 实现]({{ site.baseurl }}/docs/ai-agent/claude-code/)会进一步拆开一个成熟产品的 Context 组装、工具循环、会话压缩和子 Agent，让这些分工落到可观察的实现机制中。设计确定后，团队还要根据具体业务选择成熟 Agent 产品、SDK 或自建 Runtime，并把这些边界落实到真实应用中。
