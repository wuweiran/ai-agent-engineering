# 如何设计一个 Agent 系统

团队确认一个任务确实需要 Agent 后，还不能直接把模型、几个 API 和循环拼在一起。以登录故障调查为例，模型可以根据现场证据改变调查方向，但租户权限、生产变更和结束条件不能跟着模型自由变化。

这项工作在 AI Agent 工程师的职责中常被概括为“建设执行能力”。它可以用四个连续问题理解：

```text
模型负责判断什么？       决策边界
          ↓
作出判断时需要知道什么？ 上下文
          ↓
跨步骤需要保留什么？     状态与记忆
          ↓
可以观察和改变什么？     工具
```

**设计 Agent 系统，就是回答这四个问题，再用运行时、权限和评测把答案变成可靠的软件边界。**

## 决策边界：模型负责什么

Agent 的价值来自模型能够根据中间结果选择下一步，但这不代表所有决定都应该交给模型。

调查“欧洲客户升级后无法登录”时，模型可以理解用户描述、比较监控与发布记录，并决定下一步调查哪个系统。程序仍然检查租户权限、限制步骤和时间，并验证结案所需证据；生产回滚则由值班工程师审批。

三者的基本分工是：

| 控制者 | 更适合承担的工作 |
| --- | --- |
| 模型 | 路径难以穷举、需要理解现场信息的开放判断 |
| 程序 | 必须一致执行的权限、数据和业务规则 |
| 人工 | 高代价、不可逆或需要明确责任的决定 |

**Agent 系统设计的核心不是让模型获得尽可能多的自主权，而是为动态决策找到可靠边界。**

## 上下文：模型依据什么判断

模型不会自动知道业务规则、客户状态或此前步骤。工程师需要从当前决策反推信息需求，选择可信来源，并决定哪些信息预先加载、哪些按需查询。

上下文过少，模型缺少证据；上下文过多，关键事实会被日志和历史对话淹没；信息过期或来源不可信，后续判断则会建立在错误前提上。上下文工程不是简单拼接 Prompt，而是为当前决策持续维护一个信息工作集。

## 状态与记忆：系统保留什么

上下文只表示模型本轮看到什么，不能代替可靠存储。

- **任务状态**保存同一个任务的目标、事实、进度和外部动作，使任务能够暂停和恢复；
- **长期记忆**保存经过验证、值得在未来任务中复用的信息，并带有作用域和失效条件。

例如，“已检查身份服务，结果正常”属于当前调查的状态；“4.2.0 至 4.2.2 存在已确认的 SAML 兼容问题”才可能成为长期记忆。运行时从状态、记忆和实时数据中选择当前需要的内容，再组成上下文。

## 工具：Agent 能观察和改变什么

模型只能提出查询或操作意图，真正的动作由工具处理器和业务服务完成。工程师需要把底层系统包装成边界清楚的能力，并说明何时调用、需要什么参数、返回什么结果以及有哪些副作用。

订单查询可以自动执行，真实退款则需要资格检查和审批；任意 SQL 虽然灵活，却比“查询订单状态”更难授权、审计和评测。工具既决定 Agent 的能力，也决定它可能造成的风险。

## 软件边界贯穿四个问题

四项设计最终由运行时连接起来：

```text
目标与任务状态
      ↓
组织本轮上下文
      ↓
模型选择下一步
      ↓
运行时校验权限并执行工具
      ↓
记录结果、更新状态、判断是否继续
```

权限不能只写在 Prompt 中，工具写操作要处理幂等和超时，长任务要有步骤、时间和成本上限。Trace 记录上下文、模型决策、工具调用和状态变化，评测再判断问题出在信息、决策、执行还是终止环节。

## 用一次失败定位设计问题

| 运行现象 | 优先检查 |
| --- | --- |
| Agent 不知道关键业务事实 | 上下文来源和装配 |
| 暂停后忘记进度或重复操作 | 状态和恢复机制 |
| 选不到能力或错误理解返回值 | 工具契约 |
| 越权读取或执行高风险动作 | 决策边界、权限和审批 |
| 反复调用工具，无法结束 | 反馈、运行控制和终止条件 |
| 只能看到最终答案错误 | Trace 和分环节评测 |

这四个问题也给出了后续文章的阅读顺序：先确定模型负责什么，再设计它看到什么、系统保留什么，以及它能通过哪些工具行动。每一部分都服务同一个目标：让模型的开放判断发生在软件可以执行、恢复、观察和约束的范围内。
