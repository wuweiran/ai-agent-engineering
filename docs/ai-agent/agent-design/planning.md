# Agent Planning

用户要求 Agent 调查一项跨地区登录故障，并交付影响范围、根因证据和恢复建议。系统可以让模型每取得一条结果就决定下一步，也可以先形成调查计划，再按新证据调整。

Planning 是把目标转成可执行步骤、依赖和完成条件。它不是所有 Agent 都必须拥有的独立模块：短任务可以只决定下一步，长任务才更需要显式计划来保持方向和支持协作（[Agent 怎样进行任务规划？](../../interview/ai-agent/#agent-planning)）。

## 计划从完成条件开始

“调查登录故障”还不能直接拆步骤。系统先明确交付物：影响哪些租户和区域，根因需要哪些证据，是否只能只读调查，证据不足时怎样交接。

模型随后可以生成任务结构：

```text
确认影响范围
├─ 查询区域错误率
└─ 汇总相似工单

寻找最近变化
├─ 读取发布记录
└─ 比较配置差异

验证根因
└─ 对照错误与配置变化
```

计划不只是待办列表。步骤要说明输入、依赖、执行者和完成信号，运行时才知道哪些可以并行，哪些结果可以跳过后续步骤。

## 三种规划方式适合不同任务

顺序计划让步骤依次执行，简单可追踪，但前序错误会传播，延迟也会累积。分层计划先把大目标拆成子目标，再继续分解，适合复杂任务和多执行者，却增加状态同步成本。

动态规划每取得新结果再决定下一步，最能适应未知环境，也最接近 ReAct 循环。它不需要一开始预测完整路径，但容易重复探索，成本和完成时间也较难估计。

生产系统常组合使用：程序提供稳定阶段，模型在某个阶段内动态调查；或者模型先给出粗计划，Runtime 只把当前可执行步骤放入任务状态。

## 计划必须随着证据更新

发布记录查询失败，不代表整个调查从头重来；监控已经证明服务整体正常，计划就应删除全局故障方向，把资源转向配置和租户差异。

这种根据观察更新步骤的过程称为 Replanning。调整计划时要保留原因：哪条证据使步骤失效，哪些结果仍然可用，新步骤依赖什么。否则模型可能反复推翻方向，工程师也无法解释路径变化。

计划是候选执行结构，不是业务事实。模型写下“回滚配置”不会自动创建回滚任务，真实动作仍要经过权限、审批和业务服务。

## 并行只适用于依赖清楚的步骤

区域错误率和发布记录互不依赖，可以并行查询；比较配置差异必须等发布版本确定后才能开始。

并行能降低等待时间，却带来新的问题：多个结果怎样合并，部分失败是否阻止整体推进，两个执行者会不会写入同一状态。计划中要显式保存依赖和结果标识，不能只依赖自然语言顺序。

## 计划的运行时检查

模型可以创建计划，Runtime 仍要限制步骤、时间、Token 和工具预算，并检测连续重复。每一步应增加事实、排除假设、完成外部动作或减少待办；连续执行却没有改变任务状态，说明计划没有推进。

任务也可能等待用户信息、审批或依赖恢复。等待要保存原因、恢复条件和恢复位置，而不是让模型在循环中不断重试。

完成时，系统检查真正的交付条件，而不是只看计划中的复选框是否全部勾选。若一条更短路径已经获得充分证据，可以删除不再需要的步骤；若证据不足，即使模型宣称“计划完成”，任务也不能结束。

## Planning 和 Workflow 的边界

Workflow 的关键路径和分支由程序预先定义；Planning 让模型参与生成或调整步骤。两者可以组合：稳定的审批和发布流程使用 Workflow，未知原因调查在其中使用模型规划。

```text
程序定义不可违反的阶段与边界
→ 模型规划开放调查步骤
→ Runtime 校验依赖、预算和完成条件
→ 工具结果触发执行或重新规划
```

Planning 的价值不在于让任务看起来更自主，而在于复杂目标能够被分解、跟踪和调整，同时仍然受确定性系统约束。
