# 分布式系统常见面试题

分布式系统把一次业务拆到多个进程、节点或服务中。网络可能延迟和中断，节点可能独立失败，各处观察到的状态也可能不同。回答这类问题时，应先指出故障窗口和权威状态，再说明一致性、可用性与恢复成本之间的取舍。

## 一致性与可用性

## 分布式系统与单体进程相比，多出了哪些问题？
{: #distributed-system-failures }

同一进程中的函数调用通常能直接返回结果或异常；跨节点调用还会遇到网络延迟、请求或响应丢失、节点独立故障、时钟偏差和重复执行。

调用方看到超时时，远端可能尚未收到请求，也可能已经成功。因此，分布式系统需要显式处理结果未知、状态同步、幂等、重试、故障隔离和恢复。

## CAP 定理表达了什么？
{: #cap-theorem }

CAP 讨论发生网络分区时，系统无法同时保证线性一致的读写结果和每个请求都得到成功响应。此时必须在一致性与可用性之间作出取舍；分区容错不是可以简单放弃的选项，因为网络故障客观存在。

CAP 中的一致性不是数据库 ACID 中的“一致性约束”。没有网络分区时，系统仍可以同时提供一致结果和正常可用性，不能把所有分布式系统永久划成 CP 或 AP。

## 强一致与最终一致怎样选择？
{: #strong-vs-eventual-consistency }

强一致让读者立即看到满足系统一致性模型的最新结果，适合余额扣减、唯一资格等不能接受旧值的决策，但通常需要更多协调，并影响延迟或故障时的可用性。

最终一致允许副本短暂不同，通过复制、重试和修复最终收敛，适合商品展示、搜索索引和统计数据。选择依据不是技术偏好，而是旧值会造成什么业务后果，以及不一致能持续多久。

## BASE 是什么？
{: #base-theory }

BASE 通常概括为基本可用、软状态和最终一致。它表达一种工程取舍：分布式系统在故障或高负载下可以暂时降级，并允许中间状态经过异步处理后收敛。

BASE 不是可以忽略正确性的理由。系统仍要定义最终状态、收敛期限、失败发现、补偿与人工处置。

## 副本怎样提高可用性？又会带来什么问题？
{: #replication-tradeoffs }

副本让一个节点故障后仍有其他节点提供读取或接管写入，也能把读取流量分散到多个节点。

代价是副本同步、Leader 选举和读一致性问题。异步复制可能读取旧值或在故障切换时丢失尚未复制的数据；同步副本越多，写入延迟和无法写入的概率通常越高。

## 幂等、超时与重试

## 什么是幂等？
{: #idempotency }

幂等表示同一业务意图执行一次或重复执行，最终业务结果相同。查询通常天然幂等，创建支付、退款和发送通知则需要额外设计。

常见实现包括稳定请求 ID、数据库唯一约束、条件状态转换和结果查询。幂等键必须与业务意图绑定并持久化，不能每次重试都生成新键。

## 超时为什么不等于失败？
{: #timeout-unknown-outcome }

超时只说明调用方没有在期限内收到结果。请求可能没有到达，也可能仍在执行，甚至已经成功但响应丢失。

对有副作用的操作，应把状态记为结果未知，使用原请求 ID 查询权威系统；明确没有执行后才考虑重试。直接把超时当失败，可能产生重复支付或重复写入。

相关内容：[超时、重试与故障隔离]({{ site.baseurl }}/docs/backend/service-timeouts/)。

## 哪些错误适合重试？
{: #retryable-errors }

连接短暂失败、限流和服务临时不可用可能随时间恢复，适合在截止时间内有限重试。参数错误、权限不足和业务状态冲突不会因等待自动修复，不应重试。

有副作用的操作还要先确认再次执行是否安全。错误可能恢复和操作幂等是两个独立条件，缺一不可。

## 指数退避与随机抖动解决什么问题？
{: #backoff-jitter }

指数退避让连续失败后的等待逐步增加，给依赖恢复时间，也减少持续请求压力。随机抖动让不同调用方错开重试，避免它们在固定时间点同时醒来形成新高峰。

等待需要设置上限，重试还应受最大次数、整体截止时间和重试预算约束。退避不能让永久错误变成可恢复错误。

## 为什么不能在调用链的每一层都独立重试？
{: #retry-amplification }

如果三层调用各重试三次，一次用户请求在最坏情况下可能把最下游调用放大到二十多次。依赖已经过载时，这种乘法效应会进一步拖慢恢复。

系统应选择最合适的一层负责重试，传播整体截止时间，并限制整个调用链的重试预算。

## 截止时间和普通超时有什么区别？
{: #deadline-propagation }

单次超时限制某一阶段最多等待多久；截止时间表示整项请求或任务还剩多少时间。下游调用只能使用剩余预算的一部分，还要为状态保存和响应留出时间。

沿调用链传播截止时间，可以避免每一层重新使用完整默认超时，导致用户早已放弃而后台仍持续工作。

## 服务治理与故障隔离

## 熔断、降级和隔离有什么区别？
{: #circuit-breaker-fallback-isolation }

熔断在下游持续失败或变慢后暂时停止调用；降级牺牲非核心能力，返回缓存、简化结果或明确不可用状态；隔离限制某个依赖能够占用的线程、连接或并发量。

三者经常组合。隔离先限制故障半径，熔断阻止继续施压，降级决定用户还能得到什么结果。

## 熔断器有哪些状态？
{: #circuit-breaker-states }

Closed 状态正常放行并统计错误与慢调用；超过阈值后进入 Open，快速失败；冷却期结束后进入 Half-Open，只允许少量探测请求。

探测成功后可以关闭熔断器，失败则重新打开。Half-Open 只判断依赖是否开始恢复，不代表可以立即恢复全部流量。

## 线程池隔离和信号量隔离有什么区别？
{: #thread-pool-vs-semaphore-isolation }

线程池隔离为不同依赖使用独立执行资源，阻塞下游不会直接占满所有业务线程，但会增加线程、队列和上下文切换成本。

信号量隔离只限制同时调用数量，开销较小，但下游阻塞时仍占用原业务线程，因此必须配合超时。两者都需要有界等待和拒绝策略。

## 降级结果应该怎样设计？
{: #fallback-design }

降级结果应简单、快速、含义真实，并与正常响应保持可识别的契约。可以返回旧缓存、空结果或“稍后处理”，但不能伪造支付成功、库存充足等关键事实。

Fallback 最好避免再次同步调用另一个不可靠远端，否则会把故障转移到新的依赖。

## 服务发现与负载均衡怎样配合？
{: #service-discovery-load-balancing }

服务实例把地址和健康状态注册或上报到服务发现系统，调用方或代理取得可用实例列表，再通过轮询、随机、最少连接或一致性哈希等策略选择目标。

健康检查只能说明实例在某个时间点可用。负载均衡还要考虑慢实例、连接复用、区域亲和、灰度版本和故障摘除。

## 常见限流算法有什么区别？
{: #rate-limiting-algorithms }

固定窗口实现简单，但窗口交界处可能放行两批流量；滑动窗口统计更平滑，状态成本更高；漏桶按稳定速率输出，适合严格保护下游；令牌桶控制平均速率，同时允许一定突发。

算法之外还要选择限流维度、配额和拒绝语义。多实例共享全局配额时，需要网关或分布式状态；可以合理拆分配额时，本地限流更简单可靠。

相关内容：[容量、排队与背压]({{ site.baseurl }}/docs/backend/capacity-backpressure/)。

## 分布式令牌桶怎样实现？
{: #distributed-token-bucket }

每个限流 Key 保存令牌余额和上次补充时间。请求到达时，根据经过时间计算新增令牌，把余额限制在桶容量内；余额足够则扣减并放行，否则拒绝或返回下次可用时间。补充和扣减必须在服务端原子完成。

多实例可以在网关集中执行，也可以用 Redis Lua 脚本或原子函数维护共享状态。Key 通常包含租户、用户、接口或资源类型；桶容量决定允许的突发，请求速率决定长期平均。Redis 故障时采用本地保守配额还是拒绝流量，要根据被保护业务决定。

## 滑动窗口怎样实现？使用 Redis 时怎样选择数据结构？
{: #sliding-window-rate-limit }

精确滑动窗口可以用时间戳队列或 Redis ZSet 保存窗口内请求：成员是唯一请求 ID，Score 是时间；原子地删除窗口外记录、统计数量，再决定是否写入当前请求。核心状态包括限流 Key、窗口长度、上限和当前时间。

这种方式精确但每次要删除和统计，热门 Key 的内存与操作成本较高。滑动窗口计数器把时间划成小格，用 Hash 或多个计数 Key 汇总，精度稍低但更省资源。工程上应根据窗口精度、QPS、Key 数量和 Redis 压力选择，而不是默认使用 ZSet。

## 漏桶、滑动窗口和令牌桶怎样选择？
{: #leaky-sliding-token-bucket }

漏桶强调按近似恒定速率流出，适合严格保护固定吞吐的下游；滑动窗口限制任意连续时间段内的请求数，规则直观但要维护窗口状态；令牌桶限制长期平均速率并允许桶容量范围内的突发。

面向用户 API 通常希望容忍短时突发，令牌桶较常见；必须平滑写入速度时可用漏桶；业务合同直接规定“任意一分钟最多 N 次”时滑动窗口更容易表达。选择还要结合分布式状态成本和拒绝后的重试行为。

## 分布式事务

## 为什么需要分布式事务？
{: #distributed-transaction-purpose }

单库事务只能让一个数据库中的写入共同提交。业务跨越订单、库存、支付或消息系统后，没有一个普通本地事务能够覆盖全部状态。

分布式事务方案的目标是协调这些独立提交点，或者让失败后的状态最终收敛。选型取决于业务是否允许中间状态、资源能否预留、流程持续时间和补偿难度。

## 2PC 的机制和问题是什么？
{: #two-phase-commit }

两阶段提交先让参与者 Prepare，确认能够提交并保留所需资源，再由协调者通知所有参与者 Commit 或 Rollback。

它能提供较强协调，但 Prepare 后可能长时间持有锁和连接；协调者或网络故障还会让参与者停在不确定状态。它更适合参与者有限、事务短且一致性要求高的场景。

## XA 与 2PC 是什么关系？
{: #xa-two-phase-commit }

2PC 是协调提交的协议思想；XA 定义事务管理器与多个资源管理器之间的标准接口，通常通过两阶段提交完成跨资源事务。

使用 XA 不代表没有失败窗口或性能成本。数据库和驱动的支持、超时、恢复日志与启发式结果都需要验证。

## TCC 的三个阶段分别做什么？
{: #tcc }

Try 检查条件并预留资源，Confirm 正式生效，Cancel 释放预留资源。例如支付 Try 冻结金额，Confirm 完成扣款，Cancel 解冻。

Confirm 与 Cancel 必须幂等，还要处理未执行 Try 却收到 Cancel 的空回滚，以及 Cancel 先到、迟到 Try 又成功的悬挂问题。TCC 隔离较强，但业务侵入也高。

## Saga 适合什么场景？
{: #saga }

Saga 把长流程拆成多个本地事务，每一步提交后通过后续步骤推进，失败时按相反方向执行补偿。它适合履约、预订和审批等耗时较长、无法长期锁住资源的流程。

Saga 中间状态可能被其他请求看到，补偿也不一定能完全撤销现实动作。设计时要明确补偿语义、幂等、日志、重试与人工处理。

## TCC 和 Saga 有什么区别？
{: #tcc-vs-saga }

TCC 在 Try 阶段预留资源，Confirm 后才正式生效，隔离性通常更强，但每项业务都要提供三套接口。

Saga 每步直接提交，再通过补偿回退，适合长流程且吞吐较高，但中间状态可见、补偿更复杂。是否能够预留资源和流程持续时间是主要选择依据。

## Outbox 与事务消息有什么区别？
{: #outbox-vs-transactional-message }

Outbox 先在本地事务中共同写入业务数据和待发送事件，消息系统不可用时可以稍后补发。事务消息通常先在 Broker 创建不可见消息，再执行本地事务，并由 Broker 根据提交结果或回查决定消息是否可见。

两者都解决业务提交与消息发布的协调，不会消除重复投递，消费者仍需幂等。

## 补偿失败后怎么办？
{: #compensation-failure }

补偿也是一次可能失败的业务操作。系统要持久化流程和补偿状态，使用稳定请求 ID 幂等重试，并在超过期限或次数后告警、对账和人工修复。

不能假设所有操作都可逆。已发送通知、外部转账和实物履约可能只能追加更正动作，而不能恢复到从未发生的状态。

## 分布式锁与协调

## 分布式锁需要满足哪些基本性质？
{: #distributed-lock-properties }

同一资源在同一有效时期只能有一个持有者；持有者失联后锁最终能够释放；解锁必须验证所有者，不能删除其他客户端后来取得的锁。

可重入、公平、等待通知和自动续期是增强能力。关键写入场景还需要 Fencing Token，防止已经失去锁的旧执行者迟到写入。

## Redis 分布式锁怎样避免误删？
{: #redis-lock-safe-release }

加锁时使用原子条件写入并保存唯一持有者标识，同时设置过期时间。解锁时通过 Lua 脚本原子地比较持有者标识并删除。

直接 `DEL` 可能删除其他客户端在锁过期后新取得的锁。锁的 Key 还应绑定具体业务资源，避免全局锁串行化无关请求。

## 锁过期但业务仍未完成会发生什么？
{: #distributed-lock-expiration }

锁过期后，另一个客户端可以获得锁，而旧客户端仍可能继续执行，形成两个同时工作的持有者。长时间 GC、网络停顿和执行时间低估都可能触发这个窗口。

自动续期可以降低提前过期概率，但不能提供绝对安全。业务写入仍应幂等，关键资源还应校验 Fencing Token 或版本条件。

## Fencing Token 解决什么问题？
{: #fencing-token }

每次成功获得锁时生成单调递增的 Token，下游只接受 Token 大于已处理值的写入。锁已失效的旧客户端即使后来恢复，其较小 Token 也会被拒绝。

Fencing Token 不会阻止旧代码继续执行，只有所有关键下游都校验 Token 时才有效。

## Redis 锁和 ZooKeeper 锁怎样选择？
{: #redis-vs-zookeeper-lock }

Redis 锁延迟低，适合临界区短、允许过期并有业务幂等兜底的场景。ZooKeeper 使用会话和临时顺序节点，更适合协调与等待通知要求较强的场景，但延迟和运维成本通常更高。

组件名称不能自动证明锁安全。必须分析网络分区、故障切换、会话、租约和下游写入校验。

## 全局唯一 ID 需要满足什么？
{: #distributed-id }

全局 ID 通常要在多节点下避免冲突，并兼顾生成吞吐、可用性、趋势有序、长度和是否暴露业务规模。

数据库号段、UUID 和 Snowflake 各有取舍。Snowflake 类方案组合时间、节点和序列，依赖节点号管理与时钟处理；UUID 简单分散，但较长且随机主键可能影响索引局部性。

## 工程场景题

## 支付接口超时后能直接重试吗？
{: #payment-timeout-retry }

不能仅凭超时判断支付失败。调用可能已经成功，只是响应丢失；使用新请求 ID 重试可能创建第二笔支付。

后端应保存原支付请求 ID，把订单标为正在确认，先查询支付服务的权威结果；明确没有执行后，才使用同一请求 ID 重试。

## 服务刚恢复，为什么又被重试流量压垮？
{: #recovery-retry-storm }

故障期间积累的请求可能按相同间隔同时重试，正常流量也已经恢复，二者叠加超过服务容量。多层重试还会进一步放大输入。

应使用指数退避与随机抖动、限制重试预算、逐步恢复流量，并让服务通过 `Retry-After` 等信号提示调用方等待。

## 分布式锁自动过期后，两个执行者同时写入怎么办？
{: #expired-lock-concurrent-writers }

不能继续把锁当作唯一正确性保证。写入端应校验版本或 Fencing Token，拒绝旧持有者的迟到请求；业务操作还要具备幂等能力。

续期只能减少发生概率，无法覆盖长时间停顿、网络分区和故障切换的所有窗口。

## 跨多个服务的业务怎样保证最终完成？
{: #cross-service-workflow-completion }

先为整个业务建立持久状态机，再让每一步使用本地事务提交自己的事实，通过消息或任务推动下一步。每个动作使用稳定 ID，并保存成功、失败和结果未知状态。

流程失败时按业务选择重试、补偿或人工接管。最终完成来自状态、幂等、恢复和对账，不来自某次调用链永不失败。

## 下游持续变慢时，怎样避免拖垮上游？
{: #slow-downstream-protection }

上游应设置截止时间和并发上限，使用独立线程池或信号量隔离；持续失败时熔断，非核心功能执行真实降级，入口在容量不足时限流或拒绝。

只缩短超时可能制造更多重试，只增加线程则可能压垮下游。保护机制要与业务状态恢复共同设计。
