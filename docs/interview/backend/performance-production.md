# 性能与生产运行常见面试题

生产问题通常不是某个组件单独“变慢了”，而是请求进入速度、处理能力和有限资源之间失去平衡。回答时应从用户结果出发，识别瓶颈与等待，再说明止损、验证、恢复和长期改进。

## 性能与容量

## QPS、吞吐量、并发量和延迟有什么区别？
{: #qps-throughput-concurrency-latency }

- **QPS**：每秒到达的请求数量；
- **吞吐量**：单位时间真正完成的工作量；
- **并发量**：同一时刻正在处理或等待的工作数量；
- **延迟**：单项工作从开始到结束的时间。

并发量近似等于到达速率乘以平均响应时间。延迟升高时，即使 QPS 不变，在途请求也会增加，并继续占用线程、连接和内存。

## 平均延迟、P95 和 P99 分别说明什么？
{: #average-tail-latency }

- **平均延迟**：描述总体水平，但容易掩盖少量极慢请求；
- **P95**：95% 的请求不超过该时间；
- **P99**：99% 的请求不超过该时间，更容易暴露排队、慢节点和长尾依赖。

观察分位数时还要看样本量和直方图。低流量下的单个异常可能让 P99 剧烈波动，不能脱离请求量解释。

## CPU 不高，服务为什么仍可能过载？
{: #low-cpu-overload }

请求可能都在等待数据库连接、锁、远程响应、磁盘或线程池队列，因此 CPU 没有饱和，服务却已经无法及时完成工作。

排查重点是“请求正在等待什么”：连接池获取时间、锁等待、在途请求、队列长度、下游并发和最老任务年龄通常比全局 CPU 更有解释力。

## 利用率接近上限时，延迟为什么会陡增？
{: #utilization-latency-curve }

资源仍有较多空闲时，新请求几乎不排队。利用率接近上限后，前一个任务的微小波动就会让后续请求等待，队列和尾部延迟开始快速增长。

因此容量规划需要保留余量，不能把“平均利用率接近 100%”当成高效。实例故障、流量倾斜和依赖变慢都会消耗这部分余量。

## 线程池、连接池和队列怎样共同限制吞吐？
{: #thread-connection-pool-queue }

- **线程池**：决定同时执行多少任务；
- **连接池**：限制能够并发访问数据库或下游的调用数；
- **队列**：保存尚未获得执行资源的任务。

任何一层先耗尽，都会成为整条链路的瓶颈。线程远多于数据库连接时，大量线程只会等待；连接池过大又可能压垮数据库。容量要按整个集群和下游真实处理能力共同设计。

## 有界队列为什么比无限队列更安全？
{: #bounded-queue }

无限队列会不断保存已经接近超时、取消或失去业务价值的请求，占用内存，并让服务在很久以后才失败。

有界队列在容量达到上限时拒绝或降级，使已经接收的工作有机会完成。队列长度需要结合处理速率、允许等待时间和突发规模设计。

## 什么是背压？
{: #backpressure }

背压让下游把当前处理能力反馈给上游。消费者降低拉取速度、队列满后拒绝、流式发送方等待接收方消费，都是背压。

没有背压时，压力只是被藏进更深的队列和内存。背压可能降低入口吞吐，却能防止整条链路同时超时和崩溃。

## 容量估算应该考虑什么？
{: #capacity-planning }

容量估算可以按以下顺序进行：

1. **确定需求**：明确目标流量、请求组合和延迟目标；
2. **估算消耗**：统计每类请求使用的 CPU、内存、连接和下游配额；
3. **计算容量**：根据单实例稳定吞吐计算实例数；
4. **预留余量**：覆盖实例故障、缓存失效、发布和流量增长；
5. **压测验证**：使用接近生产的数据规模和依赖条件，找到吞吐停止增长、尾部延迟陡升的拐点。

估算只能提供起点，最终容量以验证结果为准。

## 压测最重要的结果是什么？
{: #load-testing-goal }

压测不是得到一个最高 QPS 数字，而是找出系统在哪个流量下开始排队、哪个资源先耗尽、限流是否在失控前生效，以及故障后剩余容量能否承接关键业务。

测试报告应记录版本、数据规模、请求比例、实例数、依赖容量和持续时间，否则结果无法复现，也不能用于生产容量判断。

## 可观测性

## 指标、日志和 Trace 分别解决什么问题？
{: #metrics-logs-traces }

- **指标（Metrics）**：用聚合数据发现范围与趋势，例如错误率、P99 和队列长度；
- **日志（Logs）**：还原具体事件和业务状态；
- **Trace**：展示一次请求跨组件的调用链与时间分布。

三者需要通过请求 ID、业务 ID 和版本关联。只有指标难以解释单笔失败，只有日志和 Trace 又难以判断影响是否广泛。

## 可观测性和监控有什么区别？
{: #observability-vs-monitoring }

- **监控**：针对已知指标和阈值持续观察并告警；
- **可观测性**：强调系统输出是否足以从外部推断内部状态，包括处理事先没有预设的问题。

增加更多日志不等于更可观测。关键是稳定关联、结构化字段、业务状态和版本信息能否支持判断与行动。

## 四个黄金信号是什么？
{: #four-golden-signals }

四个黄金信号包括：

- **延迟**：区分成功、失败和长尾请求的耗时；
- **流量**：系统正在承受的请求或工作量；
- **错误**：既包括明确失败，也包括协议成功但结果错误；
- **饱和度**：资源接近上限的程度。

这些信号是观察入口，不是固定仪表盘模板。业务系统还要补充订单完成率、最老等待任务等用户结果指标。

## RED 和 USE 方法分别关注什么？
{: #red-use-methods }

- **RED**：从服务请求出发，观察请求速率（Rate）、错误率（Errors）和耗时（Duration），适合发现哪个接口异常；
- **USE**：从资源出发，检查利用率（Utilization）、饱和度（Saturation）和错误（Errors），适合定位 CPU、内存、磁盘、网络或连接资源。

两者与业务指标结合，才能从用户失败走到系统原因。

## 告警应该依据什么设计？
{: #alert-design }

分页告警应代表已经或即将影响用户、需要立即行动且有明确处置方式的问题。支付完成率下降、最老任务超过承诺时间通常比 CPU 短暂升高更值得叫醒值班人员。

无法立即行动的趋势可以进入工单或仪表盘。告警过多会造成疲劳，也会掩盖真正紧急的问题。

## 为什么只监控 HTTP 500 不够？
{: #business-failure-with-http-success }

接口可能用 200 返回“处理中”，但用户长时间无法完成支付；模型或任务接口也可能正常返回，业务结果却错误。

事故严重程度应看端到端完成率、受影响用户、资金风险和中间状态最老年龄，而不只是协议错误数量。

## 怎样设计结构化日志？
{: #structured-logging }

日志应使用稳定字段记录时间、级别、操作、结果、请求 ID、业务 ID、依赖、版本和错误类别，便于搜索和聚合。

不要为了排障记录访问令牌、密码或完整个人数据。错误堆栈应与业务上下文关联，但不能把所有大对象序列化进日志。

## Trace 应该记录哪些信息？
{: #distributed-tracing }

Trace 由多个 Span 组成，每个 Span 记录操作名称、开始结束时间、状态、父子关系和必要标签。它应能区分真正执行时间、连接池等待、锁等待和远程调用。

采样需要保留错误、慢请求和高风险业务，同时控制成本。标签基数过高会增加存储和查询压力，敏感数据也不应进入 Trace。

## 生产事故

## 线上事故发生后，最先做什么？
{: #incident-first-response }

先确认用户没有得到什么结果、影响范围和当前风险，然后采取可逆动作限制新影响，例如回滚可疑配置、降低流量、关闭非核心功能或限制重试。

根因调查和止损并行进行。等待完整根因后才行动，会让事故继续扩大；未经判断清空队列或重跑任务，又可能破坏现场并制造重复副作用。

## 为什么事故处理要先止损再定位根因？
{: #mitigation-before-root-cause }

根因验证可能需要较长时间，而影响会持续累积。先用可逆措施缩小故障半径，可以为调查创造稳定环境，并保护权威业务状态。

止损不是随意修改所有参数。动作应针对当前症状、能够快速回退，并保留日志、任务状态和版本等证据。

## 怎样确定事故影响范围？
{: #incident-impact-scope }

先确定开始时间，再按业务流程、地区、租户、版本、接口、依赖和任务阶段拆分指标。选择受影响样本与正常样本进行对照。

单条显眼日志不能代表整体事故。共同特征能帮助判断问题来自某个发布、配置、数据分布还是单一客户环境。

## 怎样验证根因而不是只找到相关性？
{: #root-cause-verification }

提出能够解释现象的假设后，尽量只改变一个因素进行对照，例如少量实例恢复旧配置，并保持流量和其他参数不变。

若问题随因素移除而消失、重新引入又复现，因果证据更强。一次同时扩容、改超时、关重试和升级依赖，即使恢复也难以确认真正原因。

## 服务指标恢复后，事故为什么可能还没结束？
{: #incident-business-recovery }

新请求恢复不代表事故期间产生的积压、结果未知订单和失败任务已经处理。业务恢复通常晚于底层组件恢复。

结束事故前，应确认新流量稳定、历史积压持续下降、未知状态得到核验、没有重复副作用，并且用户完成率恢复。

## 事故复盘应该产出什么？
{: #incident-postmortem }

复盘应说明影响、时间线、触发因素、故障怎样扩散、哪些止损有效，以及系统为何允许问题扩大。改进项应让同类错误更难发生、更早发现或更快恢复。

“加强监控、谨慎操作”不可执行。更好的改进包括配置校验、独立资源预算、业务告警、自动回滚条件和故障演练。

## 发布与演进

## 什么是灰度发布？
{: #canary-deployment }

灰度发布让少量可识别流量先使用新版本，比较新旧版本的技术指标和业务结果，验证稳定后逐步扩大范围。

灰度要有明确停止条件，并覆盖真实业务周期。订单三十分钟后才产生关键结果，观察五分钟就全量无法验证完整承诺。

## 蓝绿发布、滚动发布和灰度发布有什么区别？
{: #deployment-strategies }

- **蓝绿发布**：维护两套完整环境，通过切换流量快速前进或回退，但资源成本较高；
- **滚动发布**：逐批替换实例，资源利用率高，但会经历新旧版本共存；
- **灰度发布**：按用户、流量或任务类型逐步放量，重点是比较新旧结果并限制影响范围。

实际系统可以组合使用，例如基础设施滚动替换，同时按功能开关对业务流量灰度。

## 回滚代码为什么不能恢复全部业务状态？
{: #rollback-business-state }

回滚只恢复执行逻辑。新版本已经写入的数据、发布的消息、创建的任务和调用的外部动作不会自动撤销。

回滚方案还要处理新状态的兼容、迁移和补偿。旧版本必须能读取灰度期间产生的数据，否则回滚会触发第二次故障。

## 数据库 Schema 怎样兼容发布？
{: #schema-compatible-deployment }

数据库 Schema 通常按三步兼容发布：

1. **扩展**：增加新旧版本都能使用的兼容结构；
2. **迁移**：回填数据，逐步切换读写路径；
3. **收缩**：确认旧版本、旧消息和旧任务消失后，再删除旧结构。

直接增加非空字段、立即改名或删除字段，可能让仍在线的旧实例失败。

## 配置变更为什么也要按发布治理？
{: #configuration-deployment }

连接池、超时、并发和功能开关可以瞬间影响所有实例，故障半径可能比代码发布更大。

重要配置需要版本、范围校验、审计、小流量生效、指标关联和已验证的恢复值，不能因为“不需要部署代码”就跳过灰度与回滚。

## 长任务怎样跨越版本发布？
{: #long-running-task-versioning }

任务可能由旧版本创建、新版本执行，或者在等待用户确认时跨越发布。任务应保存稳定数据和必要的执行版本，不能依赖某个进程内的临时对象解释。

新版本要兼容旧任务，或保留旧执行器直到任务结束。存在严重安全问题时，应暂停任务并明确迁移，不能在中途静默切换规则。

## 工程场景题

## CPU 占用过高怎样排查？
{: #high-cpu-troubleshooting }

按“**影响范围 → 节点与进程 → 线程与代码热点 → 变更验证**”排查：

1. 先确认开始时间、受影响实例和用户指标，并区分整体 CPU、单核热点、用户态、系统态、I/O Wait 和虚拟化 Steal；
2. 使用 `top`、`pidstat` 等找到高 CPU 进程和线程；
3. 结合运行时工具查看热点调用栈。Java 可以把高 CPU 线程 ID 对应到线程 Dump，并结合 GC 日志或性能剖析判断是业务计算、死循环、自旋、频繁 GC 还是系统调用；
4. 对照流量、发布和配置变化验证原因，再采取限流、回滚、隔离异常实例或优化热点代码。

相关内容：[监控与故障处理]({{ site.baseurl }}/docs/backend/incident-response/)、[Java 进程 CPU 100% 怎样排查？]({{ site.baseurl }}/docs/interview/backend/java-jvm/#java-high-cpu-troubleshooting)。

## 内存占用过高怎样排查？
{: #high-memory-troubleshooting }

先判断是**容量正常增长、瞬时流量压力，还是无法回收的泄漏**：

1. 确认宿主机和容器的内存、Limit、RSS、文件缓存和 Swap，找到持续增长的进程；
2. Java 服务继续区分堆、Metaspace、直接内存、线程栈和本地内存，观察 Full GC 后的内存基线是否持续上升；
3. 结合 GC 日志、堆 Dump、线程数量和 Native Memory 等现场，定位大对象、无界缓存、队列积压、类加载或线程过多；
4. 先限流、缩小批次或隔离异常实例，再根据证据修复引用、容量或资源上限。

相关内容：[监控与故障处理]({{ site.baseurl }}/docs/backend/incident-response/)、[内存泄漏和内存溢出有什么区别？]({{ site.baseurl }}/docs/interview/backend/java-jvm/#jvm-memory-leak-oom)。

## 磁盘空间或 I/O 异常怎样排查？
{: #high-disk-troubleshooting }

先区分两类问题：

- **空间不足**：使用 `df -h`、`df -i` 判断容量还是 inode 用尽，再用 `du` 定位大目录，用 `lsof` 检查已删除但仍被进程占用的文件；
- **I/O 饱和**：使用 `iostat`、`pidstat -d` 等查看吞吐、延迟、队列和高读写进程，再检查日志暴增、数据库刷盘、临时文件、备份或 Compaction。

止损可以清理或转移确认可删除的数据、恢复日志轮转、扩容磁盘，并限制非核心高 I/O 任务。处理后同时确认磁盘指标和用户请求延迟恢复。

相关内容：[监控与故障处理]({{ site.baseurl }}/docs/backend/incident-response/)。

## P99 延迟升高但错误率正常，怎样排查？
{: #high-p99-normal-error-rate }

可以按以下顺序排查：

1. **缩小范围**：确认慢请求集中在哪个接口、租户、版本和依赖；
2. **拆分耗时**：用 Trace 区分连接等待、锁等待、执行和网络时间；
3. **检查资源**：观察队列、连接池、热点数据和少数慢节点；
4. **核对用户结果**：检查用户完成率和截止时间。

错误率正常可能只是因为请求最终仍成功，或者接口用“处理中”掩盖了超时。

## 流量突增时，为什么扩大线程池可能让系统更差？
{: #thread-pool-expansion-overload }

更大线程池会把更多并发请求送到数据库和下游。如果真正瓶颈是连接、锁或固定配额，只会增加排队、上下文切换和超时。

扩容前先定位限制吞吐的资源，并设置有界队列和下游并发预算。必要时在入口限流，保护已经接收的工作。

## 一千条数据求和与百万级数据聚合应该怎样设计？
{: #large-scale-data-aggregation }

先确认五个条件：**数据位置、更新频率、精度、允许陈旧时间和 SLA**。

- **一千条内存数据**：直接遍历求和；
- **数据已在数据库**：优先使用数据库 `SUM`，避免搬到应用层；
- **百万级数据**：单机或数据库能在期限内完成时仍优先使用；超过单节点能力后，再分区计算局部和并归并；
- **持续数据流**：使用增量聚合，并持久化中间状态和处理位置。

稳定性还要检查分片遗漏或重复、数值溢出与精度、重试幂等、热点分区和结果校验。**数据量不是唯一选型依据，SLA 和现有存储能力更关键。**

## 缓存故障后，为什么数据库容易一起故障？
{: #cache-outage-cascade }

数据库容量通常按正常缓存未命中比例设计，缓存全部失效会让原本由缓存承担的读取瞬间回源。

应限制回源并发、合并热点请求、返回旧副本或降级，并逐步重建缓存。让全部请求直接访问数据库会形成级联故障。

## 消息积压恢复时，为什么不能全速消费？
{: #backlog-recovery-rate }

新流量已经恢复，历史积压再全速回放，会共同争抢下游容量，甚至让依赖再次过载。

恢复速度要受剩余容量控制，并持续观察新消息延迟、积压最老年龄、错误率和业务完成结果。

## 如何定位一次慢接口？
{: #slow-api-troubleshooting }

定位慢接口可以按三步进行：

1. **确认影响**：明确用户看到的延迟、开始时间和影响范围；
2. **拆分链路**：从入口 Trace 拆分网关、线程池、连接池、数据库、远程调用和序列化时间；
3. **验证瓶颈**：结合指标判断是普遍饱和、少数慢节点，还是特定参数和数据分布触发。

优化前要找到主要等待或工作量。只凭经验增加缓存、索引或线程，可能没有作用，甚至扩大瓶颈。
