# 数据库常见面试题

数据库面试通常围绕两件事展开：怎样在并发修改中保证数据正确，以及怎样在数据增长后保持查询高效。

前半部分讨论通用数据库机制，页面末尾的 MySQL 专项讨论 InnoDB 等具体实现。

## 事务与并发

## 什么是数据库事务？
{: #database-transaction }

事务是数据库中的一个逻辑工作单元，其中的操作要么共同提交，要么共同回滚。

事务边界应对应一项必须共同成立的业务结果。例如下单时，库存扣减、订单和订单明细都成功，才能确认下单成功。本地数据库事务不能自动覆盖支付、消息和其他远程服务。

## ACID 分别指什么？
{: #acid }

- **原子性**：事务中的操作共同成功或共同回滚；
- **一致性**：事务前后，数据仍满足约束和业务不变量；
- **隔离性**：并发事务之间的中间状态按隔离规则控制；
- **持久性**：事务提交后，结果能够在故障后保留。

ACID 描述数据库事务的性质，不表示一次跨多个服务的业务操作天然具有相同保证。

## 数据库有哪些常见事务隔离级别？
{: #transaction-isolation-levels }

SQL 标准定义了读未提交、读已提交、可重复读和串行化四种隔离级别。隔离越强，并发事务越接近依次执行，通常也会付出更多等待、冲突检测或重试成本。

不同数据库对隔离级别的具体实现和保证并不完全相同，实际判断应以所用数据库的文档和测试结果为准。

## 脏读、不可重复读和幻读有什么区别？
{: #transaction-read-anomalies }

- **脏读**：读到了另一个尚未提交、以后可能回滚的事务数据；
- **不可重复读**：同一事务两次读取同一行，得到不同值；
- **幻读**：同一事务按相同条件查询两次，结果集合新增或减少了行。

前两者主要关注已经存在的行值，幻读关注满足查询条件的行集合是否变化。

## 什么是 MVCC？
{: #mvcc }

MVCC 是多版本并发控制。数据库为数据保留可见版本，让读取操作按事务规则看到一致的快照，从而减少读写互相阻塞。

MVCC 不等于没有锁，也不会自动避免两个事务同时修改同一行的冲突。旧版本还需要清理，长事务可能延长其保留时间。

## 乐观锁和悲观锁有什么区别？
{: #optimistic-vs-pessimistic-locking }

悲观锁先取得排他性的访问权，再读取或修改数据，适合冲突概率较高、临界区较短的操作，但等待和死锁风险更高。

乐观锁读取时不独占数据，写入时通过版本号或条件更新确认数据未被别人改变。冲突较少时并发能力更好，但调用方必须正确处理更新失败和重试。

## 为什么事务不宜过大或持续太久？
{: #long-transaction }

长事务会更久地占用连接、锁和数据库版本，增加并发等待、冲突概率、日志量与回滚成本。它还可能阻碍旧版本清理或复制进度。

事务应只包含完成当前原子业务结果必需的数据库操作。耗时计算、用户交互和不受本地事务控制的远程调用通常应移出事务。

相关内容：[业务状态与并发控制]({{ site.baseurl }}/docs/backend/business-state/)。

## 索引与查询

## 什么是数据库索引？为什么能加快查询？
{: #database-index }

索引是数据库为特定字段建立的额外数据结构。它让数据库能够从较小范围定位目标记录，避免每次查询都扫描整张表。

索引会占用存储空间，插入、删除和更新时也要同步维护。因此索引应服务真实访问路径，并不是越多越好。

## 为什么数据库索引常使用 B+ 树？
{: #b-plus-tree-index }

B+ 树分支多、层级浅，可以用较少的页面读取定位数据；叶子节点按顺序组织，也适合范围查询、排序和顺序扫描。

哈希结构适合等值查找，但通常不擅长范围和有序访问。具体数据库还可能提供哈希、全文、空间等其他索引类型。

## 什么是联合索引的最左前缀？
{: #composite-index-leftmost-prefix }

联合索引按定义的字段顺序组织。例如索引 `(buyer_id, status, created_at)`，通常可以直接支持从 `buyer_id` 开始的查询，也可以继续利用后面的 `status` 和 `created_at` 缩小范围或提供顺序。

若查询跳过最左侧字段，只按 `status` 查找，这份索引通常难以直接定位连续范围。字段顺序应根据实际过滤、排序和数据分布设计，而不是把常用字段随意排列。

## SQL 中写了索引字段，为什么仍可能不走索引？
{: #index-not-used }

优化器会选择预计成本最低的执行计划，而不是看到索引就一定使用。常见原因包括：条件会命中表中大部分数据、表本身很小、函数或隐式类型转换改变了比较方式、前导通配符无法定位范围，以及统计信息与真实数据分布不符。

是否使用索引应查看实际执行计划，不能只凭 SQL 外观判断。

## 怎样使用执行计划排查慢查询？
{: #query-execution-plan }

先确认数据库选择了全表扫描还是索引访问，再看实际扫描行数、过滤位置、连接顺序、排序或临时结构，以及时间主要花在哪个算子。

执行计划要和真实参数、数据分布一起分析。一条 SQL 在多数用户下很快，仍可能因为某个大租户或时间范围产生完全不同的成本。

## 什么是 N+1 查询？
{: #n-plus-one-query }

N+1 查询是先用一次查询取得 N 条主记录，再为每条记录分别执行一次关联查询，最终产生 `1 + N` 次数据库往返。

常见解决方式是批量查询关联数据、合理使用连接或 ORM 预加载。目标不是强行只写一条 SQL，而是避免查询次数随着结果条数无意增长。

相关内容：[数据库查询性能]({{ site.baseurl }}/docs/backend/database-query-performance/)。

## 工程场景题

## 两个用户同时购买最后一件商品，怎样避免超卖？
{: #prevent-overselling }

不能只在应用中先查询库存，再根据查询结果扣减。两个请求可能同时读到库存为 1，随后各自创建订单。

应在写入发生时执行带库存条件的原子更新，并检查受影响行数。只有成功扣减库存的请求才能继续创建订单；库存扣减和订单写入共同构成下单结果时，还应放在同一个数据库事务中。

## 两个客户端同时修改订单，怎样避免后提交者覆盖先提交的结果？
{: #prevent-lost-update }

可以使用版本号进行乐观并发控制。客户端读取订单时取得版本，更新时把原版本放进条件，并在成功后增加版本号。

更新零行表示订单已经被其他请求修改，客户端应读取最新状态后重新决定，不能直接覆盖。高冲突场景也可以使用悲观锁，但需要评估等待和死锁成本。

## 为什么不应该在数据库事务中等待远程支付调用？
{: #remote-call-inside-transaction }

远程调用耗时和结果都不受本地数据库控制。把它放进事务会长时间占用连接和锁；即使本地事务回滚，也无法撤销支付平台已经完成的动作。

更稳妥的做法是先持久化支付意图和稳定请求 ID，再在事务之外调用支付服务，通过状态查询、幂等和补偿处理结果未知。

## 一条查询只返回 20 行，为什么仍可能很慢？
{: #small-result-slow-query }

返回行数不等于数据库实际处理的行数。没有合适访问路径时，数据库可能扫描数百万行、过滤、排序，最后才取前 20 行。

应查看实际执行计划，确认扫描行数、索引使用、排序和连接成本，同时区分连接池等待、锁等待与 SQL 执行时间。

## 为什么深分页会越来越慢？
{: #deep-pagination }

使用较大的 `OFFSET` 时，数据库通常仍要找到并跳过前面的记录，页码越深，扫描和丢弃的工作越多。

对于稳定排序的数据，可以使用游标分页：把上一页最后一条记录的排序字段和唯一 ID 作为下一页起点，使每页都从明确位置继续。

## 执行计划看起来正常，查询为什么仍然可能很慢？
{: #slow-query-with-good-plan }

时间可能没有花在访问数据上，而是花在等待数据库连接、等待其他事务释放锁、读取冷数据、传输大量结果或应用反序列化。

排查时应拆分各阶段耗时，并结合当时的参数、数据分布和并发状态，不能只看一份静态 `EXPLAIN`。

## MySQL 专项

MySQL 是一个客户端—服务端关系数据库。客户端发送 SQL，Server 层负责连接、解析、优化和执行，存储引擎负责表数据、索引、事务与恢复；最常用的 InnoDB 以页为基本读写单位，用 B+ 树组织索引，并通过 Buffer Pool、redo log、undo log 和 MVCC 协同完成读写。

以下问题主要针对使用 InnoDB 的 MySQL，不应直接推广到所有关系型数据库。

## InnoDB 的聚簇索引是什么？
{: #mysql-clustered-index }

InnoDB 的表数据按主键索引的叶子节点组织，因此主键索引也称聚簇索引。通过主键查询可以直接定位整行数据。

一张 InnoDB 表只有一个这样的数据组织顺序。没有显式合适主键时，InnoDB 会选择或生成内部行标识，但通常仍应由应用设计稳定、较短且递增性合理的主键。

## InnoDB 的二级索引为什么会回表？
{: #mysql-secondary-index-table-lookup }

InnoDB 二级索引的叶子节点通常保存索引字段和主键值，而不是完整数据行。查询需要索引之外的列时，数据库先从二级索引找到主键，再到聚簇索引读取整行，这称为回表。

如果查询需要的字段都能从该索引取得，就可能形成覆盖索引，避免额外回表。

## MySQL 的可重复读怎样处理幻读？
{: #mysql-repeatable-read-phantom }

InnoDB 在普通一致性读取中通过 MVCC 快照提供可重复读；在需要锁定的当前读中，会使用记录锁和间隙相关的锁定机制，防止其他事务在受保护范围内插入新记录。

具体行为受查询条件、索引、读方式和事务隔离级别影响。“MySQL 可重复读完全没有幻读”过于笼统，面试时应说明是在讨论快照读还是当前读。

## MySQL 中什么是当前读和快照读？
{: #mysql-current-read-vs-snapshot-read }

普通 `SELECT` 在 InnoDB 中通常是快照读，按照 MVCC 读取符合事务可见性规则的数据版本。

`SELECT ... FOR UPDATE`、`SELECT ... FOR SHARE` 以及更新和删除需要读取当前可修改版本，属于当前读，并会取得相应锁。两者看到的数据和并发影响可能不同。

## MySQL 为什么可能发生死锁？怎样处理？
{: #mysql-deadlock }

两个事务以不同顺序持有对方需要的锁，就可能形成循环等待。InnoDB 会检测死锁并回滚其中一个事务，让另一个继续。

应用需要捕获死锁错误，并在操作具备安全重试条件时重新执行整个事务。工程上还应统一加锁顺序、缩短事务并确保条件使用合适索引，从源头减少锁范围和等待。

## MySQL 的 binlog、redo log 和 undo log 分别做什么？
{: #mysql-binlog-redo-undo }

- **binlog**：MySQL Server 层的逻辑变更日志，常用于复制和基于时间点恢复；
- **redo log**：InnoDB 的重做日志，用于崩溃恢复并支持已提交修改的持久性；
- **undo log**：保存修改前的信息，用于事务回滚和 MVCC 读取旧版本。

三者服务的目标不同，不能把它们都简单称为“用于恢复的日志”。
