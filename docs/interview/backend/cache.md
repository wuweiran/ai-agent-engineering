# 缓存常见面试题

缓存问题考察怎样在权威数据之外维护一份可能过期的副本。回答时应先说明权威数据在哪里、允许缓存旧多久，以及缓存故障后怎样保护下游。

前半部分讨论通用缓存机制，页面末尾的 Redis 专项讨论具体产品实现。

## 缓存设计

## 为什么使用缓存？哪些数据适合缓存？
{: #why-use-cache }

缓存把经常读取或计算昂贵的结果保存在更快的位置，用额外存储和一定的数据陈旧风险换取更低延迟与更高读取容量。

适合缓存的数据通常读取频繁、变化相对较少，能够从权威系统重建，并且允许在明确时间内使用旧值。余额、支付结果等关键状态不能只依赖缓存决定写入结果。

## 什么是 Cache Aside？
{: #cache-aside }

Cache Aside 是由应用同时管理缓存和数据库的常见模式：读取时先查缓存，未命中则读取数据库并回填；写入时先更新数据库，再让对应缓存失效。

缓存只是读取副本，数据库仍是权威来源。该模式实现简单，但数据库更新和缓存失效之间仍存在短暂不一致窗口。

## 为什么通常先更新数据库，再删除缓存？
{: #update-database-then-delete-cache }

如果先删除缓存再更新数据库，中间到达的读请求可能读取旧数据库值，并把旧值重新写回缓存。先提交数据库，再删除缓存，可以避免这个更常见的覆盖顺序。

后者仍不是原子操作：数据库提交后，服务可能在删除缓存前崩溃。因此还需要删除重试、变更事件或 TTL，避免旧值永久存在。

## 更新数据库后删除缓存失败怎么办？
{: #cache-invalidation-failure }

不能假装数据库与缓存已经同时成功。常见做法是记录可重试的失效事件，通过消息或 Outbox 再次删除缓存，并用 TTL 限制旧值的最长存活时间。

如果业务要求读取立即看到新值，仅靠普通 Cache Aside 通常不够，需要更严格的读写路径或暂时绕过缓存。

## TTL 有什么作用？设置 TTL 就能保证缓存一致吗？
{: #cache-ttl }

TTL 使缓存到期回收，也限制旧数据不会无限期存在。TTL 越短，数据通常更新得更及时，但回源和重建次数也越多。

TTL 不能保证数据库更新后缓存立即变化，只能限制陈旧窗口。大量 Key 同时到期还会形成流量尖峰，通常需要给过期时间加入随机偏移。

## 缓存和数据库怎样保证强一致？
{: #cache-strong-consistency }

数据库与独立缓存通常无法通过一次普通本地事务共同提交，因此很难同时获得低延迟、高可用和严格强一致。

真正要求强一致的读取应直接访问权威存储，或者使用能够提供相应一致性保证的统一数据系统。工程中更常见的是明确允许的陈旧时间，并用失效、重试和版本机制缩短不一致窗口。

相关内容：[缓存怎样加速读取而不破坏正确性](../../backend/cache-consistency/)。

## 常见缓存故障

## 缓存穿透、缓存击穿和缓存雪崩有什么区别？
{: #cache-penetration-breakdown-avalanche }

- **缓存穿透**：查询的数据本来就不存在，请求每次都绕过缓存访问数据库；
- **缓存击穿**：一个高流量热点 Key 失效，大量请求同时回源；
- **缓存雪崩**：大量 Key 同时失效或缓存服务整体故障，大范围流量集中回到下游。

三者都会增加数据库压力，但原因和处理方法不同。

## 怎样防止缓存穿透？
{: #prevent-cache-penetration }

入口先校验明显非法的请求；权威存储确认数据不存在后，可以短时间缓存空结果。数据规模和场景合适时，也可以用布隆过滤器快速排除一定不存在的 Key。

数据库超时或临时错误不能缓存为“不存在”，否则会把故障结果持续返回给用户。恶意随机查询还需要限流，不能只依赖缓存处理。

## 怎样防止热点 Key 失效造成缓存击穿？
{: #prevent-cache-breakdown }

可以让一个请求取得回填权，其他请求短暂等待，合并同一个 Key 的数据库查询。允许短暂陈旧时，也可以继续返回旧值，由后台执行者刷新缓存。

回填锁必须有过期和等待上限，避免持有者崩溃后所有请求永久阻塞。热点 Key 还可以提前刷新，减少自然到期时的尖峰。

## 怎样减少缓存雪崩？
{: #prevent-cache-avalanche }

让过期时间分散，避免大量 Key 在同一时刻失效；缓存服务采用合适的高可用方案；应用限制回源并发，并为非核心请求准备旧值、本地副本、降级响应或快速失败。

缓存恢复时也应逐步重建热点，不能让全部请求同时回填。

## 缓存整体不可用时，为什么不能让所有请求直接访问数据库？
{: #cache-failure-database-fallback }

数据库的容量通常按正常缓存未命中比例设计，不一定能承受缓存平时承担的全部流量。无限回源可能把一次缓存故障扩大成数据库和整个服务故障。

系统需要预先设置回源预算，通过并发限制、请求合并、降级和限流保护数据库。缓存故障时，宁可拒绝部分非关键流量，也不能让权威存储失去服务能力。

## 什么是缓存预热？
{: #cache-warm-up }

缓存预热是在流量到来前，提前加载预计会被频繁访问的数据。例如发布新版本或缓存集群恢复后，先加载热门商品和配置。

预热不能保证覆盖所有热点，也可能加载很快过期的数据。它应根据真实访问统计选择对象，并限制加载速度，避免预热本身压垮数据库。

## 工程场景题

## 商品缓存显示有库存，怎样保证下单不会超卖？
{: #cached-stock-prevent-overselling }

缓存中的库存只适合展示和过滤请求，不能决定最终成交。用户读取缓存后，库存可能已经被其他订单扣减。

下单时仍要在权威库存系统执行带条件的原子扣减，并根据写入结果决定是否创建订单。缓存短暂陈旧可以影响体验，但不能突破交易约束。

## 更新数据库后删除缓存失败，怎样限制不一致时间？
{: #cache-deletion-failed-after-update }

数据库提交后，旧缓存可能继续被读取。系统应保存可重试的失效事件，通过消息或 Outbox 再次删除，并用 TTL 限制旧值不会永久存在。

如果某类读取不能接受陈旧数据，应暂时绕过缓存或使用更严格的读取路径，不能只依赖后台重试最终成功。

## 热点 Key 失效后，大量请求同时到达怎么办？
{: #hot-key-expiration }

不能让所有请求同时回源。可以只允许一个执行者加载新值，其他请求短暂等待；允许陈旧时，也可以继续返回旧值，由后台刷新。

回填需要等待上限和失败退路。还可以提前刷新或错开过期时间，减少失效瞬间的流量尖峰。

## 缓存服务整体故障后，怎样避免数据库被压垮？
{: #cache-outage-protect-database }

先限制回源并发，只让数据库承担预先评估过的流量。热门请求可以合并，非关键内容可以返回旧副本、降级结果或快速失败。

缓存恢复后应逐步放量和重建热点。让全部缓存流量直接访问数据库，通常会把单点故障扩大为整个系统故障。

## 缓存命中率很高，为什么业务结果仍可能有问题？
{: #high-hit-rate-wrong-business-result }

命中率只说明多少请求没有访问下游，不说明返回的数据是否及时和正确。缓存可能长期返回旧价格、旧权限或错误的空结果，同时保持很高命中率。

除了命中率，还要观察数据更新延迟、回源压力以及陈旧数据造成的用户结果，并明确权威状态与允许的陈旧时间。

## 哪些数据不适合通过普通缓存路径决定结果？
{: #data-not-suitable-for-cache-decision }

不能容忍陈旧、涉及权限或资金、并且会直接决定写入结果的数据，不应只依赖普通异步失效缓存。例如账户余额、支付确认和最终库存资格。

这些数据可以为展示提供缓存副本，但执行交易或授权时必须回到权威系统确认。是否适合缓存，取决于旧值被使用后会造成什么后果。

## Redis 专项

以下问题针对 Redis 的实现和生产使用，不代表所有缓存系统都采用相同机制。

## Redis 为什么快？
{: #why-redis-is-fast }

Redis 的主要数据操作在内存中完成，避免了大多数磁盘随机访问；核心命令执行路径较短，常用数据结构也针对相应操作进行了优化。

“单线程”不是完整答案。Redis 还会使用后台线程或 I/O 线程处理部分网络、持久化和清理工作。大 Key、慢命令、网络传输和持久化仍然可能造成延迟。

## Redis 常见数据结构适合什么场景？
{: #redis-data-structures }

- String：普通缓存值、计数器和简单状态；
- Hash：保存对象的多个字段；
- List：有序序列和简单队列；
- Set：去重集合和交并集；
- Sorted Set：排行榜、按分数或时间排序；
- Stream：带消费位置的消息流。

选择结构要看应用需要执行什么操作，而不只是数据长什么样。复杂或高可靠消息场景仍应评估专门的消息系统。

## Redis 怎样删除过期 Key？
{: #redis-expiration-deletion }

Redis 结合惰性删除和定期删除。读取 Key 时发现它已过期会将其删除；后台也会周期性抽样清理过期 Key，避免从未再次访问的数据长期占用内存。

过期时间到达不表示 Key 一定在那个瞬间完成物理删除，但对正常读取来说它已经不可见。

## RDB 和 AOF 有什么区别？
{: #redis-rdb-vs-aof }

RDB 定期生成数据快照，文件紧凑、恢复较快，但故障时可能丢失最近一次快照后的数据。AOF 记录写命令，通过刷盘策略在持久性和性能之间取舍，文件通常更大，也需要重写。

两者可以组合使用。即使启用持久化，Redis 能否作为某项业务的权威存储，仍取决于复制、故障切换和业务数据语义。

## 什么是 Redis 大 Key？有什么影响？
{: #redis-big-key }

大 Key 是占用大量内存或包含大量元素的单个 Key。读取、删除、序列化或迁移它时会产生较长执行时间、较大网络流量和内存压力，也可能阻塞其他请求。

应限制单个值或集合的规模，按业务维度拆分，并使用渐进式扫描和删除能力，避免在线执行一次性大操作。

## 什么是 Redis 热 Key？有什么影响？
{: #redis-hot-key }

热 Key 是访问量远高于其他 Key 的少数数据。它可能使某个 Redis 分片、网络连接或应用回填路径成为瓶颈，即使集群总体资源仍然充足。

常见处理包括本地短期缓存、请求合并、只读副本、拆分可拆的数据，以及避免热点在同一时刻失效。具体方式取决于是否允许旧值，以及操作是否只读。
