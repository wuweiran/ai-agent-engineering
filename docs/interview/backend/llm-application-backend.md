# 大模型应用后端常见面试题

大模型应用后端把一个慢、昂贵、输出不完全确定的计算环节接入真实产品。面试重点不是 Prompt 技巧，而是流式交付、工具执行、持久状态、幂等恢复、容量成本和安全边界。

## 流式生成与事件

## 什么是流式生成和流式返回？
{: #streaming-generation-return }

- **流式生成**：模型按照自回归过程逐步产生 Token；
- **流式返回**：应用收到文本片段后立即转发给客户端，不等待完整回答。

链路通常是“模型逐步生成 → 模型 API 返回 Delta → 应用后端转发 → SSE 等协议推送 → 客户端拼接展示”。流式返回是交付方式，不代表模型一次性生成了完整内容，也不代表结果已经持久化。

相关内容：[流式生成与任务事件]({{ site.baseurl }}/docs/backend/llm-backend/streaming-generation/)。

## 为什么大模型回答通常使用流式返回？
{: #llm-streaming-purpose }

模型逐 Token 生成，完整回答可能需要数秒或更久。流式返回让用户提前看到内容和任务阶段，降低感知等待时间，也能展示工具调用与确认状态。

流式传输只改善过程交付，不缩短模型完整计算，也不能代替最终结果持久化。

## SSE 和 WebSocket 怎样选择？
{: #sse-vs-websocket }

- **SSE**：通过普通 HTTP 让服务器持续向客户端发送事件，协议简单、支持自动重连，适合文本和任务进度的单向推送；
- **WebSocket**：提供全双工长连接，适合双方都需要频繁、低延迟发送消息的场景。

大模型输出通常是客户端用普通 POST 提交请求，服务器持续推送结果，因此 **SSE 往往足够**。需要低延迟双向控制、音视频或复杂协作时再考虑 WebSocket。

## SSE 事件应该包含哪些字段？
{: #sse-event-schema }

SSE 事件至少包含：

- **事件类型**：区分文本增量、工具开始、工具完成、等待确认、任务完成和失败；
- **任务或生成 ID**：关联同一次任务；
- **任务内序号**：用于发现缺失、排序和去重；
- **负载**：保存当前事件需要传递的数据。

结束事件应指向已经持久化、能够重新读取的最终结果。

## Token Delta、草稿和最终消息有什么区别？
{: #delta-draft-final-message }

- **Token Delta**：当前连接收到的即时片段，可能只有半句话；
- **草稿**：后端已经合并、但尚未提交的内容；
- **最终消息**：模型正常结束并通过必要检查后持久化的结果。

页面可以临时展示 Delta，但其他业务逻辑不应把它当作完整回答。**最终消息保存成功后，才能发布完成事件。**

## SSE 断线后怎样恢复？
{: #sse-reconnect-replay }

服务端为事件分配单调序号或事件 ID，客户端重连时携带最后已接收位置，服务端补发后续持久事件，再继续推送新事件。

高频 Token Delta 不一定全部长期保存，也可以返回当前草稿快照；关键是避免重复拼接和缺失内容。任务已经完成时，直接读取最终消息即可。

## 客户端断开连接后，Agent 任务应该停止吗？
{: #disconnect-vs-cancel }

断线只说明当前订阅消失，可能来自刷新页面或网络切换；取消才是用户明确要求停止任务。短回答可以选择断线即取消，长 Agent 任务通常应独立继续。

执行生命周期由持久任务控制，SSE 只是观察通道。产品必须明确策略，不能偶然取决于哪个进程持有连接。

## Nginx 或网关为什么可能让 SSE 看起来不实时？
{: #sse-proxy-buffering }

代理可能缓冲响应，积累到一定大小后才发送；空闲超时也可能关闭长连接。服务端和网关需要正确设置流式响应、关闭不合适的缓冲，并在长时间无事件时发送心跳。

心跳只维持连接，不代表任务仍有进展。任务存活和进度仍要从持久状态判断。

## Tool Call 与业务执行

## 模型输出 Tool Call 后，后端要做什么？
{: #tool-call-backend-flow }

后端收到 Tool Call 后按以下链路处理：

1. **查找工具**：只从已注册工具中匹配执行器；
2. **校验参数**：按 Schema 检查字段、类型和枚举；
3. **绑定身份**：从认证上下文绑定当前用户和租户；
4. **业务检查**：重新验证权限、资源状态和业务条件；
5. **确认与落库**：高风险动作等待确认，并保存执行意图；
6. **执行与回传**：调用业务服务，持久化结果，再把受控 Tool Result 交给模型。

**模型提出动作，软件决定能否执行，领域服务确认真实结果。**

## Tool Call 的 Schema 校验为什么不等于业务校验？
{: #tool-schema-vs-business-validation }

- **Schema 校验**：只能证明字段、类型和枚举符合工具契约；
- **业务校验**：确认订单属于当前用户、金额合法，并且订单当前仍允许修改；
- **授权校验**：使用可信认证上下文，在服务端重新检查资源权限。

Prompt 中的“只处理当前用户订单”不能代替授权。

## 模型为什么不能直接调用数据库或内部 HTTP？
{: #tool-business-boundary }

任意 SQL 和 HTTP 能力难以限制权限、参数和副作用，也会把底层结构暴露给模型。更安全的工具应表达业务动作，例如查询订单、检查退款资格和创建待审申请。

专用工具让业务服务继续掌握规则，并为审批、幂等、审计和错误语义提供稳定边界。

## 哪些 Tool Call 需要用户确认？
{: #tool-approval }

- **低风险动作**：读取和可安全重做的操作可以自动执行；
- **有副作用动作**：修改数据、支付、退款、部署和删除，需要根据金额、可逆性和责任边界决定用户确认或人工审批。

确认前应保存具体方案和参数。**用户确认的是已经展示的动作，不能确认后再让模型重新生成另一组参数。**

## 有副作用的 Tool Call 怎样实现幂等？
{: #tool-call-idempotency }

先看业务动作能否天然幂等。设置标签、把状态改成指定值等接口，重复使用相同资源 ID 和参数可以得到同一结果；涉及并发覆盖时再增加资源版本条件。

创建支付、退款申请等可能产生第二份结果的动作，需要稳定业务请求 ID 或 Idempotency-Key。需要跨进程恢复时，再持久化 Tool Call、参数摘要和处理状态。只依赖模型生成的 `tool_call_id` 不够，因为模型重新规划后可能产生新的调用 ID。

## 工具调用超时后，模型能否自行重试？
{: #tool-timeout-retry }

重试方式取决于工具契约：

- **只读查询或天然幂等写入**：可以在预算内有限重试；
- **支持幂等键的写入**：重试时必须复用同一业务请求 ID；
- **无法安全重做的高风险动作**：超时后先查询权威系统的真实结果。

模型可以提出重试建议，但由 Runtime 根据幂等性、结果查询能力和副作用风险决定是否执行。**高风险写入仅凭超时不能直接重复。**

## 工具结果应该怎样返回给模型？
{: #tool-result-contract }

Tool Result 应包含：

- **稳定业务字段**：只返回当前判断真正需要的信息；
- **资源 ID 和状态**：让模型能够关联业务对象；
- **明确结果语义**：区分成功、拒绝、暂时失败和结果未知；
- **错误码与下一步**：说明是否可以重试、改参数或停止。

不应把整份内部响应、堆栈和敏感数据直接放入 Context。

## Prompt Injection 怎样影响 Tool Call 安全？
{: #prompt-injection-tool-security }

网页、文档和工具结果中的文本可能诱导模型忽略原任务、泄露数据或调用高权限工具。这些内容只是外部数据，不能改变系统权限和业务规则。

系统应隔离不可信内容，限制工具集合和参数范围，在执行层重新鉴权，并为高风险动作保留确认。仅在 System Prompt 中写“忽略恶意指令”不够。

## Agent 任务持久化

## 为什么 Agent 任务不能只保存聊天记录？
{: #agent-task-vs-conversation }

聊天记录说明用户和模型说过什么，却不能可靠证明工具是否执行、用户确认了哪组参数、外部写入是否结果未知，以及 Worker 应从哪里恢复。

任务状态需要独立保存目标、总体阶段、执行尝试、稳定步骤和 Tool Call。领域服务继续掌握订单、支付等业务真相。

## Task、Run、Step 和 Tool Call 分别表示什么？
{: #task-run-step-tool-call }

- **Task**：用户目标和总体状态；
- **Run**：一次具体执行尝试，以及使用的模型、Prompt 和工具版本；
- **Step**：可恢复的模型、工具或人工步骤；
- **Tool Call**：单独记录远程副作用、执行状态和确认结果。

这四层让系统区分“同一任务的新尝试”和“不能重复的旧业务动作”。表名可以变化，恢复所需事实不能丢失。

## 为什么 Run 要记录模型、Prompt 和工具版本？
{: #agent-run-versioning }

模型、Prompt、Skill、工具 Schema 和执行策略都会改变 Agent 行为。只保存应用提交号，失败后无法重现模型当时实际看到和使用的组合。

任务跨越发布时，Run 还要明确继续使用原版本还是经过专门迁移，不能随 Worker 当前配置静默漂移。

## 什么样的 Step 才能作为恢复点？
{: #agent-step-checkpoint }

只有输入、输出和外部结果都已持久化的稳定步骤才能作为恢复点。半句流式文本、仍在执行的工具和结果未知的写操作都不能标记成功。

并行步骤还需要依赖关系和独立调用 ID，不能只按数据库写入顺序推断执行关系。

## Worker 重启后怎样恢复 Agent 任务？
{: #agent-worker-recovery }

Worker 重启后按以下步骤恢复：

1. **取得执行权**：读取 Task 并获取租约；
2. **创建新尝试**：创建新的 Run，找到最近一个稳定 Step；
3. **核验副作用**：检查 `executing` 或 `unknown` 的 Tool Call，向领域服务查询真实结果；
4. **重建 Context**：根据持久状态重新组装模型所需信息；
5. **继续执行**：跳过已经稳定完成的步骤，继续未完成部分。

恢复不是重放全部聊天，也不是从头执行所有工具。**状态决定哪些步骤可以跳过，权威业务结果决定哪些动作不能重做。**

## 租约和业务幂等分别解决什么问题？
{: #agent-lease-vs-idempotency }

- **租约和心跳**：决定哪个 Worker 暂时拥有推进任务的权利，减少多个实例长期并发执行；
- **业务幂等和条件写入**：保证即使短暂重复执行，也不会产生重复业务结果。

网络分区和租约过期边缘仍可能出现短暂重叠。因此，**租约不能替代幂等，幂等也不能替代任务调度所有权。**

## Agent 任务取消怎样处理已经发生的动作？
{: #agent-cancellation-compensation }

取消可以阻止尚未执行的步骤，也可以尽力中止等待中的模型或网络调用。已经发送的消息、完成的退款和修改的数据不会自动撤销。

需要撤销时，领域系统必须提供合法补偿。任务取消和业务补偿是两个不同状态与操作。

## Event 和 Artifact 分别有什么作用？
{: #agent-event-artifact }

- **Task**：回答任务当前处于什么状态；
- **Event**：记录状态怎样变化，支持前端展示、断线续传和审计；
- **Artifact**：保存报告、文件、变更回执等可读取的最终交付物。

不是每个 Token Delta 都值得长期保存。

## Context 与任务状态有什么区别？
{: #context-vs-task-state }

- **Context**：下一轮模型当前能看到的信息工作集，包括目标、规则、近期结果和工具定义；
- **任务状态**：系统持久保存的目标、事实、进度和恢复依据。

**Context 可以压缩和重建，任务状态必须可靠。**不能把数据库全部历史塞进模型，也不能把完整对话当作任务数据库。

## 成本、容量与生产运行

## 大模型应用怎样控制 Token、延迟和成本？
{: #llm-token-latency-cost }

可以从四处控制成本和延迟：

- **Context**：删除无关历史，压缩过大的工具结果；
- **执行预算**：限制输出长度、模型轮次和工具调用次数；
- **模型选择**：按任务复杂度选择满足质量要求的模型；
- **缓存**：对稳定 Prompt 前缀使用 Prompt Cache，对低风险重复结果使用结果缓存。

优化要同时观察**任务成功率、严重错误、输入输出 Token、首个结果等待和完整耗时**。成本下降但任务质量退化不算有效优化。

## 模型并发限制应该放在哪里？
{: #llm-concurrency-limits }

模型并发需要分层限制：

1. **入口层**：按用户、租户和任务类型控制接收量；
2. **任务调度层**：限制同时运行的 Agent 数量；
3. **模型网关层**：按供应商、模型和配额限制请求；
4. **工具层**：为昂贵或容量有限的工具设置独立并发预算。

只限制 SSE 连接数不能代表生成容量，因为一个连接可能没有模型调用，一项后台任务也可能在没有连接时持续调用模型。

## 模型 API 限流后怎样重试？
{: #llm-rate-limit-retry }

遵循供应商的重试提示，只对暂时性限流和连接失败进行有限退避，并加入随机抖动。重试要受任务截止时间、尝试次数和全局重试预算限制。

大量任务同时醒来会形成重试风暴。任务可以进入等待状态，由调度器错峰恢复，而不是占住 Worker 睡眠。

## 模型调用、Tool Call 和整个 Run 的超时怎样分层？
{: #llm-timeout-layers }

- **模型调用超时**：限制一次生成最多等待多久；
- **Tool Call 超时**：限制某个外部动作最多等待多久；
- **Run 截止时间**：限制整个任务最多运行多久。

下层只能使用上层剩余预算的一部分。模型超时通常可以开始新的生成尝试；有副作用工具超时则必须先确认真实结果。**不同层不能使用同一个默认超时和重试规则。**

## 怎样缓存大模型应用？
{: #llm-application-caching }

- **精确结果缓存**：复用完全相同输入的结果；
- **语义缓存**：在低风险、允许陈旧的场景复用相似问答；
- **Prompt Cache**：复用稳定 Prompt 前缀的计算。

缓存键要包含模型、Prompt、工具定义、权限和关键 Context 版本。动态订单、个性化权限和高风险建议不能只按用户问题复用。

## Redis 在 AI Agent 系统中可以承担什么职责？
{: #redis-agent-use-cases }

Redis 适合承担**高频、临时、可丢失或可重建**的状态：

- 限流计数和并发令牌；
- 精确结果缓存、语义缓存和短期 Tool Result；
- 租约、热点任务进度和 SSE 订阅位置；
- 使用 Stream 做较轻量的事件传递。

边界是：任务目标、审批、Tool Call 副作用和最终业务状态仍进入可靠数据库；锁和租约不能代替业务幂等。缓存 Key 要包含租户、权限、模型、Prompt、工具和知识版本，TTL 由允许的陈旧时间与恢复能力决定。

## 消息队列在 AI Agent 系统中解决什么问题？为什么不只用数据库轮询？
{: #message-queue-agent-tasks }

职责要分开：

- **数据库**：保存 Task、Step、Tool Call 和领域状态，是权威事实；
- **消息队列**：分发任务 ID，吸收突发流量，并提供确认、重试、分区和背压；
- **Worker**：收到消息后读取最新任务状态，再决定是否推进。

数据库轮询适合小规模方案；大量 Worker 高频扫描会带来空查询、热点锁和调度延迟，还要自己实现领取、公平和可见性超时。队列也不能代替数据库，因为消息可能重复或过期，消费者仍需幂等和状态校验。

## Kafka 在 AI Agent 系统中适合哪些场景？怎样保证消息顺序？
{: #kafka-agent-use-cases }

**适用场景：**Kafka 适合高吞吐、需要保留和重放、并由多个消费组独立读取的 Agent 事件流，例如任务状态变化、Tool Call 审计、知识索引更新、监控、计费和分析。只需要简单任务队列语义时，不必因为系统使用 Agent 就选择 Kafka。

**顺序保证：**

- Kafka 只保证单个 Partition 内有序；
- 同一 Task 或业务对象使用 `task_id`、`order_id` 等稳定 Key，写入同一 Partition；
- 一个消费组内，该 Partition 由单一消费序列处理；
- 重试、重放、跨 Topic 和扩容仍可能产生迟到消息，因此事件要携带事件 ID、业务版本和前置状态，消费者做幂等与状态机校验。

Task、Step 和业务事实仍保存在数据库中，Kafka 负责事件传输和重放。

相关内容：[怎样保证消息顺序？]({{ site.baseurl }}/docs/interview/backend/message-queue/#message-ordering)、[Kafka 怎样选择 Partition Key？]({{ site.baseurl }}/docs/interview/backend/message-queue/#kafka-partition-key)。

## 大模型应用应该记录哪些可观测数据？
{: #llm-observability }

记录任务和 Run ID、模型与配置版本、每次调用延迟、输入输出 Token、停止原因、工具调用状态、错误和最终任务结果。

Prompt 与工具结果可能包含敏感信息，Trace 需要脱敏、访问控制和保留期限。接口成功率正常不代表任务完成率正常。

## 模型、Prompt 和工具变化怎样安全发布？
{: #llm-release-versioning }

模型、Prompt 和工具变更按同一发布链路治理：

1. **固定版本组合**：记录模型、Prompt、Skill、工具 Schema、权限策略和应用版本；
2. **回归评测**：使用固定任务验证质量、安全和工具行为；
3. **小流量灰度**：先在低风险任务或少量流量中运行；
4. **按业务指标放量**：观察越权、重复写入和任务完成率，而不只是模型 API 错误；
5. **处理存量任务**：回滚新任务后，继续处理已经由新版本创建的 Run、Artifact 和外部动作。

发布单元不是某一个 Prompt 文件，而是整组实际运行配置。

## 工程场景题

## 模型说“退款成功”，系统能否直接把任务标记完成？
{: #model-text-vs-business-fact }

不能。模型文本只是候选说明，只有支付服务回执、必要审批和持久 Tool Call 状态共同确认后，退款才是业务事实。

运行时还要检查没有 executing 或 unknown 的副作用，并先保存最终结果，再发布任务完成事件。

## Tool Call 已成功，但 Agent 在保存结果前崩溃怎么办？
{: #tool-succeeded-before-persist-crash }

本地状态应视为结果未知。恢复时使用调用前保存的业务请求 ID 查询领域服务，确认成功后补记 Tool Call 和 Step，再继续模型循环。

没有稳定请求 ID 和下游查询能力时，系统无法安全判断是否重做，只能暂停自动处理或转人工。

## SSE 正常返回文字，但任务状态没有更新怎么办？
{: #stream-state-inconsistency }

流式文字只是临时传输。服务端应先提交关键任务状态和最终消息，再发送完成事件；状态写入失败时不能假装任务已完成。

客户端重连后从任务状态和事件位置恢复，而不是把屏幕上已经看到的文本当作权威结果。

## 用户刷新页面后，怎样继续观察同一个 Agent 任务？
{: #agent-task-resubscribe }

任务创建时返回稳定 Task ID。页面重新打开后先读取任务当前状态和最终结果，再从最后事件序号订阅后续事件。

新的 SSE 连接只恢复观察，不创建新的任务，也不重新执行已经完成的 Tool Call。

## Agent 不断调用同一个工具，后端怎样停止？
{: #agent-loop-control }

Runtime 从两层停止循环：

- **进展检测**：识别相同工具和近似参数的重复调用，并检查任务事实、待办和外部状态是否发生变化；
- **硬预算**：达到轮次、Token、时间或费用上限时停止或转人工。

只设置最大轮数是最后防线，**完成条件与进展检测**才能更早阻止无意义循环。
