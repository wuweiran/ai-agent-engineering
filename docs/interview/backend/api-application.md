# API 与应用层常见面试题

API 是客户端与业务系统之间的契约。好的设计既要准确表达资源和业务动作，也要处理校验、并发、重复请求、长任务、错误语义和版本演进。

## 资源与 HTTP 语义

## REST 的核心是什么？
{: #rest-core }

REST 的核心可以概括为：

- **资源中心**：URI 标识资源，客户端操作资源表示；
- **统一接口**：使用一致的方法语义、状态码和表示形式；
- **无状态**：每个请求携带处理所需信息，服务端不依赖隐含会话过程；
- **可缓存与分层**：允许中间层参与缓存、代理和扩展。

HTTP 与 JSON 是常见实现方式，但“HTTP + JSON”不自动等于 REST。业务动作也不必为形式纯粹全部硬塞进 CRUD，可以使用清楚命名的子资源或自定义方法。

## URI 应该怎样表达资源？
{: #resource-uri-design }

URI 通常使用名词和层级表达资源，例如 `/orders/{order_id}/items`。查询参数适合过滤、排序、分页和可选视图，不应把权限信息交给客户端自行声明。

资源层级要服务稳定身份和访问关系，不能把数据库表结构直接暴露成 API。过深嵌套会让路径和资源生命周期难以理解。

## GET、POST、PUT、PATCH 和 DELETE 怎样选择？
{: #http-method-selection }

- **GET**：读取资源；
- **POST**：创建由服务端分配标识的资源，或执行非幂等动作；
- **PUT**：使用完整表示创建或替换已知 URI 的资源；
- **PATCH**：部分修改资源；
- **DELETE**：删除资源。

方法语义要与重试和缓存一致。不能因为客户端只支持 POST，就把所有读取与更新都包装成没有明确语义的 POST。

## HTTP 方法的安全性和幂等性有什么区别？
{: #http-safe-idempotent }

- **安全性**：客户端只期望读取信息，不要求改变资源状态，例如 GET 和 HEAD；
- **幂等性**：同一请求重复执行，预期服务端效果与执行一次相同。

**安全方法一定幂等，幂等方法不一定安全。**PUT 和 DELETE 会改变资源，但按定义应当幂等；多次 DELETE 返回不同状态码也不影响其效果幂等。

## HTTP 幂等和业务幂等有什么区别？
{: #http-vs-business-idempotency }

- **HTTP 幂等**：描述 HTTP 方法的预期语义；
- **业务幂等**：要求同一购买、支付或退款意图重复提交时，只产生一份业务结果。

POST 默认不幂等，却可以通过 Idempotency-Key、唯一约束和结果查询实现业务幂等。**方法名称不能代替服务端并发控制。**

## `PUT` 和 `PATCH` 有什么区别？
{: #put-vs-patch }

- **PUT**：通常提交目标资源的完整新表示，未提供字段可能被视为替换后的缺失值；
- **PATCH**：只描述需要修改的部分，需要额外定义字段删除、空值和并发冲突语义。

二者都不应直接信任客户端提交的服务端字段，例如资源所有者和内部状态。

## 状态码与错误契约

## `200`、`201`、`202` 和 `204` 怎样选择？
{: #http-success-status }

- **`200 OK`**：操作成功，并返回响应内容；
- **`201 Created`**：资源已经创建，通常携带新资源或 `Location`；
- **`202 Accepted`**：请求已接受，但处理尚未完成；
- **`204 No Content`**：操作成功，且无需返回响应内容。

**状态码应与真实完成点一致。**任务尚未完成时返回 `200 succeeded` 会让客户端误判结果。

## `400`、`409` 和 `422` 有什么区别？
{: #http-client-error-status }

- **`400 Bad Request`**：请求格式、解析或通用语义错误；
- **`409 Conflict`**：请求与资源当前状态冲突，例如版本过期或重复名称；
- **`422 Unprocessable Content`**：内容格式正确，但具体指令无法按当前语义处理。

团队应建立一致约定，避免相同错误在不同接口随机返回不同状态。业务错误码再补充稳定机器语义。

## `401` 和 `403` 有什么区别？
{: #http-401-403 }

- **`401 Unauthorized`**：缺少有效认证凭据，客户端通常需要登录或刷新凭据；
- **`403 Forbidden`**：服务端已经识别身份并理解请求，但拒绝该操作。

对象级授权场景也可以使用 `404` 隐藏资源是否存在，但团队需要一致处理，避免错误信息泄露权限边界。

## `429` 和 `503` 有什么区别？
{: #http-429-503 }

- **`429 Too Many Requests`**：调用方超过速率或配额，常针对用户、客户端或接口维度；
- **`503 Service Unavailable`**：服务整体或依赖暂时无法处理请求。

两者都可以使用 `Retry-After` 提示等待时间，但客户端仍应限制尝试次数并使用退避，不能假设稍后一定成功。

## API 错误响应应该包含什么？
{: #api-error-response }

错误响应通常包含：

- **稳定错误码**：供客户端识别和分支处理；
- **简洁说明**：说明发生了什么；
- **重试语义**：指出是否可以重试；
- **请求关联 ID**：支持服务端定位 Trace；
- **字段错误**：指出具体字段和违反的约束。

内部堆栈、SQL 和敏感配置不能暴露给客户端。**HTTP 状态码表达协议层类别，业务错误码表达领域语义**，不能只用一个永远为 200 的包装对象替代二者。

## 入参校验和业务校验有什么区别？
{: #input-vs-business-validation }

- **入参校验**：检查类型、格式、长度、必填和基本范围，回答输入能否被程序处理；
- **业务校验**：结合当前身份和权威状态，判断库存、优惠资格和状态转换是否成立。

Schema 合法不能证明操作被授权，也不能证明业务条件仍然有效。并发会改变的条件还要在最终写入时重新确认。

## 分页与查询

## Offset 分页和 Cursor 分页有什么区别？
{: #offset-vs-cursor-pagination }

- **Offset 分页**：使用页码或偏移，简单且支持跳页；但深分页通常需要扫描并丢弃前面的记录，数据持续变化时还可能重复或遗漏；
- **Cursor 分页**：使用上一页最后一条记录的稳定排序字段继续，成本更稳定，也更适合动态数据；但不方便任意跳页，并要求确定性排序。

**需要跳页且数据规模较小时用 Offset；连续浏览大规模动态数据时优先 Cursor。**

## Cursor 应该包含什么？
{: #cursor-design }

Cursor 应包含继续查询所需的排序位置，例如创建时间和唯一 ID。多个排序字段共同决定顺序时，都要进入 Cursor。

对外 Cursor 可以编码或签名，避免客户端依赖内部字段。编码不是加密，若包含敏感信息仍需保护。

## 列表接口怎样设计过滤和排序？
{: #api-filter-sort }

只开放服务端明确支持且有索引或容量保障的字段，排序要有稳定的唯一补充字段。不能把任意 SQL、字段名和表达式直接交给客户端。

过滤条件还要进入权限范围和资源配额。一个合法查询如果能扫描全库，也可能成为拒绝服务入口。

## 重复请求与并发

## 创建订单接口怎样防止重复提交？
{: #create-api-idempotency }

通常采用三层防重：

1. **前端防抖**：提交后禁用按钮，减少用户连续点击；
2. **Redis 拦截**：使用 `SET NX` 短期拦截正在处理的重复请求，降低数据库压力；
3. **数据库幂等**：客户端为一次购买意图生成稳定的 Idempotency-Key，服务端在幂等表中对 `(用户或租户, Idempotency-Key)` 建立唯一约束，并在同一事务中创建订单、记录订单 ID 和处理状态。

相同 Key 和参数再次到达时，处理中就返回处理中状态，已成功就根据订单 ID 返回第一次创建的订单；相同 Key 携带不同参数时返回冲突。涉及远程支付时，再使用稳定业务请求 ID 查询支付的真实结果。

## Idempotency-Key 应该保存多久？
{: #idempotency-key-retention }

保存时间至少覆盖客户端可能重试、网络超时恢复和业务结果查询的窗口。支付、订单等高风险操作可能需要保存到业务对象完整生命周期或审计期限结束。

过早删除会让迟到重试产生新结果；永久保留会增加存储和唯一键范围。系统应根据业务语义定义过期与归档。

## 怎样避免并发更新互相覆盖？
{: #api-optimistic-concurrency }

读取资源时返回版本号或 ETag，更新时通过 `If-Match` 或请求字段提交原版本。服务端在同一写入条件中检查版本，成功后增加版本号。

版本不匹配时返回冲突或前置条件失败，让客户端读取最新状态后重新决定，不能无条件覆盖其他请求的修改。

## 批量接口应该怎样设计？
{: #batch-api-design }

批量接口要限制数量和总请求大小，明确是全批原子提交，还是每项独立返回结果。响应应能关联每个输入的成功、失败和错误原因。

一个超大批次会长时间占用事务、内存和连接。需要长时间处理时，应拆成异步任务，而不是无限提高 HTTP 超时。

## 长任务与异步接口

## 什么工作适合异步接口？
{: #async-api-boundary }

耗时超过正常请求期限、依赖外部条件、可以独立追踪，且不要求客户端立即得到最终结果的工作适合异步处理，例如报表、批量导入和 Agent 长任务。

当前请求仍要验证目标并可靠创建任务。把工作扔进内存线程后直接返回，不等于建立了可恢复的异步接口。

## 为什么异步接口通常返回 `202 Accepted`？
{: #http-202-async }

`202` 表示服务端已经接受请求，但最终处理尚未完成。响应应返回任务 ID 和状态查询地址，必要时提供预计时间、取消能力或回调配置。

`202` 不能在任务创建都未持久化时提前返回。客户端还需要区分等待、成功、失败和取消。

## 长任务状态接口应该返回什么？
{: #long-running-operation-status }

长任务状态接口至少返回：

- **任务标识**：稳定的任务 ID；
- **当前状态**：区分等待用户、等待依赖、正在执行和已经完成；
- **时间信息**：创建时间和最近更新时间；
- **失败信息**：失败时返回稳定错误码和原因；
- **结果位置**：完成后返回可读取的结果或链接。

进度百分比只有在总工作量可估计时才可靠。无法估计时，阶段和最近事件比虚假的精确百分比更有价值。

## 轮询、回调和 SSE 怎样选择？
{: #polling-webhook-sse }

- **轮询**：实现最简单，适合状态变化不频繁的任务，但要设置间隔、退避和停止条件；
- **Webhook**：适合服务到服务通知，需要签名验证、重试和幂等；
- **SSE**：适合浏览器持续接收服务端单向进度事件。

**连接断开不应自动等于任务取消。**任务生命周期应由持久状态控制，传输通道只负责观察。

## 取消异步任务意味着什么？
{: #async-task-cancellation }

取消表示系统停止尚未开始或后续可停止的步骤，并尽力中断正在等待的操作。它不能自动撤销已经发送的短信、完成的支付或写入的数据。

需要撤销真实结果时，业务系统必须提供补偿操作。取消请求本身也应幂等，并返回已经确定的任务状态。

## 版本与兼容

## API 版本应该放在哪里？
{: #api-versioning }

常见方式包括：

- **URL 版本**：直观、易路由，但会改变资源 URI；
- **Header 版本**：保持 URI 稳定，但调试和缓存配置更复杂；
- **媒体类型版本**：通过内容协商区分表示，表达能力强，但客户端和运维成本更高。

版本位置不是最重要的，关键是**兼容策略、调用方迁移证据和旧版本退出条件**。不能每次新增可选字段就创建新版本。

## 哪些 API 变化通常向后兼容？
{: #api-backward-compatible-changes }

增加可选请求字段、增加响应字段或新增资源通常可以兼容会忽略未知字段的客户端。删除或重命名字段、改变已有语义、把可选改成必填、收紧枚举范围通常具有破坏性。

兼容性还取决于真实客户端行为。有些客户端会严格反序列化未知字段，因此发布前需要契约测试和使用量观察。

## API 字段替换怎样安全迁移？
{: #api-field-migration }

先同时提供新旧字段，让生产者和消费者都能理解两种表示；观察旧字段使用情况，调用方完成迁移后再停止写入并最终删除旧字段。

迁移期间要定义两个字段冲突时的优先级，并避免长期双写形成两份权威状态。

## 工程场景题

## 客户端请求超时后，应该直接重新提交吗？
{: #client-timeout-recovery }

读取请求通常可以重新执行；创建订单、支付等写操作可能已经成功，只是响应丢失，不能使用新请求 ID 直接重做。

客户端应使用同一 Idempotency-Key 查询或重试，服务端返回原结果或正在处理状态。无法确认时向用户显示“正在确认”，而不是最终失败。

## 文件上传接口怎样处理大文件？
{: #large-file-upload }

限制文件大小和类型，使用流式读取避免一次载入内存；大文件可以分片上传，每片校验，完成后按上传 ID 合并。对象存储直传还能减少应用服务器带宽和连接占用。

系统要处理断点续传、重复分片、内容哈希、恶意文件扫描和未完成上传清理。上传完成不代表业务处理完成，后续校验可以成为异步任务。

## API 已经返回成功，后续异步任务失败怎么办？
{: #api-success-background-failure }

先确认 API 当时承诺的是什么。如果只承诺“任务已创建”，返回 `202` 后任务失败是合法终态；如果返回“订单已完成”，关键异步步骤就不应留到响应之后。

任务失败后要保存原因、支持查询与重试，并通过通知或界面把结果交给用户。不能让失败只存在于后台日志中。
