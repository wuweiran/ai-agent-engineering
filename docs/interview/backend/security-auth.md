# 后端安全与鉴权常见面试题

后端安全问题首先要区分身份、权限和资源边界，再讨论令牌、浏览器攻击与接口防护。安全不能只依赖前端隐藏按钮或 Prompt 约束，关键检查必须在服务端每次请求中执行。

## 认证与会话

## 认证、授权和鉴权有什么区别？
{: #authentication-authorization }

- **认证（Authentication）**：确认调用者是谁，例如验证密码、Session 或访问令牌；
- **授权（Authorization）**：判断该身份能否对特定资源执行特定操作；
- **鉴权**：工程中常作为认证和授权的统称，但排查问题时仍要分清身份无效和权限不足。

用户已登录，不代表可以访问任意订单；服务身份合法，也不代表它可以代替最终用户执行所有操作。

## Cookie、Session 和 Token 分别是什么？
{: #cookie-session-token }

- **Cookie**：浏览器保存并按域、路径等规则自动发送的小段数据，是一种客户端存储与传输机制；
- **Session**：服务端保存的会话状态，浏览器通常只在 Cookie 中携带难以猜测的 Session ID；
- **Token**：客户端携带的凭据，服务端可以本地验证，也可以交给认证服务校验。

三者不是互斥概念：Session ID 可以放在 Cookie 中，JWT 也可以放在 Cookie 中。安全性取决于**传输、存储、过期、撤销和 Cookie 属性**，而不只取决于名称。

## 服务端 Session 和 JWT 怎样选择？
{: #session-vs-jwt }

- **服务端 Session**：便于立即撤销、控制并发登录和保存认证状态；代价是多实例需要共享或路由会话；
- **JWT**：各服务可以本地验签，减少认证服务查询；代价是难以即时撤销，权限变化后旧令牌仍可能有效。

选择重点是**撤销要求、扩展方式、数据敏感度和客户端类型**。内部系统和高风险业务通常更看重集中控制；跨服务 API 可以使用短期访问令牌。JWT 不是天然更先进。

## JWT 的结构是什么？签名代表什么？
{: #jwt-structure-signature }

JWT 通常由三部分组成：

- **Header**：声明令牌类型和签名算法；
- **Payload**：保存身份、签发方、受众和过期时间等声明；
- **Signature**：检测内容是否被篡改，并确认令牌由可信密钥签发。

**签名不等于加密。**普通 JWT 的 Header 和 Payload 只是编码，拿到令牌的人可以读取其中内容，因此不能放密码和不必要的敏感数据。

## 验证 JWT 时需要检查什么？
{: #jwt-validation }

验证 JWT 时至少检查：

- **签名与算法**：验证签名，并固定服务端允许的算法；
- **签发方与受众**：确认令牌由可信身份系统签发，且确实发给当前服务；
- **时间声明**：检查过期时间和生效时间；
- **令牌用途**：区分 Access Token、ID Token 和其他专用令牌；
- **密钥来源**：只使用可信配置中的密钥，不能盲信 Header 中的外部地址。

**只验签不等于完成验证。**不检查受众，可能把发给另一个服务的令牌错误接受。

## JWT 怎样实现注销和权限变更后失效？
{: #jwt-revocation }

常见做法是缩短 Access Token 生命周期，配合可撤销的 Refresh Token；高风险场景可以维护撤销列表、用户令牌版本或会话状态。

完全无状态的长效 JWT 很难即时撤销。权限变化的生效速度、每次查询认证服务的成本和系统可用性之间需要取舍。

## Access Token 和 Refresh Token 有什么区别？
{: #access-refresh-token }

- **Access Token**：生命周期较短，用于访问资源服务；
- **Refresh Token**：生命周期较长，只交给授权服务器，用来换取新的 Access Token。

Refresh Token 价值更高，应安全保存、限制使用客户端和范围，并使用轮换与重放检测。它不应发送给普通业务 API。

## 密码应该怎样存储？
{: #password-storage }

密码不能明文保存，也不应只做普通快速哈希。应使用专门的慢密码哈希算法，为每个密码生成独立盐，并根据系统能力设置合理成本。

登录时对输入执行同样的算法后比较。还应防止撞库与暴力破解，支持 MFA，并在泄露或高风险变更后撤销现有会话。

## 为什么登录失败不应该提示“用户不存在”？
{: #account-enumeration }

不同错误文字、状态码或响应时间会帮助攻击者枚举有效账户。登录和找回密码通常返回统一的外部提示，同时在内部日志保留真实原因。

统一提示不能代替账户级限速、MFA 和异常行为检测，也要避免攻击者利用锁定策略批量锁死其他用户。

## OAuth 与权限模型

## OAuth 2.0 解决什么问题？
{: #oauth2-purpose }

OAuth 2.0 是授权框架，让用户允许第三方客户端在限定范围内访问资源，而不把账号密码交给客户端。

它定义资源所有者、客户端、授权服务器和资源服务器等角色。OAuth 本身不是登录协议；客户端要确认用户身份时，通常使用 OpenID Connect。

## OAuth 2.0 和 OpenID Connect 有什么区别？
{: #oauth-vs-oidc }

- **OAuth 2.0**：解决授权，回答“客户端能访问什么资源”，主要凭据是 Access Token；
- **OpenID Connect**：在 OAuth 2.0 上增加身份层，使用 ID Token 和 UserInfo 回答“用户是谁”。

**Access Token 给资源 API，ID Token 给客户端确认身份。**不能把二者混用。

## Authorization Code 与 PKCE 解决什么问题？
{: #authorization-code-pkce }

- **Authorization Code**：前端先取得短期授权码，再由客户端换取令牌，避免令牌直接暴露在浏览器跳转 URL 中；
- **PKCE（Proof Key for Code Exchange）**：客户端发起授权前生成校验值，换令牌时证明自己是同一个客户端，降低授权码被截获后的滥用风险。

现代公共客户端应使用 Authorization Code + PKCE。

## OAuth 中 `state` 和 OIDC 中 `nonce` 分别做什么？
{: #oauth-state-oidc-nonce }

- **`state`**：把 OAuth 授权响应与客户端发起的请求关联，主要防止登录 CSRF 和响应混淆；
- **`nonce`**：写入 OIDC 的 ID Token，把令牌与本次认证请求关联并降低重放风险。

它们都需要不可预测、绑定当前会话并在使用后失效，不能用固定字符串代替。

## RBAC 和 ABAC 有什么区别？
{: #rbac-vs-abac }

- **RBAC（Role-Based Access Control）**：通过角色聚合权限，易于理解和管理，适合职责较稳定的系统；
- **ABAC（Attribute-Based Access Control）**：根据用户、资源、动作和环境属性动态决策，例如租户、部门、订单归属、设备和时间。

复杂系统常组合两者：**角色决定基础能力，属性继续限制具体对象**。只使用角色容易出现角色爆炸，也无法自动解决对象级授权。

## 最小权限和默认拒绝是什么？
{: #least-privilege-default-deny }

- **最小权限**：只授予完成职责所需的权限；
- **默认拒绝**：没有明确允许规则时一律拒绝。

权限检查要覆盖每个入口和资源，不能因为接口是“内部调用”或按钮被隐藏就默认放行。权限还应定期审查，避免角色和服务账户长期积累不再需要的能力。

## 资源与多租户授权

## 什么是对象级授权？
{: #object-level-authorization }

用户能够访问“订单”接口，不代表能够访问每一张订单。服务端还要检查当前订单是否属于该用户、租户或允许的业务范围。

资源 ID 不可猜测只能降低碰撞概率，不能代替授权。路径参数、请求体和隐藏字段中的对象 ID 都必须在服务端校验。

## 多租户系统怎样防止跨租户访问？
{: #multi-tenant-isolation }

认证后建立可信租户上下文，所有查询和写入同时带上 `tenant_id`，并校验用户、租户和资源之间的关系。不能先按全局 ID 取出记录，再相信客户端声明归属。

高风险系统还可以在数据库策略、独立 Schema 或独立实例层增加隔离。日志、缓存 Key、消息和搜索索引同样要带租户边界。

## 服务身份与用户身份有什么区别？
{: #service-vs-user-identity }

- **服务身份**：说明哪个应用或工作负载正在调用，常用于服务间认证；
- **用户身份**：说明这次操作最终代表谁，以及该身份能访问哪些业务资源。

订单服务持有合法机器凭据，只证明它可以调用支付服务；若调用代表用户支付，还要携带并校验用户、租户、订单和金额等授权上下文。

## 权限检查应该放在网关还是业务服务？
{: #gateway-vs-service-authorization }

权限检查应分层执行：

- **网关**：验证令牌、基础路由权限和通用配额，尽早拒绝明显无权请求；
- **业务服务**：根据资源归属和当前业务状态实施对象级授权，守住最终业务规则。

不能只在网关鉴权后，让内部服务完全信任所有参数。

## Web 与接口安全

## SQL 注入是怎样发生的？如何防止？
{: #sql-injection }

应用把不可信输入拼接进 SQL 结构时，攻击者可以改变查询语义。防止 SQL 注入应使用参数化查询或 PreparedStatement，并限制数据库账户权限。

参数化只适用于值。表名、排序字段等动态结构要从白名单映射，不能把用户字符串直接拼接。

## XSS 是什么？后端能做什么？
{: #xss }

XSS 让攻击者提供的内容被浏览器当作脚本执行。存储型 XSS 会把恶意内容保存后影响其他用户，反射型 XSS 则通过当前响应触发。

核心防护是在输出位置按 HTML、属性、URL 或脚本上下文正确编码。后端还应验证富文本、设置安全响应头，并避免把未经处理的内容拼入页面；CSP 是纵深防御，不是唯一防线。

## CSRF 攻击成立需要什么条件？
{: #csrf }

CSRF 成立通常需要三个条件：

1. 用户已经登录，浏览器会自动携带目标站点 Cookie；
2. 攻击页面能够诱导浏览器发送修改状态的请求；
3. 服务端没有验证请求来源或 CSRF Token。

攻击者通常不需要读取响应。常见防护是 SameSite Cookie、CSRF Token、Origin/Referer 校验，以及不使用 GET 修改状态。XSS 可能绕过许多 CSRF 防护，因此两者都要处理。

## CORS 是什么？为什么不能代替鉴权或 CSRF 防护？
{: #cors }

**CORS 是浏览器执行的跨源访问策略。**服务端通过响应头声明允许的 Origin、方法和 Header，主要控制前端脚本能否读取响应或发送非简单请求。

它不能替代安全边界：

- 非浏览器客户端不受 CORS 限制，所以它不是鉴权；
- 简单跨站请求仍可能被发送，所以它不是完整的 CSRF 防护；
- 携带凭据时不能使用任意来源通配符。

## 接口签名怎样防止请求被篡改和重放？
{: #api-signature-replay }

客户端对方法、路径、参数摘要、时间戳和 Nonce 等规范化内容计算签名，服务端使用共享密钥或公钥验证。内容被修改后签名不再匹配。

时间窗口限制旧请求，Nonce 或请求 ID 防止窗口内重复使用。签名不负责加密请求内容，仍需 TLS；密钥分发、轮换和泄露处置同样重要。

## HTTPS 是否足以保证接口安全？
{: #https-security-boundary }

HTTPS 保护传输中的机密性、完整性和服务器身份，防止普通网络窃听与篡改。

它不能判断登录者是否有权限，不能阻止合法客户端发送恶意参数，也不能修复 SQL 注入、越权和业务逻辑漏洞。TLS 是安全通信基础，不是完整应用安全方案。

## 工程场景题

## 用户修改权限后，旧 Token 仍能访问怎么办？
{: #permission-change-token-revocation }

缩短 Access Token 有效期，使权限变化在可接受窗口内生效；高风险操作可以实时查询权限或校验用户令牌版本。Refresh Token 应撤销或轮换，防止继续换取旧权限令牌。

需要即时撤销时，系统就不能保持完全无状态，必须引入会话、撤销列表或集中策略检查。

## 微服务之间是否可以完全信任内网请求？
{: #zero-trust-internal-service }

不能。内网地址可能被错误配置、横向移动或其他服务滥用。服务间调用仍应认证工作负载身份、限制权限，并验证业务参数和最终用户上下文。

网络边界只能作为一层防御，不能替代最小权限、审计和领域校验。

## 缓存权限结果有什么风险？
{: #authorization-cache-risk }

缓存可以降低策略查询成本，但权限撤销、角色变化和租户调整后，旧结果可能继续放行请求。

需要定义短 TTL、版本或主动失效，并让高风险操作回到权威策略确认。缓存 Key 必须包含用户、租户、资源和操作，不能只按角色缓存。

## 如何防止登录接口被暴力破解和撞库？
{: #login-brute-force }

应结合账户与来源限速、渐进等待、泄露密码检查、MFA、异常设备检测和告警。错误响应不要泄露账户是否存在。

固定次数永久锁定容易被攻击者利用来拒绝服务。防护要在阻止自动化攻击和避免误伤正常用户之间取舍。
