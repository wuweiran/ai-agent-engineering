# RAG 与知识检索面试题

这些问题覆盖 RAG 的离线索引、在线检索、Query 改写、混合召回、Rerank、Context 选择、评测和性能。RAG 是 AI Agent 岗位常见的应用基础，但并不要求模型自主调用工具，也不等于 Agent。

## RAG 的原理是什么？怎样缓解幻觉和知识过期？
{: #rag-principle }

**RAG（Retrieval-Augmented Generation，检索增强生成）在模型生成前，从外部知识源检索当前问题需要的证据，再把证据和问题一起交给模型。**模型参数负责理解和生成，外部索引负责提供可更新、可引用的事实。

它通过两点改善结果：

- **缓解幻觉**：让回答依据检索材料生成，并保留 Citation；证据不足时返回无答案，而不是只靠模型训练记忆补齐；
- **更新知识**：新增、修改和删除文档时更新外部索引，不必为每次知识变化重新训练模型，并通过版本和时间元数据避免使用过期资料。

RAG 不能消除幻觉。检索可能漏召回，Chunk 可能丢失上下文，文档本身也可能错误或冲突；即使回答忠实复述错误材料，业务结论仍然可能错误。因此还要评测检索质量、证据忠实度和无答案处理，并对高风险事实访问权威系统。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)、[大模型幻觉]({{ site.baseurl }}/docs/llm/hallucination/)。

## RAG 的完整流程是什么？
{: #rag-end-to-end-flow }

RAG 包含离线索引和在线回答两条链路。离线先把文档解析、清洗、切成 Chunk，保留来源、权限和版本元数据，生成 Embedding 并建立向量与关键词索引。

在线流程是：

1. **理解 Query**：结合必要会话历史完成指代消解，提取实体、时间、版本和权限范围，复杂问题按需改写或拆分；
2. **召回候选**：执行向量检索、BM25 关键词检索和元数据过滤，优先保证正确证据进入候选集；
3. **融合与重排**：合并多路结果，去重后使用 Reranker 对 Query 与 Chunk 重新打分；
4. **选择 Context**：在 Token 预算内兼顾分数、来源多样性、问题覆盖和父子块映射，保留 Citation；
5. **生成回答**：要求模型只依据证据回答，处理冲突资料，证据不足时明确未知；
6. **记录与评测**：保存 Query、索引版本、候选、排序和引用，分别评测检索、生成与端到端结果。

Query 理解和检索路径不一定都由模型决定。固定问答可以由程序执行同一套流程；Agent 场景才可能根据初次结果决定是否改写 Query 或继续检索。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)。

## 企业知识库怎样构建，并接入 RAG？
{: #rag-pipeline }

可以按**知识来源、离线索引、在线检索和持续维护**组织：

1. **接入知识源**：明确文档来源、权限、版本和更新时间，完成格式解析、正文清洗与去重；
2. **建立索引**：按文档结构切成 Chunk，保留来源和权限元数据，生成 Embedding，并建立向量、关键词和元数据索引；
3. **在线检索**：处理 Query，执行召回、权限过滤、融合、Rerank 和 Context 选择，再让模型依据证据回答；
4. **持续更新**：新增和修改文档增量索引，删除文档同步删除 Chunk；Embedding 模型或 Chunk 策略变化时重建新版本并切换；
5. **分层评测**：分别验证索引新鲜度与权限、证据 Recall 和排序，以及回答的忠实度和引用。

知识库不是“把文档写入向量库”就结束，源文档、索引和访问权限必须在整个生命周期内保持一致。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)。

## RAG 索引怎样动态更新？
{: #rag-index-update }

**日常变化增量更新，索引结构变化全量重建并切换版本。**

- **增量更新**：按文档 ID、更新时间或变更事件，只重新处理新增、修改和删除的文档；
- **去重与同步**：使用内容哈希跳过未变化内容，保证源文档删除、权限和版本变化同步到所有 Chunk；
- **版本管理**：Embedding 模型、Chunk 策略或元数据结构变化时构建新索引，评测后切换读流量，保留旧版本回滚；
- **一致性校验**：监控更新延迟、失败和源索引差异，通过重试与定期校准修复漏更。

定时扫描实现简单但更新有延迟；事件或 CDC（Change Data Capture，变更数据捕获）更及时，却要处理重复、乱序和失败恢复。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)。

## Naive RAG、Advanced RAG 和 GraphRAG 有什么区别？
{: #naive-advanced-graph-rag }

- **Naive RAG**：Chunk 向量索引 + Query 向量检索 Top-K + 生成，是最小文本检索链路；
- **Advanced RAG**：加入 Query 改写、混合检索、过滤、父子索引、Rerank 和 Context 选择，重点是提高文本召回与排序质量；
- **GraphRAG**：显式建立实体和关系图，通过子图、关系路径和社区摘要取得多跳或全局证据。

三者不是必须依次升级的版本。**Advanced RAG 优化文本检索链路，GraphRAG 改变知识表示和检索路径。**生产系统也可以组合使用。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)。

## GraphRAG 适合什么问题？什么时候不值得使用？
{: #graph-rag-use-cases }

**适合：**实体关系密集、多跳关联、跨文档证据组合和资料库全局主题概括，例如投资关系查询和多份报告的风险归纳。

**不适合：**简单 FAQ、局部事实查询和小型知识库。这些场景通常先用混合检索与 Rerank，因为图谱抽取、关系维护和社区摘要都有额外成本与错误。

判断标准是：只有评测证明普通文本检索持续败在**关系推理或全局视角**上，GraphRAG 才值得引入。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)。

## RAG 为什么需要 Query 改写？
{: #rag-query-rewriting }

Query 改写主要解决三类问题：

- **语义鸿沟**：用户口语与文档术语不同；
- **上下文依赖**：多轮问题存在指代和省略；
- **复杂目标**：一个问题包含多个可独立检索的子问题。

它把原问题转换成一个或多个独立、明确、适合检索的 Query。简单查询直接检索更快；复杂、多轮或初次召回不足时才值得改写。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)。

## Query 改写有哪些常见策略？
{: #rag-query-rewriting-strategies }

- **上下文补全**：完成指代消解，把多轮追问改成独立 Query；
- **查询扩展**：补充同义词、实体、错误码和领域术语，或生成多个改写；
- **任务分解**：把多跳问题拆成多个子查询；
- **HyDE**：生成假设性相关文档，用它的 Embedding 检索真实材料；
- **Step-back**：先提出更抽象的原则性问题，再辅助回答原问题。

HyDE 的假设文档是检索表示，不是事实答案；Step-back 与细粒度任务分解解决的也不是同一个问题。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)。

## Query Drift 怎样发现和控制？
{: #rag-query-drift }

**Query Drift 是改写后的 Query 偏离用户原意。**常见表现是丢失实体、时间、租户或业务约束。

控制方法包括：

1. 保留原始 Query，与改写 Query 一起召回；
2. 检查关键实体和约束是否仍然存在；
3. 比较语义相似度与两路检索结果；
4. 改写不满足约束时拒绝使用。

固定相似度阈值没有通用答案，最终要用原问题的召回和回答指标验证。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)。

## Query 改写怎样控制延迟和成本？
{: #rag-query-rewriting-cost }

- **按需启用**：先做意图或复杂度路由，只改写多轮、组合或召回不足的查询；
- **降低单次成本**：使用较小模型完成分类和改写；
- **减少等待**：将独立的改写 Query 并行检索；
- **限制扩散**：控制改写数量和候选集规模，避免后续召回与 Rerank 成本成倍增长。

优化目标是**端到端回答质量与延迟**，不是生成尽可能多的 Query。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)。

## RAG 有哪些分块策略？Chunk 大小怎样选择？
{: #rag-chunk-size }

常见策略包括：

- **固定长度与重叠窗口**：实现简单，适合结构弱的文本，但可能截断语义或产生重复；
- **按文档结构切分**：沿标题、段落、列表、表格、代码函数或类切分，能够保留原有语义边界；
- **递归切分**：先按较大结构切分，块仍过大时再按段落、句子等边界继续拆分；
- **语义切分**：根据相邻内容的语义变化寻找主题边界，质量依赖 Embedding 和阈值，成本也更高。

Chunk 太小容易缺少回答所需上下文，太大则会混入多个主题并增加 Token。最终要根据文档结构、问题类型、证据 Recall、回答忠实度和 Context 成本评测，不套用固定大小。需要“小块匹配、大块回答”时使用父子索引。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)。

## 父子索引什么时候值得使用？
{: #rag-parent-child-index }

**适用条件：定位需要细粒度匹配，回答又需要完整段落、表格说明或上下章节。**

- 子块参与检索，提高定位精度；
- 命中后返回所属父块，补充回答上下文；
- 多个子块指向同一父块时要去重。

代价是额外的索引、映射和 Context；父块过大仍会引入噪声。是否采用要比较证据 Recall、回答忠实度、Token 和延迟。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)。

## 文档怎样高效索引和检索？Retriever 应怎样选择？
{: #hybrid-search-rerank }

离线阶段先按结构切成 Chunk，保存文档 ID、来源、权限、版本和时间等元数据，再生成 Embedding 并建立向量、关键词和元数据索引。在线 Retriever 负责根据 Query 从索引取得候选：

- **Dense Retriever** 使用 Embedding 进行向量检索，擅长语义近似；
- **Sparse Retriever** 使用 BM25 等关键词检索，擅长错误码、编号和专有名词；
- **Hybrid Retriever** 分别召回后，通过加权或 RRF 融合，再交给 Reranker 精排。

数据规模增大后通常使用 HNSW 等近似最近邻索引，在 Recall、延迟、内存和构建成本之间取舍。向量可以存入专门向量数据库、支持向量能力的关系数据库或检索引擎；选型主要看数据规模、元数据过滤、更新频率、可用性和团队运维能力，而不是默认必须引入独立向量库。

Retriever 和 Chunk 策略要联合评测。分块过小可能缺少上下文，过大则降低匹配精度；父子索引可以用子块匹配、父块回答。最终比较 Recall@K、MRR、NDCG、过滤正确性、延迟和索引更新成本，而不是只看向量相似度。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)。

## Chunk 过多或质量参差时，怎样 Rerank 和融合？
{: #rag-rerank-context-budget }

**先过滤明显无效内容，再对有限候选精排，最后按证据覆盖而不是固定 Top-K 组装 Context。**常见过程是：

1. 按权限、版本、时间、语言和文档类型过滤，并删除空块、模板噪声和重复内容；
2. 向量和 BM25 分别召回，使用 RRF 或归一化加权融合候选；
3. 对较小候选集使用 Cross-Encoder、专用 Reranker 或模型，将 Query 与每个 Chunk 一起重新打分；
4. 合并高度重叠的 Chunk，同一父块多次命中时去重，保留来源和 Citation；
5. 在 Token 预算内按相关性、来源多样性和子问题覆盖选择证据，必要时取回父块或压缩片段。

相关性分数接近时，不应只保留同一文档的相邻片段；复杂问题还要保证不同子问题都有证据。多个来源相互冲突时，不要提前融合成单一结论，而应把来源、版本和时间一起交给生成阶段处理。

过少会降低证据 Recall，过多会增加噪声、Lost in the Middle、延迟和费用。参数要通过 Recall@K、NDCG、回答忠实度、完整性、Token 和 P95/P99 联合确定。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)、[长 Context、压缩与缓存]({{ site.baseurl }}/docs/llm/context-cost-cache/)。

## RAG 系统怎样评测？评测集包含什么？
{: #rag-evaluation-dataset }

评测分三层：

- **检索层**：Recall@K、Precision@K、MRR、NDCG，以及权限、版本和时间过滤；
- **生成层**：答案相关性、证据忠实度、引用正确性和无答案处理；
- **端到端**：任务完成、延迟和成本。

评测样本至少包含：原始问题、应召回证据、允许答案或判断规则、无答案标记、资料版本和权限范围。样本应覆盖术语差异、多跳、过期或冲突文档和权限隔离，并保留独立测试集。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)。

## 知识检索如何提高回答正确率？
{: #rag-optimization }

核心是让**正确、最新且有权限的证据**进入模型 Context，并让回答忠于证据：

1. 通过结构化 Chunk、合适的 Embedding、关键词与向量混合召回，提高正确证据的 Recall；
2. 使用元数据过滤排除无权限、过期或版本错误的内容；
3. 通过 Query 改写、Rerank、去重和来源多样性，把真正相关的证据排到前面；
4. 在生成时保留来源，证据不足就明确无答案，不让模型用自身记忆补齐；
5. 分开评测检索和生成，从第一次失败的位置优化。

证据没进入候选时检查 Query、Chunk、Embedding 和召回；已召回但排名靠后时检查融合与 Rerank；已经进入 Context 仍答错时再检查 Context、Prompt、模型和结果验证。平均相似度提高不等于最终回答更正确。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)、[Agent 质量改进]({{ site.baseurl }}/docs/ai-agent/agent-quality/improvement/)。

## RAG 的端到端性能怎样优化？
{: #rag-performance }

先拆分延迟：**Query 处理、各路召回、融合、Rerank、Context 组装、模型首 Token、完整生成**，再优化占比最大的阶段。

常见手段包括：

- 并行执行相互独立的检索；
- 使用元数据过滤缩小范围；
- 限制候选集和 Rerank 数量；
- 批量或缓存 Embedding；
- 按 Query 复杂度决定是否改写和 Rerank；
- 缓存带租户、权限、知识版本和检索配置的结果。

并行会增加瞬时并发，缓存会引入陈旧和污染风险，因此要同时观察回答质量、P95/P99、资源饱和度与成本。

相关内容：[RAG 与知识检索]({{ site.baseurl }}/docs/llm/rag/)、[长 Context、压缩与缓存]({{ site.baseurl }}/docs/llm/context-cost-cache/)。

## 为什么代码搜索不一定优先使用 RAG？
{: #code-search-rag-vs-grep }

- **Grep 或符号索引**：适合函数名、变量名、错误码和调用关系等精确查询；
- **语义检索**：适合自然语言描述、跨文件概念和相似实现。

工程系统可以组合两者。**技术更复杂不代表在所有查询上更准确。**
