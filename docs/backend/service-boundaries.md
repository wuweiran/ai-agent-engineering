# 什么时候应该拆分服务

电商系统最初由一个团队维护，一个应用同时处理商品、订单、库存和支付。代码按模块划分，共享一个数据库，部署一次即可完成下单功能。业务增长后，促销库存更新成为热点，支付接入由专门团队维护，订单团队又希望独立发布。

团队开始讨论把系统拆成订单服务、库存服务和支付服务。拆分会建立更清楚的容量和责任边界，也会把原来的函数调用变成网络调用，把同一数据库事务变成跨服务协作。

服务数量不会自动提高系统成熟度。拆分只有持续解决真实约束时，才值得承担新增的分布式成本。

## 单体应用并不等于混乱

单体只表示主要功能在同一个部署单元中运行。它仍然可以有清楚的内部模块：

```text
commerce-app
├─ product
├─ order
├─ inventory
├─ payment
└─ notification
```

每个模块拥有自己的业务逻辑和数据访问接口，其他模块通过明确方法调用，不能随意修改内部表。测试、日志和权限也可以按模块组织。

同一进程带来几个实际优势：

- 函数调用快速，失败语义清楚；
- 一个数据库事务可以保护订单和库存；
- 本地调试容易还原完整业务链；
- 一次部署保持代码版本一致；
- 团队不必维护大量服务模板、网络和告警。

小型团队和中等规模业务通常可以从模块化单体开始。代码边界混乱，应先整理模块职责；直接拆成远程服务只会把混乱的依赖搬到网络上。

## 什么问题可能推动拆分

系统发展到一定阶段，某些模块会出现与整体不同的约束。拆分通常由几类压力推动。

### 独立容量

商品读取需要大量缓存和水平扩容，库存扣减受热点写入影响，支付连接又有严格并发限制。把资源特征差异很大的模块独立部署，可以分别扩容和配置资源。

拆分不能消除底层瓶颈。库存服务独立后，热门商品的归属仍然需要按库存规则串行决定，只是这一容量边界更容易隔离和观测。

### 独立发布

支付团队需要快速接入新的渠道，订单团队则希望保持稳定。所有模块共用一次发布，会让无关变化互相阻塞，并扩大回滚范围。

服务拆分后，团队可以独立测试和发布支付能力，前提是接口保持兼容。若每次支付变化仍要求订单服务同步修改和同时部署，物理拆分并没有产生真正的独立性。

### 团队责任

一个团队持续拥有支付规则、值班和生产结果，可以让支付成为清楚的服务边界。服务由谁负责，比代码目录叫什么更重要。

若多个团队都可以直接修改同一服务和数据库，事故发生后仍然无法确定责任，拆分只会增加沟通接口。

### 故障隔离

推荐任务耗尽资源时，不应拖慢支付回调；一个支付渠道故障也不应阻塞所有订单查询。独立进程、资源池和部署单元可以限制部分故障影响。

很多隔离也能先在单体内部实现，例如独立连接池、有限队列和后台 Worker。只有进程级和部署级隔离确实有价值时，才需要服务边界。

### 数据与安全边界

支付凭据、个人地址和普通商品数据可能有不同访问要求。独立服务可以只向其他模块提供必要接口，减少直接访问敏感表的范围。

服务边界仍需真正的身份、授权和网络策略。把代码放进另一个仓库，却继续共享高权限数据库账号，不会形成有效安全边界。

## 拆分订单与库存会失去什么

在单体中，订单创建和库存扣减可以处在同一个数据库事务：一起提交或一起回滚。

库存拆成独立服务后，下单过程变成：

```text
订单服务请求库存服务保留库存
库存服务返回 reservation_id
订单服务创建订单
订单过期后释放 reservation_id
```

现在会出现新的故障窗口：

- 库存保留成功，订单创建失败；
- 订单创建成功，响应丢失，调用方不知道结果；
- 订单取消消息延迟，库存没有及时释放；
- 库存服务可用，订单服务不可用；
- 两边对订单状态理解不同。

系统需要资源 ID、幂等键、状态机、事件和补偿处理这些情况。拆分前的一条事务，变成了跨时间和网络的业务协议。

**服务边界会把本地一致性问题转化成分布式协作问题。**只有拆分收益足够明确，才值得接受这项成本。

## 数据所有权要跟着服务边界变化

库存成为独立服务后，应由它掌握库存记录和保留状态。订单服务通过接口查询或请求保留，不能继续直接更新库存表。

若两个服务仍然共享数据库并互相读写表，会产生隐性耦合：

- Schema 修改需要跨团队协调；
- 一方绕过另一方业务规则；
- 查询负载无法按服务隔离；
- 故障和审计责任不清楚。

迁移数据所有权不一定一步完成。可以先禁止新增跨模块写入，再提供库存接口；随后迁移读取，观察调用量；最后收回旧表权限。迁移期间需要清楚记录哪个系统是权威来源。

每个服务都拥有独立数据库也不代表每份数据只能出现一次。订单服务可以保存 `sku_id`、商品名称和成交价格快照，但不能维护一份自己可修改的“当前库存”。复制数据的用途和更新来源要明确。

## 服务接口需要表达业务能力

库存服务不应只把数据库 CRUD 暴露成远程 API。订单真正需要的是：

```text
reserveInventory(order_id, sku_id, quantity)
releaseInventory(reservation_id, reason)
confirmInventory(reservation_id)
getReservation(reservation_id)
```

这些接口表达库存生命周期，也给服务留下实施并发、幂等和审计规则的位置。

若订单服务先远程读取库存，再根据结果远程写入，原来的“先查再改”并发问题会跨网络再次出现。条件扣减仍要在库存服务内部原子执行。

服务接口还要有稳定错误语义：库存不足、保留已存在、保留已确认和结果未知需要不同处理。仅返回 500 会迫使调用方猜测是否应该重试。

## 同步调用与事件怎样选择

下单需要立即知道库存是否保留成功，通常使用同步请求。订单创建后通知推荐和分析系统，不影响当前响应，可以发布事件。

选择依据是当前步骤是否需要对方结果才能继续：

- 必须立即取得决定，使用同步调用，并处理超时和结果未知；
- 只需告知已经发生的事实，使用事件，让消费者独立处理；
- 长流程可以同步取得关键资源，再通过事件推动后续状态。

事件降低生产者对消费者可用性的依赖，也带来重复、乱序和延迟。同步调用更直观，却会把下游延迟和故障传回上游。二者没有统一优劣，要根据业务依赖选择。

## 分布式事务应该怎样理解

订单与库存拆开后，团队可能希望重新获得“一切共同提交”的效果。通用分布式事务协议可以协调部分数据库和资源，但会增加可用性、性能与运维成本，也未必覆盖外部支付等系统。

更常见的设计是把长交易表达成状态机：

```text
库存保留成功
订单创建为 pending_payment
支付成功后确认库存
订单取消后释放库存
```

每一步有稳定请求 ID 和可重试操作，状态转换保留证据，失败后通过补偿回到可接受状态。这类流程有时称为 Saga。

补偿不等于时间倒流。释放库存可以补偿订单取消，已经发送的短信和已经被用户看到的价格却无法完全撤回。业务需要定义哪些结果可逆、怎样修复、什么时候转人工。

## 拆分后可观测性必须跨服务

单体中的一次调用可以在一个进程日志中追踪。拆分以后，下单请求经过订单、库存、支付和消息系统，排查需要统一关联：

- Trace ID 和 Request ID；
- 订单 ID、库存保留 ID 和支付请求 ID；
- 每个服务与接口版本；
- 同步调用耗时和错误；
- 事件生产、投递和消费记录；
- 最终业务状态。

服务各自健康不代表交易成功。订单服务返回 200、库存服务也没有错误，订单仍可能因为事件丢失无法释放库存。观测必须从用户交易连接到跨服务执行链。

值班责任也要明确。一次故障由哪个团队先响应，跨服务事故谁负责协调，依赖方怎样升级问题，都属于服务化后的工程成本。

## 什么时候应该继续保持单体

出现下面情况时，继续维护模块化单体通常更合适：

- 一个团队能够理解和发布整个系统；
- 各模块扩容方式相近；
- 本地事务简化了关键业务；
- 发布频率没有互相阻塞；
- 当前故障可以通过内部资源隔离解决；
- 服务平台、观测和自动化还不成熟；
- 业务边界仍在频繁变化。

服务拆分不是架构升级的固定阶段。过早拆分会让每次开发都处理接口版本、远程故障、数据同步和本地环境搭建，而业务收益尚未出现。

## 怎样从单体逐步拆出一个服务

若支付模块已经具备明确团队、独立容量和接口边界，可以按阶段迁移：

1. 在单体内部先整理支付模块，禁止其他模块直接访问其内部表；
2. 定义支付能力、请求身份、错误语义和超时；
3. 建立独立服务和数据所有权，先让少量流量调用；
4. 比较新旧路径的业务结果与延迟；
5. 逐步扩大流量，同时保留回切能力；
6. 清除旧调用和跨模块数据访问；
7. 明确独立发布、值班、容量和事故流程。

先整理逻辑边界，再移动部署边界，可以减少一次同时改变代码、数据和调用方式的风险。

拆分完成的标志也不只是新服务上线。原单体不再绕过服务规则，团队能够独立发布和承担结果，故障与容量也确实得到隔离，拆分才实现了目标。

## 判断服务边界的几个问题

考虑拆分时，可以依次询问：

1. **当前具体约束是什么？**容量、发布、团队、安全还是故障隔离。
2. **内部模块化能否先解决？**独立队列、连接池和代码边界可能已经足够。
3. **拆分后谁拥有数据和业务结果？**没有明确所有权，服务边界不会稳定。
4. **本地事务会变成什么协议？**超时、幂等、事件、补偿和人工处理都要设计。
5. **团队能否承担运行成本？**部署、观测、值班、兼容和本地开发需要基础设施。
6. **收益能否被验证？**独立扩容、发布速度和故障范围应该有实际改善。

服务拆分是一项工程投资。它用网络和分布式状态的复杂度，换取独立容量、责任、发布和故障边界。没有明确收益时，结构清楚的单体更简单；约束持续存在且边界已经稳定时，服务化才开始值得。

## 从一笔订单回看后端工程

一笔订单从进入系统到长期运行，已经覆盖后端工程的四类责任：

- 接收请求：识别用户与目标，返回明确结果；
- 管理状态：用约束、事务、并发控制和缓存维护可信事实；
- 协调工作：让任务、消息和远程调用跨越时间与服务继续；
- 保障运行：控制容量，观测事故，安全发布并决定系统边界。

这些责任不会因为系统采用单体或微服务而变化。架构选择决定责任落在哪个进程、数据库和团队中，工程师仍要检查正确性、可靠性、性能、安全与可演进性。

传统后端主线至此形成完整骨架。大模型进入应用后，会带来更长且流式的调用、按 Token 计费、不确定输出和新的上下文管理问题；这些变化仍然建立在请求、状态、任务、故障和生产运行的后端基础之上。
