# 后端系统怎样安全发布和演进

订单团队准备增加库存保留期限。新版本创建待支付订单时写入 `reservation_expires_at`，过期 Worker 根据它取消订单并释放库存。开发和测试都已完成，生产环境却不能在同一时刻替换所有应用实例和数据库结构。

发布期间，旧实例仍然不会写这个字段，新实例已经依赖它；队列中还有旧版本创建的订单，回滚后也可能留下新格式数据。后端演进的难点，是让系统在新旧代码、Schema、消息和任务同时存在时继续正确运行。

这篇文章属于“保障运行”，重点检查可演进性：变化怎样逐步进入生产，影响怎样限制，出错后怎样恢复，并保持已经产生的业务事实可处理。

## 一次发布改变了哪些契约

“发布订单服务”听起来只是部署一份新代码，真实行为可能同时依赖：

- HTTP 请求与响应字段；
- 数据库表、约束和索引；
- 队列中的事件 Schema；
- 与库存、支付服务之间的接口；
- 超时、连接池和并发配置；
- 定时任务与 Worker 版本；
- 功能开关和灰度范围。

库存保留期限至少涉及订单 API、`orders` 表、订单过期消息和 Worker。只记录应用提交号，事故后仍然无法解释某笔订单由哪版代码创建、使用什么 Schema、由哪个 Worker 处理。

发布记录需要关联真正共同决定行为的版本和配置。请求日志、消息与 Trace 也应带上应用版本和关键契约版本，使失败可以重现。

## 数据库迁移为什么不能一步完成

团队最初计划直接执行：

```sql
ALTER TABLE orders
ADD COLUMN reservation_expires_at TIMESTAMP NOT NULL;
```

已有订单没有值，旧实例创建新订单时也不会写入字段。数据库若立即要求非空，迁移或旧版本写入都会失败。

更稳妥的过程分成几个阶段。

### 先扩展 Schema

先增加允许为空的新字段：

```sql
ALTER TABLE orders
ADD COLUMN reservation_expires_at TIMESTAMP NULL;
```

此时旧代码可以继续工作，新代码也能开始识别字段。这个变化只扩展能力，没有立即删除旧版本依赖的结构。

### 再让新代码兼容读写

新版本读取订单时处理两种情况：

- 新订单已有 `reservation_expires_at`，按明确时间过期；
- 旧订单字段为空，根据 `created_at` 和旧规则计算，或交给兼容流程处理。

新版本开始写入字段，但不能假设数据库里的每条记录都已经具备新格式。

### 回填旧数据

后台任务分批为仍有效的待支付订单补充期限。回填要限制批量大小和数据库负载，记录进度，并能够暂停恢复。一次更新整张大表可能持有长事务、制造锁等待并影响线上请求。

回填完成后，团队验证空值数量、过期任务和订单结果，再考虑收紧约束。

### 确认迁移完成后收缩旧结构

所有应用实例、Worker 和脚本都已经使用新字段，旧订单也已处理后，才能增加非空约束或删除旧兼容逻辑。

这类过程常被概括为“先扩展，再迁移，最后收缩”。顺序的价值在于任何中间阶段都允许新旧版本同时运行。

## 接口变化也要兼容新旧调用方

订单详情准备增加：

```json
{
  "reservation_expires_at": "2026-07-14T10:50:00Z"
}
```

增加可选字段通常不会破坏忽略未知字段的客户端。直接把旧字段改名或改变含义则可能让尚未升级的移动端失败。

需要替换字段时，可以先同时返回旧字段和新字段，观察旧字段使用量；调用方全部迁移后，再停止返回旧字段。请求参数也可以先接受两种格式，在服务端转换为统一内部表示。

兼容不意味着永久保留所有历史接口。每项兼容逻辑都应有迁移范围、使用数据和移除条件，否则系统会积累无法删除的旧行为。

服务端也不能假设客户端发布速度与自己相同。移动应用可能数周没有升级，外部合作方可能按季度变更。接口的兼容期要根据真实调用方决定。

## 消息比同步接口存活得更久

队列中的消息可能等待数小时，死信消息甚至几天后才重新投递。生产者升级后，消费者仍然可能收到旧格式；消费者升级后，也可能先处理新生产者发出的消息。

假设新版过期事件增加 `reservation_expires_at`：

```json
{
  "event_type": "order.expiration_scheduled",
  "event_version": 2,
  "order_id": "O-1042",
  "reservation_expires_at": "2026-07-14T10:50:00Z"
}
```

迁移期间，新消费者要同时理解 v1 和 v2。若新增字段可以从订单数据库重新查询，也可以先把它设为可选。只有确认队列、重试和死信中不再存在 v1，才移除旧解析逻辑。

消息 Schema 应遵守几个原则：新增字段优先采用可选形式；字段含义不能悄悄改变；枚举增加新值时，旧消费者要有明确处理；事件版本与生产者版本要可追踪。

“当前生产者已经不发旧消息”无法证明旧消息已经消失。队列保留时间和死信重放都要计入兼容期。

## 长任务应该使用哪个版本

订单过期任务可能在新版本发布前创建，发布后才执行。若任务每一步自动使用最新代码，前半段按旧规则保存状态，后半段却按新规则解释，结果可能不一致。

有两种常见处理方式：

- 任务状态保存所需规则和数据快照，使新 Worker 能按照原语义继续；
- 任务绑定处理版本，旧 Worker 保留到相关任务结束或迁移完成。

订单过期规则较简单，更适合在任务中保存明确的 `reservation_expires_at`，让任何兼容 Worker 根据数据处理。复杂工作流若中间状态与代码结构紧密相关，可能需要版本化状态机和专门迁移逻辑。

发布前要清点正在运行、等待和失败重试的任务。新请求切换成功，不代表旧任务能够被新版本安全接管。

## 怎样让少量流量先使用新版本

离线测试无法覆盖全部生产数据、流量和依赖状态。新版本可以从小范围开始：

- 内部测试账户；
- 指定地区或商品；
- 少量用户流量；
- 只读请求；
- 新创建任务，不处理中途任务。

灰度范围要能够识别。每项请求、订单和指标应知道自己使用了哪个版本或开关分组，团队才能比较新旧结果。

灰度观察不能只看部署是否成功。库存期限功能至少要检查：

- 下单成功率与延迟；
- 新订单字段写入比例；
- 订单按预期时间取消；
- 已支付订单没有被过期任务取消；
- 库存只释放一次；
- 数据库写入与回填负载；
- 旧版本读取新数据是否正常。

少量流量持续一个合理业务周期后，再逐步扩大。只运行几分钟可能看不到三十分钟后的订单过期结果。

## 发布前要定义停止条件

团队不能等异常发生后临时争论是否回滚。发布计划需要提前定义停止信号：

- 出现已支付订单被取消，立即停止；
- 库存重复释放或变成非法值，立即停止；
- 下单成功率持续低于约定范围，暂停扩大并调查；
- 数据库锁等待明显上升，降低回填速度；
- 少量旧订单缺少期限但兼容逻辑正常，可以继续观察。

严重正确性错误与普通性能波动不能只看同一个总分。资金、库存和权限边界一旦破坏，即使整体成功率仍然很高，也要停止发布。

## 回滚代码不等于回到发布前

灰度异常后，团队可以把新请求切回旧应用版本。此时生产环境可能已经存在：

- 带有 `reservation_expires_at` 的新订单；
- v2 过期消息；
- 部分回填后的旧订单；
- 新版本创建但尚未执行的任务；
- 新索引和数据库字段。

旧版本如果不能读取这些状态，单纯回滚会产生第二次故障。因此，新增数据库字段通常不会在代码回滚时立即删除，新格式消息也要确保旧消费者可以忽略或处理。

回滚方案包含两部分：阻止问题继续扩大；处理新版本已经留下的数据、任务和外部动作。已经取消的订单、释放的库存和发送的通知不会因部署旧代码自动撤销，必要时要通过业务补偿修复。

**代码回滚恢复执行逻辑，数据迁移和补偿处理已经发生的现实状态。**

## 配置变化也要像代码一样发布

连接池、超时、限流和功能开关可以不部署代码直接修改，影响却可能更快覆盖所有实例。前面的支付事故就是并发配置提高后耗尽连接池。

重要配置需要：

- 版本记录与变更说明；
- 类型、范围和组合校验；
- 小范围生效；
- 指标关联和审计；
- 一键恢复到已知值；
- 与应用版本的兼容检查。

动态配置的便利不能绕过发布治理。一次全局配置推送可能比应用灰度具有更大的影响范围。

## 发布完成后怎样清理兼容逻辑

迁移长期停在中间状态，会留下可空字段、双写、双读和多版本消息解析。它们增加理解成本，也让后续修改更容易出错。

每次兼容设计都要记录退出条件：

- 旧客户端调用量归零；
- 队列中旧事件超过最长保留期；
- 旧任务全部结束或迁移；
- 数据回填验证通过；
- 回滚观察窗口结束。

达到条件后，团队删除旧写入路径和读取分支，再收紧数据库约束。清理同样要经过测试和灰度，不能因为它“只是删旧代码”就降低发布要求。

## 安全演进的判断顺序

后端变化可以沿四个问题检查：

1. **哪些契约正在变化？**接口、Schema、消息、任务和配置都要识别。
2. **新旧版本能否同时运行？**先扩展能力，再迁移数据和流量，最后移除旧路径。
3. **怎样限制影响并判断停止？**灰度范围、业务周期、指标和停止条件要明确。
4. **回滚后已有状态怎么办？**新数据、旧任务和外部副作用需要继续兼容或补偿。

安全发布允许变化出现失败，同时要求每次变化可识别、影响有限、结果可验证，并留有清楚的恢复路径。

传统后端主线还需要回答一个架构问题：订单、库存和支付什么时候值得拆成独立服务。服务拆分可以解决容量、团队和发布边界，也会把本地事务和函数调用变成网络通信与分布式状态。
