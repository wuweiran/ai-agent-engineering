# 后端工程

用户在电商应用中选好商品和收货地址，点击“提交订单”，很快看到订单编号。这个简单结果背后，后端已经接收 HTTP 请求、识别用户、读取商品价格、检查库存、写入订单，并把业务结果转换成响应。

随着交易量增加，问题会继续出现：两名用户同时购买最后一件商品，谁能成功？支付回调重复到达，会不会重复确认订单？发送通知需要十秒，下单请求是否一直等待？一台服务重启，已经接收的工作怎样继续？

后端工程要让请求在多用户、多服务和故障中仍然产生可信的业务结果。可以沿四个问题逐步理解：

1. 一个请求进入系统后，怎样得到明确响应；
2. 多个请求同时改变数据时，业务状态怎样保持正确；
3. 一次请求做不完的工作，怎样交给后台继续；
4. 服务和依赖发生故障时，系统怎样限制影响并恢复。

## 先看一次下单请求

创建订单可以从一个 HTTP 请求开始。API 取得当前用户，校验商品和地址，业务层读取可信价格并检查库存，再通过数据库事务保存订单与明细。成功后返回订单 ID，失败则给出客户端能够处理的错误。

这一过程要区分几种责任：入口层处理 HTTP，认证组件确认调用者，业务层判断交易是否合法，数据访问层读写数据库。责任清楚后，接口协议、交易规则和存储实现才能分别变化。

## 数据库保存的是业务事实

订单创建成功以后，数据库中的订单、价格快照和库存变化成为权威状态。商品详情页缓存、日志和消息都可以复制这些事实，却不能各自决定最后一件库存属于谁。

当两个请求同时购买一件商品时，只在代码中先查询“库存大于零”再执行扣减并不安全。另一个请求可能在两步之间修改库存。数据库约束、事务和并发控制要把交易条件落实到真正写入的时刻。

## 长工作离开请求继续运行

发送短信、更新推荐数据和生成交易报表不适合长期占用下单请求。后端可以先创建任务或事件，再由 Worker 在后台执行。用户得到的是已经提交的订单，后续工作是否完成则通过任务状态、通知或重试机制继续追踪。

队列提高了工作继续执行的机会，也带来重复投递、乱序和恢复问题。业务动作需要幂等，任务进度需要持久化，失败还要区分重试、等待和终止。

## 多个服务会一起失败

真实交易可能经过订单、库存、支付和优惠服务。一次超时只表示调用方没有及时取得结果，无法单独证明对方没有执行。盲目重试可能重复创建支付，完全不重试又会放大短暂故障。

后端需要为调用设置超时，按错误类型决定重试，通过幂等键和状态查询处理结果未知，并用日志、指标和 Trace 还原请求经过的服务。发布时还要兼容新旧版本，避免一次变更同时影响所有交易。

## 这套知识怎样展开

本章先用两篇文章建立起点：[一个后端请求如何完成](request-lifecycle/)跟随提交订单的完整路径，[后端怎样保存业务状态](business-state/)再处理最后一件库存的并发竞争和事务边界。

后续内容会沿同一条交易链进入异步任务、消息队列、缓存、服务调用和生产运行。每次只解决当前业务链中已经出现的问题，再逐步补齐完整的后端知识体系。
