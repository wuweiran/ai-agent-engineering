# 后端工程

用户在电商应用中点击“提交订单”，几百毫秒后看到订单号。后端在这段时间里接收了来自网络的不可信输入，确认用户身份，计算可信价格，争抢库存，写入订单，并返回与数据库事实一致的结果。

订单创建以后，工作还没有结束。用户可能稍后支付，订单会在超时后取消，消息可能重复到达，支付请求可能只返回一个无法判断结果的超时。促销流量、依赖故障和新版本发布又会改变系统的运行条件。

这些问题可以沿一条主线理解：

```text
接收不可信请求
        ↓
形成可信业务状态
        ↓
让未完成的工作跨越请求继续
        ↓
在故障、流量和版本变化中守住结果
```

HTTP、数据库、缓存和消息队列是完成这条主线的工具。工程判断不能从“应该使用什么组件”开始，而要先确定系统承诺了什么结果，以及哪些失败可能破坏它。

## 从请求到业务结果

客户端提交的商品 ID、数量和地址都只是请求。后端从认证信息取得当前用户，在他的权限范围内查询资源，使用服务端价格和规则重新计算订单，然后通过数据库事务提交。

这条链路建立了后端最基本的边界：客户端描述自己想做什么，业务代码判断操作是否合法，数据库约束和事务使结果在并发请求中仍然可信，API 只返回已经能够确认的状态。

[后端请求链路](request-lifecycle/)从一次下单解释这条边界。[业务状态与并发控制](business-state/)继续处理两个用户同时争抢最后一件库存时，为什么每个请求单独看都正确，系统整体仍可能出错。

数据正确以后，还要高效找到它。[数据库查询性能](database-query-performance/)说明数据量增长后查询为何退化，以及工程师怎样通过访问路径、索引和执行计划定位问题。

## 服务拆分后的调用边界

支付、库存等能力独立部署后，进程内函数调用变成远程调用。系统要找到当前可用实例、分配请求，并处理实例变化、网络超时和慢节点。[服务调用中的超时与容错](service-timeouts/)从服务发现和负载均衡开始，继续解释结果未知、幂等重试、熔断、降级和隔离。

## 跨越请求的异步工作

下单接口不能保持连接等待用户三十分钟后支付。订单的过期时间和当前状态必须持久化，后台执行者才能在用户离开、服务重启后继续处理。

队列让工作跨越进程，却不能保证消息只出现一次。发送方可能在业务提交后、消息发送前崩溃；消费者也可能在完成业务操作后、确认消息前崩溃。因此，可靠系统不假设网络完美，而是让每项业务意图有稳定身份，使重复执行得到同一个结果。

[异步任务](async-tasks/)区分请求、持久任务和业务状态。[可靠消息的发送与消费](message-delivery/)解释本地消息表、重复到达和业务幂等来自哪些故障窗口。

## 多个节点与服务怎样协调状态

数据有多个副本时，要明确能够接受多旧的结果；订单、库存和支付分属不同服务时，要决定怎样推进、补偿和恢复；多个实例争抢同一任务时，还要处理锁过期与迟到写入。[分布式系统中的状态协调](distributed-state-coordination/)把 CAP、BASE、共识算法、分布式事务、锁和租约放进这三类状态问题中解释。

## 生产压力与故障应对

缓存通过保存可重建的读取副本减少数据库负担，但副本可能过期，缓存故障也可能让全部流量突然回到数据库。[缓存一致性与故障应对](cache-consistency/)把性能优化放回权威状态和故障退路中讨论。

当请求进入速度持续高于处理速度，等待会不断增加，继而触发超时和重试。此时 CPU 不高也不代表系统仍有容量，请求可能都在等待连接池、数据库锁或下游服务。[容量、排队与过载保护](capacity-backpressure/)解释吞吐、排队、背压和限流之间的关系。

生产故障发生后，工程师要从用户结果出发，用指标确认范围，通过日志和 Trace 找到异常位置，在根因完全明确前先限制影响，并处理已经积压或结果未知的业务状态。[监控与故障处理](incident-response/)用一次支付退化串起这段过程。

系统还要持续变化。生产发布不是旧版本瞬间替换成新版本，而是一段新旧代码、数据、消息和任务同时存在的时期。[安全发布与系统演进](release-evolution/)说明兼容、灰度与回滚怎样保护已经发生的业务事实。

## 大模型进入应用以后

大模型给后端增加了一个生成较慢、按 Token 消耗资源、依赖 Context 且输出不完全确定的计算环节。原有边界没有消失：身份仍由认证系统确认，订单仍由业务服务保存，模型提出的动作仍要经过权限与业务规则。

[大模型应用后端](llm-backend/)从三项变化展开：怎样流式交付模型和 Agent 的执行过程，怎样把模型提出的工具调用变成受控业务动作，以及怎样保存一次多轮 Agent 任务使它能够恢复。

## 后端工程的判断方法

面对一个设计，先说明用户最终要得到什么结果，再沿执行链寻找权威状态和故障窗口：

```text
哪些步骤必须共同成功？
故障发生后，结果是失败还是未知？
重复请求和并发修改会不会改变结果？
等待会占用什么资源，何时必须拒绝？
版本变化时，旧数据和旧任务由谁继续处理？
```

专家式判断不是背出组件清单，而是指出业务不变量、故障窗口和恢复依据，再选择刚好能够守住这些边界的机制。
