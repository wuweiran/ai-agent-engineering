# 一个后端请求如何完成

用户在购物车中选中一双运动鞋，确认收货地址后点击“提交订单”。应用发送请求，稍后显示订单编号 `O-1042` 和待支付金额。对用户来说，下单已经完成；对后端来说，它必须把来自网络的商品 ID 和数量转化成一笔可信的交易记录。

这里使用一个简化的电商系统说明请求链路，不对应淘宝等平台的内部实现。真实大型电商还会拆分更多服务，但一次请求中的基本责任没有改变。

这一步属于后端的“接收请求”：识别谁想做什么，把输入变成合法业务操作，并返回与真实结果一致的响应。重点先看正确性和安全边界，性能优化留到请求链清楚之后。

## 客户端先发出一份协议明确的请求

客户端调用创建订单接口：

```http
POST /api/orders HTTP/1.1
Host: shop.example.com
Authorization: Bearer <access-token>
Idempotency-Key: checkout-7f3c
Content-Type: application/json

{
  "items": [
    {"sku_id": "SKU-8812", "quantity": 1}
  ],
  "shipping_address_id": "ADDR-52",
  "coupon_id": "CP-20"
}
```

请求包含方法、路径、认证信息和下单参数。`Idempotency-Key` 标识用户的这次提交，后续网络重试可以用它识别同一个创建请求。

请求体只提交商品 SKU、数量、地址和优惠券。客户端页面显示的商品价格不能作为成交依据，否则调用者可以篡改请求，把 `699` 元改成 `6.99` 元。后端要从商品与营销系统取得当前可信价格，再计算订单金额。

浏览器或移动应用并不直接调用某个业务函数。DNS 将域名解析为地址，客户端建立 HTTPS 连接；负载均衡器或 API Gateway 接收流量，再把请求交给一台可用的应用实例。此处先把这些网络设施看成请求到达后端入口的过程。

HTTP 规定怎样传递请求与响应，却不知道库存是否充足、优惠券能否使用，也不会自动保证地址属于当前用户。

## 入口层把 HTTP 转成应用数据

Web Server 收到字节流后解析 HTTP，Web 框架再根据方法和路径找到处理函数。例如，`POST /api/orders` 会匹配 `create_order`。

路由处理函数先完成与接口有关的工作：解析 JSON，检查必要字段和基本格式，取得认证结果，再调用创建订单的业务能力。

`quantity` 必须是正整数，`items` 不能为空，这些可以在输入校验阶段发现。输入结构合法仍不等于订单可以创建：`SKU-8812` 可能已经下架，地址可能属于其他用户，优惠券也可能不适用于这件商品。

**格式校验让程序能够读取请求，业务校验才决定请求能否成立。**

## 用户身份不能由请求体指定

访问令牌经过认证后，服务取得当前身份：

```text
user_id: U-204
account_status: active
```

客户端不需要提交 `buyer_id: U-204`。即使提交，服务也不能直接相信，因为调用者可以改成别人的用户 ID。

认证回答“调用者是谁”，授权回答“他能否操作这项资源”。后端要检查 `ADDR-52` 是否属于 `U-204`，优惠券 `CP-20` 是否已经发放给他，以及账号是否允许交易。

来自客户端的 SKU、地址和优惠券 ID 都只是请求参数。服务端必须在当前身份范围内重新查询和验证，才能将它们用于订单。

## 业务层计算真正的订单

路由函数可以把经过结构校验的数据交给应用服务：

```text
createOrder(
  buyer = U-204,
  items = [{ sku = SKU-8812, quantity = 1 }],
  shippingAddress = ADDR-52,
  coupon = CP-20,
  idempotencyKey = checkout-7f3c
)
```

业务层随后完成交易判断：

- 商品是否仍然可售；
- 当前库存能否满足数量；
- 从价格服务取得的销售价是多少；
- 优惠券能否用于当前用户和商品；
- 运费、折扣和应付金额怎样计算；
- 同一个幂等键是否已经创建过订单。

这些规则不属于 HTTP。以后即使创建订单还可以由内部客服或批量接口触发，同一交易能力仍应执行可信定价、库存检查和优惠规则。

路由层关心 Header、JSON 和状态码；业务层关心买家、商品和交易规则；数据访问层负责保存订单。清楚的责任边界可以避免接口或数据库变化迫使所有代码一起修改。

## 数据库保存订单与成交快照

业务检查通过后，后端要保存订单主记录和商品明细。简化的事务可能包含：

```sql
INSERT INTO orders (
  order_id,
  buyer_id,
  shipping_address_id,
  status,
  total_amount_minor,
  idempotency_key,
  created_at
) VALUES (
  'O-1042',
  'U-204',
  'ADDR-52',
  'pending_payment',
  65900,
  'checkout-7f3c',
  CURRENT_TIMESTAMP
);

INSERT INTO order_items (
  order_id,
  sku_id,
  quantity,
  unit_price_minor,
  discount_amount_minor
) VALUES (
  'O-1042',
  'SKU-8812',
  1,
  69900,
  4000
);
```

订单明细保存成交时的商品、单价和折扣快照。商品以后改名或涨价，历史订单仍应显示用户当时购买的内容，不能每次都用商品表中的最新价格覆盖历史事实。

真正实现会使用参数化查询或 ORM，不能把客户端输入直接拼进 SQL。参数化避免输入被解释成 SQL 结构。

如果订单主记录创建成功、明细写入失败，系统会留下无法解释的空订单。两项写入必须共同成立时，应放在同一个数据库事务中：全部提交，或全部回滚。

库存扣减也需要与订单创建协调。为了先讲清一次请求，本篇只确认它属于下单成功条件；下一篇会处理两个用户同时购买最后一件商品时，库存怎样安全扣减。

## 响应要与真实结果一致

事务提交后，服务返回：

```http
HTTP/1.1 201 Created
Content-Type: application/json
Location: /api/orders/O-1042

{
  "order_id": "O-1042",
  "status": "pending_payment",
  "payable_amount_minor": 65900,
  "currency": "CNY",
  "expires_at": "2026-07-14T10:50:00Z"
}
```

`201 Created` 表示订单资源已经创建。`pending_payment` 明确说明下单成功和支付成功是两个不同状态；客户端接下来可以进入收银台。

失败也需要稳定语义：

| 情况 | HTTP 状态 | 客户端可以怎样处理 |
| --- | --- | --- |
| 商品数量格式错误 | `400 Bad Request` | 修正输入后重新提交 |
| 访问令牌无效或过期 | `401 Unauthorized` | 重新登录或刷新身份 |
| 地址不属于当前用户 | `403 Forbidden` | 停止提交并重新选择地址 |
| 商品不存在或已下架 | `404 Not Found` | 刷新购物车 |
| 库存不足或优惠券已使用 | `409 Conflict` | 展示最新状态，让用户调整订单 |
| 服务内部故障 | `500 Internal Server Error` | 告知提交未确认，并按幂等键查询或重试 |

HTTP 状态码提供通用类别，业务错误码可以进一步区分 `OUT_OF_STOCK` 和 `COUPON_ALREADY_USED`。错误响应不应把数据库堆栈和内部路径直接暴露给客户端；完整细节进入服务日志，并通过请求 ID 与用户看到的错误关联。

## 为什么下单接口需要幂等键

用户点击提交后，网络可能在服务成功创建 `O-1042` 之前断开响应。客户端只看到超时，无法判断订单有没有创建。如果它直接再次提交，可能产生两个待支付订单。

服务可以对 `buyer_id + idempotency_key` 建立唯一约束。同一个键再次到达时：

- 已有订单创建成功，返回同一个订单结果；
- 第一次请求仍在处理中，返回明确的处理中状态；
- 上次确定失败且没有产生订单，才允许重新执行。

幂等键不能只放在日志中，它必须参与持久状态和并发判断。客户端也不能每次重试都生成新键，否则服务无法识别两次请求来自同一次用户操作。

## 一次请求应该做到哪里

订单、明细与必要的库存结果提交后，接口可以返回。发送短信、更新搜索统计、生成推荐特征不必阻塞用户。

判断某项工作是否留在同步请求中，可以问：用户取得“订单已经创建”这个结果是否必须等待它？保存订单和确认库存必须完成；推荐系统晚几秒得知新订单，通常不影响用户进入支付。

若把所有后续工作都塞进请求，任何外部依赖变慢都会拖慢下单。若过早把订单创建也交给后台，接口虽然很快，却只能告诉用户“系统以后可能会生成订单”。同步边界应围绕当前业务承诺确定。

## 代码还要处理横切问题

一个可运行的下单接口还需要：

- 为请求分配或接收请求 ID；
- 记录耗时、结果、用户和订单标识；
- 对异常进行统一转换；
- 限制请求体大小和处理时间；
- 对用户与设备实施合理限流；
- 在服务停止时完成或安全终止正在处理的请求。

这些能力通常由中间件、框架和基础设施提供，不应由每个路由重复实现。中间件可以处理通用认证、日志和限流，商品是否可售仍然属于业务层。

## 请求成功代表什么

一次下单请求可以概括为：HTTP 入口解析参数，认证组件确认买家，业务层重新取得可信价格并检查交易规则，数据库事务提交订单事实，接口把结果转换为稳定响应。

请求的完成点，是订单已经可靠保存，客户端也收到了与实际状态一致的订单编号和金额。处理函数完整运行，本身不能证明交易成立。

下一篇会继续处理更困难的情况：用户小林和小周几乎同时购买 `SKU-8812` 的最后一件库存。两个请求分别看都合法，系统仍然只能接受其中一个结果。
