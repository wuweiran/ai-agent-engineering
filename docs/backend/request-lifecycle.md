# 一个后端请求如何完成

用户在购物车中选中一双运动鞋，确认地址后点击“提交订单”。应用发送 SKU、数量、地址和优惠券，稍后显示订单号 `O-1042`。

客户端表达了购买意图，却不能决定成交事实。价格可能被篡改，地址可能属于其他用户，库存也可能已经变化。后端请求的任务，是把这些不可信输入变成一项经过授权、符合业务规则并可靠保存的结果。

以下使用一个简化的模拟电商系统说明这条链路。

## 客户端提交意图，不提交事实

创建订单接口可以接收：

```http
POST /api/orders HTTP/1.1
Authorization: Bearer <access-token>
Idempotency-Key: checkout-7f3c
Content-Type: application/json

{
  "items": [{"sku_id": "SKU-8812", "quantity": 1}],
  "shipping_address_id": "ADDR-52",
  "coupon_id": "CP-20"
}
```

请求里没有成交价、买家 ID 和订单状态。页面显示的价格可以被修改，`buyer_id` 也可以被伪造成别人。后端必须从自己的系统取得这些事实。

HTTP 只规定请求怎样传输。它不会判断库存是否充足、优惠券能否使用，也不会保证地址属于当前用户。

## 入口先让请求变得可处理

Web 框架根据 `POST /api/orders` 找到处理函数，解析 JSON，并检查基本结构：商品列表不能为空，数量必须是正整数，地址 ID 必须存在。

这些检查只能证明程序能够读取输入。`SKU-8812` 可能已经下架，`CP-20` 也可能不适用于这件商品。

**格式校验回答“输入长什么样”，业务校验回答“这次操作能不能成立”。**

参数要通过框架绑定、参数化查询或 ORM 进入数据库，不能拼接成 SQL。错误输入应该成为稳定的客户端错误，而不是泄露内部堆栈。

## 身份决定资源范围

访问令牌经过认证后，后端取得当前身份：

```text
user_id: U-204
tenant_id: TENANT-7
account_status: active
```

认证确认调用者是谁，授权判断他能否操作具体资源。后端必须在 `U-204` 的范围内查询 `ADDR-52` 和 `CP-20`，不能先按全局 ID 取出数据，再相信客户端声明的归属关系。

这个习惯可以阻止一类常见漏洞：攻击者把地址 ID 或订单 ID 换成别人的值，接口因为只检查“记录存在”而返回或修改了不属于他的资源。

多租户系统还要把 `tenant_id` 带入数据访问边界。服务之间使用的机器凭据证明“哪个服务在调用”，不能自动代替最终用户对订单的权限。

## 业务层重新计算订单

入口把身份和结构化参数交给创建订单能力：

```text
createOrder(
  buyer = U-204,
  items = [{sku = SKU-8812, quantity = 1}],
  shippingAddress = ADDR-52,
  coupon = CP-20,
  idempotencyKey = checkout-7f3c
)
```

业务层从可信系统取得商品售价，检查商品状态、优惠资格、地址范围和库存条件，再计算运费、折扣与应付金额。即使以后下单还可以由客服后台或内部任务触发，这些规则仍然必须执行。

路由层理解 Header、JSON 和状态码；业务层理解买家、商品和交易规则；数据访问层负责把结果落实到存储。边界清楚的目的不是追求固定分层，而是避免网络协议、业务规则和数据库细节互相混杂。

## 事务定义请求承诺的完成点

简化订单至少包含主记录和成交明细：

```text
orders
├─ order_id
├─ buyer_id
├─ status
├─ total_amount_minor
└─ idempotency_key

order_items
├─ order_id
├─ sku_id
├─ quantity
├─ unit_price_minor
└─ discount_amount_minor
```

明细保存成交快照。商品以后调价，历史订单仍要显示用户当时接受的价格。

订单主记录、明细和当前请求必须确认的库存结果要么共同成功，要么共同失败。如果主记录写入成功而明细失败，系统会留下无法解释的订单。因此，它们应在数据库事务中提交。

事务提交以前，程序“执行到了最后一行”仍不代表订单成立。事务成功提交后，数据库中才出现可以被其他请求读取的业务事实。

## 响应必须与真实状态一致

提交成功后，接口返回：

```http
HTTP/1.1 201 Created
Content-Type: application/json
Location: /api/orders/O-1042

{
  "order_id": "O-1042",
  "status": "pending_payment",
  "payable_amount_minor": 65900,
  "currency": "CNY"
}
```

`pending_payment` 说明订单已经创建，但支付尚未完成。API 不应把中间过程包装成比数据库更确定的结果。

错误也要让调用方知道下一步能做什么。格式错误可以修正后重试，身份失效需要重新登录，库存冲突需要刷新状态。内部故障或网络超时则可能让客户端不知道订单是否已经创建，不能一律提示“请再次提交”。

## 幂等键识别同一次用户操作

订单已经提交，响应却在返回途中丢失。用户再次点击时，如果后端把它当作新操作，就可能创建两张订单。

`Idempotency-Key: checkout-7f3c` 为这次购买意图提供稳定身份。数据库可以对 `buyer_id + idempotency_key` 建立唯一约束：原订单已经创建就返回同一个结果；原请求仍在处理中就返回可查询状态；明确没有产生订单时才重新执行。

幂等键必须进入持久状态和并发约束。每次重试生成新键，或者只把键写在日志里，都无法阻止重复结果。

## 哪些工作留在请求中

订单与必要库存结果必须在响应前确认，因为用户看到的承诺是“订单已经创建”。短信、推荐更新和报表不影响这个结果，可以在请求结束后继续。

同步边界不是由代码执行时间决定，而由当前 API 的业务承诺决定：

```text
用户必须立刻知道其结果
        → 当前请求中确认
可以稍后完成且能够独立追踪
        → 保存为后台任务
```

把所有后续工作塞进请求，会让非关键依赖拖慢下单；把订单创建本身也过早放到后台，又只能告诉用户“系统以后可能创建订单”。

## 一次请求的知识骨架

一次下单可以压缩为这条链：

```text
不可信输入
→ 身份与资源权限
→ 服务端业务规则
→ 数据库事务
→ 与事实一致的响应
```

客户端提出意图，后端验证并计算，数据库确认事实，API 表达结果。任何一层都不能用前一层的声明代替自己的判断。
