# 怎样观测并处理一次后端事故

运动鞋促销开始二十分钟后，客服收到用户反馈：订单已经创建，支付页面却长时间显示“正在确认”。接口监控中的 HTTP 成功率仍然正常，支付服务的平均延迟也没有明显变化。

值班工程师继续查看，发现处于 `payment_confirming` 的订单快速增加，少数支付请求需要十几秒才返回。订单服务的支付连接池已经接近耗尽，新请求虽然仍返回 200，却无法及时完成用户交易。

生产观测要从用户结果进入系统，再用指标、日志和 Trace 缩小范围。事故处理则要在根因完全明确之前限制影响，并在恢复后处理已经积压的业务状态。

## 先确认用户受到了什么影响

“支付系统异常”范围太大，不能直接指导止损。值班工程师先回答：

- 用户无法完成哪一步；
- 问题从什么时候开始；
- 影响所有订单还是特定商品、地区或支付方式；
- 请求完全失败，还是完成时间明显增加；
- 是否存在重复支付、错误金额或订单丢失；
- 有多少订单停在中间状态，最早一笔等待多久。

这次事故集中在新创建的订单。用户可以进入支付，但订单服务在创建支付单超时后进入 `payment_confirming`，确认任务又迟迟得不到结果。资金尚未确认错误，主要影响是用户无法继续支付，订单占用库存。

影响描述可以整理为：

```text
开始时间: 10:20
受影响流程: 创建订单后的支付确认
受影响订单: 约 18%，集中在支付方式 W
用户结果: 长时间停在“正在确认”，无法继续支付
当前风险: 库存被待支付订单占用；尚未发现重复扣款
```

事故严重程度应依据用户与业务影响，而不是某个服务是否返回 500。HTTP 200 也可能承载“处理中”或降级结果，技术成功率无法代替交易完成率。

## 指标先告诉工程师哪里在变坏

指标是按时间聚合的数值，适合观察趋势和范围。订单系统可以从三层指标开始。

| 层面 | 示例 | 回答的问题 |
| --- | --- | --- |
| 用户结果 | 下单完成率、支付完成率、支付确认时长 | 用户是否完成交易，影响有多大 |
| 业务状态 | `payment_confirming` 数量、最老等待时间、订单取消率 | 工作积压在哪个业务阶段 |
| 系统运行 | 请求延迟、错误率、连接池等待、队列深度、数据库锁等待 | 哪个组件或资源正在异常 |

值班工程师看到：支付完成率从 82% 降到 61%；`payment_confirming` 订单持续增长；订单服务取得支付连接的 99 分位等待时间从 20 毫秒升到 1.8 秒。

平均延迟没有明显变化，是因为大多数请求仍然很快，少量极慢请求被平均值稀释。高分位延迟和积压年龄更早暴露尾部故障。

指标不应为每个可测量数值都设置告警。告警要指向需要采取行动的变化，例如用户支付完成率持续下降、确认订单最老等待超过期限、连接池接近耗尽。CPU 短暂升高但用户结果正常，通常只需观察。

## 日志解释一项具体操作发生了什么

指标把范围缩小到支付确认路径，工程师再选择几笔受影响订单查看结构化日志。

一次支付创建日志可以包含：

```json
{
  "timestamp": "2026-07-14T10:27:12Z",
  "level": "WARN",
  "service": "order-service",
  "request_id": "REQ-301",
  "order_id": "O-1042",
  "payment_request_id": "PAYREQ-7001",
  "dependency": "payment-service",
  "operation": "create_payment",
  "duration_ms": 2000,
  "result": "timeout"
}
```

结构化字段让工程师按订单、支付方式、服务版本和结果聚合，而不必搜索自然语言句子。

日志要记录发生了什么，避免写入完整银行卡信息、访问令牌和其他敏感数据。订单 ID 与支付请求 ID足以关联权威系统，必要内容由有权限的工程师再查询。

错误日志也不能只写“支付失败”。连接超时、连接池等待超时、下游返回限流和金额冲突需要不同处理。稳定错误类别可以直接连接告警和恢复策略。

## Trace 还原时间花在了哪里

一笔用户请求经过网关、订单服务、支付服务和数据库。Trace 使用同一个 Trace ID 连接各段操作，每段称为 Span。

受影响请求的轨迹可能是：

```text
POST /orders/O-1042/pay            2.4s
├─ validate_order                  12ms
├─ acquire_payment_connection      1.8s
├─ POST payment-service/payments   580ms
└─ save_payment_confirming         8ms
```

支付服务自身只处理了 580 毫秒，主要时间消耗在订单服务等待连接。此时盲目扩容支付服务不会解决问题。

对比正常 Trace 后，工程师发现正常请求几乎不等待连接。异常从促销版本发布后开始，新版本把支付调用的并发上限从每实例 20 提高到 100，连接池容量却仍是 30。大量请求同时进入后，连接池开始排队，超时又触发确认查询，进一步占用相同连接。

Trace 将“支付很慢”拆成了具体等待位置。它不自动给出根因，仍要与版本、配置和资源指标对照。

## 版本与配置也属于事故现场

相同代码在不同配置下会表现不同。每个请求的日志与 Trace 应能关联实际生效的：

- 应用版本；
- 运行环境和实例；
- 超时、连接池和并发配置版本；
- 依赖地址与路由策略；
- 功能开关和灰度分组。

这次事故只影响新版本实例，旧版本和同一支付服务仍然正常，问题范围进一步收敛。工程师不需要重启全部服务，也不需要假设支付平台整体故障。

若发布记录只保存代码提交号，却没有记录动态配置，就无法重现为什么同一版本在不同时间表现不同。

## 根因未明时先限制影响

调查完整根因需要时间，系统资源却正在耗尽。团队先执行几项可逆止损：

- 暂停继续扩大新版本流量；
- 把支付并发上限恢复到已验证值；
- 限制新的支付确认查询；
- 保留订单与 `payment_request_id`，不重复创建支付；
- 非核心报表和推荐任务暂时停用，为交易释放资源；
- 页面明确显示支付系统繁忙，允许用户稍后继续。

这些动作保护新请求和权威状态，同时保留现场数据。直接清空确认任务或把订单标记失败，虽然能让积压数字下降，却可能丢失已经创建的支付单。

涉及资金错误、重复扣款或金额不一致时，止损应更严格：暂停相关支付方式，冻结自动状态转换，保存审计信息并进入人工核对。

**事故处理先控制新的用户影响，再继续确认完整根因。**止损不等于根因修复，它只是为调查和恢复争取稳定环境。

## 怎样从现象确认根因

团队提出假设：新版本提高并发后，连接池排队引发了创建超时；确认任务继续使用同一连接池，形成自我放大。

验证时尽量只改变一个因素：

1. 灰度实例恢复旧并发配置；
2. 连接池大小保持不变；
3. 使用相同支付测试流量；
4. 比较连接等待、超时和确认任务增长。

恢复配置后，连接等待迅速下降，新订单不再大规模进入 `payment_confirming`。测试环境中重新应用高并发配置，又能复现排队。这为根因提供了证据。

如果团队同时扩容连接池、修改超时、关闭重试并升级支付 SDK，即使指标恢复，也无法判断哪项真正有效，后续更难制定长期修复。

## 回滚以后还要处理已有状态

新请求恢复不代表事故结束。事故期间已经积累数千笔 `payment_confirming` 订单，其中可能存在三种结果：

- 支付单已经创建，只缺少订单侧记录；
- 支付服务明确没有收到创建请求；
- 支付服务仍无法确认结果。

恢复任务根据原 `payment_request_id` 查询：找到支付单就保存同一个 `payment_id`；明确不存在时才重新创建；仍然未知则保留状态并按退避继续查询。

订单可能接近支付过期时间。系统需要决定是否延长库存保留，并通知用户继续支付；不能因为平台事故让用户的订单在确认过程中自动过期。

积压清理也要控制速度。一次性并发查询所有旧订单，会让刚恢复的支付服务再次过载。恢复流量通常要低于依赖剩余容量，并持续观察新请求与积压任务的共同负载。

## 什么情况下可以宣布恢复

团队不能在配置回滚命令成功后立即结束事故。恢复需要多层证据：

- 新支付请求的高分位延迟恢复；
- 连接池等待和超时下降；
- 新增 `payment_confirming` 数量回到正常范围；
- 历史积压持续减少，最老等待时间下降；
- 用户支付完成率恢复；
- 抽查订单与支付平台记录，没有重复支付和金额错误；
- 限流与熔断没有掩盖新的失败。

底层资源通常较早恢复，业务积压随后清理，用户完成率还要经过一段时间才会回升。事故结束时间应以用户和业务状态恢复为依据。

## 复盘怎样改变系统

有效复盘要形成具体改动，而不是只写“加强监控”或“谨慎修改配置”。这次事故可以产生：

- 为支付并发和连接池容量建立一致性校验；
- 发布前压测高分位支付延迟；
- 灰度观察连接等待和支付完成率；
- 为 `payment_confirming` 数量与最老年龄设置告警；
- 将确认查询与创建支付使用独立并发预算；
- 明确支付故障时的页面降级与库存延期策略；
- 编写一键恢复并发配置的操作手册；
- 演练支付响应丢失和连接池耗尽。

事故暴露的也可能是组织问题：配置没有评审，值班人员看不到发布记录，或者订单与支付团队对超时语义理解不同。复盘要修复真正让事故发生或扩大的一组条件。

责任追究不能替代系统改进。个人操作失误若可以轻易影响所有流量，发布机制本身也缺少保护。

## 三类观测怎样配合

一次后端事故中，三类信息形成不同视角：

- 指标发现何时、哪里和多大范围正在变坏；
- 日志说明具体订单和调用记录了什么结果；
- Trace 还原一项请求跨服务的顺序与耗时。

它们还需要版本、配置和业务状态作为上下文。只有技术指标，没有订单结果，团队可能在用户仍然无法支付时宣布恢复；只有业务失败，没有服务观测，又难以找到应该修改哪一层。

## 一次事故的处理主线

后端事故可以沿一条稳定路径处理：

1. 确认用户受到的具体影响；
2. 用指标按时间、版本和业务阶段缩小范围；
3. 通过日志与 Trace 找到异常发生的位置；
4. 在根因完整确认前先限制新影响；
5. 用对照实验验证根因并修复；
6. 处理积压和结果未知的业务状态；
7. 以用户结果恢复作为结束依据；
8. 把事故转化成容量、发布、告警和演练规则。

观测服务于行动。生产系统真正恢复时，新交易重新完成，已有状态得到处置，同类问题也增加了更早发现和更快止损的防线。
