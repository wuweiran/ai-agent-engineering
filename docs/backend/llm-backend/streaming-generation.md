# 一次模型回答怎样流式返回

用户打开订单助手，输入：“我买的鞋为什么还没发货？”普通订单接口可以很快返回一段 JSON，模型却需要读取订单与物流信息，再逐步生成自然语言解释。若后端一直等到全部内容完成，用户可能数秒内看不到任何反馈。

团队决定流式返回回答。这个变化看似只是边生成边显示，实际增加了生成状态、事件协议、取消、断线恢复和最终消息等后端需求。

这篇文章属于“接收请求”和“协调工作”：后端要把一次持续生成管理成可以观察、取消和恢复的业务过程。

## 普通 HTTP 响应怎样工作

传统订单查询可以一次返回完整结果：

```http
GET /api/orders/O-1042 HTTP/1.1
Authorization: Bearer <access-token>
```

服务验证用户、查询订单，然后返回：

```json
{
  "order_id": "O-1042",
  "status": "paid",
  "shipment_status": "waiting_pickup"
}
```

响应提交后，请求结束。客户端拿到的是一个完整结构，后端也有清楚的成功或失败状态。

订单助手仍要执行身份和订单查询，却多了一段持续生成。模型可能先输出“你的订单已经支付”，稍后补充“仓库正在等待快递揽收”。传输中的片段并不构成完整回答。

## 先创建一项生成记录

用户提交问题时，应用后端先绑定身份与会话，并创建生成 ID：

```text
generation_id: G-2048
conversation_id: C-81
user_id: U-204
user_message_id: M-301
assistant_message_id: M-302
status: starting
created_at: 2026-07-14T11:20:00Z
```

这项记录让后端能够回答：哪位用户发起了哪次生成，目前处于什么状态，最终内容保存在哪里。

客户端重试提交时，应携带稳定请求 ID，避免网络抖动创建两项相同生成。生成记录与模型供应商的请求 ID 还可以关联，用于排查超时和计费。

用户问题进入数据库后，后端再查询他有权查看的 `O-1042`。模型不会从客户端传来的订单 ID自动获得权限，订单事实也不能由历史回答代替。

## SSE 怎样把事件推给浏览器

Server-Sent Events，简称 SSE，允许服务器在一条 HTTP 连接上持续向浏览器发送文本事件。订单助手可以使用：

```text
event: generation.started
data: {"generation_id":"G-2048"}

event: message.delta
data: {"generation_id":"G-2048","sequence":1,"text":"你的订单已经支付，"}

event: message.delta
data: {"generation_id":"G-2048","sequence":2,"text":"目前正在等待快递揽收。"}

event: message.completed
data: {"generation_id":"G-2048","message_id":"M-302"}
```

事件类型比直接发送裸文本更容易演进。客户端能够区分内容片段、完成、错误和取消，也可以用 `sequence` 检查顺序与缺失。

SSE 适合服务器持续向客户端单向推送。若应用需要频繁双向实时控制，WebSocket 可能更合适。订单助手的用户消息仍可通过普通 POST 提交，回答通过 SSE 返回，结构通常已经足够。

技术选择应由交互需求决定，不需要因为“AI 是实时的”就统一使用 WebSocket。

## 流式片段为什么不能直接成为最终消息

模型输出可能在中途失败或被用户取消：

```text
“你的订单已经支付，但由于——”
```

这段文字不完整，也可能缺少随后出现的重要限制。若后端收到每个片段就覆盖数据库中的最终消息，客户端重连后可能把半句话当作正式回答。

更清楚的做法是区分：

- 流式片段用于当前连接中的实时展示；
- 生成缓冲保存后端目前收到的内容；
- 只有模型正常完成并通过必要检查后，才提交最终 Assistant Message；
- 取消或失败时，记录终止原因，决定是否保留部分内容供用户参考。

生成状态可以包括：

```text
starting
streaming
completed
cancelled
failed
```

状态变化由后端确认，客户端看到的动画不能成为权威记录。

## 什么时候把完整回答写进数据库

后端收到模型完成信号后，还要确认：

- 流中所有片段已经按序合并；
- 输出没有超过产品允许长度；
- 必要的结构化部分能够解析；
- 回答关联正确会话和用户；
- 内容符合当前产品的保存规则。

检查通过后，后端在事务中写入完整消息，并把生成状态改为 `completed`。SSE 随后发送 `message.completed`。

若先通知客户端完成，数据库写入却失败，用户刷新页面后会找不到刚才的回答。因此，完成事件应该对应一个已经持久化、可以重新读取的结果。

有些产品会持续保存草稿，以支持很长的生成和恢复。草稿也要与最终消息明确区分，不能让其他业务逻辑误用。

## 用户点击停止时发生什么

用户发现问题问错了，点击“停止生成”。客户端发送：

```http
POST /api/generations/G-2048/cancel
```

后端先检查 `G-2048` 属于当前用户，并将取消意图写入状态，再通知正在执行的模型调用停止读取或取消下游请求。

取消有几个边界：

- 模型可能在取消到达前已经完成；
- 供应商可能只能停止后续输出，已消耗 Token 无法撤回；
- 下游请求取消不代表立刻停止计算；
- 已经发送给客户端的片段无法收回；
- 若生成同时触发业务动作，还要分别处理那些动作的结果。

因此，取消接口要具备幂等性。重复取消已经结束的生成，可以返回当前最终状态，而不是制造错误。

对于纯回答，产品可以保留部分文本并标记“已停止”，也可以不写入会话。选择要保持一致，并明确这段文本是否会进入未来 Context。

## 浏览器断线后是否停止模型

用户切换网络或关闭页面，SSE 连接断开。后端有两种合理策略。

### 连接断开就取消

即时问答且生成成本较高时，用户已经离开，继续生成可能没有价值。后端检测连接关闭后取消下游调用，并把状态标记为 `cancelled`。

这种方式节省资源，却无法支持用户刷新后继续查看结果。短暂网络抖动也可能浪费已经完成的大部分生成。

### 生成作为独立任务继续

长报告或重要任务可以在连接断开后继续。模型调用由后台任务持有，SSE 只是订阅生成事件。用户重新进入页面后，根据 `generation_id` 查询状态并重新订阅。

这种方式更可靠，也需要事件缓冲、状态持久化和后台容量。用户明确取消时，任务仍要停止。

策略取决于产品承诺。后端必须显式选择，不能让生成是否继续偶然取决于哪个进程持有客户端 Socket。

## 断线重连怎样补回内容

SSE 连接不保证客户端收到断线期间的事件。若产品支持重连，服务可以保留一段事件记录，客户端带上最近收到的序号：

```http
GET /api/generations/G-2048/events?after=17
```

后端先返回序号 18 之后的已保存事件，再继续推送新事件。事件应有稳定 ID 或序号，并设置合理保留期。

另一种简化方式是在重连后直接返回当前完整草稿，再从当前位置继续。关键是客户端不能把重复片段拼接两次，也不能在缺少中间内容时假装完整。

如果生成已经完成，重连不必重放所有 Delta。客户端可以直接读取已持久化的最终消息。

## 模型调用失败时怎样表达

失败可能发生在不同阶段：

- 调用前的身份或订单查询失败；
- 模型服务限流或不可用；
- 流传输中断；
- 输出超过上限；
- 最终消息保存失败；
- 用户取消。

SSE 可以发送稳定错误事件：

```text
event: generation.failed
data: {
  "generation_id": "G-2048",
  "code": "MODEL_TIMEOUT",
  "retryable": true
}
```

客户端不需要看到供应商异常堆栈。后端日志与 Trace 保存真实错误、供应商请求 ID、模型版本和已接收 Token。

重试也不能默认追加到原回答后面。新尝试应有独立 `attempt_id`，并确认是从头重做、继续生成，还是向用户展示一个新的回答版本。

## 流式连接怎样影响容量

普通接口几十毫秒结束，一台实例可以快速复用连接和执行资源。模型生成持续数秒甚至更久，会显著增加同时在线连接数。

容量规划要观察：

- 活跃 SSE 连接；
- 同时进行的模型生成；
- 等待模型配额的请求；
- 每次生成的输入输出 Token；
- 客户端断开后仍在运行的任务；
- 生成缓冲和事件记录占用的内存；
- 网关与负载均衡器的空闲超时。

服务可以限制每名用户的并发生成，设置总生成上限和有限队列。达到容量时，明确返回繁忙或排队状态，比让所有连接长时间无响应更可靠。

SSE 连接也需要心跳，避免代理因长时间没有文本而关闭；客户端离开后要及时释放订阅资源。心跳只证明连接仍在，不代表模型生成有进展。

## 日志和 Trace 要增加什么

一次生成需要关联：

```text
request_id
generation_id
conversation_id
user_message_id
assistant_message_id
model_request_id
model_and_config_version
```

工程师还要区分几段耗时：准备业务数据、等待模型首个 Token、持续生成、最终检查、数据库提交。用户觉得“回答很慢”，可能是订单查询慢，也可能是模型首字延迟高，或者内容生成完成后保存阻塞。

生产指标可以包括：

- 生成成功、失败和取消比例；
- 首个 Token 延迟与完整回答延迟；
- 输出 Token 速度；
- 客户端断线率；
- 重连恢复成功率；
- 未提交草稿数量；
- 用户在回答完成前离开的比例。

模型调用成功率仍然不等于用户取得答案。最终消息是否成功保存与展示，才完成这次后端交付。

## 从普通请求到流式生成，后端增加了什么

传统 HTTP 请求通常一次返回完整结果。流式生成把一次调用扩展成持续一段时间的状态过程：

- 用生成 ID 关联请求、模型调用与最终消息；
- 使用事件协议传递开始、片段、完成、取消与失败；
- 区分实时片段、草稿和权威最终消息；
- 明确用户取消与客户端断线的语义；
- 通过持久状态和事件记录支持恢复；
- 按长连接、模型并发和 Token 重新规划容量；
- 在 Trace 中分别观察数据准备、首字与完整生成。

**流式传输改善等待体验，生成状态和最终消息保证结果可以确认与恢复。**

订单助手下一步还要决定模型本轮看到哪些订单、物流、政策和会话内容。数据库可以保存完整历史，模型请求却不能无限装入全部数据；Context 的选择会直接影响回答质量、权限与成本。
