# 大模型应用怎样改变后端

电商系统增加了订单助手。用户不再只点击“查询物流”，还可以问：“这双鞋为什么没发货？如果还来得及，把地址改到公司。”

原有订单、物流和地址服务仍然掌握业务事实。大模型增加的是一种新的计算方式：它根据 Context 生成文字，也可以根据中间结果提出下一项工具调用。这个计算过程持续更久、按 Token 消耗资源，输出也不像普通函数那样完全确定。

应用后端要把这种不确定能力放进确定的业务边界：

```text
后端选择模型本轮能看到什么
        ↓
模型生成文字或提出工具请求
        ↓
后端验证、执行并确认真实结果
        ↓
持久状态记录任务已经进行到哪里
```

身份、权限、订单状态和真实业务动作仍然由程序控制。模型看到一段订单描述，不代表它获得了订单权限；模型说“地址已修改”，也不能代替订单服务的成功回执。

## 模型调用为什么不是普通接口调用

传统接口通常在较短时间内返回完整 JSON。模型可能持续数秒生成，前端希望逐步看到内容；用户还可能取消、断线并重新进入页面。

生成过程因此需要稳定任务或生成 ID、事件协议和最终状态。[一次模型回答怎样流式返回](streaming-generation/)说明临时的文本与进度事件怎样交付，以及为什么连接断开不能自动决定任务是否停止。

流式传输只解决“过程怎样被看见”。最终消息、任务状态和业务结果仍要可靠保存，才能在刷新页面后重新读取。

## 模型提出动作以后发生什么

订单助手查询物流后，可能请求 `change_shipping_address`。这段模型输出只是候选动作：参数需要 Schema 校验，订单和地址要按当前用户重新授权，订单状态必须仍允许修改，高影响操作还可能等待用户确认。

[模型发出工具调用后，后端怎样继续执行](tool-execution-loop/)沿着一轮工具请求展开：

```text
模型提出请求
→ 后端校验权限与业务条件
→ 确定性程序执行
→ 查询或保存真实结果
→ 把结果交给模型继续判断
```

模型可以选择下一步，软件决定这一步能不能做，并确认它是否真的成功。这条边界使 Agent 能处理开放任务，又不能绕过已有业务系统。

## 多轮任务为什么需要独立状态

一次 Agent 任务可能进行多轮模型调用、工具执行和用户确认。只保存最后一条聊天消息，服务重启后无法判断已经执行过哪些动作，尤其无法处理“外部修改可能成功但响应丢失”的情况。

[一次 Agent 任务需要保存哪些状态](agent-task-state/)从恢复场景倒推最小状态：Task 保存目标和总体阶段，Run 区分执行尝试，Step 标记稳定执行边界，Tool Call 记录有副作用动作是否已确认。

数据库状态和模型 Context 承担不同职责：

- 持久状态说明系统已经确认了什么，支持调度、恢复和审计；
- Context 是下一轮从任务目标、业务事实和近期结果中选择出的输入；
- 订单、支付等领域服务继续掌握真实业务状态。

不能把完整对话当作任务数据库，也不能把数据库里的所有历史无选择地塞给模型。

## 后端工程的新判断

大模型没有推翻请求、状态、异步任务和生产运行的基础。它让这些问题以新的形式同时出现：

```text
流式输出是临时传输，什么结果才算最终提交？
模型输出不确定，哪些内容可以展示，哪些动作必须由程序验证？
任务包含多轮调用，怎样在断线和重启后继续？
每轮都消耗 Token，怎样限制时间、轮次、并发和预算？
模型、Prompt、工具和 Context 变化后，怎样重现与评测行为？
```

其中前三项构成当前主线。模型路由、Context 检索、行为评测和组合发布，应在真实任务需要时继续展开，而不是先建设一套脱离产品的 AI 平台分类。

**大模型应用后端的核心职责，是让概率性的模型生成经过确定性的权限、状态和执行控制，最终成为可以确认、恢复和运营的产品结果。**
