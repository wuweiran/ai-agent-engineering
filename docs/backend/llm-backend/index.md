# 大模型应用怎样改变后端

电商系统准备增加订单助手。用户不再只点击“查询物流”或“修改地址”，还可以直接询问：“我买的鞋为什么还没发货？如果今天发不了，能改到公司地址吗？”

后端原有的订单、物流和地址服务仍然存在。大模型增加了一种新的交互与计算能力：理解自然语言，综合非结构化信息，生成解释和候选动作。这项能力也带来长时间生成、上下文依赖、不确定输出和按 Token 消耗资源等新约束。

后端的基本责任没有改变。身份仍由认证系统确认，订单与库存仍由业务服务保存，修改地址仍要校验订单状态，故障仍要超时、恢复和观测。变化发生在这些责任的实现方式与团队分工上。

## 大模型带来了哪些新需求

传统订单查询接收订单 ID，程序执行确定逻辑并返回结构化结果。订单助手接收自然语言问题，后端要先准备模型能够理解的输入，再管理一段逐步产生的回答。

| 新需求 | 后端要解决的问题 |
| --- | --- |
| 长时间生成 | 怎样流式返回、取消、超时，并在客户端断线后保留可确认结果 |
| Context 组装 | 从会话、订单、物流、政策和检索资料中选择本轮真正需要的信息 |
| 输出约束 | 怎样区分自然语言解释、结构化意图和真实业务结果 |
| 行为验证 | 同一任务表达可能不同，怎样用样本与成功标准持续评测 |
| 模型依赖治理 | 怎样管理模型路由、限流、Token、成本、版本和供应商故障 |
| 组合发布 | 模型、Prompt、检索数据和应用代码变化时，怎样一起灰度与回滚 |

这些需求不会把后端替换成一套全新技术。它们分别落回传统后端中的请求、状态、异步任务、远程调用、容量和发布，只是原有设计需要处理更长、更大且不完全确定的计算。

## 后端职能怎样发生分化

早期项目通常由一个应用团队完成模型接入、Prompt、接口和页面。产品规模扩大后，工作会逐渐分化。分化依据是持续责任，不要求每家公司都建立同名团队。

### AI 应用后端

应用后端把模型能力接进具体业务。订单助手团队会负责：

- 绑定用户、会话和订单范围；
- 查询订单与物流事实；
- 组织本轮 Context；
- 管理生成状态和流式协议；
- 校验模型给出的意图与参数；
- 调用真实订单服务；
- 把回答和业务结果交付给用户。

它最接近产品，对“用户是否解决问题”负责。传统业务规则、事务与权限仍然是这项工作的底座。

### 模型接入与网关

多个团队调用不同模型后，常见能力会沉淀成共享网关：

- 统一认证、模型路由和供应商接入；
- 按租户与功能限制并发和 Token；
- 记录调用量、延迟与费用；
- 管理超时、重试和故障降级；
- 版本化模型与基础调用策略；
- 保护模型凭据并实施数据边界。

网关负责模型依赖的通用治理，不理解每个订单问题应该怎样回答。若把所有 Prompt 和业务规则都塞进网关，它会变成难以演进的中央业务服务。

### Context 与知识系统

订单助手需要物流术语、售后政策和商品说明。随着资料增加，会出现专门的数据与检索工作：

- 文档怎样接入、清洗和分段；
- 权限怎样传播到检索阶段；
- 内容怎样更新和失效；
- 召回结果怎样评测；
- 哪些信息来自权威业务服务，哪些来自知识资料。

这部分经常由后端、数据工程和搜索团队共同承担。模型每轮实际看到什么，仍由应用根据当前任务装配。

### AI 质量与平台能力

模型行为无法只用传统单元测试覆盖。规模增长后，团队还会建设：

- 真实任务样本和回归评测；
- Prompt、模型和检索版本记录；
- 生成 Trace 与失败归因；
- 灰度实验和质量看板；
- 内容安全、人工审查和事故响应；
- 供多个业务复用的评测与发布平台。

这些工作可能由专门平台团队负责，业务团队仍要定义什么叫任务成功。平台无法替订单领域专家决定一段回答是否正确承诺了物流时间。

## 后端工程师开始关注哪些问题

传统后端常用请求成功率、延迟和资源消耗衡量服务。大模型应用还要增加新的观察维度。

### 请求是否真正完成用户任务

模型 API 返回 200，只说明生成请求被服务接受并返回。用户可能得到错误物流解释，等待太久后离开，或者模型承诺了业务系统不允许的地址修改。

因此，后端观测要连接三层信息：模型调用是否成功，生成内容是否满足约束，用户是否完成查询或修改任务。

### 模型实际看到了什么

相同问题可能因为订单信息缺失、检索到旧政策或会话历史过长而得到不同结果。Trace 需要记录 Context 来源、版本和作用域，而不只是最终回答。

这也改变了排查顺序。错误回答不一定来自模型能力，可能是后端没有取得当前物流状态，或把另一个用户的资料错误装入 Context。

### 输出能否被程序使用

自然语言回答适合展示，业务动作需要结构化意图。例如模型识别出 `change_shipping_address` 后，程序仍要校验订单归属、发货状态和目标地址。

模型输出是候选解释与参数，不自动成为数据库事实。越接近资金、权限和不可逆操作，越需要确定性校验与用户确认。

### 成本由什么决定

传统接口容量常按每秒请求和 CPU 估算。模型请求的消耗还受到输入长度、输出长度、模型选择和调用轮次影响。

每秒同样十个请求，短分类与长文档分析可能相差几十倍。后端开始记录输入输出 Token、Context 长度、模型、租户和业务功能，并管理预算、并发和排队。

### 行为变化怎样发布

应用代码没有改变，切换模型或修改 Prompt 也可能改变回答。检索索引更新还会改变模型看到的资料。

一次 AI 功能发布因此是一组行为版本：应用、模型、Prompt、Context 组装、检索数据、输出 Schema 和业务规则。发布前需要评测，生产中需要灰度，问题出现后要能重现与回滚这组组合。

## 工程师的工作方式怎样变化

### 需求从接口规格扩展到任务样本

“返回订单状态”可以写成明确接口。“正确回答用户关于发货与改地址的问题”还需要一批真实场景：未发货、已经出库、地址不可修改、物流资料延迟、用户表达含糊等。

产品、领域专家和工程师共同把这些场景变成成功条件。样本帮助团队明确产品承诺，也成为后续评测与回归依据。

### 测试之外增加持续评测

身份、权限、状态转换和 JSON Schema 继续由普通自动化测试检查。回答是否忠于订单事实、是否误导用户，则需要规则、人工或经过校准的模型评分。

模型和数据持续变化，离线评测也要持续运行。生产中出现的新失败经过脱敏后加入回归集，使修复可以长期验证。

### 代码评审扩展到行为评审

工程师除了审查 SQL、事务和错误处理，还要审查 Prompt、Context 来源、输出边界和评测变化。一次看似简单的 Prompt 修改可能影响多个任务，评审需要看到相邻场景和回归结果。

### 排障从调用链扩展到信息链

传统 Trace 说明请求在哪个服务超时。AI Trace 还要说明模型使用了哪版配置、取得哪些业务事实和检索资料、输出经过哪些校验，以及第一次错误在哪里出现。

工程师需要同时排查服务链与信息链。

### 协作范围扩大

后端工程师会更频繁地与领域专家定义正确答案，与数据团队维护知识资料，与前端团队设计流式交互，与 Agent 工程师划分模型和程序的控制边界。

模型接入使后端更接近产品行为，但没有取消既有工程责任。

## 哪些知识需要补充

传统后端知识仍然直接适用，同时要理解几组新概念：

| 传统基础 | 新增知识 |
| --- | --- |
| HTTP、异步任务 | 流式生成、取消、断线恢复和最终消息 |
| 数据库与会话 | 会话历史、当前 Context、生成记录和长期记忆的区别 |
| 输入输出校验 | Prompt、结构化输出和模型候选结果校验 |
| 远程调用与容量 | Token、模型并发、路由、缓存与供应商限流 |
| 自动化测试 | 评测样本、语义评分、重复运行和回归集 |
| 灰度与观测 | 模型、Prompt、检索数据的组合发布与生成 Trace |

后端工程师需要理解模型作为应用依赖的特殊契约。模型训练、Transformer 和 GPU 推理属于大模型基础与推理基础设施，可以在对应章节深入。

## 与其他章节的边界

大模型应用后端回答：模型能力怎样进入真实产品，并在身份、状态、容量和生产约束下长期运行。

AI Agent 工程回答：模型怎样根据反馈选择工具与下一步路径。如果订单助手只生成解释，它仍是普通大模型应用；当模型开始连续查询物流、检查规则并选择后续动作时，才进入 Agent 设计。

数据工程回答知识资料与检索索引怎样生产和更新；大模型基础解释 Token、上下文窗口和模型能力；推理基础设施解释 GPU 和模型服务怎样运行。

视角边界清楚后，同一个术语可以自然连接而不重复。

## 从回答进入 Agent 执行

[一次模型回答怎样流式返回](streaming-generation/)从订单助手的一次提问开始，解释生成状态、SSE 事件、用户取消、客户端断线和最终消息怎样配合。它建立了单次模型调用的请求边界。

当模型开始根据订单和物流结果选择工具，应用后端还要运行一段持续循环。[模型发出工具调用后，后端怎样继续执行](tool-execution-loop/)说明工具请求怎样经过参数校验、授权、审批和幂等控制，成为可以确认的业务动作。

多轮模型调用和工具执行还会跨越页面刷新与服务重启。[一次 Agent 任务需要保存哪些状态](agent-task-state/)进一步区分 Task、Run、Step、Tool Call、Event 和 Artifact，使系统能够确认已经发生了什么，并从稳定边界安全继续。
