# Agent 工具执行

用户对订单助手说：“查一下这双鞋为什么还没发货；如果还来得及，把地址改到公司。”模型查询订单和物流后，请求修改配送地址。

工具应该怎样表达业务动作、参数和返回值，属于 Agent 设计问题，详见[Agent 工具](../../ai-agent/agent-design/tools/)。这里假设 `change_shipping_address` 的工具契约已经确定，只讨论模型产生 Tool Call 后，应用后端怎样把它变成一次安全、可恢复的业务执行。

**Tool Call 是模型提出的候选动作，只有业务服务确认成功后，动作才成为事实。**

## 模型输出先进入确定性边界

模型可能返回：

```json
{
  "type": "tool_call",
  "id": "TC-408",
  "name": "change_shipping_address",
  "arguments": {
    "order_id": "O-1042",
    "address_id": "ADDR-73"
  }
}
```

这段 JSON 不是已经发出的业务请求。后端先从当前 Agent 版本的工具注册表中查找执行器，并按既定 Schema 解析参数。不存在的工具、缺少字段和非法枚举直接成为稳定的工具错误，不能通过反射调用任意内部函数。

接下来的一轮执行具有固定边界：

```text
模型提出工具请求
        ↓
后端绑定身份，校验参数和当前业务条件
        ↓
保存执行意图，必要时等待确认
        ↓
调用业务服务并确认结果
        ↓
保存结果，再交给模型判断下一步
```

模型可以根据反馈改变计划，但不能跳过这条执行链。

## 身份和业务条件由后端重新取得

模型参数只包含它想操作的对象。用户、租户、角色和会话权限来自认证上下文，不能接受模型生成的 `user_id` 或 `tenant_id`。

后端用当前身份重新查询订单和地址，并确认：

- 订单与地址是否属于当前用户；
- 订单是否仍处于允许修改的状态；
- 地址是否在配送范围内；
- 这项操作是否需要用户确认或人工审批。

工具 Schema 只能证明参数形状合法，不能证明操作被授权。Prompt 中的“只处理当前用户订单”也不能代替服务端检查。

业务条件还必须在真正写入时再次确认。模型读取订单后，仓库可能已经发货；订单服务应使用当前状态执行条件更新，而不是相信 Agent 几秒前取得的快照。

## 需要确认时先保存确定方案

修改地址会改变真实业务状态。Agent 可以先形成待确认方案：

```json
{
  "change_request_id": "AC-9021",
  "order_id": "O-1042",
  "from_address": "北京市朝阳区……",
  "to_address": "北京市海淀区……"
}
```

后端把方案和参数保存下来，任务进入 `waiting_for_confirmation`。用户确认后，系统执行已经展示的 `AC-9021`，不能让模型临时重新生成另一组参数。

确认本身也是一次带身份和版本的状态转换。用户确认与取消同时到达时，只有一个操作可以把任务从等待状态推进；迟到请求读取并返回已经确定的结果。

## 调用前先给副作用一个稳定身份

后端在调用订单服务前保存 Tool Call：

```text
tool_call_id: TC-408
task_id: T-301
tool_name: change_shipping_address
status: executing
business_request_id: AC-9021
idempotency_key: address-change-AC-9021
```

订单服务以 `AC-9021` 识别同一个地址修改意图，并拒绝相同 ID 携带不同参数。这样，Worker 重试或请求重复到达时，不会创建第二份业务结果。

Tool Call 和任务状态应先提交，再发起远程调用。否则进程可能在订单已经修改后重启，却没有任何记录说明这项调用发生过。

即使先保存记录，本地数据库和远程订单服务之间仍然存在故障窗口：

```text
订单服务修改成功
→ 响应在网络中丢失
→ Agent 后端只看到超时
```

此时状态是 `unknown`，不是 `failed`。恢复程序使用同一 `business_request_id` 查询订单服务：找到成功结果就补记完成；明确没有执行才允许再次提交；仍无法确认则保持等待或转人工。

模型不能因为“调用超时”自行决定再执行一次。重试副作用是后端根据幂等能力和真实结果作出的决定。

## 先保存结果，再让模型继续

订单服务确认成功后，后端在持久状态中保存业务回执，再把受控结果放进下一轮 Context：

```json
{
  "tool_call_id": "TC-408",
  "status": "succeeded",
  "result": {
    "change_request_id": "AC-9021",
    "order_id": "O-1042",
    "shipping_address_changed": true
  }
}
```

业务拒绝也应转换成稳定含义：

```json
{
  "tool_call_id": "TC-408",
  "status": "rejected",
  "error": {"code": "ORDER_ALREADY_SHIPPED"}
}
```

模型可以据此解释结果或选择下一步。数据库异常堆栈、内部凭据和无关个人信息只进入受控日志，不应直接成为模型输入。

保存顺序很重要。如果先把成功结果交给模型，进程随后在持久化前重启，模型可能生成“地址已经修改”的最终回答，而任务记录仍无法证明这件事。

## 最终回答不能代替业务完成条件

工具结果进入下一轮后，模型可能继续调用工具，也可能生成最终回答。模型说“地址已修改”不代表任务已经完成。运行时仍要检查：

```text
订单服务是否已有成功回执？
必要的用户确认是否已经保存？
是否还有 executing 或 unknown 的副作用？
最终消息和任务状态是否已经提交？
```

时间、轮次、Token 和工具调用预算也由运行时控制。模型重复相同调用却没有获得新信息，或者达到截止时间时，后端应停止、等待或转人工。

用户取消只能阻止尚未执行的步骤，不能撤销已经完成的地址修改。任务取消与业务补偿是两项不同操作。

## 后端在工具循环中守住什么

模型每轮根据 Context 和工具结果提出下一步；后端把下一步约束为一次合法业务调用，确认结果后再反馈给模型：

```text
模型提出候选动作
→ 后端绑定身份和当前业务状态
→ 持久化执行意图
→ 领域服务确认真实结果
→ 后端保存结果并推进任务
```

Agent 工具设计解决“模型应该看到哪些能力”；应用后端解决“这次候选动作怎样安全地成为现实”。出现多轮工具执行后，系统还必须保存任务已经走过的稳定步骤，才能在服务重启后继续。
