# 模型发出工具调用后，后端怎样继续执行

用户对订单助手说：“查一下这双鞋为什么还没发货；如果还来得及，把地址改到公司。”模型不能只生成文字，它需要查询订单和物流，再根据结果决定是否请求修改地址。

以下使用一个模拟订单 Agent 说明应用后端的执行边界：**模型提出候选动作，确定性程序判断能不能做、执行动作，并确认现实结果。**

## 工具调用只是模型输出

模型首先可能返回：

```json
{
  "type": "tool_call",
  "id": "TC-401",
  "name": "get_order",
  "arguments": {"order_id": "O-1042"}
}
```

这段 JSON 表示模型希望查询订单。它不是已经发出的 HTTP 请求，也不能证明当前用户有权查看 `O-1042`。

应用后端识别工具请求后，执行一轮固定控制：

```text
模型提出工具请求
        ↓
后端验证工具、参数、权限和当前状态
        ↓
确定性执行器调用业务系统
        ↓
确认并保存工具结果
        ↓
把结果交给模型决定下一步
```

如果后续顺序完全由代码预先规定，这是一条 LLM Workflow；如果模型根据工具反馈动态选择下一步，就形成 Agent 循环。无论哪一种，真实动作都必须经过同一后端边界。

## 模型只能请求显式开放的能力

后端维护当前任务可用的工具集合，例如 `get_order`、`get_shipment` 和 `prepare_address_change`。每项工具绑定参数 Schema、执行器和权限要求。

不能从模型返回的名称通过反射调用任意内部函数。模型即使生成 `delete_order_database`，运行时也只会得到未知工具错误。

工具契约面向业务动作，而不是暴露底层数据库和任意 HTTP。`change_shipping_address` 让订单服务继续实施状态和审计规则，也使底层实现变化时模型接口保持稳定。

## Schema 合法仍不等于允许执行

模型随后请求：

```json
{
  "type": "tool_call",
  "id": "TC-408",
  "name": "prepare_address_change",
  "arguments": {
    "order_id": "O-1042",
    "address_id": "ADDR-73"
  }
}
```

JSON 可以解析，只说明字段形状正确。后端仍要检查：订单和地址是否属于当前用户，订单是否尚未出库，地址是否在配送范围内，以及这个动作是否需要用户确认。

用户和租户身份来自认证上下文，不能由模型自行指定。提示词要求“只操作当前用户订单”也不能代替服务端授权。

**Schema 约束输入结构，权限和业务规则决定动作是否合法。**

## 高影响动作先形成可确认方案

查询订单可以在授权范围内自动执行；修改地址会改变真实业务状态。订单助手可以先生成待确认方案：

```json
{
  "change_request_id": "AC-9021",
  "order_id": "O-1042",
  "from_address": "北京市朝阳区……",
  "to_address": "北京市海淀区……",
  "requires_confirmation": true
}
```

任务进入 `waiting_for_confirmation`，前端展示具体变化。用户确认后，后端使用已经保存的 `change_request_id` 执行，不能让模型临时重新生成一组可能不同的参数。

读取工具、创建草稿和退款等动作风险不同，但不需要依赖一套固定等级表。后端根据副作用是否可逆、是否涉及资金或权限，决定自动执行、用户确认或人工审批。

## 有副作用的执行需要稳定身份

后端在调用订单服务前保存工具调用：

```text
tool_call_id: TC-408
task_id: T-301
tool_name: change_shipping_address
status: executing
idempotency_key: address-change-AC-9021
```

订单服务仍然在最新状态上判断是否允许修改，并用 `AC-9021` 识别重复请求。

订单服务可能已经修改成功，Agent 后端却在保存响应前崩溃。此时本地结果是 `unknown`，不是明确失败。恢复程序先用同一业务请求 ID 查询：已成功就补记结果；明确未执行才重新提交；仍无法确认则暂停自动执行。

模型不能决定“超时了，再调用一次”。这会把网络不确定性变成重复业务副作用。

## 工具结果要让下一轮能够判断

业务服务的内部响应可能包含大量字段和敏感信息，不应完整放入 Context。工具只返回当前任务需要、具有稳定含义的结果：

```json
{
  "tool_call_id": "TC-401",
  "status": "succeeded",
  "result": {
    "order_status": "paid",
    "shipment_status": "waiting_pickup",
    "address_change_allowed": true
  }
}
```

拒绝也要有稳定语义：

```json
{
  "tool_call_id": "TC-408",
  "status": "rejected",
  "error": {"code": "ORDER_ALREADY_SHIPPED"}
}
```

模型可以据此解释限制或选择下一项合理工具。数据库异常堆栈、内部凭据和无关个人数据只进入受控日志，不能成为模型输入。

权限拒绝、用户取消、预算耗尽和结果未知等边界由运行时处理，模型不能通过换一种说法绕过。

## 运行时确认任务是否真的完成

工具结果进入下一轮后，模型可能继续调用工具，也可能生成最终回答。模型说“地址已修改”不等于任务完成；运行时要检查订单服务是否已有成功回执，必要的用户确认是否存在，以及是否还有结果未知的动作。

循环还要受时间、轮次、Token 和工具调用预算约束。连续重复同一调用却没有新信息，或者达到截止时间，都应由运行时停止或转人工。

用户取消可以阻止尚未执行的步骤，却不能撤销已经完成的地址修改。任务取消与业务补偿是两项不同操作。

## 一次 Agent 循环怎样复述

模型每轮根据 Context 和工具结果提出下一步，后端把下一步约束为一个合法业务调用，确认结果后再反馈给模型：

```text
开放判断由模型完成
确定性校验和执行由程序完成
业务事实由领域服务确认
任务边界由运行时控制
```

只有文字生成时，模型调用结束就可以提交回答；出现多轮工具执行后，系统还必须保存已经走过的步骤和结果，才能在服务重启后安全继续。
