# 模型发出工具调用后，后端怎样继续执行

用户对订单助手说：“查一下这双鞋为什么还没发货；如果还来得及，把收货地址改到公司。”模型无法仅靠已有文字完成任务。它需要查询订单和物流，检查地址是否还能修改，必要时等待用户确认，再调用订单服务。

这不再是一次模型调用生成一段回答。应用后端要在模型和业务系统之间运行多轮循环，让工具结果进入下一轮判断，同时守住权限、状态和副作用边界。

以下使用一个简化的模拟订单助手说明这条执行链路。Agent 章节关心模型为什么选择某个工具；这里关心的是：**模型发出工具请求以后，应用后端怎样把它变成一次可控、可确认的业务执行。**

## 一次工具调用怎样出现

后端首先完成用户认证，取得当前会话和任务，然后把目标、必要的订单线索以及本轮可用工具交给模型。模型可能先返回一项工具请求：

```json
{
  "type": "tool_call",
  "id": "TC-401",
  "name": "get_order",
  "arguments": {
    "order_id": "O-1042"
  }
}
```

这段输出只表达“模型希望查询订单”。它不是已经执行的 HTTP 请求，更不能证明当前用户有权查看 `O-1042`。

应用后端接下来要完成四件事：

1. 识别并记录这项工具请求；
2. 验证工具、参数、权限和当前任务状态；
3. 调用真正的业务服务并确认结果；
4. 把结构化结果交给模型，开始下一轮判断。

```text
用户目标
   ↓
模型提出工具请求
   ↓
后端验证并执行工具
   ↓
工具结果进入下一轮 Context
   ↓
模型继续选择工具或给出最终回答
```

模型可以连续查询订单和物流，也可以根据结果改变计划。若每一步都由程序预先固定，它更接近 LLM Workflow；当模型根据中间反馈选择下一步时，才形成 Agent 式循环。两种系统都需要相同的后端执行边界。

## 后端不能直接按名称调用函数

一种危险实现是从模型输出中取得 `name`，再通过反射或字符串拼接调用同名函数。这样会把应用内部能力暴露给不可信输出，也难以实施逐项权限和风险控制。

更稳妥的做法是维护显式工具注册表：

```text
get_order
├─ 参数 Schema
├─ 执行器：OrderQueryTool
├─ 风险级别：read
├─ 所需权限：orders.read
├─ 超时：2 秒
└─ 结果 Schema

change_shipping_address
├─ 参数 Schema
├─ 执行器：AddressChangeTool
├─ 风险级别：write
├─ 所需权限：orders.address.write
├─ 需要用户确认：是
├─ 幂等策略：按 change_request_id 去重
└─ 结果 Schema
```

注册表是模型能力与应用代码之间的允许列表。模型只能请求当前任务实际开放的工具；即使它生成了 `delete_order_database`，运行时也会把它识别为未知工具，而不是尝试执行。

工具版本也应和一次 Agent 运行关联。若执行过程中更换了参数含义或返回结构，恢复任务时可能无法解释已经保存的调用。

## 参数通过 Schema 以后仍然不可信

假设模型随后请求修改地址：

```json
{
  "type": "tool_call",
  "id": "TC-408",
  "name": "change_shipping_address",
  "arguments": {
    "order_id": "O-1042",
    "address_id": "ADDR-73"
  }
}
```

JSON 能够解析，只说明格式正确。后端仍要逐层检查：

- `order_id` 和 `address_id` 是否符合参数 Schema；
- 当前 Agent 是否开放了这个工具；
- 当前用户是否拥有目标订单和地址；
- 订单是否仍处于允许修改地址的状态；
- 地址是否属于订单支持的配送范围；
- 当前动作是否需要用户确认或人工审批；
- 同一修改请求是否已经执行过。

用户身份和租户范围应来自认证上下文，不能让模型通过参数自行指定。即使工具参数包含 `user_id`，服务端也要与当前身份比对，或干脆不向模型开放这个字段。

**参数 Schema 解决“输入长什么样”，授权与业务校验解决“当前调用者能不能这样做”。**

## 查询工具和副作用工具要区别处理

`get_order` 读取订单，失败后通常可以安全重试。`change_shipping_address` 会修改真实业务状态，响应丢失时再次执行可能产生冲突。后端不能把所有工具都当成普通函数调用。

可以先按业务影响划分执行策略：

| 工具类型 | 示例 | 后端控制 |
| --- | --- | --- |
| 只读查询 | 查询订单、物流和售后政策 | 授权、超时、结果裁剪、有限重试 |
| 可逆写入 | 创建草稿、更新待确认方案 | 幂等、版本检查、审计、必要时撤销 |
| 高影响动作 | 退款、发信、部署、删除 | 明确确认、审批、额度限制和结果核验 |

订单地址修改适合拆成准备与执行两个阶段。后端先检查订单并生成一份待确认方案：

```json
{
  "change_request_id": "AC-9021",
  "order_id": "O-1042",
  "from_address": "北京市朝阳区……",
  "to_address": "北京市海淀区……",
  "expires_at": "2026-07-14T12:10:00Z",
  "requires_confirmation": true
}
```

任务进入 `waiting_for_confirmation`，前端显示具体变化。用户确认后，后端根据 `change_request_id` 执行，而不是让模型重新生成一份可能不同的参数。

这条边界尤其适用于资金、权限和不可逆动作：模型可以帮助形成候选方案，真实副作用由确定性接口在最新业务状态上再次校验。

## 执行前记录，执行后确认

工具请求通过校验后，后端先为它创建稳定记录：

```text
tool_call_id: TC-408
task_id: T-301
run_id: R-17
step_no: 4
tool_name: change_shipping_address
status: ready
idempotency_key: address-change-AC-9021
```

随后执行器调用订单服务。订单服务仍然要执行最终状态条件，例如只允许尚未出库的订单修改地址，并通过 `idempotency_key` 识别重复请求。

先记录再调用可以让服务重启后知道曾准备执行什么，却不能消除所有故障窗口。订单服务可能已经修改成功，应用后端却在保存响应前崩溃。此时工具调用的真实结果是 `unknown`，不能简单标记失败并再次修改。

恢复程序应先使用 `change_request_id` 查询订单服务：

- 已经成功：补记结果并继续任务；
- 明确未执行：在仍满足条件时重新提交；
- 仍无法确认：暂停任务并交给恢复流程或人工处理。

**数据库中的工具记录说明应用后端知道什么，拥有业务数据的服务决定动作实际上是否成功。**

## 工具结果要适合模型继续判断

订单服务返回的内部对象可能包含几十个字段、调用链信息和敏感数据。全部塞回模型既浪费 Context，也可能越过数据边界。只返回“成功”又不足以支持下一步。

查询订单的工具结果可以裁剪为：

```json
{
  "tool_call_id": "TC-401",
  "status": "succeeded",
  "result": {
    "order_id": "O-1042",
    "order_status": "paid",
    "shipment_status": "waiting_pickup",
    "address_change_allowed": true
  }
}
```

稳定错误也应成为可解释的工具结果：

```json
{
  "tool_call_id": "TC-408",
  "status": "rejected",
  "error": {
    "code": "ORDER_ALREADY_SHIPPED",
    "message": "订单已经出库，不能修改收货地址"
  }
}
```

模型据此可以向用户解释限制或选择其他合理工具。数据库连接串、异常堆栈和内部服务拓扑只进入日志与 Trace，不应作为工具结果暴露。

并非所有错误都交给模型决定。权限拒绝、任务取消、预算耗尽和结果未知等情况应由运行时暂停或终止；模型不能通过换一种说法绕过控制规则。

## 下一轮必须保留调用与结果的对应关系

后端把模型上一轮的工具请求和本轮工具结果一起加入 Context。`tool_call_id` 用于明确哪个结果对应哪个请求，特别是在一轮产生多个并行查询时。

```text
模型请求 TC-401：get_order
模型请求 TC-402：get_shipment
工具结果 TC-401：订单已支付，可修改地址
工具结果 TC-402：等待快递揽收
```

只有互不依赖的只读调用适合并行。若第二项动作依赖第一项结果，或两项写入可能修改同一订单，运行时应按确定顺序执行，而不是为了缩短延迟盲目并发。

工具结果进入下一轮后，模型可能：

- 请求另一个工具；
- 生成需要用户确认的方案；
- 根据失败结果改换路径；
- 给出最终回答；
- 声称任务已经完成。

最后一种仍需运行时检查。模型说“地址已修改”不等于订单服务已经确认修改，最终回答必须与保存的工具结果一致。

## 循环什么时候停止

Agent 循环不能只依赖模型自行决定结束。订单助手的运行时至少检查：

- 模型是否返回最终回答，而不是尚未执行的工具请求；
- 必需的业务结果是否已经确认；
- 是否正在等待用户补充信息或确认；
- 用户是否发出取消请求；
- 模型调用次数、工具调用次数、运行时间和 Token 是否达到预算；
- 是否连续重复相同调用而没有取得新信息；
- 是否出现权限拒绝或无法确认的副作用。

一个任务可能进入这些结果：

```text
completed
waiting_for_confirmation
waiting_for_input
cancelled
failed
limit_reached
```

产品可以为简单订单查询设置较小的轮次和时间预算，为生成售后报告设置不同上限。限制应来自任务价值、正常分布和依赖容量，并通过生产数据调整。

模型负责在开放问题中提出下一步，运行时负责确认下一步是否有效，以及任务是否具备结束资格。

## 取消不能撤回已经发生的动作

用户在 Agent 查询物流时取消，后端可以停止后续模型调用和未开始的工具。已经完成的地址修改不会因为任务状态变成 `cancelled` 自动回滚。

运行时需要区分：

- 尚未执行的工具：阻止开始；
- 正在执行的只读请求：尽力取消或丢弃结果；
- 结果未知的写入：先确认真实状态；
- 已成功的写入：保留业务结果，必要时启动明确的补偿动作。

因此，任务取消和业务撤销是两项不同操作。前者停止继续工作，后者需要领域服务提供合法的反向流程。

## 运行过程怎样被观察

一次订单助手任务需要把这些标识串联起来：

```text
task_id
run_id
model_call_id
tool_call_id
business_request_id
order_id
user_id / tenant_id
tool_and_config_version
```

Trace 可以按轮次记录模型调用、工具请求、校验结果、业务服务耗时和状态变化。敏感参数应脱敏，访问日志仍要遵循租户与数据保留规则。

生产指标不只包括模型延迟，还包括：

- 每项任务的模型轮次和工具调用数；
- 各工具成功、拒绝、超时与结果未知的比例；
- 等待用户确认的任务数量和时长；
- 重复工具调用与循环达到上限的比例；
- 从工具请求到业务结果确认的耗时；
- 模型声称完成但业务结果未确认的拦截次数。

这些数据能够区分模型选错工具、运行时拒绝、业务服务故障和用户没有确认，而不是把所有失败归为“模型效果不好”。

## 后端怎样托住一次 Agent 循环

模型开始使用工具后，应用后端增加了一条持续执行的控制链：

- 显式注册本轮允许使用的工具；
- 把模型输出当作候选请求，验证参数、权限和业务条件；
- 按查询、可逆写入和高影响动作实施不同控制；
- 在副作用前记录调用，并用幂等键与状态查询确认结果；
- 裁剪并结构化工具结果，让模型能够继续判断；
- 由运行时执行取消、预算和结束条件；
- 用任务、运行与工具标识还原每一轮发生了什么。

**模型选择可能的下一步，应用后端把这一步约束成合法、可执行且能够确认的业务动作。**

一次任务经过多轮模型调用和工具执行后，只保存最后的自然语言回答已经不够。[一次 Agent 任务需要保存哪些状态](../agent-task-state/)继续说明系统怎样记录当前步骤、已经发生的动作，以及服务重启后的恢复边界。
