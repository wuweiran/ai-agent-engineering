# 后端怎样保存业务状态

运动鞋 `SKU-8812` 只剩一件。用户小林和小周几乎同时打开商品页面，两个人都看见“有货”，随后分别提交订单。

如果两个请求都成功，系统就卖出了不存在的库存。每个请求都可能正确读取到库存为 1，问题发生在它们共同修改同一份状态的时候。

后端保存状态的难点，不只在于把订单写进数据库，还要让库存、金额和订单状态在并发、失败与缓存存在时继续可信。

这一步属于“管理状态”：数据库要表达系统相信的事实，并在多个请求同时写入时守住约束。本篇以正确性为主，也会看到缓存与索引怎样在性能和写入成本之间取舍。

## 先确定系统需要相信哪些事实

简化的下单系统至少保存三组数据：

```text
orders
├─ order_id
├─ buyer_id
├─ status
├─ total_amount_minor
├─ shipping_address_id
├─ idempotency_key
├─ version
└─ created_at

order_items
├─ order_id
├─ sku_id
├─ quantity
├─ unit_price_minor
└─ discount_amount_minor

sku_inventory
├─ sku_id
├─ available_stock
└─ version
```

订单主表记录买家、状态与总金额，订单明细保存成交时的 SKU、数量和价格快照，库存表记录当前还能出售多少件。

数据库 Schema 不是下单页面的复制。页面显示商品名称、图片和最新售价，历史订单却要保留成交时的信息。商品以后改名或调价，`O-1042` 仍应显示用户当时以什么价格购买了什么。

数据模型首先要说清两件事：哪些字段表达当前业务事实，哪些字段要保留发生交易时的历史快照。

## 约束要尽量靠近数据

应用代码可以检查购买数量大于零、订单金额不能为负、同一次提交不能创建两份订单。仅靠某个接口的代码还不够，旧版本服务、后台脚本或另一个入口都可能写入同一数据库。

数据库可以执行一部分不会随场景改变的底线：

- 主键保证 `order_id` 和 `sku_id` 唯一；
- 非空约束保证买家、状态和成交价格存在；
- 检查约束保证金额与库存不会小于零；
- 唯一约束保证同一用户的 `idempotency_key` 只对应一个订单；
- 同库关系中的外键避免订单明细引用不存在的订单。

“这张优惠券能否用于当前商品”仍由业务层判断，“库存不能成为负数”则适合由数据库一起守住。

**能够明确表达且必须始终成立的规则，应尽量由最终保存数据的系统执行。**

## “先查询再扣减”为什么会超卖

一种直观实现是：

```text
读取 SKU-8812 的库存
如果 available_stock >= 1
    把库存减 1
    创建订单
```

两个请求可能按下面的顺序发生：

1. 小林读取库存，结果为 1；
2. 小周读取库存，结果仍为 1；
3. 小林把库存改成 0；
4. 小周也根据旧结果把库存改成 0；
5. 两个人各自创建一张订单。

最终库存看起来没有变成负数，却出现了两张都声称买到商品的订单。单条 SQL 没有语法错误，真正的问题是“检查库存”和“扣减库存”分成了两个可能被其他请求插入的步骤。

修复的关键，是让库存条件与扣减在数据库执行写入的同一刻成立：

```sql
UPDATE sku_inventory
SET available_stock = available_stock - 1,
    version = version + 1
WHERE sku_id = 'SKU-8812'
  AND available_stock >= 1;
```

数据库会返回受影响行数：

- 更新一行，当前请求取得这一件库存；
- 更新零行，说明 SKU 不存在或库存已经被另一个请求拿走。

两个请求即使同时执行，也只有一个能在 `available_stock >= 1` 时完成扣减。失败的一方重新读取最新状态，并向客户端返回 `409 Conflict` 和 `OUT_OF_STOCK`。

页面上曾经显示“有货”不能保证下单成功。它只是用户浏览时的状态；真正的库存判断必须发生在提交交易时。

## 订单和库存要共同成功

成功扣减库存后，系统还要创建订单主记录与订单明细。如果库存已经减 1，写订单时数据库故障，商品就会无故少一件；如果先创建订单、扣减库存失败，又会留下无法履约的订单。

当订单和库存由同一个数据库管理时，可以把它们放在一个事务中：

1. 条件扣减库存，并检查受影响行数；
2. 写入订单主记录；
3. 写入订单明细与成交价格；
4. 全部成功后提交。

任何一步失败，事务回滚，库存和订单都回到执行前的状态。这里的事务边界来自一项业务承诺：系统只有在保存订单并取得库存后，才能告诉用户“下单成功”。

事务不能自动覆盖数据库之外的支付、短信和物流服务。下单以后调用支付平台，已经进入跨服务状态。它需要事件、任务、状态查询和补偿机制，不能通过延长本地数据库事务解决。

## 版本号避免其他修改被悄悄覆盖

库存扣减有明确的数值条件，可以直接用 `available_stock >= quantity`。其他并发修改可能需要版本号。

例如，待支付订单允许修改收货地址。用户在手机和电脑上同时打开 `O-1042`，两个页面都读取到 `version = 7`。手机先提交新地址：

```sql
UPDATE orders
SET shipping_address_id = 'ADDR-73',
    version = version + 1
WHERE order_id = 'O-1042'
  AND buyer_id = 'U-204'
  AND status = 'pending_payment'
  AND version = 7;
```

更新成功后版本变成 8。电脑再用版本 7 提交旧页面中的地址时，更新零行。服务返回冲突，让客户端读取最新订单，而不是覆盖手机刚完成的修改。

这称为乐观并发控制：读取时取得版本，写入时确认状态没有变化。它适合冲突不频繁、失败后可以刷新重试的操作。若大量请求持续争用同一资源，数据库锁、队列串行处理或重新划分数据所有权可能更合适。

## 状态变化也要遵守允许的路径

订单不会只有一个可任意修改的 `status` 字符串。一个简化生命周期可能包括：

```text
pending_payment
paid
cancelled
shipped
completed
```

每项操作只允许从特定状态出发。支付确认可以把 `pending_payment` 改为 `paid`，用户取消也只能处理仍待支付的订单，发货必须从已支付状态开始。

更新时应把前置状态写进条件：

```sql
UPDATE orders
SET status = 'cancelled',
    version = version + 1
WHERE order_id = 'O-1042'
  AND buyer_id = 'U-204'
  AND status = 'pending_payment';
```

如果支付回调已经把订单改成 `paid`，取消请求会影响零行。业务层读取当前状态后，决定拒绝取消还是进入退款流程。

这比先读取状态、在代码中判断、再无条件写入更可靠。订单状态的合法路径也应在业务代码中集中表达，不能让每个接口自由赋值。

## 什么是权威状态

下单成功后，订单数据库中的记录和库存数据库中的可售数量是当前权威状态。下面这些信息可能复制它们，却不能单独决定交易结果：

- 用户几秒前打开的商品页面；
- Redis 中的一份商品详情缓存；
- “创建订单成功”的应用日志；
- 消息队列中的 `order.created` 事件；
- 搜索索引里的商品文档。

它们可能延迟、重复或丢失。出现争议时，系统要知道向哪里确认订单是否存在、库存是否扣减、支付是否成功。

大型系统中，每项事实可能有不同负责人：订单服务掌握订单状态，库存服务掌握可售库存，支付服务掌握真实支付结果。其他服务通过订单 ID、支付 ID 和事件引用这些事实，不能各自维护一套互相冲突的最终答案。

## 缓存为什么会继续显示“有货”

商品详情读取量很大，团队可能缓存可售状态：

```text
key: sku:SKU-8812
value: { price_minor: 69900, in_stock: true, version: 36 }
```

小林下单后，数据库库存已经变成 0。缓存若没有及时失效，小周仍会看到“有货”。这不应导致第二次下单成功，因为提交订单时仍会执行数据库中的条件扣减。

更新库存后，服务可以删除缓存，让下一次读取重新加载；也可以更新缓存，但要处理数据库提交成功、缓存更新失败的情况。常见设计允许商品页在短时间内显示旧信息，同时把最终交易条件留在权威存储中再次检查。

不同数据允许的陈旧程度不同。商品介绍晚几秒更新通常可以接受，已支付金额和资金结果则需要更严格的读取路径。

加入缓存前，先要明确：允许旧多久，谁负责失效，缓存不可用时系统怎样退回权威存储。边界清楚后，缓存带来的性能收益才有意义。

## 索引让常用订单查询更快

用户经常按状态和时间查看自己的订单：

```sql
SELECT order_id, status, total_amount_minor, created_at
FROM orders
WHERE buyer_id = 'U-204'
  AND status = 'pending_payment'
ORDER BY created_at DESC
LIMIT 20;
```

合适的联合索引可以避免扫描整张订单表。库存条件更新则需要快速定位 `sku_id`，通常由主键或唯一索引支持。

索引并非越多越好。每次创建订单、修改状态或更新库存时，相关索引也要维护，占用存储并增加写入成本。工程师应根据真实查询和查询计划设计索引，为每个字段都创建索引只会把成本转移到写入路径。

数据模型决定保存什么，约束决定什么不能被破坏，索引决定常见请求能否高效找到这些数据。

## 订单结构也会随业务变化

系统后来准备为待支付订单增加库存保留期限 `reservation_expires_at`。如果新字段一上线就要求非空，尚未升级的旧服务不会写入它，下单请求会立即失败。

更稳妥的迁移分阶段进行：先增加可空字段或合理默认值，让新旧代码同时工作；新版本开始写入保留期限，并回填仍有效的旧订单；所有服务适配后，再决定是否收紧非空约束。

Schema 也是服务契约。数据库迁移必须考虑发布期间新旧实例并存，不能假设所有服务会在同一瞬间完成升级。

## 状态正确依赖哪些边界

电商下单要避免超卖和订单覆盖，需要这些边界配合：

- 数据模型区分当前状态和成交快照；
- 数据库约束守住库存、金额和幂等键等底线；
- 读取后可能变化的条件，在写入时再次确认；
- 订单与本地库存等必须共同成立的修改放进事务；
- 版本号和状态前置条件阻止并发请求互相覆盖；
- 缓存与索引服务读取，权威存储决定交易结果；
- Schema 变化兼容发布期间的新旧服务。

数据库会参与执行约束、并发控制与事务规则，使多个请求最终得到一致的交易结果。

接下来，待支付订单会过期，库存需要释放，短信和推荐更新也要离开下单请求继续运行。状态跨越队列和多个服务后，还要处理消息重复、任务恢复和本地事务之外的业务结果。
