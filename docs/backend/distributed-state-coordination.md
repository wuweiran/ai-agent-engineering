# 分布式系统中的状态协调

订单服务、库存服务和支付服务分别保存自己的业务事实。每个服务为了高可用又可能保存多个数据副本，多台实例还可能同时争抢同一项后台任务。

单数据库事务能够让一组本地写入共同提交，跨过节点和服务边界后，却不存在一个天然的共同提交点。分布式系统需要协调三类状态问题：同一份数据在多个副本上怎样保持可接受的一致，多项业务事实怎样跨服务完成，以及多个执行者怎样避免同时修改同一资源。

```text
同一份状态有多个副本
→ 一致性与共识

一项业务分布在多个服务
→ 分布式事务与补偿

多个实例争抢执行权
→ 锁、租约与版本校验
```

## 先说明系统承诺哪种一致性

“一致”不是只有一个等级。余额扣减和唯一库存资格通常要求读写立即服从同一顺序；商品介绍、搜索索引和统计报表可以允许短暂陈旧，只要最终收敛并能说明最长延迟。

[强一致与最终一致]({{ site.baseurl }}/docs/interview/backend/distributed-system/#strong-vs-eventual-consistency)之间还存在更具体的业务承诺，例如用户提交修改后自己必须立即读到、同一会话内读取不能倒退、订单状态不能从已支付退回待支付。

设计前应先回答：

```text
旧值会造成什么业务后果？
不一致最长可以持续多久？
谁有权确认最终状态？
冲突发生后按什么规则合并或拒绝？
```

只有承诺明确后，才能决定同步副本、异步复制、版本检查或补偿修复。

## CAP 讨论网络分区时的选择

数据复制到多个节点后，网络可能让节点之间暂时无法通信。此时一个节点无法同时确认其他副本的最新状态，又要决定是否继续处理请求。

[CAP]({{ site.baseurl }}/docs/interview/backend/distributed-system/#cap-theorem)表达的是：发生网络分区时，系统无法同时保证线性一致的结果和每个请求都成功响应。

- 继续接受各分区写入，可以提高可用性，但副本可能产生冲突或旧值；
- 拒绝无法获得足够确认的写入，可以守住一致顺序，但部分请求会失败或等待。

网络分区是客观故障，不能靠“选择不支持分区”消除。CAP 也不是把系统永久贴成 CP 或 AP：没有分区时，一致和可用可以同时满足；不同操作还可以采用不同策略。

CAP 中的一致性指副本对读写顺序的保证，不是数据库 ACID 中“数据满足业务约束”的一致性。

## BASE 描述允许中间状态的工程取舍

[BASE]({{ site.baseurl }}/docs/interview/backend/distributed-system/#base-theory)通常概括为基本可用、软状态和最终一致。它允许系统在故障或高负载下暂时降级，并通过异步复制、消息、重试和修复让状态最终收敛。

例如商品价格更新后，搜索索引晚几秒生效可以接受；支付成功后订单长时间仍显示未支付则不可接受。最终一致必须同时定义：

- 最终要收敛到什么状态；
- 正常情况下多久收敛；
- 更新丢失或持续失败怎样被发现；
- 自动修复、补偿和人工处理怎样进行。

BASE 不是放弃正确性，而是把正确性从“每个瞬间完全相同”改为“中间状态受控，最终结果可确认”。

## 多数派让副本认可同一顺序

一个服务的状态复制到多个节点后，写入要决定何时算成功。常见做法是由 Leader 接收写入，复制到其他节点，并在获得多数派确认后提交。

```text
客户端写入
→ Leader 追加日志
→ 复制到 Follower
→ 多数节点确认
→ 提交并应用状态
```

多数派 Quorum 的价值在于，任意两个多数集合必然有交集，新 Leader 可以从交集节点中找到已经提交的信息。它不能让网络和磁盘变快，副本越多、同步要求越高，写入等待通常越长。

[Raft、Paxos 等共识算法]解决的是多个节点在故障和网络变化下，怎样对同一条日志顺序或同一个决定达成一致。Raft 把问题拆成 Leader 选举、日志复制和安全性规则，工程实现更容易解释；Paxos 是更基础的共识思想，实际系统常使用 Multi-Paxos 或其他变体。

应用开发通常使用数据库、Kafka、etcd 或 ZooKeeper 已经提供的复制与协调能力，不自己实现共识算法。还要区分：共识能够让副本认可同一顺序，却不会自动判断“订单是否应该退款”这类业务规则。

## 跨服务业务没有一个普通本地事务

创建订单可能需要订单服务保存订单、库存服务预占库存、支付服务创建支付。每个服务都能在自己的数据库中使用本地事务，但普通 Spring `@Transactional` 无法让这些独立服务共同提交或回滚。

```text
订单创建成功
→ 库存预占成功
→ 支付调用超时
```

此时系统需要知道每个服务的权威状态、当前流程走到哪一步，以及支付结果是失败还是未知。所谓[分布式事务]({{ site.baseurl }}/docs/interview/backend/distributed-system/#distributed-transaction-purpose)，就是协调这些独立提交点，或者让失败后的状态按业务规则收敛。

## 不同事务方案服务不同约束

### XA 与两阶段提交

[两阶段提交]({{ site.baseurl }}/docs/interview/backend/distributed-system/#two-phase-commit)先让参与资源 Prepare 并保留提交条件，再由协调者通知全部 Commit 或 Rollback。XA 是事务管理器与多个资源管理器之间的标准接口，常通过两阶段提交协调多个支持 XA 的资源。

它适合参与者有限、事务较短且需要较强共同提交语义的场景。Prepare 会持有锁和连接，协调者或网络故障还可能让参与者等待恢复，因此不适合长时间业务流程。

### TCC 的资源预留

[TCC]({{ site.baseurl }}/docs/interview/backend/distributed-system/#tcc)把业务操作显式拆成：

```text
Try：检查并预留资源
Confirm：正式生效
Cancel：释放预留资源
```

例如库存 Try 预占数量，Confirm 正式扣减，Cancel 释放预占。它提供较强的业务隔离，但每个参与服务都要实现三套语义，并处理 Confirm、Cancel 的幂等、空回滚和悬挂。

### Saga 的本地提交与补偿

[Saga]({{ site.baseurl }}/docs/interview/backend/distributed-system/#saga)把长流程拆成多个本地事务，每一步完成后提交，后续失败时执行补偿。

它适合履约、预订、审批等持续时间较长的流程，不需要一直占用数据库锁。代价是中间状态会被外界看到，补偿也不一定能把现实恢复成从未发生。例如邮件已经发送，只能再发送更正，无法撤销收件人已经看到的内容。

### 本地消息表连接业务提交与消息发送

业务数据库提交后还需要可靠发布事件时，可以在同一个本地事务中写入业务数据和[本地消息表]({{ site.baseurl }}/docs/backend/message-delivery/)，再由发布程序发送消息。

它不协调所有远程服务共同回滚，而是保证“这条业务事实已经成立，因此对应消息最终不能忘记发送”。发布可能重复，消费者仍要幂等。

## 状态机是跨服务流程的恢复依据

很多系统不会使用一个覆盖全流程的分布式事务框架，而是保存持久业务状态机，让每一步在自己的本地事务中提交，再通过消息或任务推进。

```text
order_created
→ inventory_reserved
→ payment_confirming
→ paid / payment_failed / manual_review
```

状态机应保存稳定业务 ID、步骤结果、尝试次数和结果未知状态。恢复时查询权威服务，决定继续、重试、补偿还是人工接管，而不是从头重放整个调用链。

这种方式的可靠性来自本地事务、消息、幂等、状态查询、补偿和对账的组合。它允许中间状态存在，也要求产品和运维能够看见并处理这些状态。

## 分布式锁协调临时执行权

多个实例可能同时扫描到同一批任务，或者同时刷新同一个缓存 Key。[分布式锁]({{ site.baseurl }}/docs/interview/backend/distributed-system/#distributed-lock-properties)用于给某个执行者一段有期限的操作权。

一个基本锁需要：

- 获取时原子写入唯一持有者标识；
- 设置有效期，使持有者失联后锁能释放；
- 释放时验证持有者，避免删除别人后来取得的锁；
- 需要等待时定义通知、公平和超时语义。

锁通常表现为租约，因为系统无法永久确认持有者仍然存活。自动续期只能降低锁提前过期的概率，不能消除长时间停顿和网络分区。

## 锁之外还需要迟到写入保护

旧持有者可能因为 GC 停顿或网络卡顿失去锁，恢复后仍继续写入；此时新持有者也已经开始工作。单靠锁无法阻止旧请求到达下游。

[Fencing Token]({{ site.baseurl }}/docs/interview/backend/distributed-system/#fencing-token)为每次获得执行权生成单调递增编号，下游只接受比已处理编号更大的写入：

```text
旧持有者 token=41，恢复后迟到
新持有者 token=42，已经写入
下游拒绝 token=41
```

关键业务写入还应使用状态条件、版本号和幂等键。锁减少同时执行，写入约束守住最终结果；两者不能互相替代。

## 状态协调的判断顺序

面对分布式一致性问题，不应先选择组件或算法。先判断状态属于哪一类：

```text
同一份数据的多个副本
→ 明确一致性承诺，再选择复制与共识方式

一项业务的多个服务状态
→ 明确权威状态和中间状态，再选择 XA、TCC、Saga、本地消息表或状态机

多个执行者争抢同一项工作
→ 使用锁或租约减少重叠，并用版本与幂等拒绝迟到写入
```

所有方案最终都要回答：状态由谁确认，故障后怎样发现，重复执行是否安全，以及长期无法自动收敛时由谁处理。

**分布式协调不是让所有节点在每个瞬间完全一致，而是让系统在网络和节点都可能失败时，仍然知道哪些状态可信、哪些状态需要等待，以及怎样恢复到可以确认的结果。**
