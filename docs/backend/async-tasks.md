# 一个请求做不完时怎么办

订单 `O-1042` 已经创建，状态为 `pending_payment`。用户有 30 分钟完成支付；到期仍未支付，系统要取消订单并释放为它保留的库存。

下单接口不能保持连接等待 30 分钟。用户可能关闭应用，服务实例可能重启，订单仍然必须在正确时间继续处理。这里出现了同步请求之外的另一种对象：**业务任务**。

这一步开始进入“协调工作”：请求、任务和 Worker 分别承担一段生命周期。工程师主要检查任务能否在实例重启后恢复，重复执行时是否仍然安全，以及积压增加时系统能否承受。

## HTTP 请求为什么不能一直等待

HTTP 请求适合完成一段较短、有明确响应的工作。创建订单需要在返回前确认订单和库存已经保存，因此属于当前请求。等待支付则取决于用户未来是否操作，持续时间也远大于普通接口。

让请求等待会产生几个问题：

- 客户端连接可能因网络或超时断开；
- 应用实例需要长期占用连接和内存；
- 服务发布或重启会中断等待；
- 用户无法关闭页面后再回来查看状态；
- 大量待支付订单会消耗并发处理能力。

因此，下单接口只返回当前阶段的结果：订单已经创建，正在等待支付。整笔交易还会在后续请求和任务中继续。

```http
HTTP/1.1 201 Created
Content-Type: application/json

{
  "order_id": "O-1042",
  "status": "pending_payment",
  "expires_at": "2026-07-14T10:50:00Z"
}
```

支付、取消和发货会在后续请求或后台任务中继续推动订单状态。

## 进程里的定时器为什么不可靠

一个简单实现是在创建订单后启动定时器：

```text
等待 30 分钟
如果订单仍未支付
    取消订单并释放库存
```

这在单机演示中可以工作，生产环境却无法回答几个问题：

- 定时器所在的服务进程重启后，谁记得它；
- 服务扩容到十台实例后，哪一台负责执行；
- 两台实例是否会同时处理同一订单；
- 订单已经支付，旧定时器是否仍会取消；
- 程序发布时，内存中的等待任务怎样迁移。

进程内存适合保存临时计算，不适合承载持续时间超过进程生命周期的业务承诺。

订单的过期时间和当前状态需要先写入持久存储：

```text
order_id: O-1042
status: pending_payment
reservation_expires_at: 2026-07-14T10:50:00Z
```

只要这项事实仍在数据库中，任何获得执行权的 Worker 都可以在服务恢复后继续处理。

## API、队列和 Worker 各自做什么

一种常见结构包含三个部分：

| 部分 | 责任 |
| --- | --- |
| API 服务 | 验证下单请求，创建订单，尽快返回当前结果 |
| 消息队列 | 保存等待处理的工作，把消息交给可用消费者 |
| Worker | 执行业务任务，更新状态，处理失败与重试 |

订单创建后，系统可以安排一条延迟消息：

```json
{
  "event_id": "EV-7001",
  "type": "order.expiration_due",
  "order_id": "O-1042",
  "not_before": "2026-07-14T10:50:00Z"
}
```

队列不需要理解订单。它只负责保存消息，并在到达处理时间后交给 Worker。Worker 收到 `order_id`，再从订单数据库读取权威状态，决定能否取消。

有些消息系统直接支持延迟投递；另一种实现是由调度器定期查询已经到期的订单，再把任务放进普通队列。具体组件可以变化，责任关系相同：过期时间持久化，任务由独立执行者在请求结束后继续。

## 消息只是提醒，状态仍要重新读取

30 分钟内，订单可能发生变化：

- 用户已经支付；
- 用户主动取消；
- 客服延长了支付时间；
- 风控冻结了订单；
- 支付回调正在处理中。

因此，过期消息不能等同于“现在取消订单”的命令。它只提醒 Worker：这张订单可能已经到期，请检查当前状态。

Worker 可以执行条件更新：

```sql
UPDATE orders
SET status = 'cancelled',
    cancel_reason = 'payment_timeout',
    version = version + 1
WHERE order_id = 'O-1042'
  AND status = 'pending_payment'
  AND reservation_expires_at <= CURRENT_TIMESTAMP;
```

更新一行，说明订单仍待支付且已经到期，可以继续释放库存。更新零行则说明条件不成立；Worker 读取当前订单，记录它已经支付、已经取消或过期时间发生变化，然后安全结束这次任务。

**任务消息决定什么时候检查，权威状态决定现在能不能执行。**

这条边界可以防止旧消息覆盖新状态。即使过期消息晚到几分钟，也不会取消已经支付的订单。

## Worker 怎样取得一项任务

队列把消息交给某个 Worker 时，通常会等待确认。Worker 完成处理后确认消息，队列才认为这次投递结束。

如果 Worker 在处理过程中崩溃，没有确认，队列可以把消息交给另一台 Worker。这样，任务不依赖某个服务实例的生命期。

同一时刻还要避免多个 Worker 无限制处理同一任务。消息队列的可见性超时、消费者分组或数据库中的任务租约，可以让一名 Worker 暂时取得执行权。执行者失联后，其他实例才能接管。

这里要区分两件事：

- 谁暂时负责处理消息，由队列或租约管理；
- 订单是否可以取消，由数据库中的状态和条件更新判断。

即使队列偶尔把同一消息交给两个 Worker，数据库条件仍应阻止业务结果重复发生。

## 任务状态怎样被用户看到

订单过期任务在后台运行，用户仍然只关心订单现在是什么状态。订单页面可以查询：

| 状态 | 用户看到的含义 |
| --- | --- |
| `pending_payment` | 订单已创建，等待支付，并显示剩余时间 |
| `paid` | 支付成功，等待履约 |
| `cancelled` | 订单已取消，并说明是用户取消还是支付超时 |
| `payment_confirming` | 已收到支付线索，正在确认最终结果 |

“Worker 正在执行第几步”通常是内部任务状态，不必全部暴露给消费者。但当后台处理时间较长时，产品仍要明确告诉用户当前结果和等待原因，避免用户重复提交。

后台任务也可以有独立记录：

```text
job_id: J-8801
job_type: expire_order
resource_id: O-1042
status: waiting
attempts: 0
not_before: 2026-07-14T10:50:00Z
```

订单状态表达业务事实，任务状态表达系统怎样继续工作。二者不能混成一个字段：任务执行失败，不一定意味着订单失败；订单已经支付后，过期任务可以正常结束为“无需处理”。

## 失败以后能否重试

Worker 调用数据库时可能遇到连接中断，释放库存时也可能遇到依赖超时。是否重试要看失败的性质。

- 数据库临时不可用，等待后重试可能成功；
- 消息格式错误，重复执行不会自动修复；
- 订单已经支付，不需要重试取消；
- 释放库存的结果未知，要先查询或通过幂等设计确认，不能直接增加一次库存。

重试还需要退避。依赖刚恢复时，成千上万个到期任务同时立即重试，会再次把它压垮。Worker 可以逐步增加等待时间，并让恢复时间略微错开。

达到最大重试次数后，消息不能悄悄消失。系统可以把它放入待人工排查的队列，保存错误、尝试次数和订单 ID，并触发告警。

## 取消后台任务意味着什么

用户主动支付后，没有必要一定从队列中找到并删除那条过期消息。延迟队列中的消息可能难以精确移除，也可能已经交给 Worker。

更可靠的方式是让消息到达后重新检查订单。状态为 `paid` 时，它自然变成无操作。这种设计把正确性放在业务状态上，不依赖所有旧消息都能被及时撤回。

真正昂贵的后台任务可能值得支持显式取消。此时，取消通常只阻止尚未开始的步骤；已经发送的短信、已经提交的支付请求或已经写入的外部数据不会因任务状态变为 `cancelled` 自动消失，除非外部系统提供对应补偿能力。

## 异步带来的核心变化

同步请求中，调用栈还在，异常可以一路返回客户端。工作进入队列后，请求早已结束，执行时间、服务实例和重试次数都可能变化。

系统因此要显式保存：

- 要处理哪个业务对象；
- 什么时候可以开始；
- 当前由谁处理；
- 已经尝试多少次；
- 失败后是否可以重试；
- 最终业务状态在哪里确认。

**HTTP 请求负责接收目标并返回当前结果，持久任务负责在请求结束后继续兑现系统承诺。**
