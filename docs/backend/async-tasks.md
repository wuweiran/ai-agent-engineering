# 异步任务

订单 `O-1042` 已经创建，用户有三十分钟完成支付。到期仍未支付，系统要取消订单并释放库存。

下单接口不能保持连接等待三十分钟。用户会关闭应用，服务会发布重启，但订单过期这项承诺仍然存在。工作一旦长于请求和进程的生命周期，就需要成为可以持久保存、由其他执行者接管的任务。

## 请求只能返回当前已经确认的结果

HTTP 请求适合完成一段较短、具有明确响应的工作。创建订单需要在返回前确认订单和库存已经保存；等待支付则取决于用户未来的操作，持续时间也无法由当前请求控制（[什么工作适合异步接口？]({{ site.baseurl }}/docs/interview/backend/api-application/#async-api-boundary)）。

下单接口因此只返回当前阶段：

```json
{
  "order_id": "O-1042",
  "status": "pending_payment",
  "expires_at": "2026-07-14T10:50:00Z"
}
```

订单已经创建是当前事实，支付和过期处理会在后续请求与后台任务中继续推动状态。

## 进程内定时器承载不了业务承诺

最简单的实现是在创建订单后启动定时器：

```text
等待三十分钟
如果仍未支付
    取消订单并释放库存
```

演示程序可以这样工作，生产环境却无法回答：定时器所在进程重启后谁来继续，多台实例中哪一台负责，旧定时器是否会取消已经支付的订单。

进程内存只能保存临时执行现场。持续时间超过进程生命周期的事实要先写入持久存储：

```text
order_id: O-1042
status: pending_payment
reservation_expires_at: 2026-07-14T10:50:00Z
```

只要期限和订单状态仍在数据库中，任何后来获得执行权的 Worker 都能重新判断这项工作。

## API、队列和 Worker 分担不同生命周期

订单创建后，API 可以安排一项到期检查：

```json
{
  "task_id": "J-8801",
  "type": "check_order_expiration",
  "order_id": "O-1042",
  "not_before": "2026-07-14T10:50:00Z"
}
```

API 负责验证请求、提交订单与任务，然后尽快返回。队列保存等待处理的工作，在合适时间交给可用消费者。Worker 取得 `order_id` 后读取数据库，执行真正的业务判断。

有些队列支持[延迟消息]({{ site.baseurl }}/docs/interview/backend/message-queue/#delayed-message-scheduled-task)；另一种方式是调度器定期查询已经到期的任务，再放入普通队列。组件可以变化，关键边界不变：**执行意图和时间被持久化，工作不依赖原请求或某台实例继续存活。**

## 消息提醒 Worker 检查，不替业务状态作决定

三十分钟内，订单可能已经支付、被用户取消，或者由客服延长了期限。到期消息产生于过去，不能直接证明现在仍应该取消。

Worker 应在权威数据库上执行条件更新：

```sql
UPDATE orders
SET status = 'cancelled',
    cancel_reason = 'payment_timeout',
    version = version + 1
WHERE order_id = 'O-1042'
  AND status = 'pending_payment'
  AND reservation_expires_at <= CURRENT_TIMESTAMP;
```

更新一行，说明订单现在仍待支付且已经过期；更新零行，Worker 读取最新状态后安全结束或等待新的期限。

> **任务决定什么时候再次检查，权威状态决定现在能不能执行。**

这条边界使旧消息无法覆盖新事实。

## Worker 崩溃后，任务要能够被接管

队列把任务交给 Worker 后通常等待确认。处理完成才确认；Worker 中途崩溃，没有确认的任务会在可见性超时后再次投递（[消费者确认前崩溃会怎样？]({{ site.baseurl }}/docs/interview/backend/message-queue/#consumer-crash-before-ack)）。

任务记录还可以使用租约：

```text
leased_by: worker-12
lease_until: 2026-07-14T10:51:30Z
```

租约表示某个 Worker 暂时拥有推进任务的权利。执行时间较长时，它持续发送心跳延长租约；失联后，其他 Worker 才能接管。

租约减少同时执行，却不能成为正确性的唯一防线。崩溃、网络分区和租约到期边缘仍可能让两个 Worker 短暂重叠，最终业务写入必须继续通过状态条件和幂等键阻止重复结果。

## 任务状态和订单状态回答不同问题

后台任务可以记录：

```text
task_id: J-8801
status: waiting
attempts: 0
not_before: 2026-07-14T10:50:00Z
```

订单的 `pending_payment` 表示交易处于什么阶段；任务的 `waiting` 表示系统准备怎样继续工作。[长任务状态接口]({{ site.baseurl }}/docs/interview/backend/api-application/#long-running-operation-status)应把这些业务阶段表达给客户端。

订单已经支付后，过期任务仍可能被队列投递。它最终可以成功结束为“无需处理”，而不是把订单标记失败。反过来，Worker 临时故障也不表示订单本身失败。

把两类状态混在一个字段里，会让基础设施故障篡改业务含义。

## 重试前先判断上一次可能做了什么

数据库临时不可用，稍后重试可能成功；消息格式错误，等待不会让它自行修复。更重要的是，Worker 可能已经完成业务写入，只在确认任务前崩溃。

恢复时要看业务结果：条件更新尚未发生，可以使用同一任务身份重试；订单已经取消，重复任务直接结束；外部库存释放结果未知，则先按稳定请求 ID 查询，不能直接再增加一次库存。

重试还要退避并受整体期限约束。依赖刚恢复时，所有任务同时醒来会再次压垮它。持续失败达到上限后，应保存错误并进入人工处置，不能让任务静默消失。

## 取消任务不等于撤销现实动作

用户支付后，不一定需要从延迟队列精确删除旧任务。消息到达时重新检查订单，它会自然成为无操作。

昂贵任务可以支持[显式取消]({{ site.baseurl }}/docs/interview/backend/api-application/#async-task-cancellation)。取消能阻止尚未开始的步骤，也可以尽力中止正在等待的网络调用；已经发送的短信、已经提交的支付和已经写入的数据不会自动消失。撤销这些结果需要业务系统提供合法的补偿流程。

## 异步执行的系统变化

工作离开 HTTP 调用栈以后，执行时间、服务实例和尝试次数都会变化。系统必须显式保存要处理哪个业务对象、什么时候开始、当前是否有人执行，以及最终结果向哪里确认。

```text
请求接收目标并返回当前结果
        ↓
持久任务保存尚未完成的承诺
        ↓
队列把任务交给可用 Worker
        ↓
Worker 重新读取权威状态再行动
```

**异步任务的可靠性不来自某个 Worker 永不失败，而来自任务可以被重新发现、业务条件可以重新判断、重复执行不会改变正确结果。**
