# 缓存怎样加速读取而不破坏正确性

某款运动鞋进入促销页面后，商品详情的读取量迅速上升。每次请求都查询商品、价格和库存数据库，连接池很快接近上限。团队把商品详情放进 Redis，页面延迟明显下降。

与此同时，`SKU-8812` 的库存已经归零，部分用户仍然在页面上看到“有货”。缓存提高了读取能力，也让系统同时存在数据库状态和缓存副本。

这篇文章主要属于“保障运行”：缓存用存储空间与短暂旧数据换取延迟和容量，但权威交易结果仍要由能够实施约束的系统决定。

## 缓存解决什么问题

商品详情具有几个特点：

- 读取次数远多于修改次数；
- 许多用户查询相同 SKU；
- 组装详情可能需要读取多个表或服务；
- 页面允许少量信息短暂过期；
- 读取失败会直接影响浏览体验。

服务可以把组装后的结果保存到缓存：

```text
key: product-detail:SKU-8812
value:
  title: 某款运动鞋
  display_price_minor: 69900
  in_stock: true
  product_version: 42
expires_in: 60s
```

下一次请求先查缓存。命中时直接返回，减少数据库查询与组装计算；未命中时再从权威数据源读取，将结果写回缓存。

这种模式称为 Cache Aside。应用知道缓存和数据库的关系，并负责在未命中时回源、在数据变化时失效。

缓存最适合保存**能够从权威数据重新构造的读取副本**。订单支付结果、账户余额等不能凭缓存独立修改，丢失缓存也不能意味着业务事实消失。

## 页面显示“有货”为什么仍可能下单失败

缓存中的 `in_stock: true` 是某个时刻的展示信息。从缓存写入到用户点击下单之间，其他请求可能已经买走库存。

下单服务仍要在权威库存中执行条件扣减：

```sql
UPDATE sku_inventory
SET available_stock = available_stock - 1
WHERE sku_id = 'SKU-8812'
  AND available_stock >= 1;
```

更新零行，当前订单返回库存不足。页面显示与交易结果出现短暂差异，用户体验并不完美，但系统没有超卖。

如果下单接口直接相信缓存中的 `in_stock` 并减少缓存数值，多个缓存节点、过期副本和并发请求会让库存失去统一约束。缓存可以帮助快速展示和过滤明显无货请求，最终写入仍要回到权威库存。

价格也需要区分。页面显示价格可以来自缓存，创建订单时应重新取得当前可成交价格，并将它保存为订单快照。价格已变化时，产品可以提示用户确认新金额，不能悄悄按过期缓存成交。

## 数据改变后怎样更新缓存

商品标题从“经典款”改为“夏季轻量款”后，缓存中的旧值要被处理。常见做法是先提交数据库，再删除缓存：

```text
更新数据库成功
删除 product-detail:SKU-8812
下一次读取未命中
从数据库加载新值并写回缓存
```

选择删除而非直接更新，可以避免每个写入方都重新实现复杂的详情组装。下一次读取负责生成完整新副本。

顺序也有原因。如果先删除缓存，再更新数据库，两个动作之间恰好有读取请求，它会从数据库读到旧值并重新写入缓存；随后数据库更新成功，旧缓存却继续存在。

先更新数据库再删除缓存仍有故障窗口：数据库已经提交，服务在删除缓存前崩溃，旧值会保留到 TTL 到期。系统可以通过较短 TTL 限制持续时间，并使用事件或重试任务补偿删除失败。

没有一种简单双写能自动让数据库和 Redis 原子变化。设计时要明确业务可以容忍多长时间的旧数据，以及失效失败怎样被发现和修复。

## TTL 不是完整的一致性方案

给缓存设置 60 秒过期，可以保证旧值不会永久存在，也能回收不再访问的 Key。它无法保证数据修改后立即更新。

TTL 的取值包含业务与容量权衡：

- 时间短，数据更新更快，但回源频繁；
- 时间长，数据库压力更低，旧数据持续更久；
- 大量 Key 同时过期，还可能形成流量尖峰。

可以给过期时间增加随机偏移，让缓存分散失效。核心商品还可以由更新事件主动删除缓存，TTL 作为最终兜底。

对几乎不能容忍旧值的读取，可能需要直接访问权威服务，或使用带版本校验的缓存。不能只因为“缓存快”就统一延长 TTL。

## 热门商品过期时会发生什么

促销开始后，`SKU-8812` 每秒有数千次查询。缓存 Key 到期的一瞬间，大量请求同时未命中并访问数据库，这种现象常被称为缓存击穿。

缓存已经保护数据库很久，却在最热门的数据失效时突然撤掉保护。数据库连接池可能瞬间耗尽。

服务可以只允许一个请求负责回填：

```text
第一个未命中请求取得回填权
    从数据库加载
    写入缓存
其他请求短暂等待或使用旧值
```

实现可以使用进程内合并、分布式锁或专用加载器。无论采用哪种机制，都要设置等待上限和锁过期，避免负责回填的实例崩溃后所有请求永久等待。

另一种做法是逻辑过期：缓存保留旧值和数据时间。到期后，一名 Worker 异步刷新，普通请求暂时继续读旧值。这适合允许短暂陈旧、但不能让热点读取阻塞的商品信息。

选择等待新值还是返回旧值，要看数据类型。商品描述通常可以短暂使用旧值，用户权限和支付状态则不适合这样处理。

## 不存在的数据也会压垮数据库

攻击者或错误客户端持续查询随机 SKU，每次缓存都未命中，数据库也查不到。若“不存在”不被缓存，请求会反复穿透到数据库。

服务可以短时间缓存空结果：

```text
key: product-detail:SKU-NOT-FOUND
value: not_found
expires_in: 15s
```

空值 TTL 通常较短，避免商品刚创建后仍长时间显示不存在。系统还可以在入口校验 SKU 格式、限制异常调用频率，或使用更适合大规模集合判断的数据结构。

缓存空结果只是保护手段。返回 `404` 的语义仍来自权威查询，不能把一次临时数据库错误缓存成“不存在”。

## 缓存整体不可用时怎么办

Redis 故障后，所有请求直接回源数据库，看似是自然降级，却可能把缓存承受的全部流量瞬间压到数据库上。

系统需要预先决定退路：

- 限制回源并发，保护数据库连接池；
- 非核心详情返回短期本地缓存或静态降级内容；
- 热门请求合并，避免同一 SKU 重复查询；
- 超过容量后快速失败，不能让请求无限排队；
- 缓存恢复时逐步放量，避免一次性回填所有热点。

缓存提升了正常容量，也改变了系统的故障模型。数据库必须能承受一定比例的回源，却通常不可能承受全部缓存流量。容量规划要同时考虑正常命中率和缓存故障时允许保留多少服务能力。

## 多级缓存会增加更多副本

大型读取链路可能同时使用浏览器缓存、CDN、应用本地缓存和 Redis。每一层都能减少下一层压力，也会增加失效传播时间。

```text
浏览器或 CDN
应用实例本地缓存
共享 Redis
商品数据库与库存服务
```

商品图片和静态介绍适合在 CDN 长时间缓存；促销价格和可售状态需要更短 TTL 或主动失效；下单交易仍然回到权威服务确认。

多级缓存要根据每项字段的正确性要求设计，而不是追求层数。把变化速度和容忍旧值程度不同的数据塞进同一个大对象，会迫使所有字段使用相同 TTL。适当拆分商品介绍、价格和库存摘要，可以让它们采用不同更新策略。

这不是要求一开始就建设多级缓存。只有读取量和延迟目标形成真实压力时，增加一层缓存才有收益。

## 怎样判断缓存是否真的有效

缓存监控不能只看命中率。团队还要观察：

- 命中、未命中和回源请求量；
- 缓存与数据库读取延迟；
- 热点 Key 的请求分布；
- 回填等待和锁竞争；
- 删除或更新缓存失败；
- Redis 内存、淘汰和连接使用；
- 缓存故障时数据库连接池与错误率；
- 页面旧数据最终是否造成用户反复下单失败。

高命中率也可能掩盖错误：若缓存长期保存过期价格，性能数据很好，业务结果却不可信。缓存指标要和商品更新延迟、库存冲突以及用户结果一起看。

## 使用缓存前要回答什么

缓存设计可以从四个问题开始：

1. **权威状态在哪里？**缓存丢失或冲突时，向哪里确认真实结果。
2. **允许旧多久？**不同字段的陈旧容忍度可能不同。
3. **怎样失效和回填？**数据变化、热门 Key 过期和删除失败分别怎样处理。
4. **缓存不可用时怎么办？**回源容量、限流和降级路径是否明确。

这四个问题把正确性、性能和可靠性放在同一项设计中。缓存保存的是可重建副本，交易约束仍由权威系统执行；缓存故障时，系统也要限制流量，避免保护层失效后拖垮数据库。
