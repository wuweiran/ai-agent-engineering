# 缓存怎样加速读取而不破坏正确性

运动鞋进入促销页面后，商品详情读取量迅速上升。团队把组装结果放进 Redis，页面延迟明显下降。与此同时，`SKU-8812` 的库存已经归零，部分用户仍然看到“有货”。

缓存减少了重复查询，也增加了一份可能过期的数据副本。**缓存适合保存能够重建、允许短暂陈旧的读取结果；真正的交易约束仍由权威系统执行。**

## 缓存用副本减少重复工作

商品详情被大量用户重复读取，修改频率相对较低。[Cache Aside](../interview/backend/cache/#cache-aside) 模式让应用先查询缓存，未命中时从数据库或业务服务加载，再写回缓存：

```text
读取 product-detail:SKU-8812
├─ 命中：直接返回
└─ 未命中：查询权威数据 → 组装 → 写入缓存 → 返回
```

缓存保存的是读取副本。Redis 数据全部丢失时，系统可以从权威来源重新构造；账户余额和订单支付结果不能因为缓存中有一份值，就允许在缓存里独立修改。

## 页面显示和成交结果可以不同

缓存中的 `in_stock: true` 只代表某个时刻的展示信息。从缓存写入到用户下单之间，其他请求可能买走库存。

下单仍要在库存数据库执行条件扣减：

```sql
UPDATE sku_inventory
SET available_stock = available_stock - 1
WHERE sku_id = 'SKU-8812'
  AND available_stock >= 1;
```

更新零行，当前订单返回库存不足。页面短暂陈旧会影响体验，却不会造成超卖。

价格也有同样边界：商品页可以显示缓存价格，创建订单时要重新取得当前可成交价格并保存快照。缓存可以帮助展示和过滤请求，不能替最终写入实施约束。

## 数据变化后通常让旧副本失效

商品标题更新后，一种常见顺序是：

```text
提交数据库修改
    ↓
删除缓存 Key
    ↓
下一次读取回源并生成新副本
```

删除而不是直接更新，可以避免每个写入方都重做复杂的详情组装。

如果先删除缓存、再更新数据库，中间到达的读取可能取得旧数据并重新写回缓存。先更新数据库、再删除仍有故障窗口：数据库已经提交，服务却在删除前崩溃，旧副本会继续存在。

数据库和 Redis 没有一个普通本地事务可以让两者共同变化。系统通常用删除重试或变更事件缩短窗口，再用 TTL 限制旧数据不会永久存在。

## TTL 限制陈旧时间，不保证立即一致

给 Key 设置六十秒的 TTL，表示旧值最长大致保留多久，也使无人访问的数据最终被回收。TTL 指缓存项的存活时间。它不能保证数据库更新后缓存立即变化。

TTL 是业务正确性和容量之间的取舍：时间越短，数据更新越快，回源也越多；时间越长，数据库压力越低，旧值持续越久。大批 Key 在同一时刻过期还会制造流量尖峰，因此常给过期时间加入随机偏移。

商品介绍可以允许短暂陈旧，权限和支付状态通常需要更严格的读取路径。TTL 应按数据含义选择，不能由一个全局默认值决定。

## 热点失效会让保护突然消失

`SKU-8812` 每秒被查询数千次。Key 到期的一瞬间，大量请求同时未命中并访问数据库，这就是缓存击穿。

服务可以合并同一个 Key 的回源：

```text
第一个请求取得回填权
    → 读取数据库并写入缓存
其他请求
    → 短暂等待，或在允许时继续使用旧值
```

回填锁必须有等待上限和过期时间，避免负责加载的实例崩溃后所有请求永久阻塞。允许陈旧的内容还可以使用逻辑过期：普通请求继续返回旧值，只由一个后台执行者刷新。

选择等待新值还是返回旧值，不是缓存组件的通用答案，而由数据能否短暂过期决定。

## 缓存不可用时，不能让全部请求无限回源

Redis 故障后，所有请求直接查询数据库，看似自然降级，却可能把缓存平时承受的全部流量瞬间压到数据库。

系统需要预先规定回源预算：限制同时访问数据库的请求，合并热门查询，非核心详情返回短期本地副本或降级内容，超过数据库能力时快速失败。缓存恢复后也应逐步放量，避免所有热点同时重建。

缓存提升了正常容量，也改变了故障模型。数据库通常需要承受一定比例的回源，不一定能承受全部缓存流量。

随机查询不存在的 SKU 也会持续穿透数据库。系统可以在确认 `not_found` 后短时间缓存空结果，并配合输入校验和入口限流；数据库临时错误则不能被缓存成“不存在”。

## 命中率高不代表缓存设计正确

长期返回过期价格时，命中率可能非常漂亮，业务结果却不可信。观察缓存要把技术指标和用户结果连接起来：命中与回源量说明节省了多少工作，热点回填和数据库连接说明故障压力，商品更新延迟和下单冲突说明旧数据是否产生了实际影响。

缓存设计最终要明确四件事：

```text
权威状态在哪里？
允许旧多久？
数据变化后怎样失效和回填？
缓存失效时，下游能承受多少回源？
```

**缓存通过可过期副本换取延迟和容量；正确性来自系统始终知道副本可以旧到什么程度，以及何时必须回到权威状态确认。**
