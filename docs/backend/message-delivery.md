# 消息为什么会丢失或重复

订单 `O-1042` 创建后，系统准备发布一条延迟消息，让 Worker 在支付期限到达时检查订单。数据库已经提交，应用却在消息发送前崩溃。用户能看到订单，过期任务却从未进入队列。

团队修复这个问题后，又遇到另一种故障：Worker 已经取消订单并释放库存，正准备确认消息时进程重启。队列没有收到确认，于是再次投递同一条消息。

这两个事故分别揭示了消息系统最重要的两个边界：业务写入和消息发布之间可能出现空隙；消息消费与业务处理之间也可能出现空隙。

这仍属于“协调工作”，关注点从任务生命周期推进到跨组件交付。消息队列提升可靠性，也引入重复、乱序和积压；判断设计时要同时看业务结果、恢复路径和队列容量。

## 数据库成功以后，消息为什么仍会丢

一种直接写法是：

```text
1. 提交订单事务
2. 向消息队列发布 order.expiration_scheduled
3. 返回下单成功
```

若服务在步骤 1 和步骤 2 之间崩溃，订单存在，消息不存在。重启后的服务也不知道刚才还有一条消息没发。

交换顺序同样不安全：

```text
1. 发布消息
2. 提交订单事务
```

消息发出后，如果数据库事务失败，Worker 会收到一个不存在的订单。它可以把这条消息忽略，但若消息代表“给新用户发优惠券”等动作，可能已经产生了不该发生的结果。

问题来自两个系统各自提交。数据库和消息队列都可以可靠完成自己的操作，却没有一个共同事务自动保证二者一起成功。

## Outbox 把待发送消息先写入数据库

订单服务可以在创建订单的同一个数据库事务中，再写一条待发布事件：

```sql
BEGIN;

INSERT INTO orders (
  order_id,
  buyer_id,
  status,
  reservation_expires_at
) VALUES (
  'O-1042',
  'U-204',
  'pending_payment',
  '2026-07-14T10:50:00Z'
);

INSERT INTO outbox_events (
  event_id,
  event_type,
  aggregate_id,
  payload,
  status,
  created_at
) VALUES (
  'EV-7001',
  'order.expiration_scheduled',
  'O-1042',
  '{"order_id":"O-1042","not_before":"2026-07-14T10:50:00Z"}',
  'pending',
  CURRENT_TIMESTAMP
);

COMMIT;
```

订单和 Outbox 记录使用同一个数据库事务，因此只有两种结果：两者都存在，或两者都不存在。应用在事务提交后崩溃，待发送事件仍留在数据库中。

独立的发布进程持续读取 `pending` 事件，发送到消息队列，再更新发送状态。它可以在服务重启后继续扫描，不依赖原下单请求仍然存在。

这种做法称为 Transactional Outbox。名称并不重要，真正解决的是：**先把“还有一条消息需要发送”变成与业务数据共同提交的持久事实。**

## 标记已发送之前仍然可能崩溃

Outbox 发布进程也存在故障窗口：

1. 从数据库读取 `EV-7001`；
2. 成功发布到消息队列；
3. 尚未把 Outbox 标记为已发送，进程崩溃。

重启后，它会再次读取 `EV-7001` 并重新发布。Outbox 避免消息永久缺失，却不能天然保证只发布一次。

发布进程可以携带稳定的 `event_id`，让下游识别重复消息；也可以使用支持生产者幂等的消息系统减少重复。无论中间件提供什么保证，业务消费者仍应准备处理同一个事件再次到达。

这是一条贯穿可靠消息的现实约束：为了不丢消息，系统通常宁可重复投递。

## 队列为什么会再次投递已经处理的消息

队列把过期消息交给 Worker 后，会等待确认。Worker 的正常流程是：

```text
读取订单
取消到期订单
释放库存
确认消息
```

如果 Worker 在确认之前崩溃，队列无法判断业务处理是否已经完成。它只能在可见性超时或消费者失联后重新投递。

队列只看到“没有确认”，无法知道数据库里的订单是否已经取消。结果未知时重新投递，可以避免一条未完成的消息永久消失。

因此，许多消息系统提供的是**至少一次投递**：消息至少会有机会被处理，也可能被处理多次。消费者要保证重复处理不会产生重复业务结果。

## 条件更新让重复取消安全退出

过期 Worker 收到消息后，先通过条件更新取消仍待支付且已经到期的订单：

```sql
UPDATE orders
SET status = 'cancelled',
    cancel_reason = 'payment_timeout',
    version = version + 1
WHERE order_id = 'O-1042'
  AND status = 'pending_payment'
  AND reservation_expires_at <= CURRENT_TIMESTAMP;
```

第一次处理更新一行。重复消息再次到达时，订单已经是 `cancelled`，更新零行，Worker 不会再次执行后续释放。

这是一种基于业务状态的幂等：同一状态转换只能成功一次。它比简单判断“我好像见过这条消息”更接近真实结果，因为即使消费记录丢失，订单状态仍能阻止重复取消。

不过，库存释放也必须与订单取消协调。如果二者在同一个数据库，可以在一个事务中完成：只有订单从 `pending_payment` 成功变成 `cancelled`，才把库存加回一次。

```sql
BEGIN;

UPDATE orders
SET status = 'cancelled',
    cancel_reason = 'payment_timeout'
WHERE order_id = 'O-1042'
  AND status = 'pending_payment';

-- 只有上一步实际更新订单时，才执行库存释放
UPDATE sku_inventory
SET available_stock = available_stock + 1
WHERE sku_id = 'SKU-8812';

COMMIT;
```

真实代码必须检查第一条更新的受影响行数，不能无条件执行第二条 SQL。

## 消费记录解决没有天然状态门槛的动作

并非每个消费者都能靠订单状态去重。发送“订单已取消”短信时，即使订单早已是 `cancelled`，系统也无法从订单字段判断这条短信是否已经发送。

消费者可以保存处理记录，并对 `consumer_name + event_id` 建立唯一约束：

```text
consumer_name: cancellation-sms
message_id: EV-7102
processed_at: 2026-07-14T10:51:04Z
```

再次收到 `EV-7102` 时，先发现消费记录已经存在，直接确认消息。

如果业务写入和消费记录位于同一个数据库，应放在同一事务中。以发放站内优惠券为例：创建优惠券和记录事件已消费共同提交，重复消息无法创建第二张。

外部短信服务则不在本地事务内。即使本地记录和发送顺序设计得再仔细，仍可能出现短信已发送、记录未提交。此时最好把 `event_id` 作为短信服务的幂等键；若外部服务不支持幂等，系统只能在“可能重复发送”和“可能漏发”之间作出业务取舍，并通过查询或人工补偿降低影响。

## Exactly Once 为什么容易被误解

一些中间件会提供 Exactly Once 或事务消息能力。它们通常能保证特定边界内的消息生产与消费语义，例如同一消息不会在 Broker 日志中被消费者提交两次。

但一条消息处理还可能调用数据库、支付平台和短信服务。Broker 无法自动让所有外部系统共同参与它的事务。只要跨过边界，就仍然需要幂等键、条件更新、状态查询或补偿。

因此，工程中更实用的目标是：

- 消息可以至少一次到达；
- 消费者能够识别同一意图；
- 业务状态转换不会重复发生；
- 外部结果未知时先查询，再决定是否重试。

所谓“业务上只发生一次”，来自整条链路的设计，不能只依赖消息中间件的一个配置项。

## 消息顺序也可能变化

同一订单可能产生：

```text
order.created
order.paid
order.cancelled
```

由于并行分区、网络重试或不同生产者，这些消息不一定严格按业务顺序到达。消费者若先收到 `order.paid`，随后又处理一条迟到的 `order.created`，不能把支付状态退回待支付。

常见防线包括：

- 同一订单的消息使用 `order_id` 作为分区键，尽量进入同一有序分区；
- 事件携带订单版本或发生序号；
- 消费者只接受符合当前状态和版本的转换；
- 发现缺失前置事件时暂存、重试或回查权威状态。

Broker 的分区顺序可以减少乱序，业务状态检查仍然要守住最终边界。历史消息迟到时，权威订单状态不能倒退。

## 无法处理的消息去哪里

一条消息可能因为格式错误、字段缺失或消费者缺陷反复失败。无限重试会占用 Worker 并阻塞后续消息。

系统可以在有限重试后，把消息转入死信队列或异常任务表，同时保存：

- 原消息和事件 ID；
- 消费者名称与版本；
- 最后错误和失败次数；
- 对应订单等业务资源；
- 首次与最近失败时间。

值班人员修复代码或数据后，可以从明确位置重新投递。重新处理仍要使用原事件 ID，不能生成一个新身份绕过去重规则。

死信队列不是消息最终归宿。它只是把持续失败从主消费链隔离出来，系统仍需告警、排查和清理积压。

## 怎样观察消息链路

只看队列长度不足以判断订单任务是否正常。团队还需要关联：

- Outbox 中等待发布的事件数量和最老等待时间；
- 消息从创建到发布、从发布到消费的延迟；
- 消费成功、重试和死信数量；
- 同一事件被重复投递多少次；
- 订单 ID、事件 ID 和消费者处理记录；
- 订单最终是否过期取消并释放库存。

队列指标正常，Outbox 发布进程可能已经停止；消息全部确认，消费者也可能错误地跳过了业务操作。最终仍要回到用户和订单结果验证链路。

## 可靠消息真正保证什么

业务写入和消息发布之间无法用普通代码消除故障窗口，Outbox 先把待发送意图与业务数据共同保存。发布和消费都可能在确认前崩溃，所以消息需要稳定事件 ID，消费者需要条件更新、消费记录和外部幂等。

**可靠消息允许网络层出现丢失与重复，同时让业务结果保持正确并且可以恢复。**

订单过期链路建立后，下一类问题会发生在服务调用上：订单服务请求支付服务，超时后究竟应该重试、查询，还是告诉用户支付失败。消息与远程调用都会遇到“没有收到结果”，处理方式却不完全相同。
