# 服务之间怎样处理超时和故障

用户准备支付订单 `O-1042`。订单服务向支付服务发送创建支付请求，支付服务成功创建了支付单 `P-8821`，响应却在返回途中丢失。订单服务等待两秒后只得到一个超时错误。

超时没有告诉订单服务支付单是否存在。若它直接宣布失败，用户可能已经进入支付流程；若它立即重新创建，又可能出现两笔支付。

这篇文章开始从“协调工作”进入“保障运行”：远程依赖变慢或失联时，系统既要确认业务结果，也要防止一个故障占满自身资源。

## 远程调用多出了一种结果

同一进程中的函数调用通常返回结果或抛出异常。远程调用还要经过域名解析、连接、网络、负载均衡器和对方服务，其中任何环节都可能在调用方等待期限内没有回应。

订单服务看到超时时，实际情况可能是：

1. 请求没有到达支付服务；
2. 请求到达，但支付服务处理失败；
3. 支付单已经创建，响应没有返回；
4. 支付服务仍在处理，只是耗时超过上游期限。

调用方只能确认“没有及时收到结果”。这种状态称为**结果未知**。

读取商品信息超时通常可以再次查询，因为重复读取不会改变业务状态。创建支付单具有副作用，重复调用可能产生新资源，处理方式必须更谨慎。

## 一次调用可以有多个时间边界

订单接口允许用户等待三秒，不代表支付服务可以独占三秒。订单服务还需要验证请求、保存状态、转换错误并返回响应。

远程调用常见三个时间边界：

| 时间边界 | 回答的问题 |
| --- | --- |
| 连接超时 | 多久仍无法与支付服务建立连接，就停止本次尝试？ |
| 响应超时 | 请求发出后，多久仍未收到响应，就不再占用当前调用？ |
| 整体截止时间 | 当前用户请求或后台任务最晚什么时候必须结束这一轮？ |

假设支付接口的整体截止时间是三秒。订单服务前面已经消耗 600 毫秒，传给下游的剩余期限就不能仍是三秒。调用链越深，每一层都重新使用完整超时，会让最上游等待时间不断膨胀。

截止时间应沿调用链向下传递。下游发现剩余时间不足以完成工作时，可以尽早拒绝，让上游进入明确的恢复路径。

超时太长会让线程、连接和内存长期被占用；超时太短则会中断本来能够成功的请求，并制造更多重试。合理数值要来自真实延迟分布和业务等待上限，不能只复制框架默认值。

## 创建支付单需要稳定的请求身份

订单服务为这次支付意图生成稳定标识：

```text
payment_request_id: PAYREQ-7001
order_id: O-1042
buyer_id: U-204
amount_minor: 65900
currency: CNY
```

创建与重试都使用 `PAYREQ-7001`。支付服务对它建立唯一约束：

- 第一次请求成功，创建 `P-8821`；
- 相同 ID 再次到达，返回原支付单；
- 相同 ID 携带不同订单或金额，拒绝为参数冲突。

幂等键表达的是同一个业务意图，不能在每次网络重试时重新生成。它也不能只存在于订单服务内存中；服务重启后仍要从订单或支付尝试记录中找回。

支付服务仍然需要独立校验订单引用、金额和币种。订单服务已经检查过，不代表下游可以相信所有输入。

## 超时后先把未知状态保存下来

创建支付请求超时后，订单服务不能把订单直接标记为 `payment_failed`。更准确的状态是：

```text
order_id: O-1042
status: payment_confirming
payment_request_id: PAYREQ-7001
payment_id: null
```

它表示系统正在确认创建结果，用户暂时不应再发起另一笔支付。

后台任务随后根据 `PAYREQ-7001` 查询支付服务：

- 找到 `P-8821`，保存 `payment_id`，恢复为待支付或继续确认支付状态；
- 支付服务明确回答该请求不存在，才安全地重新创建；
- 支付服务仍不可用，保持 `payment_confirming` 并稍后重试；
- 已有支付单的订单或金额不一致，停止自动处理并告警。

用户页面应显示“正在确认支付结果”，而不是诱导用户重复点击。状态命名需要表达业务事实，不能把底层 `socket timeout` 直接变成用户看到的最终结论。

**结果未知的写操作先查询权威状态，确认没有发生后再重做。**

## 哪些错误值得重试

重试有价值需要同时满足两个条件：再次执行是安全的，失败原因也可能随时间消失。

| 结果 | 是否重试 | 原因 |
| --- | --- | --- |
| 连接建立失败 | 可以在限制内重试 | 请求通常尚未发出，依赖可能短暂抖动 |
| 服务返回限流 | 按建议等待后重试 | 容量可能恢复，不能立即连续轰炸 |
| 服务返回参数错误 | 不重试 | 相同输入不会自行变正确 |
| 服务返回权限不足 | 不重试 | 等待不会改变权限 |
| 创建支付超时 | 先查询 | 动作可能已经成功，直接重做不安全 |
| 查询支付状态超时 | 可以有限重试 | 查询本身没有业务副作用 |

即使请求具有幂等键，也不能无限重试。每次尝试都会占用下游容量，错误持续时还可能放大故障。

重试应设置次数和整体截止时间，并使用退避：第一次稍等，连续失败后逐步延长等待，再加入少量随机错峰。所有请求同时在固定一秒后醒来，只会形成新的流量尖峰。

## 慢支付服务怎样拖垮订单服务

支付服务平时 200 毫秒返回，故障时需要十秒。订单服务若允许每个请求持续等待，大量线程或异步任务会停在支付调用上，连接池也会被占满。

随后出现连锁反应：

- 新支付请求无法取得连接；
- 用户查询订单也排队等待；
- 网关开始超时并重试；
- 实例内存和待处理任务增加；
- 一个支付故障扩散成整个订单服务不可用。

保护上游需要几种配合手段。

### 限制并发

订单服务限制同时调用支付服务的请求数量。达到上限后，新请求可以快速进入“支付服务繁忙”，或者创建后台确认任务，不能继续无限排队。

并发上限保护线程池和连接池，也给下游留下恢复空间。它必须与实例数量和下游容量一起计算；每台实例允许一百个并发，扩容到五十台后仍可能把支付服务压垮。

### 隔离资源

支付调用可以使用独立连接池、执行队列和并发预算。这样支付服务变慢时，不会耗尽查询订单、取消订单等其他路径的全部资源。

隔离并不一定要求拆成微服务。一个应用内部也可以为不同依赖设置独立资源边界。

### 暂时熔断

当支付调用在一段时间内持续超时，继续尝试只会增加压力。熔断器可以暂时拒绝新调用，让请求快速失败或进入稍后处理。

等待窗口过去后，系统只允许少量探测请求。探测成功，再逐步恢复流量；仍然失败，则继续保持断开。

熔断依据的是依赖健康状况，不能替代单笔订单的状态判断。支付服务恢复后，`O-1042` 仍要通过 `PAYREQ-7001` 查询真实结果。

### 提供业务降级

支付暂时不可用时，系统可以保留订单和库存一段时间，提示用户稍后支付。若库存保留成本太高，也可以停止创建新的待支付订单。

降级方案由业务决定。展示缓存页面可能适合内容系统，支付系统不能伪造成功。一个功能可以暂时不可用，资金和订单状态不能被猜测。

## 状态查询也可能承受压力

创建支付超时后，所有订单同时轮询状态，会把故障从创建接口转移到查询接口。

确认任务需要控制节奏：

- 首次等待短时间，给支付服务完成处理的机会；
- 连续未知时逐步延长间隔；
- 合并用户刷新与后台查询，避免两套轮询叠加；
- 支付平台支持回调时，用回调推动状态，同时保留主动查询作为补偿；
- 超过业务期限后进入人工核对或异常订单流程。

回调也可能重复、延迟或丢失。它带来的信息仍要通过支付 ID、请求 ID 和订单状态进行幂等更新，不能因为来源是支付服务就跳过状态检查。

## 怎样观察一次远程调用故障

订单服务需要记录用户请求、订单和支付请求之间的关联：

```text
request_id: REQ-301
order_id: O-1042
payment_request_id: PAYREQ-7001
payment_id: P-8821
```

观测至少回答：

- 调用支付服务的成功、超时和错误比例；
- 连接耗时、服务处理耗时和总耗时；
- 当前并发、连接池等待和拒绝数量；
- 重试次数与首次成功、重试成功的比例；
- 熔断状态和探测结果；
- 停在 `payment_confirming` 的订单数量与最老等待时间；
- 最终有多少订单确认成功、失败或进入人工处理。

接口超时率下降，不代表问题已经解决。如果大量订单长期停在确认中，用户仍然没有获得支付结果。

## 远程故障中的判断顺序

服务调用失败后，可以按三个问题处理：

1. **结果确定吗？**明确失败、明确成功和结果未知需要不同路径。
2. **重复执行安全吗？**查询通常安全，写操作需要幂等与状态条件。
3. **继续尝试会不会放大故障？**重试必须受截止时间、退避、并发和熔断约束。

这三个判断把正确性与容量保护连接起来。幂等和查询避免重复支付，隔离与熔断避免慢依赖拖垮整个订单服务。
