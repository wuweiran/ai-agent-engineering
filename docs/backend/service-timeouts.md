# 服务调用中的超时与容错

用户为订单 `O-1042` 创建支付。支付服务已经生成支付单 `P-8821`，响应却在返回途中丢失。订单服务等待两秒，只得到一个超时错误。

超时没有说明支付失败。请求可能没到达，也可能仍在处理，甚至已经成功而只有响应丢失。若订单服务直接重试，可能创建两笔支付；若直接宣布失败，又会与真实支付状态冲突。

**没有按时收到结果，不等于远端没有执行。**

## 服务发现与负载均衡把请求送到实例

支付能力独立部署后，订单服务调用的是一个稳定服务名，而不是长期固定的机器地址。DNS、Kubernetes Service 或注册中心提供当前实例列表，调用方或代理再通过轮询、随机、最少连接等负载均衡策略选择目标实例。

健康检查用于摘除已经不可用的实例，也要关注仍能响应但持续变慢的节点。负载均衡只能在实例之间分配请求；所有支付实例都已饱和时，仍需要限流、并发控制或扩容真正的瓶颈。

服务发现和负载均衡解决“调用哪一个实例”，不能回答“调用是否成功”。请求真正跨过网络后，才会出现下面的结果未知、超时和重试问题（[服务发现与负载均衡怎样配合？]({{ site.baseurl }}/docs/interview/backend/distributed-system/#service-discovery-load-balancing)）。

## 远程调用多出“结果未知”

同一进程中的函数调用通常返回结果或抛出异常。远程调用还要经过连接、网络和对方服务，调用方看到超时时，真实情况可能是：

```text
请求没有到达
请求到达但明确失败
远端仍在处理
操作已经成功，响应没有返回
```

调用方只能确认自己没有在期限内得到答案。这种状态是[结果未知]({{ site.baseurl }}/docs/interview/backend/distributed-system/#timeout-unknown-outcome)，不能擅自折叠成失败。

读取商品超时通常可以再次读取，因为重复查询不改变业务状态。创建支付、退款和发信具有副作用，重复执行是否安全取决于接口能否识别同一个业务意图。

## 调用超时与截止时间共同限制下游

订单接口允许用户等待三秒，不表示支付服务可以独占三秒。认证、订单检查、状态保存和响应都需要时间。

调用方首先根据支付服务的真实延迟和 SLA 配置单次调用超时，同时不能超过当前请求的剩余时间：

```text
订单请求总截止时间：3.0s
完成订单检查后剩余：2.4s
支付服务常规调用上限：1.5s
本次支付调用超时：min(2.4s - 返回预留时间, 1.5s)
```

调用超时首先由调用方在本地控制。只有协议或框架支持截止时间与取消传播时，调用方才会把绝对 Deadline、剩余时间或取消信号传给支付服务。支付服务可以据此把执行时间限制为“上游剩余时间”和“自身处理上限”中的较小值，并停止尚未开始或可以取消的工作。普通 HTTP 调用如果没有这套机制，下游并不会自动知道调用方已经超时。

连接超时限制建立连接的等待，响应超时限制一次下游调用，整体截止时间限制当前请求或任务还能运行多久。数值既要参考下游延迟分布，也要服从上游业务等待上限；调用链中的每一层不能重新获得一份完整时间预算。

## 稳定请求 ID 代表同一个支付意图

订单服务为本次支付生成：

```text
payment_request_id: PAYREQ-7001
order_id: O-1042
amount_minor: 65900
currency: CNY
```

第一次创建和后续恢复都使用 `PAYREQ-7001`。支付服务对它建立唯一约束：相同 ID 再次到达时返回原支付单；相同 ID 携带不同订单或金额时拒绝冲突请求。

这个标识必须持久化。若每次网络重试都生成新 ID，支付服务看到的就是多个不同业务意图，无法防止重复创建。

这类设计让接口具备[幂等性]({{ site.baseurl }}/docs/interview/backend/distributed-system/#idempotency)：这里的 `payment_request_id` 本质上就是支付业务的幂等 Key，用于标识“一次支付意图”。同一业务意图重复提交，最终只得到一份业务结果。幂等不表示请求可以无限重试。它主要解决重复结果，不能创造下游容量，也不能修复永久参数错误。

## 超时后先保存“正在确认”

创建支付超时后，订单不能直接进入 `payment_failed`。更准确的状态是：

```text
order_id: O-1042
status: payment_confirming
payment_request_id: PAYREQ-7001
payment_id: null
```

这说明系统正在确认结果，用户暂时不应创建另一笔支付。后台任务随后按 `PAYREQ-7001` 查询支付服务：

```text
查到 P-8821
    → 保存 payment_id，继续支付流程
明确回答不存在
    → 才允许使用同一请求 ID 重新创建
仍然无法确认
    → 保持 confirming，退避后再查
返回订单或金额冲突
    → 停止自动处理并告警
```

用户界面也应显示“正在确认”，而不是把底层 `socket timeout` 翻译成最终支付失败。

**有副作用的写操作结果未知时，先查询权威状态；确认没有发生后，才考虑重新执行。**

## 重试要同时满足安全和可能恢复

参数错误、权限不足和业务状态冲突不会因为等待而自动变好，不应重试。连接短暂失败、限流或无副作用查询可能在稍后恢复，可以在整体截止时间内有限尝试。

每次重试都要回答两个问题：

```text
再次执行会不会产生第二份业务结果？
失败原因是否可能随时间消失？
```

满足以后仍要使用[退避和随机错峰]({{ site.baseurl }}/docs/interview/backend/distributed-system/#backoff-jitter)。所有调用方在固定一秒后同时醒来，会形成新的流量尖峰。

## 慢依赖会通过等待拖垮调用方

支付服务平时两百毫秒返回，故障时需要十秒。如果订单服务让每个请求持续等待，执行线程、连接池和内存会逐渐被占满。随后订单查询等无关路径也无法取得资源，一个依赖故障扩散成整个服务不可用。

上游需要在等待失控前建立边界：限制同时调用支付服务的数量；使用独立连接池和并发预算进行[故障隔离]({{ site.baseurl }}/docs/interview/backend/distributed-system/#circuit-breaker-fallback-isolation)，避免支付占尽订单查询资源；持续失败时暂时熔断，只允许少量探测请求；提供真实业务降级，例如保留订单并提示稍后支付，绝不能伪造支付成功。

这些机制保护的是调用方容量。支付服务恢复后，`O-1042` 仍然要按 `PAYREQ-7001` 确认真实结果。

状态查询本身也必须退避。大量未知订单同时高频轮询，会把压力从创建接口转移到查询接口。支付回调可以推动状态，但回调同样可能延迟和重复，仍需按请求 ID 和当前订单状态幂等处理。

## 观测要从接口错误连接到业务结果

一次支付调用需要关联用户请求、订单和支付意图：

```text
request_id
order_id
payment_request_id
payment_id
```

调用超时率和连接等待说明依赖运行情况；处于 `payment_confirming` 的订单数量、最老等待时间和最终确认比例说明用户是否得到结果。

接口不再超时，但大量订单长期停在确认中，故障仍未结束。

## 远程故障的判断顺序

服务调用没有得到正常响应时，先区分明确成功、明确失败和结果未知；再判断重复执行是否安全；最后判断继续等待和重试会不会耗尽系统容量。

```text
结果确定吗？
    ↓
同一业务意图能否被识别？
    ↓
应该查询、重试、等待还是停止？
    ↓
这些尝试是否受截止时间和并发上限保护？
```

幂等键和状态查询守住业务正确性，超时、退避、隔离和熔断限制故障扩散。两类机制缺一不可。
