# 流量增加时服务为什么会被压垮

运动鞋促销开始后，商品详情缓存承担了大部分读取，下单请求却仍要创建订单、扣减库存并写入消息。流量从每秒 300 个下单请求增长到 900 个，最初只是延迟上升，随后应用开始大量超时。

监控显示应用 CPU 只有 40%。团队增加实例后，错误反而更多，因为每台实例都把更多并发请求送进同一个订单数据库。

这篇文章属于“保障运行”：系统要知道自己能够持续完成多少工作，并在资源耗尽之前控制入口。判断容量要同时看请求进入速度、处理速度和等待数量，单独一个 CPU 数字无法说明系统还能承受多少业务。

## 处理速度跟不上时，等待一定会增加

假设订单系统每秒最多完成 500 个请求，促销期间却持续进入 800 个。每秒多出的 300 个请求只能等待、被拒绝，或者在超时后失败。

短时间的高峰可以进入队列，流量很快回落时，系统再逐步处理积压。若输入速度长期高于处理能力，队列只会越来越长：

```text
每秒进入 800 个请求
每秒完成 500 个请求
每秒新增 300 个等待请求
```

一分钟后就会增加 18,000 个等待请求。它们继续占用内存、连接或队列空间，即使最终全部超时，也已经消耗了系统资源。

因此，队列可以平滑短时波动，不能创造处理能力。持续过载只有三个方向：提高真正的处理吞吐，降低入口流量，或减少每项工作的资源消耗。

## 一次下单会占用哪些资源

下单不是一个抽象的“请求”。它在不同阶段占用不同资源：

1. 网关保存客户端连接并执行认证与限流；
2. 应用实例使用执行线程或异步任务处理业务；
3. 订单服务从连接池取得数据库连接；
4. 数据库使用 CPU、内存、磁盘和锁完成事务；
5. 应用向消息队列写入事件；
6. 依赖调用结束后，资源才逐层释放。

任何一层达到上限，后面的能力都无法被充分使用。CPU 使用率不高可能是因为线程都在等待数据库连接；数据库 CPU 不高也可能是热点库存行的锁让事务排队。

排查接口变慢时，应继续问：**请求正在等待哪一种资源，它会等待多久，等待数量是否持续增加？**

常见线索包括：

| 资源 | 接近上限时的现象 |
| --- | --- |
| 应用执行池 | 活跃任务到达上限，入口排队时间增加 |
| 数据库连接池 | 获取连接耗时增加，业务 SQL 还没执行就超时 |
| 数据库锁 | 单条事务耗时上升，热点 SKU 更新排队 |
| 消息生产连接 | 订单事务完成后，事件发布长期等待 |
| 内存 | 等待请求和大响应积累，垃圾回收变频繁 |
| 外部依赖并发 | 支付或库存服务超时，重试进一步增加请求量 |

吞吐由最先达到容量边界的环节决定。这一环节就是当前瓶颈。

## 延迟为什么会在接近上限时突然恶化

系统负载较低时，新请求很快取得资源。资源利用率接近上限后，请求开始排队。即使单次 SQL 的执行时间没有变化，总延迟也会因为等待连接和锁而增长。

平均延迟可能仍然正常，少量请求却已经等待很久。工程师还要观察高分位延迟，例如 95% 或 99% 请求在多久内完成。促销流量中，尾部请求最先暴露排队和资源竞争。

超时会进一步放大问题。客户端看不到结果后重新提交，网关或服务框架也可能自动重试。原请求仍在后台执行，新请求又加入系统，于是实际负载高于用户操作数量。

这形成一个危险循环：

```text
请求增加
排队与延迟上升
更多请求超时
客户端和服务重试
实际请求继续增加
```

重试只有在系统仍有余量时才可能帮助恢复。过载期间，无限制重试会把短暂拥堵变成持续故障。

## 为什么增加应用实例可能更糟

应用执行能力不足、CPU 接近饱和时，水平扩容通常有帮助。但下游数据库已经是瓶颈时，增加应用实例会放大进入数据库的并发。

假设原来有 10 台实例，每台连接池最多 40 个连接，应用最多向数据库建立 400 个连接。扩容到 30 台后，理论连接数变成 1,200。数据库若只能稳定处理 500 个并发事务，扩容只是让更多请求同时争用它。

扩容前要确认瓶颈：

- 应用 CPU 或执行池不足，增加实例可能提高吞吐；
- SQL 扫描过多数据，先优化查询和索引；
- 热门 SKU 的库存行必须串行更新，增加应用不会消除锁竞争；
- 支付服务有固定容量，上游需要限制并发；
- 消息消费者跟不上，扩容 API 只会让积压增长更快。

扩容还会改变共享依赖的总负载。连接池和并发上限要按整个服务集群计算，不能每台实例独立使用一个看似安全的数字。

## 有限队列比无限等待更可靠

入口队列可以吸收几秒钟的流量高峰。它必须有容量和等待上限。

无限队列会带来三种结果：

- 请求在队列中等到客户端已经超时；
- 内存随着积压持续增长；
- 系统恢复后仍要处理大量已经没有价值的旧请求。

有限队列满后，服务应快速拒绝新工作，并给出可重试的错误或明确繁忙状态。快速拒绝看起来降低了成功率，却能让已经接收的请求继续完成，并避免整个实例失去响应。

队列中的任务还要携带截止时间。用户请求已经取消或超过业务期限时，Worker 不应继续执行昂贵的下游调用。

## 背压让上游感知下游能力

消息消费者每秒只能处理 500 条库存更新，生产者却持续发送 1,000 条，积压会不断增加。背压的含义是让上游根据下游处理能力减慢，而不是把压力藏在更大的队列中。

背压可以表现为：

- 队列达到阈值后，API 降低接收速度；
- 消费者只预取自己能够并发处理的消息数；
- 下游返回限流信号和建议重试时间；
- 应用停止非核心异步任务，为交易任务保留资源；
- 流式传输中，发送方等待接收方消费后再继续。

每一层都需要有界缓冲。若上游完全忽略下游速度，积压最终会转移到内存、连接池或数据库。

## 限流要保护系统，也要分配容量

促销库存只有一件，却有大量用户反复提交同一个 SKU。系统可以在网关和业务层实施不同限流：

- 按用户或设备限制异常高频请求；
- 按接口保护昂贵的下单路径；
- 按热门 SKU 限制无意义的并发竞争；
- 为支付回调、订单查询等关键流量保留容量；
- 在系统整体接近上限时执行全局负载保护。

限流不能代替业务校验。即使只有一个请求通过，数据库仍要条件扣减库存。限流减少资源浪费，业务约束保证最终正确。

拒绝响应还要告诉客户端是否值得重试，以及建议等待多久。所有客户端立即重试会绕过限流的保护效果。

## 过载时要保留最重要的工作

容量不足时，不同工作价值不同：

- 支付回调会决定资金和订单状态；
- 已提交订单的状态查询影响用户确认结果；
- 推荐特征更新可以晚一些处理；
- 批量统计报表可以暂停。

系统可以为关键路径预留连接、线程和队列，把非核心工作延迟或丢弃。这称为负载丢弃或优先级调度。

优先级不能只写在消息字段里。消费者、队列和资源池都要真正实施，否则高优先级任务仍会排在大量普通任务后面。

## 热点资源可能让整体容量失真

数据库每秒可以处理大量不同订单，不代表它能同时更新同一个库存行。热门 SKU 会让事务争抢相同锁，形成局部串行瓶颈。

这时，集群总 CPU 和平均数据库延迟可能并不高，`SKU-8812` 的请求却大量等待。工程师需要按商品、分区或租户观察热点，而不能只看全局平均值。

缓解方式取决于业务：

- 在入口提前拒绝已经明确无库存的请求；
- 将同一 SKU 的扣减请求串行化；
- 对库存进行合理分片，但增加汇总和一致性成本；
- 使用库存令牌或预分配机制减少直接锁竞争；
- 促销开始前确认热点路径的独立容量。

这些设计会增加复杂度。普通商品没有真实热点时，数据库条件更新通常已经足够。优化应针对观测到的瓶颈，不能因为可能有高并发就提前引入复杂分片。

## 压测要寻找拐点

压测要找出系统从稳定走向失控的过程。脱离延迟、错误率和资源条件的最高 QPS 数字，无法代表可持续容量。

测试可以逐步增加流量，观察：

- 吞吐是否继续随请求量增长；
- 平均和高分位延迟何时开始陡升；
- 哪个资源最先接近上限；
- 队列和连接池等待是否持续增长；
- 限流是否在资源耗尽前生效；
- 超时和重试有没有放大输入流量；
- 流量下降后，积压能否在合理时间内清理；
- 扩容后瓶颈是否转移到其他组件。

压测场景还要接近真实业务比例。只测试商品读取，会高估下单能力；全部请求都购买同一个 SKU，又可能只测到热点锁而看不到普通订单吞吐。

最终容量基线应包含条件：当前版本、数据规模、请求组合、实例数量和依赖容量下，在某个延迟与错误率目标内可以持续完成多少业务请求。

## 容量规划看持续能力，也看突发空间

日常流量每秒 300 单、压测上限每秒 500 单，并不意味着系统可以安全上线每秒 480 单。发布波动、单机故障、依赖变慢和数据增长都会占用余量。

团队需要保留容量空间，并按业务增长提前扩容。容量计划还应考虑降级后的能力：一台实例或一个可用区退出后，剩余系统能否承担关键流量；缓存故障时，数据库允许多少比例回源；消息消费者停机十分钟后，需要多久清理积压。

这里关心的是在真实故障条件下仍能提供多少服务，而不只是理想环境中的峰值。

## 过载保护的判断顺序

面对流量增长，可以沿四个问题判断：

1. **瓶颈在哪里？**请求具体在等待哪种资源。
2. **高峰是短暂还是持续？**队列能否在业务期限内清空。
3. **哪些工作必须保留？**为关键交易预留资源，延迟非核心任务。
4. **何时拒绝新工作？**在连接、内存和数据库耗尽前实施背压与限流。

容量治理允许系统主动拒绝一部分超出能力的请求，以保护已接收的工作和核心交易。若所有请求都进入系统并在资源耗尽后一起超时，失败范围会更大，恢复时间也会更长。

下一篇会从一次真实退化开始：下单接口的 HTTP 成功率仍然正常，用户支付完成率却开始下降。工程师需要把用户结果、服务指标、日志和 Trace 连接起来，先限制影响，再找到最早出现异常的位置。
