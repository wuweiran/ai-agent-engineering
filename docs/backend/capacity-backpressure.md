# 流量增加时服务为什么会被压垮

运动鞋促销开始后，下单流量从每秒 300 个增加到 900 个。最初只是响应变慢，随后大量请求超时。监控显示应用 CPU 只有 40%，团队增加实例后，错误反而更多：新增实例把更多并发请求送进了同一个订单数据库。

以下使用模拟促销说明容量问题的主线：**当工作进入速度持续高于完成速度，等待一定增加；等待占满有限资源后，系统会从变慢转为失控。**

## 队列只能吸收短时波动

假设订单系统每秒最多完成 500 个请求，促销期间持续进入 800 个。每秒多出的 300 个只能等待或被拒绝：

```text
每秒进入 800
每秒完成 500
每秒新增 300 个等待请求
```

一分钟后会多出 18,000 个请求。它们即使最终超时，也已经占用连接、内存或队列空间。

流量很快回落时，有限队列可以平滑短时高峰；进入速度长期高于处理速度时，更大的队列只会推迟故障。持续过载只能通过提高真实吞吐、减少单项工作成本，或降低入口流量解决。

## 容量由最先耗尽的资源决定

一次下单依次占用网关连接、应用执行槽位、数据库连接、数据库 CPU 与锁、消息生产连接。任何一层到达上限，整条链路都无法继续提高吞吐。

CPU 不高可能是因为请求都在等待数据库连接；数据库 CPU 不高，也可能是热门库存行上的锁把事务串行化。

排查变慢时，最有价值的问题不是“哪项资源使用率最高”，而是：

> **请求现在等待什么，每次等多久，等待数量是否持续增长？**

连接池获取时间、锁等待、执行队列长度和下游并发通常比一个全局 CPU 数字更接近瓶颈。

## 接近上限时，延迟会突然恶化

负载较低时，新请求很快取得资源。利用率接近上限后，即使单次 SQL 的执行时间没变，请求也要先等待连接和锁，总延迟开始陡升。

平均值容易掩盖少量极慢请求，因此要观察 95% 或 99% 请求的完成时间。尾部延迟通常最先暴露排队。

超时会形成自我放大：

```text
请求增加
→ 排队与延迟上升
→ 更多请求超时
→ 客户端和上游重试
→ 实际输入继续增加
```

重试只有在系统仍有余量、失败可能短暂恢复时才有价值。过载期间无限重试会把拥堵变成持续故障。

## 扩容上游可能加重下游瓶颈

应用 CPU 已经饱和时，增加实例可能提高吞吐；数据库已经是瓶颈时，增加实例只会提高进入数据库的并发。

原有 10 台实例，每台最多使用 40 个数据库连接，集群上限是 400。扩到 30 台后，理论连接数变成 1,200。数据库若只能稳定处理 500 个并发事务，扩容应用是在放大竞争。

连接池和并发上限要按整个集群计算。优化前先定位限制吞吐的环节：低效 SQL 应减少扫描，热点锁需要改变竞争方式，固定容量的下游需要上游限并发，消费积压则要处理消费者能力，而不是统一增加 API 实例。

## 有界等待让系统能够主动保护自己

无限队列会保存已经超时、取消或失去业务价值的工作，并持续消耗内存。入口和每个下游边界都应有有限的并发、队列长度和等待期限。

队列满后快速拒绝，看起来牺牲了部分请求，却让已经接收的工作有机会完成。任务还应携带截止时间；用户早已取消或业务期限已经过去，Worker 不应继续调用昂贵依赖。

背压就是让上游感知下游能力。消费者减少预取，下游返回限流信号，API 在队列达到阈值后降低接收速度，流式发送方等待接收方消费，都是把压力反馈回来源，而不是藏进更深的缓冲区。

## 限流保护容量，业务规则保证正确

限流可以按用户阻止异常高频请求，按接口保护昂贵路径，也可以在系统整体接近上限时拒绝新工作。支付回调和已经提交订单的状态查询通常比推荐更新、批量报表更重要，可以使用独立资源预算。

限流不替代库存条件扣减。它减少无意义竞争，数据库约束仍决定最后一件商品属于谁。

拒绝响应应告诉调用方是否值得重试以及何时再试。所有客户端立即重试会绕过系统刚建立的保护。

## 压测要找到失控拐点

“最高达到 1,000 QPS”没有说明系统是否稳定。有效压测逐步增加流量，观察吞吐何时不再增长、尾部延迟何时陡升、哪个等待队列先积累，以及限流能否在资源耗尽前生效。

压测条件必须包含版本、数据规模、请求组合、实例数量和依赖容量。同样的总 QPS，全部更新同一个热门 SKU 与分散购买不同商品，会形成完全不同的锁竞争。

系统还要保留余量。一台实例退出、缓存故障或依赖变慢后，剩余容量是否仍能完成关键交易，比理想环境中的峰值更重要。

## 容量问题怎样复述

流量增加并不会直接让服务崩溃。真正的过程是：进入速度超过完成速度，等待开始累积；某个有限资源最先耗尽，尾部延迟和超时上升；重试进一步增加输入，最终形成正反馈。

```text
找到请求等待的资源
        ↓
判断高峰能否在期限内消化
        ↓
提高瓶颈吞吐或减少工作量
        ↓
用有界队列、背压和限流保护入口
        ↓
优先保留最重要的业务工作
```

**容量治理不是保证所有请求都进入系统，而是在超出持续处理能力时主动减少工作，避免所有请求一起等待和失败。**
