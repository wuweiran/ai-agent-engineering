# AI Agent 工程师知识体系

大模型正在改变软件的能力边界，也在改变软件工程师的工作方式。过去的软件按照预先写好的逻辑执行；今天的系统还可以理解自然语言、处理非结构化信息，并根据任务选择下一步行动。

这种变化并没有取代传统软件工程。一个可靠的 AI Agent 仍然需要前端承载交互，后端管理业务状态，数据库保存数据，网络连接外部服务，云平台提供运行环境。大模型带来了新的能力，而完整的软件工程体系负责把这种能力变成可以长期运行的产品。

## 一、知识体系

### 1. AI Agent 工程

当大模型能够调用工具、读取环境信息并连续执行多个步骤时，普通的 AI 功能就开始具备 Agent 的特征。围绕这类系统，工程工作可以稳定地归纳为四个阶段：

- **明确业务任务**：定义用户、动作、完成条件和风险边界，并判断采用一次模型调用、固定工作流还是 Agent。
- **构建 Agent 应用**：设计决策边界、上下文、状态、工具和运行控制，再选择成熟产品、Agent SDK 或自建 Runtime，接入业务数据与用户流程。
- **验证与改进质量**：建立评测与 Trace，定位系统第一次偏离的位置，修改后进行回归验证。
- **保障生产运行**：处理真实流量、外部依赖、版本升级和故障恢复，并根据生产反馈调整任务与系统边界。

AI Agent 工程师通常与产品经理、应用工程师、平台工程师、数据工程师、安全工程师和领域专家共同完成这些工作。不同角色的边界会随着产品规模变化，但目标始终相同：让模型能力以可控、可测量的方式服务真实业务。

### 2. 大模型基础

理解大模型的运行方式，有助于判断它能做什么、为什么失败，以及应该如何改进。相关知识包括：

- Token、上下文窗口、生成与推理
- Prompt、结构化输出与模型选择
- Embedding、RAG、重排与知识更新
- 幻觉、能力边界、成本和延迟
- 评测方法与模型之间的取舍

这些概念最终都会落到工程问题上。例如，上下文窗口不仅决定一次请求能放入多少文字，还会影响检索策略、会话管理、响应时间和费用。

### 3. 软件工程基础

Agent 需要在现有的软件世界中读取信息和执行操作，因此传统工程能力仍然构成系统底座：

- **后端**：API、异步任务、消息队列、缓存、数据库与分布式系统
- **前端**：流式输出、会话状态、工具进度、审批界面与错误恢复
- **数据**：数据建模、检索管道、数据质量、权限和生命周期
- **网络**：HTTP、DNS、TLS、连接、超时、重试和限流
- **云与运维**：容器、CI/CD、监控、SRE、安全与成本治理

这些技术在 Agent 系统中常常会出现新的组合。例如，一次普通的接口超时可能导致模型重复调用工具，继而产生重复写入；解决它既需要理解 Agent 的执行循环，也需要熟悉幂等、重试和分布式状态管理。

## 二、工程实践中的不同形态

Chat、RAG、固定工作流和 Agent 并不是依次替代的技术阶段。Chat 描述交互方式，RAG 为模型补充外部知识，工作流由程序规定执行路径，Agent 则让模型参与决定下一步行动。它们解决的问题不同，也经常出现在同一个系统中。

一个聊天界面背后可以同时运行文档检索和固定工作流；一个 Agent 也可以把 RAG 作为工具，并被限制在更大的业务流程之内。随着多个团队开始使用这些能力，模型接入、工具注册、评测和安全治理还可能沉淀为共享平台。平台化是组织规模带来的选择，并不是每个 AI 应用的终点。

工程师在实际工作中会遇到不同规模、不同风险的项目。有些项目只需要为现有产品增加一次可靠的模型调用，有些需要建设企业搜索，有些适合用确定性的工作流，还有些开放任务才值得让 Agent 动态选择工具。高风险操作通常仍由程序规则和人工审批控制，而不是交给模型自由决定。

这些项目可能先后发生，也可能同时进行。工程师会更换团队、接手遗留系统、完成几周的原型，也会参与持续数年的平台建设。系统是否更复杂并不代表它是否更先进；能够用足够简单的结构解决真实问题，才是工程上的进步。
