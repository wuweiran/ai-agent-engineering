# Token、Attention 与生成

用户输入一句话时，模型并不是直接读取人类理解的“词义”。文本先变成 Token 和向量，Transformer 根据上下文计算信息之间的关系，再预测下一个 Token。输出 Token 被追加到 Context，模型继续预测，直到结束。

这条链解释了几个应用工程现象：为什么请求按 Token 计量，为什么长 Context 增加成本，为什么 Embedding 能用于检索，以及为什么模型生成的是概率结果而不是数据库查询。

## Token 是模型处理文本的基本单位

Tokenizer 把文本切成模型词表中的 Token。一个 Token 可能是汉字、词的一部分、标点或代码片段，不等于自然语言中的一个词。

常见词可以用较少 Token 表示，罕见名称和长数字可能被拆得更细。**Context Window、API 计费、输出上限和截断通常都以 Token 为基础**，因此字符数相同的两段内容，实际占用可能不同。

BPE 是常见的子词切分方法。它从较小单位开始，反复合并语料中高频出现的相邻组合，在词表规模和表示长度之间取舍。具体模型使用的 Tokenizer 和词表并不完全相同，应用需要使用目标模型提供的计数能力估算输入。

## Embedding 把离散 Token 映射成向量

Token ID 本身只是编号。模型通过 Embedding 表把它映射成高维向量，使后续计算能够表达语义和关系。

面向检索的 Embedding 模型也把句子或文档映射成向量。语义相近的文本通常距离更近，系统可以用余弦相似度等方法寻找候选内容。**生成模型内部的 Token Embedding 与检索使用的文本 Embedding 用途不同；向量相似也只表示候选相关性。**两者不能直接混为同一套索引。

## Attention 按当前问题聚合信息

Transformer 会把每个位置映射成 Query、Key 和 Value。Query 表示当前位置要寻找什么，Key 表示其他位置可以怎样被匹配，Value 承载要聚合的内容。Query 与 Key 的相关程度形成权重，模型再按权重汇总 Value。

多头注意力让模型同时学习不同关系，例如语法联系、指代和局部模式。位置编码补充词序，否则相同 Token 的不同排列难以区分。现代大模型大多采用 Decoder-only 架构，并使用因果遮罩保证当前位置只能依据此前内容预测下一个 Token。

Attention 能利用长距离信息，却不保证 Context 中每条内容都被同等可靠地使用。**内容进入 Context Window，不代表一定会被模型稳定利用。**重要规则被大量无关内容淹没，或者关键证据处于不易利用的位置，仍可能降低回答质量。

## 长文本增加注意力计算与显存成本

标准自注意力要比较序列中的 Token 位置。序列长度增加时，需要计算和保存的两两关系会快速增长，经典实现的计算与注意力矩阵空间成本通常近似随长度平方增长。不同模型会使用稀疏注意力、分块计算等优化，因此具体开销不能只用一个公式概括。

自回归生成还会逐步扩大历史。为了避免生成每个新 Token 时重新计算全部历史 Token 的 Key 和 Value，推理服务通常使用 KV Cache 保存已有结果。KV Cache 降低的是连续解码中的重复计算，并没有消除长输入在预填充阶段的注意力成本；它本身还会占用显存，输入越长、并发越高，资源压力越大。

注意力复杂度、KV Cache 和显存调度的具体实现主要属于模型推理后端。应用工程需要理解由此产生的边界：**长 Context 会增加延迟和成本，Context 选择、压缩和按需读取仍是应用责任**，不能依赖 KV Cache 解决 Context 质量。

## 训练与模型行为

预训练让模型在大规模文本上学习预测下一个 Token，从中形成语言、知识和模式能力。模型完成基础预训练后，还可以进行 CPT（Continued Pre-Training，继续预训练）：使用领域文本、代码或工具相关语料继续训练相同的下一个 Token 预测目标，使模型适应新的数据分布。

SFT（Supervised Fine-Tuning，监督微调）使用指令、参考回答或任务轨迹作为示范，让模型模仿期望输出，更习惯按任务要求作答。RL（Reinforcement Learning，强化学习）不要求为每一步都提供标准答案，而是根据人类偏好、规则、验证器或环境结果产生奖励，再优化模型行为。

**对大多数 Agent 应用岗位，理解 CPT、SFT 和 RL 的用途与边界即可，通常不要求实现微调算法。**PPO、DPO 和 GRPO 等具体优化方法主要属于模型训练岗位；只有团队自建或微调模型、设计 Agent 轨迹数据和奖励函数时，才需要进一步掌握训练数据构造、损失、训练基础设施和模型评测。

微调适合改变模型在大量任务中的稳定行为，例如固定领域表达、工具选择习惯或特定输出模式。**微调通常不是 Agent 应用出现问题后的第一选择。**缺少实时事实应补充 Context 或工具，权限和金额规则应由业务系统执行，少量任务方法可以使用 Skill，单个失败案例则应先通过 Trace 定位根因。只有问题在代表性评测集中持续出现，且 Prompt、Context、工具和确定性规则难以稳定解决时，才值得评估微调成本。

这些训练与对齐方法改变模型总体行为，不能替代应用层的业务校验。一个经过充分对齐的模型仍可能使用过期知识、误解当前状态或生成不存在的事实。

## 多模态模型把其他模态转换成语言模型可用的表示

以图像为例，常见多模态大模型先使用 ViT 等视觉编码器把图片转换成一组视觉特征，再通过线性投影、MLP 或 Resampler 完成模态对齐：调整特征维度和数量，使其成为语言模型能够处理的视觉 Token。视觉 Token 与文本 Token 共同进入语言模型，后续仍由语言模型自回归生成文本。

这只是常见结构，不是唯一标准。有些模型在输入侧拼接视觉 Token，有些在语言模型中插入 Cross-Attention；训练时也可以冻结视觉编码器、只训练连接层，或联合更新多个模块。**应用工程主要关注模型能力、输入限制、Token 成本和识别误差；视觉编码器与模态对齐的训练细节主要属于多模态模型工程。**需要理解的主链是：[视觉编码器提取特征，连接模块完成对齐，语言模型结合文本进行生成]({{ site.baseurl }}/docs/interview/ai-agent/llm/#multimodal-vision-language-connection)。

## 应用工程需要记住什么

模型生成链路可以压缩为：

```text
文本被切成 Token
→ Token 变成向量
→ Attention 聚合 Context 中的信息
→ 模型预测下一个 Token
→ 输出追加后继续生成
```

Token 决定容量和计费单位，Embedding 支持语义表示，Attention 决定模型怎样使用上下文，自回归生成带来流式输出，训练决定模型通常会怎样响应。应用系统仍要提供可信信息、验证关键结果，并把模型放在明确的软件边界中。
