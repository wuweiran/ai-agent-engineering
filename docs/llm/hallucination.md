# 大模型幻觉

用户询问某次生产事故，模型给出一段流畅解释，还附上一条看似真实的发布记录。工程师检查系统后发现，发布编号根本不存在。

这种形式可信、内容却没有事实依据的输出通常称为幻觉。它不是单一故障类型，也不能靠一句“不要编造”彻底消除。

## 错误可能发生在不同位置

事实型错误会生成错误日期、数字、实体或事件。推理型错误使用了真实材料，却推出不成立的关系。引用型错误则捏造来源，或引用了不能支持结论的文档。

在 Agent 中，模型还可能把“建议执行”说成“已经执行”，或者把结果未知的工具调用解释为失败。这些问题不一定来自模型知识，而是来自系统没有清楚区分候选判断、工具回执和权威业务状态。

## 模型首先是在生成可能的后续内容

大模型根据训练模式和当前 Context 预测输出。问题缺少关键信息时，它仍可能生成形式完整的答案；训练知识过期、Context 含有冲突资料、多步推理传播错误，也会增加没有依据的内容。

长 Context 不是自动解法。加入更多材料可能带来噪声、冲突和过期资料，关键证据也可能被忽略。模型能力决定怎样理解信息，应用设计决定它实际看到了什么。

## 先把事实放在可核验的位置

动态订单、监控和发布记录应从权威系统查询，而不是让模型凭训练记忆猜测。RAG 可以把相关文档带入 Context，工具可以取得实时状态，结构化返回可以保留来源和观察时间。

检索只会降低一部分幻觉。召回错误文档、切块丢失上下文、资料本身过期，都会让模型依据错误信息作答。引用也只提供核验入口，不能自动证明结论受到来源支持。

## 重要结论需要独立验证

系统可以要求回答指出证据来源，并检查每个关键结论是否能在材料中找到支持。金额、权限、订单状态和生产变更等可精确判断的问题，应由确定性程序或领域服务验证。

同一个模型进行自检有时能发现明显矛盾，也可能重复原错误。高风险事实更适合交叉查询权威系统、使用独立规则或人工复核。

模型输出影响越大，验证要求越严格：

```text
草稿和建议
→ 标明不确定性与来源

可审查的业务方案
→ 程序校验参数与状态

真实写入或高风险决定
→ 权威系统确认并按需人工审批
```

## 评测要覆盖没有答案的情况

若评测集只包含资料充分的问题，无法检查模型会不会在证据不足时编造。样本还应包含冲突文档、过期数据、检索为空和问题超出范围等情况，并检查系统能否明确表达未知或请求补充信息。

RAG 评测中的 Faithfulness 关注回答是否受到检索材料支持；任务评测还要检查最终业务结果。一个回答忠实复述了错误文档，Faithfulness 可能很高，业务结论仍然错误。

## 工程目标不是让幻觉率归零

任何开放生成都难以承诺绝不出错。工程系统要让错误更难进入真实业务：提供及时且有来源的信息，缩小模型承担的判断范围，保留验证和审批，在 Trace 中记录证据链，并通过评测持续发现失败模式。

**模型负责生成候选内容，证据和确定性系统决定哪些内容值得相信。**（[什么是幻觉？怎样减少？](../interview/ai-agent/#hallucination)）
