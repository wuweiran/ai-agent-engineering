# Prompt 与结构化输出

客服系统希望模型把工单整理成摘要、问题类别和建议动作。如果只说“分析这条工单”，模型可能返回一段自然语言，字段和粒度每次都不同，程序很难继续处理。

提示词负责说明任务与边界，结构化输出负责限制结果形状。两者都能提高稳定性，却不能让概率模型变成完全确定的业务程序。

## 不同消息承担不同来源

应用通常把输入分成系统规则、用户请求和已有模型输出。System Prompt 描述模型在当前应用中的职责、边界和输出要求；用户消息表达本次目标；Assistant 历史与工具结果提供此前交互。

这些来源在具体 API 中怎样表示取决于供应商和 Runtime。重要的是让模型能够区分稳定规则、当前任务、外部资料和历史结果，同时控制哪些内容有权改变行为。

System Prompt 可以要求“不得越权退款”，却不能真正实施权限。退款服务仍要根据当前身份、订单状态和金额执行授权与业务规则。

## 好的提示词先把任务说清楚

一份可执行的提示词通常需要：当前背景、要完成的目标、可使用的信息、不得违反的限制，以及预期输出。COSTAR 等框架只是帮助检查这些要素的写作方法，不是必须采用的协议。

示例比抽象形容词更具体。给出少量输入输出示例，也就是 Few-shot，可以帮助模型理解分类边界和格式。但示例过多会增加 Context 成本，也可能让模型过度模仿少数案例。

复杂任务可以要求模型先形成计划或使用工具取得证据。应用不应要求或保存模型不可见的内部推理过程；真正需要审计的是模型获得了什么信息、请求了什么动作，以及外部结果是什么。

## JSON 输出方式

让模型输出 JSON 有三个层次，保证程度不同：

| 方式 | 应用提供什么 | 通常保证什么 | 主要问题 |
| --- | --- | --- | --- |
| Prompt 约定 | 在提示词中要求“只返回 JSON”并提供示例 | 没有协议级保证 | 可能混入解释、代码块、漏字段或写错类型 |
| JSON Mode | 在模型 API 中开启 JSON 输出模式 | 通常保证结果是合法 JSON | 不保证字段、类型和枚举符合业务要求 |
| Schema 约束的结构化输出 | 在模型 API 请求中提交 JSON Schema | 输出满足该 Schema 支持的字段、类型、枚举和必填约束 | 只保证结果形状，不保证内容真实或业务判断正确 |

所谓 API **原生支持 Schema 结构化输出**，是指 API 除了接收 Prompt，还提供专门参数让应用提交 JSON Schema，并承诺正常完成的输出符合它所支持的 Schema 约束。它和 JSON Mode 是两种不同强度的能力：JSON Mode 只把输出限制为合法 JSON，Schema 结构化输出还限制 JSON 的具体结构。不同供应商的参数名称、支持的 Schema 子集和异常行为不同，不能只根据“支持 JSON”推断它支持 Schema 约束。Schema 本身按照普通 API 契约设计即可，例如明确字段、类型、枚举和必填项；这里只需额外确认模型 API 支持哪些 JSON Schema 能力。

如果模型 API 没有这些能力，应用仍可用 Prompt、Few-shot 示例、JSON 解析、Schema 校验和有限重试组合实现，但稳定性和维护成本通常更差。用正则截取第一个大括号只能掩盖输出问题，不能代替 JSON 解析和 Schema 校验。

## 生成状态与结果验证

应用收到响应后，首先检查模型调用是否正常结束。达到输出 Token 上限、安全拒绝、连接失败或尚有工具调用待处理时，不能直接把当前内容当作完整业务结果。即使 API 支持 Schema 结构化输出，截断和拒绝等异常结果也可能无法满足预期结构。

一个结构化结果要经过三层判断：

1. **语法验证**：是否为合法 JSON；
2. **结构验证**：字段、类型、枚举和必填项是否符合 Schema；
3. **语义与业务验证**：事实是否真实，字段之间是否一致，当前用户和业务状态是否允许采用这个结论。

例如：

```json
{
  "order_id": "O-1042",
  "decision": "eligible",
  "reason": "订单满足退款要求"
}
```

这段结果可以通过语法和结构验证，订单却可能不存在、已经退款，或者当前用户无权操作。结构化输出让程序稳定取得模型的**候选判断**，订单服务仍要查询真实状态并执行资格、金额和权限规则。

格式或结构失败可以在记录验证错误后有限重试；输出被截断时应调整输出空间或缩小任务；事实和业务规则不成立时，应重新查询数据或拒绝执行，不能靠重复调用模型碰一个正确答案。系统还应记录解析失败率、Schema 校验失败率、重试率、截断率和业务验证拒绝率，用真实任务评测 Schema 与 Prompt 是否稳定。

## 工具调用也是结构化输出

Function Calling 或 Tool Calling 同样使用 Schema 约束模型生成的参数，但它表达的是**模型请求应用执行动作**。普通结构化输出表达“模型返回了一份数据”，工具调用表达“模型希望调用哪个能力以及传入什么参数”。分析退款资格并返回分类适合结构化输出；查询订单或创建退款申请则适合工具调用。

模型返回 Tool Call 后，Runtime 还要检查工具是否存在、参数是否合法、当前用户是否有权限、动作是否需要审批，再执行真实函数并把 Tool Result 交给模型。工具参数符合 Schema，只说明调用请求可以被程序解析，不代表动作已经执行，也不代表业务服务必须接受。

## Prompt Engineering 与 Context Engineering

[Prompt Engineering]({{ site.baseurl }}/docs/interview/ai-agent/llm/#prompt-vs-context-engineering) 调整指令、示例和输出要求，帮助模型更稳定地使用已有能力。Few-shot 用少量输入输出示范任务边界；复杂问题还可以要求模型先形成可检查的计划或中间结构。

Context Engineering 管理本轮模型实际看到的信息：System Prompt、工具定义、任务状态、检索材料、记忆和近期结果。它要决定信息从哪里来、什么时候加载、怎样压缩，以及哪些内容已经过期。

二者不是互斥关系。指令不清时应改 Prompt；缺少事实、材料过期或噪声过多时，应修复 Context。仅靠更强措辞无法让模型知道没有进入窗口的业务事实。

## Prompt 稳定性要通过评测改进

提示词变长不等于更好。重复规则会增加成本，冲突要求会降低稳定性，过度针对单个失败案例还可能破坏其他任务。

团队应保存一批真实任务，在相同模型和环境下比较不同版本。准确性、严重错误、任务完成率、Token、延迟和工具行为共同决定新版本是否更好。A/B 测试适合比较线上用户结果，离线回归适合在发布前阻止已知问题再次出现。

对措辞过度敏感时，可以把指令和 Few-shot 示例视为待优化参数，在固定训练样本和评分函数上搜索候选。[DSPy]({{ site.baseurl }}/docs/interview/ai-agent/llm/#dspy-prompt-optimization) 等工具能够自动优化指令与示例，但结果仍依赖样本代表性、评分指标和最终保留的测试集，不能用训练集分数代替真实质量。

复杂推理容易遗漏步骤时，可以要求输出可验证的中间结构，或对多个独立候选进行一致性选择。[Self-Consistency]({{ site.baseurl }}/docs/interview/ai-agent/llm/#cot-self-consistency) 的核心是采样多条推理路径并聚合最终答案，它会增加调用成本，也只适合答案能够比较或投票的任务。业务权限和真实状态仍应由程序验证。

修改后要找到效果来自哪里。若增加一段规则后结果改善，可以通过移除该规则重新评测，判断它是否真正贡献了质量。这类消融实验能避免 Prompt、工具和流程不断堆叠却无人知道哪些仍有价值。

## 提示词的边界

提示词把任务说清楚，结构化输出让结果更容易被程序消费，评测证明修改是否稳定有效。它们无法代替：

```text
可信数据来源
服务端权限
业务状态检查
写入幂等
外部结果确认
```

模型可以生成候选判断和候选动作；确定性软件仍要决定哪些结果可以成为真实业务状态（[怎样设计和优化 Prompt？]({{ site.baseurl }}/docs/interview/ai-agent/llm/#prompt-optimization)；[结构化输出和工具调用有什么区别？]({{ site.baseurl }}/docs/interview/ai-agent/llm/#structured-output-vs-tool-calling)）。
