# 长 Context、压缩与缓存

一个支持助手把完整聊天记录、几十篇文档和所有工具 Schema 一次交给模型。信息看似更充分，回答却变慢、费用上升，关键约束还偶尔被忽略。

**Context Window** 是一次模型调用能够处理的输入和输出空间。窗口更大只代表可以容纳更多 Token，不代表应该把所有可访问数据都放进去。

## Context 预算由多种内容共同占用

一次调用可能同时包含 **System Prompt、工具定义、用户消息、历史对话、检索材料、工具结果和预留输出**。应用应知道每类内容占用多少，而不是等接口达到上限后再从最早消息机械截断。

```text
稳定规则和工具定义
+ 当前任务与近期历史
+ 检索和工具结果
+ 预留输出空间
= 本轮 Context 预算
```

输入变长会增加传输、预处理和模型计算。多轮 Agent 还会反复携带历史内容，因此一次无关的大型工具输出会影响后续每轮的延迟与成本。

## 更长不等于更容易使用

模型对 Context 中的信息利用并不均匀。关键证据夹在大量无关内容中时，可能出现 **Lost in the Middle**：内容虽然仍在窗口里，却没有被稳定用于回答。

解决方法不是只更换更大窗口模型。系统应删除过时结果、把早期过程压缩成事实与待办、按需读取原文，并把最重要的目标、限制和证据放在清楚位置。RAG 场景还可以先按相关性重排候选，再把高价值材料放在更容易利用的位置。

**摘要是有损转换。**订单状态、工具回执和资源 ID 等可核验事实应保存在外部状态中，恢复时重新读取；摘要更适合保存任务语义和已经确认的结论。

## 不同内容采用不同保留方式

不同内容应按用途选择保留方式：**近期交互保留原文，早期讨论进行压缩，大型日志和文档留在外部系统，稳定规则放在版本化配置中，长期经验经过筛选后进入记忆。**

[**滑动窗口与摘要**]({{ site.baseurl }}/docs/interview/ai-agent/llm/#context-overflow-window-summary)解决不同问题。**滑动窗口**只保留最近消息，实现简单并保留原文，却可能丢掉最初目标；**摘要**把较早历史压缩成事实、决定和待办，能够保留更远语义，却可能遗漏细节或引入错误。[长对话中的关键信息保留]({{ site.baseurl }}/docs/interview/ai-agent/context-memory/#long-context-key-information)还需要**结构化任务状态、选择性保留和按需重读**：目标、约束、未完成事项和关键结果不应只依赖向量检索或一份滚动摘要。

长任务还可以让子 Agent 使用独立 Context，只把结论、证据和未知项带回主任务。这减少信息污染，却要求委派与返回契约足够清楚。

## 三种缓存解决不同问题

**精确结果缓存**在业务允许时为完全相同的输入复用最终回答。缓存键至少要考虑模型、Prompt、工具定义、关键 Context 版本和用户权限；否则系统可能把旧版本或其他用户的结果错误复用。

**语义缓存**通过 Embedding 判断两个问题是否相近，再复用历史答案。它能提高命中范围，也更危险：措辞相近的问题可能具有不同时间、租户或业务条件。高风险和动态事实通常不适合直接使用语义缓存。

**Prompt Cache** 复用稳定提示词前缀的模型计算。它降低重复输入的成本或延迟，却不会跳过本次生成，也不表示模型记住了此前会话。稳定前缀内容或版本变化后，缓存可能失效。

```text
精确结果缓存：复用完整结果
语义缓存：复用相近问题的结果
Prompt Cache：复用相同输入前缀的计算
```

三者都要定义命中范围、失效条件和观测指标，不能只看命中率。

## 模型调用的优化从减少无效工作开始

应用可以先减少重复规则、无关历史和过大的工具结果，再考虑摘要、检索和缓存。不同复杂度任务还可以选择不同模型，但路由规则需要通过真实任务评测，不能只按输入长度决定。

观察时要同时记录**输入 Token、输出 Token、首个结果等待时间、完整耗时、缓存命中、任务成功和严重错误**。成本下降却让关键约束丢失，不是有效优化。

**Context 管理的目标不是塞入最多信息，而是让模型以可接受的成本看到当前判断真正需要的内容。**（[怎样使用缓存优化大模型应用？]({{ site.baseurl }}/docs/interview/ai-agent/quality-production/#llm-application-cache)）
