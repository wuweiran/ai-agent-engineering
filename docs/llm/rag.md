# RAG 与知识检索

企业支持助手要回答：“版本 4.2 的 SAML 证书错误怎样处理？”答案散落在产品手册、发布说明和事故复盘中，而且会持续更新。把所有文档都放进 Prompt 既昂贵，也会让无关内容淹没真正证据。

RAG 在生成前先检索与当前问题相关的材料，再把选中的内容放进 Context。它连接两条链路：离线把知识变成可检索索引，在线为一次问题选择证据。

## 离线阶段建立可检索索引

原始文档先经过解析和清洗，再切成 Chunk。每个 Chunk 保留正文以及标题、来源、版本、时间、租户等元数据。Embedding 模型把文本映射成向量，系统把向量与原文引用写入索引。

```text
文档
→ 解析与清洗
→ Chunk 切分
→ 生成 Embedding
→ 保存向量、正文引用和元数据
```

知识更新后，索引也要更新或删除旧版本。只构建一次而没有生命周期管理的 RAG，很快会把过期资料交给模型。

## Chunk 决定模型能检索到什么

Chunk 太小，匹配可能精确，却缺少解释当前段落所需的背景；Chunk 太大，包含更多上下文，却会降低匹配精度并占用更多 Context。

固定长度切分简单，但可能截断语义。按标题和段落切分能保留文档结构；递归切分在块过大时继续按句子等边界拆分。参数没有通用最佳值，应根据真实问题和检索指标调整（[Chunk 大小怎样选择？]({{ site.baseurl }}/docs/interview/ai-agent/#rag-chunk-size)）。

父子索引用小块参与检索，命中后返回所属的大块。它试图同时获得精确匹配和完整上下文，但需要维护子块到父块的映射，也会增加返回内容。

## 向量检索先找到语义候选

在线查询时，系统使用兼容的 Embedding 模型把问题转成向量，再在索引中寻找相近候选。数据规模增大后，逐个精确比较成本很高，常用近似最近邻索引在召回率、查询延迟、内存和构建成本之间取舍。

HNSW 是常见的近似最近邻算法。它用多层邻接图从稀疏高层快速接近目标区域，再逐层进入更密集的邻域完成搜索。具体参数应通过目标数据集评测，不能把某组召回率和延迟当成固定保证。

向量索引可以由专门向量数据库、支持向量能力的关系数据库或检索库实现。产品选择取决于数据规模、过滤、更新、运维和可用性要求，而不是“向量一定不能放在传统数据库”。

## 混合检索同时照顾语义与精确词

向量检索擅长语义相近，却可能忽略错误码、产品编号和人名等精确词。BM25 等关键词检索对这些内容更敏感。

混合检索分别取得向量与关键词候选，再按加权分数或排名融合。元数据过滤还可以先限定租户、版本、语言和时间，防止相似但无权访问或已经过期的内容进入候选集。

代码库检索也不必在 RAG 和 Grep 之间二选一。精确函数名适合符号或文本搜索，自然语言问题可以使用语义检索，两者可以组合。

## 重排把昂贵判断留给少量候选

第一阶段检索需要从大量文档中快速召回候选，通常使用查询和文档分别编码的 Bi-Encoder。它速度快，却难以充分建模查询与文档之间的细粒度关系。

Rerank 对较小候选集重新打分。Cross-Encoder 把问题与候选文本一起编码，排序通常更精细，但每对内容都要计算，成本更高。

常见链路是先召回较多候选，再重排，最后只把少量证据放进 Context。Top-K 不是固定参数，过少可能漏证据，过多会引入噪声和成本（[为什么 RAG 常结合关键词检索和 Rerank？]({{ site.baseurl }}/docs/interview/ai-agent/#hybrid-search-rerank)）。

## 生成阶段要保留证据边界

应用把选中的文档片段、来源和用户问题交给模型，并要求回答只依据可用证据。没有足够材料时，模型应表达未知，而不是补齐一个看似合理的结论。

RAG 与 Agent 是不同概念。固定问答接口可以在每次模型调用前自动检索；Agent 也可以把搜索知识库作为工具，根据结果决定是否继续查询。区别在于谁决定检索路径，而不在于是否使用向量数据库。

## 检索和回答要分开评测

只看最终回答无法知道错误出在检索还是生成。检索阶段可以观察：

- Recall@K：正确证据是否出现在前 K 个结果中；
- Precision@K：返回结果中有多少真正相关；
- MRR：第一个正确结果出现得有多靠前。

生成阶段可以检查回答是否切中问题，以及结论是否受到检索材料支持。评测样本要保存问题、应召回证据、允许答案和资料版本，才能比较切块、Embedding、检索和重排的变化。

## RAG 的知识骨架

```text
离线：文档 → Chunk → Embedding → 索引
在线：问题 → 召回 → 过滤与混合检索 → 重排 → Context → 生成
评测：检索是否找到证据 + 回答是否忠于证据
```

RAG 的价值不是让模型“拥有”全部企业知识，而是为当前问题选择少量、及时、有权限并可核验的材料（[一个标准 RAG 系统包含哪些步骤？]({{ site.baseurl }}/docs/interview/ai-agent/#rag-pipeline)）。
