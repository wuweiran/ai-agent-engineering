# RAG 与知识检索

企业支持助手要回答：“版本 4.2 的 SAML 证书错误怎样处理？”答案散落在产品手册、发布说明和事故复盘中，而且会持续更新。把所有文档都放进 Prompt 既昂贵，也会让无关内容淹没真正证据。

RAG 在生成前先检索与当前问题相关的材料，再把选中的内容放进 Context。它连接两条链路：离线把知识变成可检索索引，在线为一次问题选择证据。

## 离线阶段建立可检索索引

原始文档先经过解析和清洗，再切成 Chunk。每个 Chunk 保留正文以及标题、来源、版本、时间、租户等元数据。Embedding 模型把文本映射成向量，系统把向量与原文引用写入索引。

```text
文档
→ 解析与清洗
→ Chunk 切分
→ 生成 Embedding
→ 保存向量、正文引用和元数据
```

知识更新后，索引也要更新或删除旧版本。只构建一次而没有生命周期管理的 RAG，很快会把过期资料交给模型。

### 索引更新与版本

生产系统通常按文档 ID 和源系统版本识别变化，只重新解析、切分和索引新增或修改的文档；源文档删除时，对应 Chunk 也必须从索引删除。内容哈希可以跳过没有实际变化的文档，并帮助识别重复内容，但不能替代来源、权限和版本判断。

当 Embedding 模型、Chunk 策略或元数据结构发生变化时，旧索引与新索引可能不兼容。系统应构建新的索引版本，完成一致性检查和检索评测后再切换读流量，并保留回滚能力，而不是在原索引上逐步覆盖。全量重建适合这类结构性变化；日常文档变更则更适合增量更新。

无论采用定时扫描、变更事件还是 CDC（Change Data Capture，变更数据捕获），都要处理重复和乱序事件、失败重试、源文档与索引之间的差异，以及更新期间读请求看到哪个版本。**RAG 更新的目标是让可检索索引与有权访问的源知识保持一致。**（[RAG 索引怎样动态更新？]({{ site.baseurl }}/docs/interview/ai-agent/rag/#rag-index-update)）

## Naive RAG、Advanced RAG 与 GraphRAG

[Naive RAG、Advanced RAG 与 GraphRAG]({{ site.baseurl }}/docs/interview/ai-agent/rag/#naive-advanced-graph-rag) 常被用来说明不同检索架构，但它们不是严格的行业分级，也不是每个系统都必须依次经历的三个版本。文档解析、Chunk 和索引属于离线阶段，不会在每次用户提问后重新执行。

Naive RAG 是最小可用链路：

```text
离线：文档 → Chunk → Embedding → 向量索引
在线：Query → 向量检索 Top-K → Context → 生成
```

它适合直接的局部知识问答，但只按向量相似度取固定数量候选时，容易漏掉错误码等精确词，也难以处理含糊表达、噪声候选和多跳问题。

Advanced RAG 仍以文本片段检索为主，在检索前、检索中和检索后增加质量控制，例如 Query 改写、混合检索、元数据过滤、父子索引、Rerank 和 Context 选择。它不是一个固定算法，而是针对真实检索失败逐步加入的工程能力。

GraphRAG 改变了知识表示和检索路径。离线阶段从文档中提取实体与关系，建立可回溯到原文的图；在线阶段根据问题定位实体或子图，通过关系遍历取得多跳证据。部分实现还会划分社区并生成摘要，用于回答“整个资料库有哪些主要主题”一类全局问题。路径遍历与社区摘要解决的目标不同，并非每个 GraphRAG 查询都要同时执行。

```text
离线：文档 → 实体与关系 → 图索引 → 可选的社区与摘要
在线：Query → 实体或子图 → 关系与原文证据 → Context → 生成
```

**Advanced RAG 优化文本检索链路，GraphRAG 则增加显式关系结构。** GraphRAG 更适合关系密集、多跳和跨文档概括，不能普遍替代成本更低的文本检索。生产系统也可以同时使用关键词、向量和图检索，再根据问题类型选择或合并结果（[GraphRAG 适合什么问题？]({{ site.baseurl }}/docs/interview/ai-agent/rag/#graph-rag-use-cases)）。

## Query 改写让问题更适合检索

用户表达适合人与人交流，不一定适合直接检索。多轮对话中的“它为什么失败”缺少指代对象；复杂问题可能同时包含多个子目标；业务说法与文档术语之间还可能存在语义鸿沟。[Query 改写]({{ site.baseurl }}/docs/interview/ai-agent/rag/#rag-query-rewriting)把当前问题转换成一个或多个可独立检索的查询。

常见策略解决不同问题：

- **上下文补全**：结合必要对话历史完成指代消解，把依赖上下文的追问压缩成独立查询；
- **查询扩展**：补充同义词、实体、错误码和领域术语，或生成多个改写并合并召回结果；
- **任务分解**：把多跳问题拆成多个子查询，按依赖顺序或并行检索；
- **HyDE**：让模型生成一份假设性相关文档，再使用该文档的 Embedding 检索真实材料。它生成的是检索表示，不应被当作事实答案；
- **Step-back**：先提出更抽象的原则性问题，再用抽象知识辅助原问题。它与拆成多个细粒度子查询不是同一种策略。

改写也会产生 Query Drift：新查询偏离用户原意，检索到看似相关但无法回答原问题的材料。系统可以保留原始 Query 与改写 Query 做混合召回，比较两者的语义或实体约束，并在改写丢失关键名词、时间和租户时拒绝使用。

不是每个问题都值得改写。简单实体查询可以直接检索；只有多轮指代、复杂组合或召回不足时才启用改写。延迟敏感场景可以使用较小模型完成分类和改写，并行执行多个查询，但是否采用应由真实检索评测决定。

## Chunk 决定模型能检索到什么

Chunk 太小，匹配可能精确，却缺少解释当前段落所需的背景；Chunk 太大，包含更多上下文，却会降低匹配精度并占用更多 Context。

固定长度和重叠窗口实现简单，但可能截断语义并产生重复。按标题、段落、列表、表格或代码函数切分能够保留文档结构；递归切分在块过大时继续按段落、句子等边界拆分；语义切分根据相邻内容的 Embedding 变化寻找主题边界，但需要额外计算，效果也依赖模型和阈值。[Chunk 大小]({{ site.baseurl }}/docs/interview/ai-agent/rag/#rag-chunk-size)没有通用最佳值，应根据真实问题和检索指标调整。

[父子索引]({{ site.baseurl }}/docs/interview/ai-agent/rag/#rag-parent-child-index)用小块参与检索，命中后返回所属的大块。它试图同时获得精确匹配和完整上下文，但需要维护子块到父块的映射；多个命中指向同一父块时还要去重，父块过大也会重新引入噪声。

## 向量检索先找到语义候选

在线查询时，系统使用兼容的 Embedding 模型把问题转成向量，再在索引中寻找相近候选。数据规模增大后，逐个精确比较成本很高，常用近似最近邻索引在召回率、查询延迟、内存和构建成本之间取舍。

HNSW 是常见的近似最近邻算法。它用多层邻接图从稀疏高层快速接近目标区域，再逐层进入更密集的邻域完成搜索。具体参数应通过目标数据集评测，不能把某组召回率和延迟当成固定保证。

向量索引可以由专门向量数据库、支持向量能力的关系数据库或检索库实现。产品选择取决于数据规模、过滤、更新、运维和可用性要求，而不是“向量一定不能放在传统数据库”。

## 混合检索同时照顾语义与精确词

向量检索擅长语义相近，却可能忽略错误码、产品编号和人名等精确词。BM25 是常用的关键词相关性算法：查询词在一篇文档中出现越多，文档通常越相关；但重复很多次后的增益会逐渐减小。它还会提高稀有词的权重，并校正文档长度，避免长文仅因包含更多词就天然得分更高。因此，错误码和专有名词往往更容易被 BM25 找到。

混合检索分别取得向量与关键词候选，再按加权分数、归一化分数或 RRF 等排名方法融合。两路原始分数通常不在同一尺度，不能未经处理直接相加；权重与候选数量应通过带相关证据标注的查询集确定，而不是套用固定比例。元数据过滤还可以先限定租户、版本、语言和时间，防止相似但无权访问或已经过期的内容进入候选集。

代码库检索也不必在 RAG 和 Grep 之间二选一。精确函数名适合符号或文本搜索，自然语言问题可以使用语义检索，两者可以组合。

## 重排把昂贵判断留给少量候选

第一阶段检索需要从大量文档中快速召回候选，通常使用查询和文档分别编码的 Bi-Encoder。它速度快，却难以充分建模查询与文档之间的细粒度关系。

[Rerank]({{ site.baseurl }}/docs/interview/ai-agent/rag/#hybrid-search-rerank) 对较小候选集重新打分。Cross-Encoder 把问题与候选文本一起编码，排序通常更精细，但每对内容都要计算，成本更高。

常见链路是先召回较多候选，再重排，最后只把少量证据放进 Context。选择阶段不必只按固定 Top-K 截断，也可以在分数基础上按来源多样性、重复度和 [Context Token 预算]({{ site.baseurl }}/docs/interview/ai-agent/rag/#rag-rerank-context-budget)逐个加入。块数没有通用答案：过少可能漏证据，过多会引入噪声和成本。

## 生成阶段要保留证据边界

应用把选中的文档片段、来源和用户问题交给模型，并要求回答只依据可用证据。Context 组装时要保留每个片段的文档、版本、时间和 Citation，不能先把多个来源改写成一段无法追溯的摘要。

多个片段互相补充时，回答应覆盖问题的不同子目标；多个来源冲突时，应指出差异并优先使用权威、更新且适用范围匹配的材料，不能为了生成流畅答案而自行合并。没有足够材料时，模型应表达未知或要求补充信息，而不是用训练记忆补齐一个看似合理的结论。

RAG 与 Agent 是不同概念。固定问答接口可以在每次模型调用前自动检索；Agent 也可以把搜索知识库作为工具，根据结果决定是否继续查询。区别在于谁决定检索路径，而不在于是否使用向量数据库。

## 检索和回答要分开评测

只看最终回答无法知道错误出在检索还是生成。检索阶段可以观察：

- Recall@K：正确证据是否出现在前 K 个结果中；
- Precision@K：返回结果中有多少真正相关；
- MRR：第一个正确结果出现得有多靠前。

生成阶段可以检查回答是否切中问题，以及结论是否受到检索材料支持。无答案问题还要验证系统能否表达证据不足，而不是强行作答。

[RAG 评测集]({{ site.baseurl }}/docs/interview/ai-agent/rag/#rag-evaluation-dataset)至少保存原始问题、应召回证据、允许答案或判断规则、无答案标记、资料版本和权限范围。样本应覆盖术语差异、多跳问题、过期或冲突文档和权限隔离，才能比较切块、Embedding、检索、重排与生成的变化。

## RAG 的知识骨架

```text
离线：文档 → Chunk → Embedding → 索引
在线：问题 → 召回 → 过滤与混合检索 → 重排 → Context → 生成
评测：检索是否找到证据 + 回答是否忠于证据
```

RAG 的价值不是让模型“拥有”全部企业知识，而是为当前问题选择少量、及时、有权限并可核验的材料（[企业知识库怎样构建，并接入 RAG？]({{ site.baseurl }}/docs/interview/ai-agent/rag/#rag-pipeline)）。
